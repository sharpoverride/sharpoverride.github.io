An LLM in a loop, with tools.

An agent is an LLM that can take actions — call APIs, run code, search the web — observe the results, and decide what to do next. The architecture is a few hundred lines of orchestration around a model that can output structured tool calls. The hard part isn't the loop; it's keeping it productive.

The agent loop

Think → Act → Observe → repeat. The LLM reasons; emits a tool call; runtime executes; result feeds back; loop until done or budget exceeded.

What changed in 2024–25

Tool-calling went from a brittle prompt trick to a first-class model capability. Claude, GPT-4, Gemini all support structured tool calls natively. Agent frameworks (LangGraph, CrewAI, AutoGen, Microsoft AutoGen) provide the orchestration.

01What makes a system an agent.

A chatbot answers; an agent acts. The difference is the loop: the model can request actions, observe outcomes, and continue. "Book a flight from SFO to JFK next Tuesday under $400" doesn't have a one-shot answer; it requires search-the-flights → compare-prices → confirm-availability → book → return-confirmation. Each step is a tool call; each result informs the next step.

The agent shape unlocks tasks that pure chat can't. Customer service that resolves issues end-to-end. Coding agents that read files, run tests, commit fixes. Research agents that explore the web for hours. The "wow" demos are downstream of one architectural change: the model can call functions, and the runtime executes them.

"A chatbot tells you the answer. An agent does the thing."

02The agent loop.

Fig. AThe model picks a tool, the runtime executes, the result becomes the next observation. Iterate until the model returns a final answer.

ReAct: Reasoning + Acting

The 2023 paper that named the pattern. The LLM produces a free-text "thought" + a structured "action" at each step. The thought is for the model itself (visible in logs); the action is what the runtime executes. This pattern works because giving the model "thinking room" before each action improves decision quality. Most production agents use a ReAct-shaped prompt.

Tool-calling protocol

You pass the model a JSON schema describing each tool: name, params, types. The model emits a JSON tool call: {name: "search_flights", args: {from: "SFO", to: "JFK", date: "..."}}. The runtime parses, executes the corresponding function, appends the result to the conversation. This is now a first-class feature in all major LLMs; you don't need fragile parsing.

Step budget

The loop has to terminate. Hard limit on iterations (typically 10–30). Soft limit on cost (cumulative tokens). Watchdog on stuck loops (model calls the same tool with the same args twice → break). The agent's biggest failure mode is "ran for an hour, accomplished nothing, cost $50."

03Tool design.

Tools are APIs the model can call

Each tool has a name, a description, parameters with types, and a function body. Write tool descriptions for the model, not for engineers. "Searches the flights database for trips matching the criteria" — describe what it does, when to use it, what it returns. The model's tool-use quality is dominated by the description.

Granularity is the design decision

Too coarse: one tool book_trip(...) with 30 parameters. The model gets confused; partial inputs fail validation. Too fine: 50 tools for every CRUD operation. The model can't pick. Sweet spot: ~5–15 tools per agent, each with a clear single purpose. Compose more complex flows via the agent's loop, not a giant tool.

Side-effect tools need guardrails

Read-only tools are safe. Tools that book flights, send emails, run code, delete files — these need explicit controls: idempotency keys (see idempotency chapter), dollar limits, human-in-the-loop confirmation for high-stakes actions. The agent will make mistakes; build the brake before the throttle.

Multi-agent: planner + workers

For complex tasks, a "planner" agent decomposes the goal into subtasks; "worker" agents execute each. Each worker is a smaller agent with a focused tool set. This is the Anthropic-style "orchestrator + subagents" pattern and the shape behind most production multi-agent systems. Each subagent has its own context window; the planner aggregates.

04In .NET.

C#   SimpleAgent.cs (using Semantic Kernel) 
// dotnet add package Microsoft.SemanticKernel
using Microsoft.SemanticKernel;
using Microsoft.SemanticKernel.ChatCompletion;

// Define tools as plain C# methods with descriptive attributes.
public sealed class FlightTools
{
    [KernelFunction, Description("Search flights by route and date. Returns a list of options.")]
    public async Task<FlightOption[]> SearchFlights(
        [Description("3-letter origin airport code")]      string from,
        [Description("3-letter destination airport code")] string to,
        [Description("ISO date YYYY-MM-DD")]              string date)
        => await _api.SearchAsync(from, to, date);

    [KernelFunction, Description("Book a specific flight. Requires user confirmation.")]
    public async Task<BookingResult> BookFlight(
        [Description("Flight ID from SearchFlights")] string flightId,
        [Description("User confirmation token")]    string confirmationToken)
    {
        if (!await _humanLoop.ConfirmedAsync(confirmationToken))
            return new BookingResult(Status: "NeedsHumanConfirmation", ...);
        return await _api.BookAsync(flightId);
    }
}

// Configure the kernel with the tools and an LLM.
var builder = Kernel.CreateBuilder();
builder.AddOpenAIChatCompletion("gpt-4o", apiKey);
var kernel = builder.Build();
kernel.ImportPluginFromObject(new FlightTools(...));

// Agent loop — SK handles tool calling and observation injection.
var chat = kernel.GetRequiredService<IChatCompletionService>();
var history = new ChatHistory();
history.AddSystemMessage("You book flights. Always confirm before booking. Step budget: 10.");
history.AddUserMessage(userRequest);

var stepCount = 0;
while (stepCount++ < 10)
{
    var response = await chat.GetChatMessageContentAsync(history, new()
    {
        FunctionChoiceBehavior = FunctionChoiceBehavior.Auto(),
    }, kernel);

    history.Add(response);
    if (response.Content is { } answer && response.Items.Count == 0)
    {
        return answer;   // Model produced a final answer — no more tool calls.
    }
    // Otherwise SK auto-invoked tools; observations are in the history.
}
throw new InvalidOperationException("Agent exceeded step budget");

05The detail.

Evals are non-negotiable

Agents drift more than chatbots. An eval suite of (task, expected tool sequence, accepted final answers) catches regressions when you change the model, the prompt, or the tools. Run on every change. The best eval suites have 100+ adversarial scenarios.

Idempotency keys for every action

The agent will retry tool calls under retry policies and after errors. Mutating tools need idempotency keys. Booking the same flight twice because the agent retried after a network error is the failure mode that hits production once and is remembered forever.

Context-window management is operational

Long agent runs blow past context windows. Summarise older steps (the agent's own observations from 5 steps ago can usually be compressed). Truncate tool outputs — a 100-page web search result gets summarised before going back to the model. Without this, agents fail at step ~7 when the context fills.

Human-in-the-loop for high-stakes actions

"Send this email", "transfer this money", "delete this file" — make the agent emit a proposal, surface it in the UI, wait for human approval before executing. Build the human gate before you turn the agent loose. The asymmetric cost of "agent ran wild" justifies the friction.

Observability is harder

Agent runs are non-deterministic and multi-step. Log every step: model output, tool call, tool result, decision. Use a tracing tool that understands LLM steps (LangSmith, Helicone, Phoenix). When something goes wrong, you need the full trajectory.

"The agent is an LLM with footguns. Build the safety catches first, then turn it on."

06Decision guide.

Customer support automation

Agent with tools: lookup-order, refund, send-message, escalate. Human-in-loop for refunds > $X. Eval suite of canonical interactions.

Coding agent

Tools: read-file, edit-file, run-tests, search-codebase. Idempotency by version-control: changes are commits, easy to revert.

Research / browsing agent

Tools: web-search, fetch-page, summarise. Step budget ~20. Time budget the user can interrupt. Cite sources in the final answer.

Multi-agent system

Planner + workers. Planner decomposes; workers each have a focused toolkit. Cross-worker coordination via shared state, not direct messaging.

Simple Q&A over your docs

Don't reach for an agent. RAG (see RAG chapter) is the right shape. Add agents only when actions are needed.

Long-running autonomous agent

Hard cost cap. Persistent state (resumable). Watchdog for stuck loops. Telemetry. The longer the agent runs autonomously, the more the safety net matters.
