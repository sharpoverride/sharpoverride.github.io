Shipping without downtime, or fear.

Five deployment patterns, ordered by how much risk they take with users. Rolling is the default; canary is the safety net; blue-green is the rollback insurance; feature flags decouple deploy from release. Pick by failure cost.

The five patterns

Recreate (downtime). Rolling (gradual). Blue-green (parallel). Canary (small slice). Feature-flagged (deploy ≠ release).

The key insight

"Deploy" and "release to users" should be separate events. The deploy is when code reaches production. The release is when users see the change. Feature flags split the two; you can deploy daily and release weekly — or vice versa.

01The deploy-risk tradeoff.

Every deploy is a hypothesis: this version is no worse than the previous one. Sometimes the hypothesis is wrong. The question is how many users find out. Pick a deployment pattern by the cost of being wrong. If wrong = a 5% conversion drop on the homepage, you want canary with automated rollback. If wrong = three minutes of static error pages, rolling is fine.

The patterns differ on three axes: blast radius during the deploy, capacity required, and rollback speed. There's no winner; there's a portfolio. Most production systems use rolling for the boring deploys and canary for the scary ones.

"You don't roll back. You roll forward — to the version that was working ten minutes ago."

02Five patterns.

Fig. 27Recreate is the simplest, riskiest. Feature flags decouple the deploy from the release entirely.

Rolling — the default

Kubernetes' default Deployment strategy. Spin up one new pod of v2, wait for readiness, kill one v1 pod, repeat until all are swapped. Capacity stays roughly constant; users hit a mix of v1 and v2 during the rollout. Fast, cheap, and the right answer for most deploys. Rollback is "redeploy v1" — takes another rolling cycle.

Blue-green — for known-risky deploys

Keep v1 (blue) running. Deploy v2 (green) to identical capacity alongside. Switch traffic at the load balancer atomically. Rollback is one config change — flip back to blue. The cost: 2× capacity during the cutover window. Useful for schema migrations, major framework upgrades, anything you suspect might be surprising.

Canary — graduated rollout

Route a small percentage (typically 1–5%) of traffic to v2; monitor key metrics; expand on green, halt on red. Argo Rollouts and Flagger automate the ramp + halt loop based on Prometheus metrics. Best for user-facing changes where partial degradation is acceptable but a full rollout would be a Sev-2 incident.

A/B test — canary as experiment

Same shape as canary, but the goal is to measure v2 vs v1 over hours/days/weeks. The split is randomised per user (sticky), the metrics compared are product metrics (conversion, retention), the outcome decides whether v2 is the new v1. Tools: GrowthBook, Optimizely, LaunchDarkly Experimentation.

03Feature flags as a separate layer.

The fifth pattern isn't a deployment strategy at all — it's the realisation that code deployment and feature release are different events. Deploy v2 with the new feature wrapped in if (flag.IsEnabled("new-checkout")). Roll v2 to 100% of pods. The new code is in production but the flag is off; users still see the old checkout. Days or weeks later, turn the flag on — for 1% of users, then 10%, then 100%.

This is the single highest-leverage operational change a team can make. Deploys become routine. Rollback is a config change. A/B tests are flag variants. Long-running migrations can ship dark and turn on per-cohort. The pattern was invented at Flickr in 2009; it's the default at every modern engineering org that doesn't ship slowly.

04In practice.

yaml   argo-rollout-canary.yaml 
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata: { name: orders }
spec:
  replicas: 10
  strategy:
    canary:
      steps:
        - setWeight: 5      # 5% → v2
        - pause: { duration: 10m }
        - analysis:
            templates: [{ templateName: success-rate }]
            args:
              - { name: service-name, value: orders }
        - setWeight: 25
        - pause: { duration: 10m }
        - setWeight: 50
        - pause: { duration: 30m }
        - setWeight: 100
      analysis:
        templates: [{ templateName: success-rate }]
        startingStep: 1     # run analysis from step 1 onward
  selector: { matchLabels: { app: orders } }
  template:
    metadata: { labels: { app: orders } }
    spec:
      containers: [{ name: api, image: ghcr.io/example/orders:v2 }]
C#   FeatureFlag.cs 
// dotnet add package Microsoft.FeatureManagement.AspNetCore

builder.Services.AddFeatureManagement();

app.MapPost("/checkout", async (

05The detail.

Forward migrations only

Database migrations need a rollback story. The best one is "the migration is forward-only and the new code handles both old and new schemas." Two deploys: deploy v2 that handles both schemas, run the migration, deploy v3 that assumes the new schema. Rolling back from v3 requires the migration to be reversible — which most aren't. Plan for forward-only.

Pre-stop hook for graceful shutdown

When K8s kills a v1 pod during a rolling deploy, in-flight requests die mid-flight. Add a preStop hook: sleep a few seconds so the load balancer drains the pod from rotation, then exit cleanly. This is the single most-overlooked production config and the cause of "we get a few errors during every deploy" tickets.

Canary metrics need a baseline

Comparing "v2 error rate" to "absolute threshold of 1%" is wrong — what if v1 was also at 0.9%? Compare v2 to v1's concurrent error rate: if v2's rate exceeds v1's by >0.5% for >5min, halt. Tools like Flagger and Argo Rollouts do this differential analysis natively.

Feature flags decay

A 6-month-old feature flag is technical debt. Every flag adds a branch in production code. Bake in a retire-the-flag step at the end of every feature rollout. Some teams use "flag expiry dates" enforced in CI.

The flag service can't be a SPOF

If your feature-flag SaaS is down, your app should default to last-known-good state, not 500. Local caches with TTL fallbacks; circuit-break the flag fetch; never let the flag service take down your app. The flag is metadata, not control plane.

"Continuous deployment is what happens when rolling back becomes boring."

06Decision guide.

Default for routine deploys

Rolling. K8s default. Cheap. Use it 90% of the time.

Schema migration, framework upgrade

Blue-green. Pay the 2× capacity for the window; the atomic cutover and instant rollback are worth it.

User-facing UI change, conversion-critical

Canary + automated metric-based rollback. Argo Rollouts or Flagger. 1% → 5% → 25% → 100% with halts on red.

Want to test in production

Feature flag the new code. Deploy dark. Turn on for internal users, then 1% of real users, then ramp.

Mobile app

You can't roll back binaries fast — App Store review. Every server-affecting feature must be flagged. Server-side kill switch is the actual rollback.

Database-heavy change

Two-phase migration. Deploy v2 that handles both schemas → run migration → deploy v3 assuming new schema. Forward-only. Don't try to coordinate "atomic deploy + migrate".
