Microservices for the browser.

Multiple teams shipping into one UI shell, each owning a slice. Module federation, single-spa, web components. Most of the time it's overkill. The cases where it's right are real but small.

Three composition modes

Build-time (monorepo + packages). Run-time (module federation, single-spa). Iframe (browser sandbox).

The real reason to use it

Multiple teams shipping independent surfaces of one product, at different cadences. Not "we want microservices for the front-end."

01Why or why not.

One frontend codebase, many teams. Conflicting library versions. Slow CI. PRs that touch everyone's code. Release coupling — team A's bug blocks team B's launch. Sound familiar? You might want micro frontends.

But: most front-end orgs that adopt micro-frontends regret it within 18 months. The duplication tax (each MFE ships its own React copy), the consistency tax (each team's "Button" looks slightly different), the orchestration tax (who deploys the shell?). Solve the problem with a monorepo and shared packages first. If that still hurts, then micro frontends.

"The Module Federation demo looks great. The Module Federation production system looks like a service mesh in HTML."

02Composition modes.

Build-time composition (monorepo)

Each team's UI is a package. Apps depend on packages. CI builds the whole thing. Simplest and best for most teams. Nx, Turborepo. This isn't "micro frontends" by most definitions; it's the alternative most teams should pick.

Run-time composition (federation)

Webpack Module Federation, Vite's plugin, RSPack. The host app loads remote modules at runtime from URLs. Each MFE deploys independently; host picks up the new version on next page load. Cost: shared dependencies must be resolved per-page; type sharing across MFE repos is tricky.

Run-time composition (single-spa)

A router that loads frameworks-of-frameworks into the same page. Mix React + Vue + Angular if you must. Powerful, scary; useful for incremental migrations more than greenfield design.

Iframes

The oldest, simplest micro-frontend. Strong isolation (separate JS context, CSS scope, security boundary). Best for embedding untrusted content (third-party apps, payment widgets). Cost: cross-frame communication is awkward; deep linking is harder.

03Shared dependencies.

React is 50 KB gzipped. If each MFE ships its own copy, 10 MFEs = 500 KB just for React. Module Federation supports sharing: declare React as a shared singleton; all MFEs use the host's instance. The version-matching rules are subtle; one MFE on React 18 and another on React 19 breaks hooks.

Sharing turns a build-time concern into a runtime contract. When you change major versions, you have to coordinate across teams — which is the problem MFEs were supposed to solve.

Design system

If each MFE has its own button, the product looks inconsistent. Centralised design system, versioned, used by all MFEs. Versioning is now an architectural concern: when you publish v2 of the design system, you have N MFE teams to coordinate with.

04Decision guide.

Single team, single product

Don't. Monorepo with packages is plenty. Micro frontends solve org problems, not technical ones.

Multiple teams, one product, build pain

Try Turborepo / Nx monorepo first. 80% of the wins for 10% of the complexity.

Multiple teams, must deploy independently

Module Federation with carefully managed shared dependencies. Versioned design system.

Legacy migration (Angular → React)

single-spa makes the migration incremental. Migrate page-by-page; ship each migration as it completes.

Embedding third-party widgets

Iframes. Sandboxed, security-boundaried. The grandfather technique; still the right one for untrusted content.

You're being told to adopt MFEs by a vendor

Push back. The pattern has real costs. Make sure the team-level reason is real before paying them.

"Micro frontends are the answer when your monorepo has stopped being able to ship. Not before."
