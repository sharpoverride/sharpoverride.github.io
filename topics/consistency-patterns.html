Consistency patterns.

CAP is a starting point, not an answer. The real menu has five levels, each with a specific cost — and a specific bug you'll ship if you pick the wrong one.

The problem

Replication makes data fast and durable. It also means two reads of the same key can return different answers. Every consistency model is a contract about how different.

The thesis

You don't pick consistency once. You pick it per operation. The login page can be eventually consistent. The account balance cannot.

01CAP, honestly.

The CAP theorem is widely cited and widely misunderstood. The accurate version: during a network partition, a distributed system must choose between consistency (every read sees the latest write) and availability (every request gets a non-error response). That's it. Outside of a partition, you can have both.

Which is why PACELC is the more useful framing in 2026: in case of Partition, choose A or C; Else (normal operation), choose Latency or Consistency. Cosmos DB session consistency is PA/EL — fast and available, with bounded weirdness. Spanner is PC/EC — globally consistent, and you pay for it in latency.

Fig. 06 · CAP for the partition; PACELC for the rest of the time
"In the absence of network failures both consistency and availability can be satisfied. The CAP theorem is fundamentally about what happens when something already went wrong."

02The five levels.

Forget "consistent vs eventually consistent." Real systems live on a spectrum. From strongest to weakest, with the specific guarantee each gives you:

Fig. 07Stronger left, faster right. Most apps live in the middle.

Linearizable is the simplest mental model — the system behaves as if every operation happens at a single point on a global timeline. Easy to reason about, expensive to implement, requires consensus (Raft/Paxos) and round-trips. Pay for it for money movement and inventory; don't pay for it for product reviews.

Session consistency is the sweet spot for user-facing systems and it deserves its own paragraph. It gives every session two guarantees: read-your-writes (you'll always see your own last write) and monotonic reads (time never goes backward for you). Other users may see slightly older data, but you never see a contradiction with yourself. This is what humans actually want from "consistency" 95% of the time.

Why not always pick "Strong"?

In Cosmos DB, switching from Session to Strong roughly doubles your RU cost and adds 2–10× latency for multi-region reads. SQL Server reading from the primary versus a readable secondary changes your read p99 by 5–50 ms. Strong consistency is a globally rationed resource; spend it where it matters.

03Cosmos DB, five levels.

Cosmos DB is one of the few production databases that lets you pick any of the five levels per operation. Pricing and latency change with the choice, which makes it a perfect teaching example.

C#   CosmosClientFactory.cs 
// dotnet add package Microsoft.Azure.Cosmos
using Microsoft.Azure.Cosmos;

namespace Academy.Consistency;

public sealed class CosmosClientFactory(IConfiguration cfg)
{
    // Account-level default. Per-request overrides allowed only DOWN the spectrum.
    // e.g. account = Session → may request Eventual; cannot request Strong.
    public CosmosClient CreateForAccount(ConsistencyLevel accountDefault) =>
        new(
            cfg["Cosmos:ConnectionString"]!,
            new CosmosClientOptions
            {
                ConsistencyLevel          = accountDefault,
                ApplicationPreferredRegions = ["West Europe", "North Europe"],
                SerializerOptions         = new() { PropertyNamingPolicy = CosmosPropertyNamingPolicy.CamelCase },
            });
}

// Two services, two needs, two configurations. Don't make the whole app pay
// for the strictness one endpoint requires.
public sealed class BillingService(CosmosClientFactory factory)
{
    // Money movement. Strong by default. ~2x RU, multi-region writes serialised.
    private readonly Container _ledger =
        factory.CreateForAccount(ConsistencyLevel.Strong)
               .GetContainer("finance", "ledger");

    public async Task<LedgerEntry> PostAsync(LedgerEntry e, CancellationToken ct)
    {
        // Defaults to Strong from the client; no override needed.
        var r = await _ledger.CreateItemAsync(e, new PartitionKey(e.AccountId), null, ct);
        return r.Resource;
    }
}

public sealed class CatalogService(CosmosClientFactory factory)
{
    // Product reads. Session is fine; price changes settle in seconds.
    private readonly Container _products =
        factory.CreateForAccount(ConsistencyLevel.Session)
               .GetContainer("shop", "products");

    public async Task<Product?> GetAsync(Guid id, CancellationToken ct)
    {
        // Lower than session for non-critical reads — saves RU.
        try
        {
            var r = await _products.ReadItemAsync<Product>(
                id.ToString(),
                new PartitionKey(id.ToString()),
                new ItemRequestOptions { ConsistencyLevel = ConsistencyLevel.Eventual },
                ct);
            return r.Resource;
        }
        catch (CosmosException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
        {
            return null;
        }
    }

    // The interesting case: the user just edited the price and reads it back.
    // We need read-your-writes — session token carries that promise across reads.
    public async Task<Product> UpdateAndReadAsync(Product p, CancellationToken ct)
    {
        var write = await _products.UpsertItemAsync(p, new PartitionKey(p.Id.ToString()), null, ct);

        // Pass the session token from the write into the read.
        // Without this, you might read an older replica and not see your own change.
        var read  = await _products.ReadItemAsync<Product>(
            p.Id.ToString(),
            new PartitionKey(p.Id.ToString()),
            new ItemRequestOptions { SessionToken = write.Headers.Session },
            ct);
        return read.Resource;
    }
}

Two ideas this snippet quietly demonstrates:

  • Account default is the ceiling. You can request anything weaker at request time; you cannot request stronger than the account allows. So pick the account default to be the strictest level you need anywhere, and downgrade elsewhere.
  • The session token is read-your-writes made explicit. Pass it from the write response into the read request and Cosmos guarantees the read sees the write — even from a different replica.

04EF Core isolation.

Consistency at the distributed-database level has a cousin at the single-database level: transaction isolation. Same animal — different vocabulary. The four ANSI levels prevent specific anomalies; the cost is locking, MVCC versions, or retries.

Isolation level Prevents Doesn't prevent Cost Use for
Read Uncommitted — Dirty reads, non-repeatable reads, phantoms Almost zero Never. Seriously.
Read CommittedSQL Server default Dirty reads Non-repeatable reads, phantoms Low · row-level locks Most read paths
Repeatable ReadPostgres "Repeatable Read" Dirty + non-repeatable reads Phantoms (SQL Server) · serialisation anomalies (Postgres) Mid · range locks or snapshot Reports needing a stable snapshot
Snapshot / MVCCPostgres default, SQL Server opt-in Dirty + non-repeatable + phantoms (reads see a snapshot) Write skew (specific to snapshot) Mid · version tombstones Default for most apps
Serializable All anomalies — behaves as if transactions ran one at a time — High · retries on conflict Money movement, inventory reservations
C#   TransferService.cs 
// Money transfer between two accounts. Must be serializable; debits and credits
// must agree even under concurrent transfers on the same account.
public sealed class TransferService(BankDbContext db, ILogger<TransferService> log)
{
    public async Task<TransferResult> TransferAsync(
        Guid fromId, Guid toId, decimal amount, CancellationToken ct)
    {
        const int MaxAttempts = 3;
        for (var attempt = 1; ; attempt++)
        {
            await using var tx = await db.Database.BeginTransactionAsync(
                System.Data.IsolationLevel.Serializable, ct);
            try
            {
                var from = await db.Accounts.FirstAsync(a => a.Id == fromId, ct);
                var to   = await db.Accounts.FirstAsync(a => a.Id == toId,   ct);

                if (from.Balance < amount)
                    return TransferResult.InsufficientFunds;

                from.Balance -= amount;
                to.Balance   += amount;

                await db.SaveChangesAsync(ct);
                await tx.CommitAsync(ct);
                return TransferResult.Ok;
            }
            catch (DbUpdateException ex) when (IsSerializationFailure(ex) && attempt < MaxAttempts)
            {
                // Postgres SQLSTATE 40001 / SQL Server 1205 — retry with jitter.
                log.LogInformation("serialization retry {Attempt}", attempt);
                await Task.Delay(Random.Shared.Next(10, 100), ct);
            }
        }
    }

    static bool IsSerializationFailure(DbUpdateException ex) =>
        ex.InnerException is PostgresException { SqlState: "40001" } ||
        ex.InnerException?.Message.Contains("deadlock") == true;
}

Serializable on Postgres is implemented with Serializable Snapshot Isolation (SSI) — it doesn't lock, it watches for conflicts and aborts losers. So your code must retry. Skipping the retry loop is the most common Postgres-serializable bug.

05Optimistic concurrency.

The pattern that bridges the single-DB and distributed-DB worlds: optimistic concurrency control. Add a version column. Reads include it. Writes condition on it. Two concurrent writers find out at commit time; one wins, the other retries.

This is the same idea as Cosmos DB's ETag, DynamoDB's ConditionExpression, and HTTP's If-Match header. Same pattern, different names.

C#   Product.cs · EF mapping 
public sealed class Product
{
    public Guid    Id    { get; set; }
    public string  Name  { get; set; } = default!;
    public decimal Price { get; set; }

    [Timestamp]                       // SQL Server rowversion
    public byte[] RowVersion { get; set; } = [];

    // — or, for Postgres, a uint xmin column —
    // public uint XMin { get; set; }    [ConcurrencyCheck], [Column("xmin")]
}

// Mapping. SQL Server "rowversion" auto-increments on every UPDATE.
public sealed class ShopDbContext(DbContextOptions<ShopDbContext> opts) : DbContext(opts)
{
    protected override void OnModelCreating(ModelBuilder b)
    {
        b.Entity<Product>()
            .Property(p => p.RowVersion)
            .IsRowVersion();
    }
}

public sealed class ProductService(ShopDbContext db, ILogger<ProductService> log)
{
    public async Task<UpdateResult> UpdatePriceAsync(
        Guid id, decimal newPrice, byte[] expectedVersion, CancellationToken ct)
    {
        var product = await db.Products.FirstAsync(p => p.Id == id, ct);
        product.RowVersion = expectedVersion;          // the version the client read
        product.Price      = newPrice;

        try
        {
            await db.SaveChangesAsync(ct);
            return new UpdateResult.Ok(product.RowVersion);
        }
        catch (DbUpdateConcurrencyException)
        {
            // Someone else wrote between this client's read and write.
            // Re-read, present the conflict to the caller (HTTP 409).
            var current = await db.Products
                .AsNoTracking().FirstAsync(p => p.Id == id, ct);
            return new UpdateResult.Conflict(current);
        }
    }
}

// HTTP boundary — surface the conflict as ETag / If-Match (RFC 9110).
app.MapPatch("/products/{id}/price", async (
    Guid id, PriceUpdate body, HttpRequest req,
    ProductService svc, CancellationToken ct) =>
{
    if (!req.Headers.TryGetValue("If-Match", out var etag))
        return Results.Problem("If-Match required", statusCode: 428);

    var result = await svc.UpdatePriceAsync(id, body.Price,
        Convert.FromBase64String(etag.ToString()), ct);

    return result switch
    {
        UpdateResult.Ok ok        => Results.Ok(new { etag = Convert.ToBase64String(ok.NewVersion) }),
        UpdateResult.Conflict c   => Results.Conflict(new { current = c.Current }),
        _                              => Results.Problem("unknown"),
    };
});

The HTTP layer matters. If-Match + ETag turn optimistic concurrency into a contract your API consumers can program against. A client that gets a 409 knows exactly what to do: re-fetch, resolve, retry. A client that gets a "last write wins" silently overwriting their colleague's edit has no clue and no recourse.

The dimension nobody talks about is blast radius of a wrong choice. Postgres at Read Committed will surprise you with lost updates if you don't add a version column. Cosmos at Eventual will surprise you with a list-then-detail page where the detail doesn't exist yet. DynamoDB at default consistency will surprise you when a user can't see their own post for 200 ms. None of these are bugs in the database; they are mismatches between the data model and the operation.

07Decision guide.

Money movement, inventory hold

Serializable txn against a single-primary RDBMS. Retry on conflict. Don't go distributed for this until you have to.

User profile, settings, preferences

Session consistency (Cosmos default; or single Postgres). Read-your-writes is what users emotionally expect.

Product catalogue, public content

Eventual is fine — 95% of reads. Cache with TTL; staleness measured in seconds is invisible.

Analytics, reporting

Read from a replica with bounded staleness. Hours-old data is acceptable. Don't cost the primary.

Multi-tenant collaboration (Figma, Notion)

Strong per-document (CRDT/OT inside the doc), eventual across documents. Scope the strict region.

"Just put it in Cosmos at Strong"

You've doubled your RU bill and added cross-region latency for an SLA the business probably doesn't need. Pick per-op.

The questions to ask before picking a level

① What's the user-visible failure if a read is stale? Wrong product price (cheap to recover). Wrong account balance (expensive). Wrong "did my post send?" (lost trust).

② How often do reads and writes overlap on the same key? If a row is written once and read a million times, eventual is free money. If reads and writes interleave, you need stronger guarantees.

③ Is the "consistency" the user wants the same as the textbook definition? Often what users want is read-your-writes, which session-level gives you cheaply — not full linearizability.
