Case study Year 2014 (pre-acquisition) Scale 50B msgs/day · 32 engineers Stack Erlang · FreeBSD · single-purpose hardware

Thirty-two engineers. Fifty billion messages.

WhatsApp's pre-acquisition engineering team was famously tiny. The architectural choice was famously old-school: Erlang on FreeBSD, vertically scaled, no microservices, no cloud, no JS. The lessons are mostly about discipline and constraint.

The numbers

~32 engineers. ~450M MAU. ~50B msgs/day. ~12 servers. ~$1M MRR per engineer.

The choices

Erlang for concurrency. FreeBSD for kernel tuning. Single chat-server process per active million users. No build pipeline; live system patches.

01Why Erlang.

Erlang/OTP was built for telecoms: high-concurrency, fault-tolerant, soft-real-time. Each user's session is a lightweight Erlang process (few KB stack). One server can hold a million processes. The supervisor-tree model makes "let it crash and restart" the standard error-handling pattern.

The fit with chat: a user = a process, a message = a message, the language's native primitives are the application's domain. The amount of glue code WhatsApp needed was minimal.

"Erlang was a 30-year-old language with the perfect concurrency primitives for chat. It just took someone to notice."

02Why FreeBSD.

FreeBSD's network stack and kqueue were better tuned than Linux's epoll for the WhatsApp workload at the time. The team patched the kernel for their needs — TCP stack, socket allocator, scheduler. Each server held ~1M concurrent connections, on hardware that other companies would've thought needed 100.

The lesson isn't "use FreeBSD." It's "vertical scale on tuned hardware is undervalued in the cloud era." The cost of one giant carefully-tuned machine is often less than a fleet of un-tuned ones.

03The engineering culture.

No microservices

One Erlang application, one binary. The "service" boundary was the supervisor tree inside the application. Cross-service RPC was inter-process Erlang messaging — local and cheap.

Live code reload

Erlang's hot-code reload feature was used in production. Deploy a fix without restarting the server. This is how WhatsApp avoided "deploy windows"; the server kept running through thousands of code updates.

No JIRA, no standups, no architecture committee

The team was tiny enough that communication was face-to-face. No process overhead. Don't generalise this to a 500-engineer org; it worked because of the specific shape of the team.

Owning the stack

The engineers wrote Erlang, tuned the kernel, ordered the hardware, racked the servers. End-to-end ownership. Hard to staff at scale; brutally effective at small scale.

05The lessons.

Match the language to the domain

Chat is a concurrency problem. Erlang is a concurrency language. The fit was perfect. For most apps, the domain doesn't have such a sharp fit; pick boring tech. But when it does, pay attention.

Vertical scale, well-tuned, is cheap

The cloud era pushes you toward "many small commodity instances." WhatsApp's "few big tuned instances" was 10× cheaper for the workload. The optimisation requires ownership; the cloud takes it from you.

Smaller teams move faster, up to a limit

32 engineers couldn't build a hundred features in parallel. They built one feature very well. The constraint forced focus.

Not all of this scales

WhatsApp got acquired and grew the team to thousands. Live code reload, no-process culture, owning the rack — most don't survive scale. The era of "32 engineers, 50B msgs/day" is the era you can't reproduce by hiring.

"WhatsApp's engineering was the right shape for its scale. Bigger scale needs different shape."
