Consistent hashing.

The ring, virtual nodes, and why your shard rebalance shouldn't be a Friday-night incident. A full implementation in C# you can drop into a shard router.

The problem

You have N shards. You add one. With hash(key) % N, almost every key now maps to a different shard. You just invalidated 90% of your cache and need to migrate 90% of your data.

The fix

Place shards on a fixed-size ring. Each key maps to one fixed point. Adding a shard moves only 1/N of the keys.

01The modulo problem.

Most engineers' first attempt at sharding is some flavour of shard = hash(key) % N. It works. Until you change N. The day you go from four shards to five, roughly 80% of keys move. Caches go cold; the database is hammered by a wave of misses; an operation that should have been "add another box" becomes a multi-hour migration.

The pathological case is a cache cluster. You scale from 4 to 5 nodes during a traffic spike. The new modulo means cache hit rate craters from 95% to 20%. The backing store, which was sized for the cached load, falls over. You've just turned "scaling up" into "global outage."

Fig. 10 · Naive mod-N: every shard change reshuffles almost everything

02The ring.

Karger et al., 1997. The idea is one of those rare gems that you look at and think "of course."

  1. Imagine the output of your hash function as a giant circle — 0 through 2^32 - 1, wrapping back to 0.
  2. Place each node on the ring at hash(node_id).
  3. Place each key on the ring at hash(key).
  4. Each key belongs to the first node clockwise from its position.
  5. Add a node and only the arc immediately counter-clockwise from it moves. About 1/N of the keys.
Fig. 11Only the arc immediately counter-clockwise from the new node migrates

03Virtual nodes.

The naïve ring has two problems. With three nodes hashed onto a 2^32 ring, the three arcs are almost never equal — one node ends up with 50% of the keys, another with 10%. Worse, when a node fails, all of its load goes to a single neighbour, which then collapses.

The fix: each physical node owns many virtual positions on the ring. 100–200 replicas per node is a common number. Now:

  • Load smooths out. With 100 vnodes per box, the load variance between nodes is <5% (vs. up to 2× without).
  • Failures distribute. When a node dies, its 100 vnodes' arcs each go to different neighbours. No single survivor takes the full hit.
  • Weighting is trivial. A bigger box gets more vnodes — 200 instead of 100. It gets ~2× the keys, automatically.
Fig. 12 · Virtual nodes turn an unfair ring into an even one

04Full implementation.

The whole ring in C# below. About 90 lines. Thread-safe for reads, builds atomically on changes, weighted, generic over the node type. You can paste this into a shard router and start using it.

C#   ConsistentHashRing.cs 
using System.Collections.Immutable;
using System.IO.Hashing;            // xxHash — 5× MD5, deterministic, no allocation
using System.Text;

namespace Academy.Sharding;

public interface IRingNode
{
    string Id     { get; }
    int    Weight { get; }     // 1 = baseline; 2 = double share; …
}

/// <summary>
/// Thread-safe consistent hash ring with virtual nodes.
/// Reads are O(log V) where V = total virtual nodes (≈ nodes × 100).
/// Add/remove rebuilds the ring atomically; readers never see a partial state.
/// </summary>
public sealed class ConsistentHashRing<TNode> where TNode : IRingNode
{
    private const int VirtualNodesPerWeight = 100;

    // The ring: a sorted list of (hash, node).
    // ImmutableSortedDictionary gives us O(log n) reads and copy-on-write builds.
    private ImmutableSortedDictionary<uint, TNode> _ring =
        ImmutableSortedDictionary<uint, TNode>.Empty;

    private ImmutableDictionary<string, TNode> _nodes =
        ImmutableDictionary<string, TNode>.Empty;

    private readonly object _lock = new();

    public IEnumerable<TNode> Nodes => _nodes.Values;

    /// Replace the entire node set in one atomic swap. Use for bulk reloads.
    public void SetNodes(IEnumerable<TNode> nodes)
    {
        var ring  = ImmutableSortedDictionary.CreateBuilder<uint, TNode>();
        var index = ImmutableDictionary.CreateBuilder<string, TNode>();

        foreach (var node in nodes)
        {
            index[node.Id] = node;
            var vnodes = VirtualNodesPerWeight * Math.Max(1, node.Weight);
            for (var i = 0; i < vnodes; i++)
            {
                var point = HashToUInt32($"{node.Id}#{i}");
                // Collisions are vanishingly rare with xxh32 across < 10k vnodes; resolve by skipping.
                if (!ring.ContainsKey(point)) ring[point] = node;
            }
        }
        lock (_lock)
        {
            _ring  = ring.ToImmutable();
            _nodes = index.ToImmutable();
        }
    }

    public void Add(TNode node)    => SetNodes(_nodes.Values.Concat([node]));
    public void Remove(string id) => SetNodes(_nodes.Values.Where(n => n.Id != id));

    /// Map a key to its owning node. O(log V).
    public TNode Locate(string key)
    {
        var ring = _ring;        // snapshot — readers never block
        if (ring.Count == 0) throw new InvalidOperationException("empty ring");

        var h = HashToUInt32(key);

        // First node clockwise from h. Binary-search the sorted keys.
        var keys = ring.Keys.ToArray();   // cache this in production — see notes
        var idx  = Array.BinarySearch(keys, h);
        if (idx < 0) idx = ~idx;          // next greater
        if (idx == keys.Length) idx = 0;    // wrap
        return ring[keys[idx]];
    }

    /// Replication — return the next N distinct nodes clockwise. Useful for Dynamo-style
    /// "write to 3 replicas, read with quorum" patterns.
    public IReadOnlyList<TNode> LocateReplicas(string key, int count)
    {
        var ring = _ring;
        if (count > _nodes.Count)
            throw new ArgumentException("more replicas than nodes");

        var h    = HashToUInt32(key);
        var seen = new HashSet<string>(count);
        var result = new List<TNode>(count);

        // Walk clockwise — uses the keys' sorted order naturally.
        foreach (var (point, node) in ring.Concat(ring))   // concat to handle wrap
        {
            if (point < h && result.Count == 0) continue;
            if (seen.Add(node.Id))
            {
                result.Add(node);
                if (result.Count == count) return result;
            }
        }
        return result;
    }

    private static uint HashToUInt32(string s)
    {
        Span<byte> buf = stackalloc byte[Encoding.UTF8.GetMaxByteCount(s.Length)];
        var n = Encoding.UTF8.GetBytes(s, buf);
        return XxHash32.HashToUInt32(buf[..n]);
    }
}
C#   RingTests.cs · the distribution test 
// The test you must write the first time you build one of these.
// Without it, you don't know whether your ring is balanced.
[Fact]
public void Ring_distributes_within_5_percent_with_100_vnodes()
{
    var ring = new ConsistentHashRing<Shard>();
    ring.SetNodes([
        new Shard("shard-1", Weight: 1),
        new Shard("shard-2", Weight: 1),
        new Shard("shard-3", Weight: 1),
        new Shard("shard-4", Weight: 1),
    ]);

    var counts = new Dictionary<string, int>();
    for (var i = 0; i < 1_000_000; i++)
    {
        var owner = ring.Locate($"user:{i}");
        counts[owner.Id] = counts.GetValueOrDefault(owner.Id) + 1;
    }

    var expected = 250_000;
    foreach (var (id, n) in counts)
        Assert.InRange(n, expected * 0.95, expected * 1.05);
}

[Fact]
public void Adding_a_node_moves_only_a_quarter_of_keys()
{
    var before = new ConsistentHashRing<Shard>();
    before.SetNodes([new("s1",1), new("s2",1), new("s3",1)]);

    var after = new ConsistentHashRing<Shard>();
    after.SetNodes([new("s1",1), new("s2",1), new("s3",1), new("s4",1)]);

    var moved = 0;
    for (var i = 0; i < 100_000; i++)
    {
        var k = $"key:{i}";
        if (before.Locate(k).Id != after.Locate(k).Id) moved++;
    }
    // Expect ~1/4 to move. Tolerance for randomness.
    Assert.InRange(moved, 20_000, 30_000);
}

Three production caveats you'll discover the first time you ship this:

  • Cache the sorted keys array. The snippet above calls ring.Keys.ToArray() in the hot path. In production, materialise it on each SetNodes and keep it next to the ring. Saves ~80% of Locate's allocations.
  • Use xxHash, not MD5/SHA. Cryptographic hashes are huge overkill — distribution is what you want, not collision resistance against an adversary. xxHash32 is roughly 5× faster.
  • Persist the ring topology. The ring is configuration, not state. Write it to your config store (etcd, ConfigMap, Cosmos config doc) and have services subscribe; you'll want every replica to compute the same answer for the same key.

05Sharded EF Core.

The natural use in a .NET shop: pick the right DbContext based on the tenant or user id. The ring sits in DI as a singleton; each request resolves the shard, opens a context with the right connection string, and proceeds normally.

C#   ShardResolver.cs · DbContextFactory 
public sealed record Shard(string Id, int Weight, string ConnectionString) : IRingNode;

public sealed class ShardResolver(ConsistentHashRing<Shard> ring) {
    public Shard For(Guid tenantId) => ring.Locate(tenantId.ToString("N"));
}

// One DbContext class, many connection strings — one per shard.
public sealed class ShardedShopFactory(ShardResolver resolver, IDbContextFactory<ShopDbContext> baseFactory) {
    public ShopDbContext For(Guid tenantId)
    {
        var shard = resolver.For(tenantId);

        var opts = new DbContextOptionsBuilder<ShopDbContext>()
            .UseNpgsql(shard.ConnectionString)
            .EnableSensitiveDataLogging(false)
            .Options;

        return new ShopDbContext(opts);
    }
}

// Wire-up
builder.Services.AddSingleton<ConsistentHashRing<Shard>>(_ =>
{
    var ring = new ConsistentHashRing<Shard>();
    ring.SetNodes([
        new("eu-1", Weight: 1, ConnectionString: cfg["Shards:eu-1"]!),
        new("eu-2", Weight: 1, ConnectionString: cfg["Shards:eu-2"]!),
        new("eu-3", Weight: 2, ConnectionString: cfg["Shards:eu-3"]!),   // bigger box
    ]);
    return ring;
});
builder.Services.AddSingleton<ShardResolver>();
builder.Services.AddSingleton<ShardedShopFactory>();

// Usage in a handler
app.MapGet("/orders/{tenantId:guid}", async (
    Guid tenantId, ShardedShopFactory factory, CancellationToken ct) =>
{
    await using var db = factory.For(tenantId);   // ring picks the shard
    return await db.Orders
        .Where(o => o.TenantId == tenantId)
        .AsNoTracking()
        .ToListAsync(ct);
});

The pattern works for any shard-owned data — Postgres, SQL Server, Cosmos containers, Redis instances, file-storage accounts. The ring is the routing primitive; what's behind each ring node can be anything.

When a shard moves, what about its data?

The ring tells you where a key should be. Migrating the data when topology changes is a separate concern. Two common strategies: (a) double-write during migration — for the duration of a shard split, write to both old and new; reads check new first; once new is in sync, switch fully. (b) lazy migration on read — when a key's owner changes, the new owner reads it from the old owner on first access. Slow path on miss, no migration window. Most production sharding systems combine the two.

06Tradeoff matrix.

Range partitioningexplicit ranges Just the rebalanced range Hot ranges possible (recent dates!) High · range maps, splits, merges Spanner, HBase, CockroachDB For range-scan-heavy workloads
Tablet / chunk lookupconsult a directory Just the moved chunk Excellent (rebalance per-chunk) High · need a metadata service MongoDB, Bigtable, Vitess Heavy; right for elastic scale

Two pieces of context the table can't capture.

Jump consistent hash is genuinely magical: six lines of code, zero memory overhead, perfect distribution. The catch is that nodes must be addressable as integers 0..N-1, which means you can't easily add weights or pick arbitrary node ids. It's used inside Vitess and a number of Google systems but rarely in application code, because the integer-id constraint is awkward when nodes have names like shard-eu-west-3.

Range partitioning wins when you actually do range scans (analytics, time-series). Consistent hashing kills the locality you need. Spanner, CockroachDB, and TimescaleDB all use range-partitioning for this reason. The cost is hot ranges — recent dates always land on the same shards — which you mitigate with prefix sharding or salting.

07Decision guide.

Sharding a cache

Consistent hashing, 100–200 vnodes. Add/remove during operation is safe — at most 1/N keys go cold.

Sharding a database

Consistent hashing for the routing decision. Pair with a chunk catalogue for the migration tooling — you'll want to move data, not just route it.

Routing requests to backends

Envoy and HAProxy both have consistent hash LB modes. Use when you want session affinity without sticky cookies — same tenant always hits the same backend.

Workload imbalance after resize

Increase vnodes per node from 100 to 1000. Distribution variance goes from ±5% to ±2%. CPU cost is negligible.

Heterogeneous nodes

Use weights. The "2× bigger" box gets weight=2 and twice the vnodes — twice the keys, automatically.

Sharding by user, but a few users are huge

You have a hot key problem, not a hashing problem. Split the heavy tenants out — give them dedicated shards. The ring still routes for the rest.

"Modulo sharding is the kind of mistake that's easy to ship at small scale and impossible to undo at large scale. Pay the consistent-hash tax on day one."
