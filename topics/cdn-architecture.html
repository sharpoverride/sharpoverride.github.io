Chapter 20 Difficulty ★★★☆☆ Read ~12 min Stack Azure Front Door · CloudFront · Cloudflare · Fastly

The cheapest byte is the one the edge already has.

Giphy's edge cache hit rate is 99.5%. The other 0.5% is what hits their origin. Yours is 60% because nobody set Cache-Control correctly. Closing that gap is the highest-leverage performance work in most apps.

The problem in one line

Every byte your origin serves costs you compute, egress, and 200ms across an ocean. Every byte the edge serves costs you 5ms and pennies.

The thesis

Pull-through caching is the right default. Tune Cache-Control, immutable filenames, and signed URLs. Only reach for push-CDN, edge functions, or Fastly's VCL when the basics aren't enough.

01Why origin-served bytes are expensive.

An image stored in Azure Blob and served via your app, to a user in Tokyo, with no CDN: the request crosses the ocean to your East US origin, the app's pod serves the bytes, the response crosses back. Two ocean traversals. ~250ms round-trip latency. Compute cost. Egress cost (~$0.05/GB from cloud to internet). For a 200 KB image at 1M views, that's $10 in egress and ~3 minutes of cumulative pod CPU for nothing — the image hasn't changed in months.

Now put a CDN in front. First request still goes to origin, but the CDN caches the response. Every subsequent request from anyone near that edge point-of-presence (POP) is served locally. Latency drops to ~10ms; origin cost drops to ~0. Done correctly, the CDN absorbs >95% of traffic — and that's not a "scale" benefit, it's the steady state. The CDN isn't a performance optimisation; it's the only sane way to serve static content at any scale.

"Your origin's job is to be the source of truth. The CDN's job is to be everywhere else."

02Pull, push, the edge.

Two ways content gets to the edge. Pull (lazy): the edge fetches on demand and caches the response. Push (eager): you upload to the CDN's storage; it's there on first request. Pull is the modern default; push is for video-on-demand and pre-known release artifacts.

Fig. 21Most cloud CDNs default to a single edge tier. The optional "shield" / "origin shield" / "tiered cache" mode adds the second hop and is the difference between origin-mostly-idle and origin-on-fire.

Cache-Control is the contract

The CDN doesn't guess. The origin tells it what to cache for how long via the Cache-Control header. Three values matter: public (cacheable by anyone in the path), max-age=N (cache for N seconds), s-maxage=N (separate TTL for shared caches like the CDN). Without Cache-Control: public, max-age=..., the CDN defaults to "don't cache" and every byte hits origin. The "low CDN hit rate" bug is almost always missing or wrong headers.

Immutable assets win on every dimension

Hash your filenames: app.7c4a92.js not app.js. The hash changes every deploy. Now you can ship Cache-Control: public, max-age=31536000, immutable — cache forever, never revalidate. Browsers and CDNs cache aggressively. This single pattern moves typical hit rates from 60% to 95%. Old deploys' assets remain cacheable because their hashed URLs don't change; new deploys' URLs are different, fetched fresh, cached forever.

Dynamic content can be cached too

A product page that varies by user country: Cache-Control: public, s-maxage=60, max-age=0 + Vary: X-User-Country. The CDN caches 200 variants of the page (one per country) for 60 seconds; the browser doesn't cache it at all (max-age=0). Even a 60-second cache absorbs 99% of traffic during a viral spike. "Dynamic" isn't a synonym for "uncacheable" — it just means a shorter TTL and a more careful cache key.

03Cache-key engineering.

The cache key is what makes two requests "the same" to the CDN. Get it wrong and you either fragment the cache (low hit rate) or serve the wrong content to the wrong user (cache poisoning). The default key is URL + a handful of headers; the tuning is which headers go in.

Drop query strings you don't need

By default, most CDNs include the full query string in the cache key. Marketing parameters like utm_source=twitter fragment the cache: every share on Twitter has a different key for the same content. Configure the CDN to strip utm_* parameters from the cache key (Cloudfront has "QueryString allow-list", Front Door has rules engine). Hit rate jumps overnight.

Vary: User-Agent is almost always wrong

"We need to serve different HTML to Safari" → Vary: User-Agent → cache fragments by user agent string → effective hit rate drops to near-zero because every browser version is a different key. The right fix is one of: don't vary (most apps), Vary: Sec-CH-UA-Mobile (small number of values), or do the variation at the application layer with feature detection on the client. Every Vary header value multiplies the cache fragmentation.

Signed URLs for protected content

Premium video, downloadable invoices, anything you don't want hot-linked. The app generates a URL with a signature and a short expiry: https://cdn.example.com/files/invoice-7c4a.pdf?Expires=1715000000&Signature=.... The CDN validates the signature at the edge (no origin round-trip) and serves the bytes only if valid. The cached object is keyed on the path, not the signature, so 1,000 valid signed URLs for the same file are 1 cache hit. CloudFront and Front Door support this natively; Cloudflare via Workers; Fastly via VCL.

Purge is the panic button, not the strategy

"We deployed a fix, purge the CDN." Purge is slow (10s–60s globally), rate-limited, and metric-degrading on the CDN side. The right strategy is versioned URLs — purge becomes unnecessary because new content has a new URL. Reserve actual purge for one-shot incidents: a leaked file, a broken hot-fix, a takedown notice.

04In ASP.NET Core.

The origin's job is to set the right headers. ASP.NET Core's response-caching middleware is for in-process caching; for CDN integration, you write the headers yourself or use response-caching attributes / output-caching policies.

C#   CdnCacheHeaders.cs 
using Microsoft.AspNetCore.Mvc;
using Microsoft.Net.Http.Headers;

namespace Academy.Cdn;

public static class CdnCacheExtensions
{
    // Immutable hashed asset: cache forever, never revalidate.
    public static IResult ImmutableAsset(this HttpContext ctx, string path, string contentType)
    {
        ctx.Response.Headers.CacheControl = "public, max-age=31536000, immutable";
        return Results.File(path, contentType);
    }

    // Short-lived shared cache (CDN only). Browser does not cache.
    public static void ShortSharedCache(this HttpResponse resp, int seconds)
    {
        resp.Headers.CacheControl = $"public, s-maxage={seconds}, max-age=0";
    }

    // Per-user / no-CDN — Authorization / Cookie response, never cacheable.
    public static void NoCache(this HttpResponse resp)
    {
        resp.Headers.CacheControl = "private, no-store";
        resp.Headers.Append(HeaderNames.Vary, "Authorization, Cookie");
    }
}

// Output cache policy for catalog pages: CDN-cacheable for 5 minutes, vary by country.
public sealed class CatalogOutputCachePolicy : IOutputCachePolicy
{
    public ValueTask CacheRequestAsync(OutputCacheContext ctx, CancellationToken ct)
    {
        ctx.AllowCacheLookup  = true;
        ctx.AllowCacheStorage = true;
        ctx.AllowLocking      = true;       // single-flight on cache miss
        ctx.SetVary(new[] { "x-user-country" });
        ctx.ResponseExpirationTimeSpan = TimeSpan.FromMinutes(5);
        return ValueTask.CompletedTask;
    }
    public ValueTask ServeFromCacheAsync(OutputCacheContext ctx, CancellationToken ct) => default;
    public ValueTask ServeResponseAsync(OutputCacheContext ctx, CancellationToken ct) => default;
}

Wire-up in Program.cs. Static files get the immutable header automatically; product pages opt into the policy; user-specific pages opt out.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddOutputCache(o =>
{
    o.AddPolicy("catalog", new CatalogOutputCachePolicy());
});

var app = builder.Build();

// Hashed static assets: max age 1 year, immutable.
app.UseStaticFiles(new StaticFileOptions
{
    OnPrepareResponse = ctx =>
    {
        var path = ctx.File.PhysicalPath ?? "";
        if (Regex.IsMatch(path, @"\.[a-f0-9]{6,}\.[a-z]+$"))
            ctx.Context.Response.Headers.CacheControl =
                "public, max-age=31536000, immutable";
        else
            ctx.Context.Response.Headers.CacheControl =
                "public, s-maxage=300, max-age=0, stale-while-revalidate=60";
    }
});

app.UseOutputCache();

app.MapGet("/catalog", () => Results.Ok(new { items = LoadFromDb() }))
   .CacheOutput("catalog");

app.MapGet("/me", (HttpResponse r, ClaimsPrincipal u) =>
{
    r.NoCache();
    return Results.Ok(new { name = u.Identity?.Name });
});

app.Run();

05The detail.

Stale-while-revalidate is the friendly TTL

Cache-Control: public, s-maxage=300, stale-while-revalidate=60 says "fresh for 5 minutes, then for 1 more minute the CDN may serve stale while it refreshes in the background." Users get fast responses during the refresh; origin sees one fetch per CDN POP per refresh window. This is how Vercel, Netlify, and Next.js shape the cache curve.

Cache stampedes need request coalescing

Cache expires. 1,000 concurrent requests for the same key all miss simultaneously. 1,000 requests hit origin. Origin dies. Most CDNs offer request collapsing (Cloudflare's "Argo Tunnel + Tiered Cache", Front Door's caching by default, Fastly's clustering) that ensures only one request reaches origin during a miss; the other 999 wait for it. Enable it for popular keys.

CDN logs are the lookup of last resort

Ship the CDN's access logs to your SIEM or analytics store. The 5% of requests that miss are the ones you need to optimise; the 95% that hit are the ones you need to count. Without CDN logs you're running the largest part of your stack blind.

Edge functions are the new tier

Cloudflare Workers, CloudFront Functions, Fastly Compute@Edge, Front Door rules engine. The CDN can run small bits of logic at the edge — A/B testing, geo-routing, header rewriting, even small APIs. The deeper your edge logic, the higher your effective hit rate because origin doesn't see traffic that the edge can fully answer.

Don't put authenticated requests behind a CDN without care

Authenticated content with Cache-Control: public is the classic CDN-poisoning bug. One user's response is served to another user. Always include private in authenticated responses or strip the Authorization / Cookie from the cache key. Most CDNs default-skip caching of Authorization-bearing requests, but don't bet on the default.

"99% cache hit rate is not a goal. It's evidence the rest of the system is configured correctly."

07Decision guide.

SPA + static assets, on Azure

Front Door + Blob storage origin. Hashed filenames, immutable for assets, short s-maxage for HTML.

Marketing site, want a free tier

Cloudflare. Generous free tier, Workers for any logic, great DDoS protection. The "no-decision" decision.

Media / video / news at scale

Fastly. VCL gives you cache logic nobody else does. Instant purge matters for breaking-news takedowns.

API responses need geo-routing

CDN with edge functions (Workers / CloudFront Functions / Front Door rules). Route on header, geo, A/B bucket — at the edge.

Protected downloads (invoices, video)

Signed URLs with short TTL. Same CDN cache hit; signature validated at the edge; origin sees only the first miss.

Cache hit rate is 60%

Check four things: query-string stripping, Cache-Control: public, max-age, immutable filenames, Vary headers. One of them is the bug.

Three things people get wrong

One — no Cache-Control: public. CDN defaults to "don't cache." The fix is two words in a header. The bug it produces is your origin doing every byte of traffic.

Two — Vary: User-Agent. Cache fragments per browser version. Effective hit rate plummets. Use Sec-CH-UA-* client hints or don't vary at all.

Three — relying on purge for deploys. Hashed filenames make purge unnecessary. Versioned URLs let old and new content coexist; purge becomes the panic button for incidents, not the deploy mechanism.
