Charge once. Retry safely.

A network is allowed to retry. A double-click is allowed to fire. A queue is allowed to redeliver. Your money endpoint is not allowed to charge twice. The pattern Stripe shipped in 2015 and everyone copied since.

The problem in one line

"At-least-once" is the only thing the wire promises. "Exactly-once" is something your application earns.

The thesis

Idempotency is a property of the request, not the endpoint. A client-supplied key plus a server-side dedupe store turns every API into safe-to-retry. The store needs strong consistency or the trick breaks.

01The retry problem.

Your mobile client sends a POST /charges. Two seconds later the request is still in flight. The user taps again. The phone gives up on the first request, retries. Your load balancer fails over mid-deploy and the original request lands on a new pod. Three POSTs hit your service. Each one looks legitimate. Each one finds a valid card. Each one writes a row.

Without a dedupe story, your customer is charged three times. Even with the best authentication, the best validation, the best logging — the wire is allowed to do that. HTTP semantics make POST non-idempotent on purpose precisely so the client can retry safely on transient failures: if the server doesn't say it succeeded, the client should assume it didn't. The cost of that liberty is duplicates, and the fix is a key.

"The network is allowed to retry your request. Your money endpoint is not allowed to apologise."

Stripe's 2015 blog post made the pattern famous: the client generates a UUID, puts it in Idempotency-Key, and the server promises that any future request with the same key returns the same response without re-executing the side effect. Shopify adopted it. Twilio adopted it. AWS calls it ClientRequestToken. Every modern payment API has a flavour of this header. The reason: it's the cheapest thing that works, and the bug it prevents is the most expensive bug a payment system can have.

02The key as contract.

The lifecycle is straightforward — until it isn't. Three states, two storage rows, one decision per request.

Fig. 03First request reserves the key, executes, caches the response. Replays read the cache. The handler runs once.

What the key actually is

A UUID generated by the client, sent as Idempotency-Key in the request header. The server never generates it. If the server generates it, every retry generates a different one, and the dedupe story collapses. The client is the one who knows that two requests are "the same logical operation." It must keep the same key across all retries of that operation and rotate to a fresh key for the next logical operation.

Stripe's recommendation is to store the key alongside the local pending state: the moment your client wants to charge, it generates the UUID, persists it next to the user's intent, and then sends the request. If the request fails, the local state still has the same UUID for the next retry. If the app crashes, the UUID survives.

What the server stores

Two pieces of state, both keyed on the UUID: a reservation marker (atomically set on first sight) and the cached response (written after the handler completes). Typical TTL is 24 hours — long enough that mobile clients waking from a backgrounded retry can still replay, short enough that the store doesn't grow without bound. Stripe uses 24 hours. So should you.

The cached response should include the full response body, status code, and a hash of the request payload. The hash is the safety net: if a client reuses an Idempotency-Key with a different body, that's a bug, and you should reject with 409 Conflict instead of silently returning the wrong cached response.

03The concurrency wrinkle.

The hard case isn't the second request. The hard case is the second request arriving before the first request has finished. Two pods receive the same UUID within milliseconds. Both check the key store. Both see "no record." Both proceed. Both charge.

Three outcomes per request: first ever (do work, cache), in progress (poll), replay (read cache and return).

SETNX is the lock

Redis SET key value NX EX ttl returns "ok" if the key didn't exist, "nothing" if it did. That single atomic operation is your distributed lock. The pod that wins moves to execute. The pod that loses moves to poll. The trick is the in-flight state. The loser cannot just return the cached response — there isn't one yet. It also cannot proceed — that would defeat dedupe. It has to wait, briefly, polling the response cache with a backoff, until either the response shows up or the lock TTL expires.

If the winner crashes mid-charge, the lock TTL expires (typical: 60 seconds for the in-flight, 24 hours for the cached response). After that, a retry will win the SETNX and run again. If the charge succeeded but the response wasn't cached, the retry will charge a second time. That's why the actual money operation must itself be idempotent at the downstream level (Stripe's API, the payment processor, the bank), which they all are — typically via a second idempotency key passed through.

Belt and braces with a DB unique index

Even with Redis as the primary, put a unique index on (IdempotencyKey) in your Orders / Charges table. EF Core enforces it on save. If the Redis path fails open and a duplicate slips through, the database rejects the second insert. The Redis path is the fast path; the database is the truth.

04In ASP.NET Core.

A middleware that intercepts requests with an Idempotency-Key header, reserves the key in Redis, captures the response, and replays on retry. Drop into the pipeline before your endpoints. No endpoint code changes.

C#   IdempotencyMiddleware.cs 
// dotnet add package StackExchange.Redis
using System.Security.Cryptography;
using System.Text;
using System.Text.Json;
using Microsoft.AspNetCore.Http;
using StackExchange.Redis;
using Microsoft.Extensions.Logging;

namespace Academy.Idempotency;

public sealed record CachedResponse(int Status, string Body, string BodyHash);

public sealed class IdempotencyMiddleware
{
    private readonly RequestDelegate _next;
    private readonly IConnectionMultiplexer _redis;
    private readonly ILogger<IdempotencyMiddleware> _log;
    private static readonly TimeSpan InflightTtl = TimeSpan.FromSeconds(60);
    private static readonly TimeSpan CacheTtl    = TimeSpan.FromHours(24);

    public IdempotencyMiddleware(RequestDelegate next,
        IConnectionMultiplexer redis, ILogger<IdempotencyMiddleware> log)
    {
        _next = next; _redis = redis; _log = log;
    }

    public async Task InvokeAsync(HttpContext ctx)
    {
        // Only POST / PATCH / DELETE need this. Reads are naturally idempotent.
        if (!HttpMethods.IsPost(ctx.Request.Method)
            && !HttpMethods.IsPatch(ctx.Request.Method))
        { await _next(ctx); return; }

        var key = ctx.Request.Headers["Idempotency-Key"].ToString();
        if (string.IsNullOrWhiteSpace(key))
        { await _next(ctx); return; }   // caller opted out; their problem

        var db        = _redis.GetDatabase();
        var lockKey   = $"idem:lock:{key}";
        var cacheKey  = $"idem:resp:{key}";

        var bodyHash  = await HashBodyAsync(ctx.Request);

        // Path 1 — cached response exists. Validate request hash matches.
        var cached = await db.StringGetAsync(cacheKey);
        if (cached.HasValue)
        {
            await ReplayAsync(ctx, cached!, bodyHash); return;
        }

        // Path 2 — try to acquire the in-flight lock atomically.
        var acquired = await db.StringSetAsync(
            lockKey, bodyHash, InflightTtl, when: When.NotExists);

        if (!acquired)
        {
            // Path 3 — another pod is processing. Poll cacheKey with backoff.
            for (var i = 0; i < 30; i++)
            {
                await Task.Delay(100 * (i + 1));
                cached = await db.StringGetAsync(cacheKey);
                if (cached.HasValue)
                { await ReplayAsync(ctx, cached!, bodyHash); return; }
            }
            ctx.Response.StatusCode = StatusCodes.Status409Conflict;
            await ctx.Response.WriteAsync("Idempotency lock held; retry.");
            return;
        }

        // Path 4 — we own the lock. Execute and capture.
        var origBody = ctx.Response.Body;
        await using var capture = new MemoryStream();
        ctx.Response.Body = capture;

        try
        {
            await _next(ctx);

            capture.Position = 0;
            var respBytes = capture.ToArray();
            var entry = new CachedResponse(ctx.Response.StatusCode,
                Convert.ToBase64String(respBytes), bodyHash);

            // Only cache 2xx responses; client errors should be retryable.
            if (ctx.Response.StatusCode >= 200 && ctx.Response.StatusCode < 300)
                await db.StringSetAsync(cacheKey,
                    JsonSerializer.Serialize(entry), CacheTtl);

            await capture.CopyToAsync(origBody);
        }
        finally
        {
            ctx.Response.Body = origBody;
            await db.KeyDeleteAsync(lockKey);
        }
    }

    private static async Task<string> HashBodyAsync(HttpRequest req)
    {
        req.EnableBuffering();
        using var sha = SHA256.Create();
        var hash = await sha.ComputeHashAsync(req.Body);
        req.Body.Position = 0;
        return Convert.ToHexString(hash);
    }

    private static async Task ReplayAsync(HttpContext ctx, string json, string bodyHash)
    {
        var entry = JsonSerializer.Deserialize<CachedResponse>(json)!;
        if (!string.Equals(entry.BodyHash, bodyHash, StringComparison.Ordinal))
        {
            ctx.Response.StatusCode = StatusCodes.Status409Conflict;
            await ctx.Response.WriteAsync("Idempotency-Key reused with different body.");
            return;
        }
        ctx.Response.StatusCode = entry.Status;
        ctx.Response.Headers["Idempotency-Replayed"] = "true";
        await ctx.Response.Body.WriteAsync(Convert.FromBase64String(entry.Body));
    }
}

Wire it up in Program.cs — one line. The redis multiplexer is registered as singleton, the middleware as scoped. Place it after auth and before your endpoint routing:

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IConnectionMultiplexer>(
    ConnectionMultiplexer.Connect(builder.Configuration.GetConnectionString("Redis")!));

builder.Services.AddDbContext<PaymentsDbContext>(o =>
    o.UseSqlServer(builder.Configuration.GetConnectionString("Payments")));

var app = builder.Build();

app.UseAuthentication();
app.UseMiddleware<IdempotencyMiddleware>();   // after auth, before endpoints

app.MapPost("/v1/charges", async (
    ChargeRequest req, PaymentsDbContext db, CancellationToken ct) =>
{
    var charge = new Charge { Id = Guid.NewGuid(),
        Amount = req.Amount, Currency = req.Currency };
    db.Charges.Add(charge);
    try { await db.SaveChangesAsync(ct); }
    catch (DbUpdateException) when (IsUniqueViolation(ex))
    {
        // Belt-and-braces: DB unique index on (IdempotencyKey) rejects dupe.
        return Results.Conflict(new { error = "duplicate_charge" });
    }
    return Results.Created($"/v1/charges/{charge.Id}", charge);
});

app.Run();

05The detail.

Hash the body, compare on replay

If a client reuses the same key with a different request body, that's a programmer bug — almost always a UUID generated once and then reused for a different operation. Returning the cached response would be silent corruption. Reject with 409 instead. The hash check costs a few microseconds and saves a Sev-1.

Only cache 2xx responses

If the first attempt returned 400 Bad Request because the body was malformed, the second attempt with the same key but a corrected body should be allowed to succeed. Caching a 4xx response makes the key permanently poisoned. Caching a 5xx response is even worse — it freezes a transient failure. Cache 200, 201, 202, 204. Let everything else retry.

The TTL split that matters

Two TTLs, not one: in-flight lock = 60 seconds (long enough to outlast a slow handler, short enough that a crashed pod doesn't block forever), cached response = 24 hours (long enough for mobile-app retries; aligned with what Stripe does). The lock TTL is the one that bites you in production — set it shorter than your max request timeout and you'll see double-execution on slow paths.

Add a unique index. Even with Redis.

The Redis path can fail open under partition. The database is the truth. builder.HasIndex(x => x.IdempotencyKey).IsUnique(); in your EF configuration costs you nothing and saves you the day Redis is unreachable for 30 seconds.

MassTransit consumers want the same pattern

Inside a message handler, the equivalent is UseInMemoryOutbox or the MassTransit dedupe filter keyed on MessageId. Brokers redeliver — that's an at-least-once promise — so consumers need the same idempotency guarantee that POST endpoints do. Same primitives, same Redis store typically does double duty.

"Idempotency is the only feature where the absence of bugs is the entire user experience. Nobody thanks you for it. Everybody curses its absence."

07Decision guide.

Public payments / money API

Require Idempotency-Key. Reject POST without it with 400. Redis lock + DB unique index. 24 h cache TTL.

Internal microservices over HTTP

Middleware optional, default off. Producers that retry should send the header; everyone else can skip. Don't make idempotency a tax.

Mobile client → backend

Mandatory. Mobile networks retry aggressively. Generate the UUID locally, persist alongside the user intent before the first send.

Message consumer (Kafka / RabbitMQ / SB)

MassTransit dedupe filter or your own check keyed on MessageId. Brokers are at-least-once. Consumers must be idempotent.

Greenfield .NET, no Redis yet

Just use the DB. Unique index on (key, tenant). Add Redis later for the low-latency replay if the DB lookup is your hot path.

Already serverless on AWS

DynamoDB conditional Put + API Gateway. The header pattern is the same; the store changes.

Three things people get wrong

One — caching the 500. A handler throws, the middleware caches the error response, every retry replays it. Cache only 2xx, and surface the lock so the next attempt can try fresh.

Two — generating the key server-side. "We'll just use a request ID from the gateway." Every retry generates a new request ID, so dedupe is impossible. The client owns the key. Always.

Three — assuming Redis is the truth. Redis fails over, AOF lags, a flush happens. Put the unique index in the database. Belt and braces. Always.
