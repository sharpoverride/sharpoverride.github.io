Interview walkthrough Difficulty ★★★★☆ Read ~13 min Scale ~150M users · 7M listings

Design Airbnb.

Search, geo-indexing, calendar availability, booking flow. The deceptively-simple "list all listings" turns into a spatial-index problem; the deceptively-simple "is this date available" turns into a concurrency problem.

Anchor numbers

~150M users · 7M+ active listings · ~2M+ stays/night peak · search latency target <300ms · booking conflicts must be 0.

The hard parts

(1) Geo-search at low latency. (2) Calendar availability without double-bookings. (3) Pricing dynamism (host rules + ML). (4) Multi-host inventory invariants.

Out of scope

Payments (separate; off-the-shelf processor). Messaging. Reviews after the fact.

04Deep dive — geo search.

S2 cells, not lat/lon ranges

"All listings within 5km of (37.77, -122.4)" — naive SQL with bounding box and a WHERE lat BETWEEN… works for 10k listings, dies at 7M. Use S2 or geohash: each listing tagged with an S2 cell ID at multiple resolutions; the query computes the cells covering the search area and looks up by cell ID. Index hit, fast.

Pre-filter by date

Search is "in this area, available these dates." Two-stage filter: (1) area → candidate listings via S2; (2) date → subset by checking calendar. Usually the second stage is the expensive one; cache "this listing is generally available" hints (no bookings in next 90 days) to skip cold lookups.

Personalisation = re-ranking

The base query returns ~100 candidates; ML re-ranks based on user, season, price sensitivity, conversion-likelihood. Same shape as search architecture: cheap retrieval + expensive re-rank.

05Deep dive — booking without double-book.

The race condition

Two guests click "book" for the same listing-night at the same millisecond. Both can't succeed. The traditional fix: a serialisable transaction over the calendar row, with a unique constraint on (listing_id, date).

Idempotency for retries

A network glitch retries the booking. Idempotency-Key in the request (see idempotency chapter); replay returns the same booking, not a second one. The same guest's two clicks become one booking.

Atomic per-night calendar updates

One row per (listing, date). Booking inserts rows for each night in the stay. Unique constraint enforces "one booking per night"; the second booker's insert fails. Wrap in a transaction; rollback the partial; show "no longer available." The DB is the safety mechanism, not the app code.

Pre-booking hold

"Continue to payment" places a 15-minute hold. Hold rows in the same calendar table, expiring rows reclaimed by a background job (or just a TTL'd marker). Prevents the user from completing payment to find someone else booked it in the meantime.

06Scale & tradeoffs.

Search is read-heavy; bookings are write-critical

Different scaling. Search uses read replicas + caching; bookings need primary writes with strong consistency. Separate the paths.

Calendar shard by listing

Calendar table grows; shard by listing_id. All operations for one listing land on one shard; no cross-shard transactions needed.

Pricing is cacheable, briefly

Price for (listing, date range, user-segment) cached for ~5 minutes. Slightly stale on conversion; rarely matters.

Tradeoffs to mention

Pre-booking hold vs commitment — strict hold helps users; weak hold maximises inventory. Search consistency — recently-booked listings may appear in search briefly; that's fine because booking re-validates.

Open questions

Refunds & cancellations — a separate workflow with saga semantics. Host calendar sync — iCal export/import. Multi-currency. Tax computation.

"The double-booking bug exists because the application tried to be clever. The DB unique constraint is the engineering."
