Gateway, balancer, proxy.

Three layers, three jobs, three places to put a feature flag wrong. A YARP-based gateway in 100 lines and the honest answer to "do I need an API gateway?"

The names mean things

"Reverse proxy" is the protocol — accept HTTP, forward HTTP. "Load balancer" is a job — distribute load. "API gateway" is a product — proxy + LB + auth + rate limit + transform.

The thesis

One reverse proxy is non-negotiable. A load balancer is the cheapest insurance you'll buy. An API gateway is a useful pattern that becomes a liability the moment it grows business logic.

01Three layers.

The terms are used interchangeably in casual conversation and that confusion costs teams real time. Here is the precise version.

Fig. 08Three layers · same wire protocol · very different responsibilities

The relationship is one of containment:

  • An API gateway is always a load balancer, which is always a reverse proxy.
  • A reverse proxy may or may not balance load. A load balancer may or may not enforce gateway policies.
  • When in doubt, the simpler layer is the right one. The gateway is the easiest place to accumulate accidental business logic.
"Every minute of business logic in your API gateway will cost you an hour of debugging six months from now, after the person who added it has left."

02Balancing algorithms.

The load balancer's job sounds trivial — pick a backend — until you remember that "fair" is not a single thing. Five algorithms, each fair on a different axis.

Fig. 09 · Pick by what "fair" means in your system★ = modern default

Three practical rules:

  • Round-robin is wrong by default. The moment your backends have different request mixes (some doing 10 ms work, some doing 500 ms), round-robin will overload the slow ones. Least-connections or P2C is almost always better.
  • Sticky sessions are a debt. Cookie-based affinity makes deploys painful (one pod's restart drops 1/N of users), makes hot-key isolation impossible, and makes horizontal scaling lopsided. Use only when you truly cannot move session state out.
  • Health checks must reflect business health. /healthz that returns 200 even when the DB is down has caused more outages than it has prevented. Probe a critical dependency, not just liveness.

03A YARP gateway, 100 lines.

YARP (Yet Another Reverse Proxy) is Microsoft's reverse-proxy library. Same team as ASP.NET Core; battle-tested at Bing, Azure, and Office. Below: a gateway with authentication, per-tenant routing, rate limiting (via chapter 1), response transforms, and forwarded headers — every piece configured in code, no separate YAML.

C#   Program.cs 
// dotnet add package Yarp.ReverseProxy Microsoft.AspNetCore.Authentication.JwtBearer
using Microsoft.AspNetCore.Authentication.JwtBearer;
using System.Threading.RateLimiting;
using Yarp.ReverseProxy.Transforms;

var builder = WebApplication.CreateBuilder(args);

// ── Authentication: JWT, single source of truth for all backends ──
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.Authority = builder.Configuration["Auth:Authority"];
        o.Audience  = "academy-gateway";
    });
builder.Services.AddAuthorization();

// ── Rate limiting: per-tenant, per-endpoint (see chapter 01) ──
builder.Services.AddRateLimiter(o =>
{
    o.RejectionStatusCode = StatusCodes.Status429TooManyRequests;
    o.GlobalLimiter = PartitionedRateLimiter.Create<HttpContext, string>(ctx =>
    {
        var tenant = ctx.User.FindFirst("tenant_id")?.Value ?? "anonymous";
        var tier   = ctx.User.FindFirst("tier")?.Value      ?? "free";
        return RateLimitPartition.GetSlidingWindowLimiter(
            $"{tier}:{tenant}",
            _ => new() {
                PermitLimit = tier == "enterprise" ? 10_000
                            : tier == "pro"         ? 1_000
                                                       : 100,
                Window = TimeSpan.FromMinutes(1),
                SegmentsPerWindow = 6,
            });
    });
});

// ── YARP: declare routes + clusters in code (no appsettings YAML) ──
builder.Services.AddReverseProxy()
    .LoadFromMemory(
        routes: [
            new RouteConfig
            {
                RouteId   = "orders",
                ClusterId = "orders-cluster",
                Match     = new() { Path = "/api/orders/{**catch-all}" },
                AuthorizationPolicy = "authenticated",
            },
            new RouteConfig
            {
                RouteId   = "catalog",
                ClusterId = "catalog-cluster",
                Match     = new() { Path = "/api/catalog/{**catch-all}" },
                // public; no auth required, but rate-limited by IP
            }
        ],
        clusters: [
            new ClusterConfig
            {
                ClusterId = "orders-cluster",
                LoadBalancingPolicy = "PowerOfTwoChoices",
                HealthCheck = new() {
                    Active = new() {
                        Enabled = true,
                        Interval = TimeSpan.FromSeconds(5),
                        Policy = HealthCheckConstants.ActivePolicy.ConsecutiveFailures,
                        Path = "/healthz",
                    }
                },
                Destinations = new Dictionary<string, DestinationConfig>
                {
                    ["d1"] = new() { Address = "http://orders-1.internal:8080/" },
                    ["d2"] = new() { Address = "http://orders-2.internal:8080/" },
                    ["d3"] = new() { Address = "http://orders-3.internal:8080/" },
                },
            },
            new ClusterConfig
            {
                ClusterId = "catalog-cluster",
                LoadBalancingPolicy = "LeastRequests",
                Destinations = new() {
                    ["d1"] = new() { Address = "http://catalog.internal:8080/" },
                },
            }
        ])
    .AddTransforms(ctx =>
    {
        // Strip Authorization, replace with an internal token signed by the gateway.
        // Backends never see customer JWTs — they trust the gateway's token.
        ctx.RequestTransforms.Add(new RemoveRequestHeaderTransform("Authorization"));
        ctx.AddRequestHeader("X-Internal-Token", MintInternalToken(ctx));
        // Forward tenant context as a header — saves backends re-parsing JWT.
        ctx.AddRequestHeader("X-Tenant-Id",
            ctx.HttpContext?.User.FindFirst("tenant_id")?.Value ?? "");
        // Add trace context for distributed tracing.
        ctx.AddRequestHeader("traceparent",
            Activity.Current?.Id ?? "");
    });

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();
app.MapReverseProxy();

// Gateway-local health endpoint (used by the LB above it).
app.MapGet("/healthz", () => Results.Ok(new { ok = true, t = DateTimeOffset.UtcNow }))
   .AllowAnonymous();

app.Run();

What this small file is quietly buying you:

  • One auth model. Customers send JWTs; backends never see them. The gateway strips the customer token and mints an internal one — backends only trust the gateway. Auth breach blast-radius stops at the perimeter.
  • Health-aware routing. Power-of-two-choices for orders (heterogeneous load), least-requests for catalogue (uniform load). Failed health checks bleed traffic immediately.
  • Tenant context as a header. Backends don't re-parse the JWT — they read X-Tenant-Id and trust it because only the gateway can stamp it on internal-network traffic.
  • Distributed tracing. One traceparent from edge to database in your APM tool.

04Cross-cutting policies.

The honest reason API gateways exist: a handful of concerns belong between the client and the service mesh, and putting them in every service is a guaranteed source of drift.

Authentication / authorisation

One token-validation place. Bugs and key rotations happen once. Backends consume a smaller, trusted, internal claim set.

Rate limiting & quotas

The right layer for "free / pro / enterprise" tier enforcement. Don't put pricing logic in your services.

Request / response transforms

API versioning by URL prefix, payload shaping for mobile vs. web, response compression. Plays well at the edge.

Aggregation / BFF

"Get my orders + my profile in one round trip." The gateway calls 3 services and fuses. Use sparingly — see anti-patterns.

Observability

One place to log every request, attach trace headers, emit metrics. Without it, you have a different log format per service.

API contracts / docs

OpenAPI aggregated across services. One developer portal. One place to evolve external contracts independently of internal.

06Anti-patterns.

The gateway as application

The most common failure mode: business logic creeps into gateway policies. Tax calculation in an Azure APIM policy. Order validation in a Kong plug-in. The gateway has the worst dev loop of any tier — versioned config, hot-reload semantics, vendor-specific debugging. Once business logic is there, no team wants to own it. Push it back into services; the gateway should only enforce contract.

Sticky sessions to avoid sharing state

Cookie-based affinity is sometimes "free scaling" today and "no zero-downtime deploys" tomorrow. The fix is the same as it always was: move session state to Redis / Cosmos / SQL. Pay the small latency now to avoid the deploy pain forever.

Trusting X-Forwarded-For blindly

The header is writable by the client. If your gateway terminates from one trusted LB, replace the chain instead of appending. Otherwise an attacker can forge an IP for your rate-limiter or audit logs.

One gateway for everything

Internal-service-to-service traffic should generally not go through the public-facing gateway. The auth / RL / cert work doesn't make sense there, the latency tax is paid twice, and the blast radius of a gateway outage becomes "the whole company." Use service mesh (Linkerd / Istio) or direct calls for east-west; reserve the gateway for north-south.

07Decision guide.

Single backend service

NGINX or YARP as reverse proxy. Skip the gateway. Add a managed LB in front for HA.

3–10 backend services

YARP-as-gateway is enough. Code, version-controlled, deployed with your services. Add APIM only if you need a developer portal.

External partners / monetised API

APIM or Kong. Developer portal, OAuth, subscription keys, usage analytics — these are the actual features you're paying for.

Global low-latency reads

Azure Front Door / CloudFront / Cloudflare in front. Edge cache + WAF + DDoS for free, the rest of your gateway behind it.

Microservices > 30 services

Add a service mesh (Linkerd, Istio) for east-west. Gateway only on the perimeter. Don't double up.

You're a .NET shop

YARP unless something else specifically beats it. Same observability, same DI, same NuGet, same Visual Studio. Zero new ops.
