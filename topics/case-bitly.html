The simplest interview question.

"Design a URL shortener." Asked in every coding interview for a decade. The toy version fits in 30 lines. The real Bitly handles ~10 billion clicks a month, half of which are bots. The system-design lessons are in the gap.

The core problem

Map short code → long URL. Fast redirect. Click analytics. Custom branded short domains.

The size

~10⁹+ unique short links cumulative. ~10B clicks/month. Most reads are "redirect"; analytics is async.

01The simple version.

Generate a unique short code. 7 characters base-62 = 62⁷ ≈ 3.5T URLs — enough. Store in codes(code PRIMARY KEY, long_url, created_at). On request, lookup, 301 redirect.

The interesting parts: how do you generate the code? Hash of URL collides; random + retry-on-collision works but doesn't scale linearly; base-62 of an auto-increment ID is elegant but reveals creation order; a key-generation service handing out blocks works in practice.

"The URL shortener interview question is interesting because the constraints aren't visible in the prompt."

02The real version.

Key generation service

Pre-generate codes in batches. A central service mints a million codes at a time; web servers grab a batch of 1000, use them locally, ask for more when depleted. No coordination on the hot path; no race conditions; no collision retries.

Read-heavy means cache-heavy

The hot read pattern is "this code → that URL." Memcached / Redis cluster in front of MySQL; hit rate >95%. Cache TTL essentially infinite (URLs rarely change post-creation); invalidation on the rare edit.

Custom domains

Enterprise customers want brand.co/abc, not bit.ly/abc. One service, many domains: the redirect handler checks the Host header and looks up the brand's namespace. SSL via a separate cert per customer (or wildcards).

03Click analytics.

Every redirect is a click event. Store who clicked, when, where (geo from IP), referrer, user-agent. 10B clicks/month = ~4k events/sec average; peak much higher. Bursts on social-viral content can do 100× normal.

Pattern: fire-and-forget the click event to a queue (Kafka); the redirect path returns 301 within microseconds. The analytics pipeline reads from Kafka asynchronously and aggregates into a data store (Cassandra historically, now various). Redirects don't wait on analytics; analytics is best-effort.

Bot traffic

Up to half of Bitly's redirect traffic is bots. Filter early: per-IP rate limiting, user-agent classification, CAPTCHA on suspicious patterns. Bots inflate the analytics if you don't filter; customers see misleading dashboards.

05The lessons.

Pre-generate keys

Hot-path code-generation = bottleneck. Pre-mint in batches; serve from cache. Same pattern as Twitter Snowflake IDs, Stripe charge IDs.

Async analytics from day one

Don't block the redirect on logging. Queue → consumer → aggregate. The redirect SLA is microseconds; the analytics SLA is minutes.

Cache aggressively; URLs are immutable

A short link rarely changes. Cache forever. Invalidate on the rare edit. Cache hit rate is the system's heartbeat.

Bots inflate every metric

~50% of Bitly traffic is bots. Filter at ingest. Without filtering, every customer metric is a lie; with filtering, they're not.
