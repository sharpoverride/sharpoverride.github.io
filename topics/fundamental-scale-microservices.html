The technique nobody calls by name.

"Scale microservices" usually means one specific thing: take the synchronous fan-out, replace it with an asynchronous fan-out, and put a capability boundary in front. The fancy name is "Backend for Frontend + async aggregation"; the shape is what every team eventually arrives at.

The default problem

The API gateway calls 12 services synchronously to render a page. Latency = sum of slowest path. One slow downstream = whole page slow.

The technique

BFF (purpose-built per-client API). Async fan-out. Per-capability ownership. Result: latency = max, not sum. Robustness: degraded slices, not total failure.

01The sync fan-out trap.

Page load: gateway → 12 internal service calls in sequence (or in parallel waiting for the slowest). Latency is bounded by the slowest path; if any one service is degraded, the page is degraded. Failure of any one service = failure of the page.

This is what most "microservices are broken" complaints actually describe. The architecture isn't wrong; the synchronous fan-out is.

"Sum of latencies is the failure mode. Max of latencies is the goal."

02BFF.

Backend for Frontend. One BFF per client (web, iOS, Android, partner API). The BFF is the only thing the client talks to. It fans out to internal services; aggregates; transforms; returns the per-client-shaped response.

Why per-client: the iOS app needs different fields, different cardinality, different error shapes than the web app. A single "API for everything" forces compromises that hurt every client. Per-client BFFs let each surface optimise.

The BFF's job

(1) Authenticate the client. (2) Decide which internal services to call. (3) Fan out with timeouts. (4) Compose the response. (5) Cache aggressively where possible. The BFF is the client's pseudonym for "the backend"; everything behind it is invisible.

03Async aggregation.

Push, don't pull

Instead of "fetch the user's home feed on every refresh," maintain the user's home feed as a materialised view, updated by events from upstream services. Reads become cache lookups; writes flow through queues. Same shape as Twitter's home timeline (see Instagram for the same pattern).

CQRS at the BFF

Writes go through the slow path (validate, save, publish event). Reads come from a denormalised store optimised per-client-screen. The write store and read store are different shapes. Each can scale independently. See Event sourcing & CQRS.

Streaming responses

Some sections of the page can render before others. Server streams the response (HTTP/2 push, Server Components, SSE). User sees something within 100ms even if the full payload takes 800ms. The perceived-performance win is enormous.

04Capability splitting.

"User service" tends to grow until it does everything user-shaped. Split by capability, not by entity: identity, profile, preferences, billing, audit. Each is its own service, owned by its own team, scaled independently.

The result: a "user" is reassembled from N capability services at the BFF. Each capability has its own data model, its own DB, its own deployment cadence. No giant user-service monolith.

Service boundaries follow team boundaries

Conway's law: software shape matches communication structure. Microservices with team boundaries that match service boundaries are sustainable. Microservices that span team boundaries become coordination nightmares.

05Decision guide.

Page is slow due to fan-out

Build a BFF. Move the aggregation server-side. Pre-materialise expensive views via events.

One service is owned by three teams

Split by capability. Conway's law isn't optional; rest your service boundaries on real team boundaries.

Two services share a database

You don't have two services. Either merge them or split the DB. Shared DBs are the most common microservices anti-pattern.

Sync chain > 3 hops

Async-ify. The 4th hop is where the latency compounds beyond rescuable; queue the work; return on event.

GraphQL as BFF

Valid choice (see gRPC vs REST vs GraphQL). Client picks fields; server resolves from N services with DataLoader to batch. The "single endpoint" sells the pattern to product.

You're back at a monolith

Fine. Modular monolith (monolith vs microservices) is usually the right place. Most teams don't need microservices.

"You don't scale microservices by adding more services. You scale them by paying attention to the seams."
