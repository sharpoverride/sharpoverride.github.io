One write. One truth. One stream.

Dual-writes are how silent data drift starts. The outbox stops the drift by making the broker write transactional with the database write. Change data capture goes further: tail the database's write-ahead log and skip the outbox entirely.

The problem in one line

"Save the row, then publish the event" is two writes. One can succeed and the other can fail. The database and the broker disagree forever.

The thesis

Pick one source of truth — the database — and derive the event stream from it. Either via an outbox table written in the same transaction, or via CDC tailing the WAL. Never two writes pretending to be one.

01Why dual-write is broken.

The naive pattern is the most common production bug. Save the order to the database. Publish OrderPlaced to Kafka. Two operations, two systems, no shared transaction. The pod crashes between them. The order is in the database. The event was never published. Downstream services — inventory, shipping, recommendations — never hear about it. The data is silently inconsistent.

The other ordering is no better: publish first, then save. Now the event is on Kafka, downstreams react, but the database write failed. Phantom event. Inventory decremented stock for an order that was never created. Either ordering produces drift; neither ordering is acceptable; both are shipped routinely as "best effort". The fix is to remove the "two writes" entirely.

"Two systems with two writes is not a transaction. It's a coin flip with reputation damage on the tails side."

02Outbox and CDC.

Two patterns. Both rest on the same insight: one transaction, one source of truth. The difference is who tails the source.

Fig. 18Outbox keeps the event shape under app control. CDC streams the raw row shape — closer to the truth, further from the contract.

Outbox keeps events at the right abstraction

An OrderPlaced event has fields the app chose to publish. A WAL change event has every column of the Orders table, including internal ones (SyncToken, RowVersion, etc.). The outbox event is the contract; the WAL event is the schema. Most domains want the contract — the published event shouldn't change when an internal column is added or renamed. The outbox is the right pattern.

CDC wins when you can't change the app

Legacy app, can't add an outbox table, can't deploy a dispatcher. CDC tails the database without the app's knowledge. It also wins when multiple consumers need different views of the same change — let Debezium populate a Kafka topic of raw row changes, and let downstream services map each change into the abstraction they need. It's a one-to-many fan-out where the source of truth was always the schema.

Outbox + CDC is the rare-but-real third option

Write to the outbox table as part of your transaction. Let CDC pick the outbox table up from the WAL. You get the contract control of the outbox and the no-dispatcher operational simplicity of CDC. Debezium has explicit support for this — the "outbox event router" SMT (single message transform) reads the outbox columns and emits events in the right shape on the right topic. This is the pattern Shopify, Wayfair, and others have published as "the one we settled on."

03Ordering, schema, replay.

Per-aggregate ordering, not global

CDC streams in transaction order at the database level. Kafka partitions break that ordering — events for the same order_id go to the same partition (via key hashing), but events across order_ids can be reordered relative to each other on the consumer side. The consumer's contract is per-key ordering, not global. Design downstream handlers accordingly: an OrderShipped for order 7 can land before OrderConfirmed for order 9, but never before OrderConfirmed for order 7.

Schema evolution is the elephant

CDC streams the table's schema. Add a column, every consumer sees a new field. Drop a column, consumers fail to deserialise. Use the Confluent schema registry or equivalent, evolve schemas additively (Avro forward-compat), and version your consumers. The outbox pattern has the same problem at the application level — but the app controls the timing and can ship the producer change before the consumer change.

Initial snapshot + ongoing tail

Debezium starts with a snapshot: lock the table briefly, read every row, emit each as a "create" event, then switch to tailing the WAL from the matching log position. The snapshot is the slow part — for a 1 TB table, hours. Plan the rollout: snapshot in low-traffic windows, parallelise across tables, monitor the snapshot.mode and pick initial_only, schema_only, or when_needed based on whether existing data needs to be ingested.

Replay is the killer feature, and the killer footgun

CDC topics on Kafka can be replayed from offset 0. A new consumer comes online, replays the full history, builds its state from scratch. This is glorious for projections (Chapter 13) and disastrous for side-effects — replay your OrderPlaced consumer and you'll send three years of refund emails. Make consumers idempotent (Chapter 08), or gate them behind a "replay-safe" flag for non-idempotent operations.

04In EF Core.

The outbox in EF Core is a table + an ISaveChangesInterceptor that captures the events emitted during a SaveChanges and inserts them into the table inside the same transaction. A separate hosted service polls the table, publishes to the broker, and marks rows dispatched.

C#   OutboxInterceptor.cs 
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Diagnostics;
using System.Text.Json;

namespace Academy.Outbox;

public sealed class OutboxMessage
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public string Type   { get; set; } = "";
    public string Payload{ get; set; } = "";
    public DateTimeOffset OccurredAt { get; set; } = DateTimeOffset.UtcNow;
    public DateTimeOffset? DispatchedAt { get; set; }
}

// Any aggregate that wants to emit events implements this — internal to the bounded context.
public interface IHasDomainEvents
{
    ICollection<object> DomainEvents { get; }
}

// Interceptor — captures DomainEvents on every SaveChanges and writes them to the outbox.
public sealed class OutboxInterceptor : SaveChangesInterceptor
{
    public override ValueTask<InterceptionResult<int>> SavingChangesAsync(
        DbContextEventData data, InterceptionResult<int> r, CancellationToken ct = default)
    {
        var ctx = data.Context!;

        var events = ctx.ChangeTracker.Entries<IHasDomainEvents>()
            .SelectMany(e => e.Entity.DomainEvents)
            .ToList();

        foreach (var ev in events)
        {
            ctx.Add(new OutboxMessage
            {
                Type    = ev.GetType().FullName!,
                Payload = JsonSerializer.Serialize(ev, ev.GetType())
            });
        }

        // Clear after capture so re-entrancy doesn't double-write.
        foreach (var e in ctx.ChangeTracker.Entries<IHasDomainEvents>())
            e.Entity.DomainEvents.Clear();

        return base.SavingChangesAsync(data, r, ct);
    }
}

// Hosted dispatcher — polls the outbox and publishes. Single instance per env (leader-elect — see Ch. 16).
public sealed class OutboxDispatcher : BackgroundService
{
    private readonly IServiceProvider _sp;
    private readonly IPublishEndpoint _bus;
    private readonly ILogger<OutboxDispatcher> _log;

    public OutboxDispatcher(IServiceProvider sp, IPublishEndpoint bus,
        ILogger<OutboxDispatcher> log)
    { _sp = sp; _bus = bus; _log = log; }

    protected override async Task ExecuteAsync(CancellationToken stop)
    {
        while (!stop.IsCancellationRequested)
        {
            try
            {
                using var scope = _sp.CreateScope();
                var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();

                var batch = await db.Set<OutboxMessage>()
                    .Where(m => m.DispatchedAt == null)
                    .OrderBy(m => m.OccurredAt)
                    .Take(100)
                    .ToListAsync(stop);

                foreach (var m in batch)
                {
                    var type = Type.GetType(m.Type)
                        ?? throw new InvalidOperationException($"unknown type {m.Type}");
                    var ev = JsonSerializer.Deserialize(m.Payload, type)!;

                    await _bus.Publish(ev, type, stop);
                    m.DispatchedAt = DateTimeOffset.UtcNow;
                }

                if (batch.Count > 0) await db.SaveChangesAsync(stop);
                await Task.Delay(TimeSpan.FromMilliseconds(500), stop);
            }
            catch (Exception ex) when (!stop.IsCancellationRequested)
            {
                _log.LogError(ex, "outbox dispatcher iteration failed");
                await Task.Delay(TimeSpan.FromSeconds(2), stop);
            }
        }
    }
}

Wire-up: register the interceptor with EF Core, register the dispatcher as a hosted service, gate it behind leader election (see Chapter 16) so only one pod publishes at a time.

C#   Program.cs 
builder.Services.AddDbContext<AppDbContext>((sp, o) => o
    .UseSqlServer(builder.Configuration.GetConnectionString("App"))
    .AddInterceptors(sp.GetRequiredService<OutboxInterceptor>()));

builder.Services.AddSingleton<OutboxInterceptor>();
builder.Services.AddHostedService<OutboxDispatcher>();   // leader-elected externally

builder.Services.AddMassTransit(x =>
    x.UsingAzureServiceBus((ctx, cfg) => cfg.Host(builder.Configuration["ServiceBus"])));

05The detail.

Outbox dispatcher must be single-threaded per partition

Two dispatchers running concurrently will publish the same row twice. Either elect a single leader (Chapter 16) or claim rows with UPDATE TOP(N) ... SET ClaimedBy = @pod, ClaimedAt = SYSUTCDATETIME() WHERE ClaimedBy IS NULL. The claim approach avoids elections but adds SQL complexity. Either works.

Index the unsent rows

A filtered index on WHERE DispatchedAt IS NULL keeps the dispatcher query fast even as the outbox table grows to millions of dispatched rows. Without it, your dispatcher does a full scan every poll and the table becomes the bottleneck.

Purge dispatched rows

An outbox table is a buffer, not a journal. After dispatch + a safety window (24 hours typical, for replay or audit), delete the rows. A background job that DELETE TOP(10000) FROM Outbox WHERE DispatchedAt < @cutoff on a schedule keeps the table small.

CDC means privileged DB access

Debezium needs REPLICATION rights on Postgres, VIEW SERVER STATE on SQL Server, or equivalent on every DB. Security teams will ask. The mitigation is a dedicated CDC user, a tight network, and audit logging on the connector — but the access is genuinely privileged. Outbox-only deployments don't have this footprint.

Consumer idempotency is non-negotiable

Both outbox and CDC give at-least-once delivery. Every consumer must dedupe by event id (typically the row id in the outbox / the LSN in CDC). The MassTransit outbox / inbox helps; for raw consumers, a Redis SETNX on the event id is the cheap answer. If your consumer isn't idempotent, "exactly-once" is a lie everyone in the room is telling.

"The outbox is the contract you publish. The schema is the contract the database publishes. The bug is acting like they're the same."

07Decision guide.

.NET app you own, modest volume

EF Core outbox + a hosted dispatcher. Leader-elect the dispatcher so only one pod publishes. The cheapest correct answer.

Legacy app you can't change

CDC via Debezium or Azure SQL CDC. The app stays untouched; the WAL becomes the stream.

Multiple consumers, different views

Outbox + Debezium router (or pure CDC). One source of truth, many transforms, replay safety.

On Cosmos DB

Change Feed. First-party, scalable, has a hosted processor library. Don't bolt a generic CDC on top.

On Azure SQL, small scale

Azure SQL CDC + a hosted reader that publishes to Service Bus. Lower op cost than Debezium for <1k changes/sec.

Event sourcing already

Your event log is your CDC. Don't add a second layer. Subscribe directly to the stream (see Chapter 13).

Three things people get wrong

One — running the dispatcher in every pod. Concurrent dispatchers double-publish. Leader-elect or claim-tokens. Always one publisher per partition.

Two — treating CDC events as the application event contract. CDC events have raw column shapes. Add a column for internal use and every consumer sees it. Outbox + router gives you a stable contract.

Three — forgetting to retain the WAL. Debezium needs the WAL from the last committed offset. If your DB's WAL retention is shorter than your downtime, you'll resnapshot the entire database to recover. Set replication slot retention generously.
