Design ChatGPT.

A chat product that serves LLM responses. The interesting parts: GPU-bound inference scheduling, token streaming over SSE, conversation memory, and the cost ledger that makes "free tier" a real engineering constraint.

Anchor numbers

~200M MAU · ~100M daily messages · model serves 100–500 tokens/sec per GPU · A100-class GPU costs ~$2/hr · per-message cost: ~$0.001 – $0.10.

The hard parts

(1) Streaming tokens with low TTFB. (2) Batching requests onto GPUs efficiently. (3) Conversation context that exceeds the model's window. (4) Cost — each conversation is dollars, not microseconds.

What we won't model

The training pipeline. The model itself. We're designing the serving infrastructure.

The GPU bill is the architecture. Every design choice translates into "more efficient GPU utilisation = lower cost" — request batching, KV cache reuse, prompt caching, smaller models for cheap queries.

03High-level architecture.

Fig. AConversation service stitches history; inference router schedules onto the GPU fleet; safety filter scans output as it streams.

Components

Conv service: persists conversation history, formats the prompt (system message + history + new turn). Inference router: picks a GPU instance and a model variant, manages batching. GPU fleet: runs the model. Safety filter: scans tokens as they stream for policy violations.

04Deep dive — GPU serving.

Batching is everything

A GPU running one request at a time is wildly under-utilised. Continuous batching (also called dynamic batching, "iteration-level scheduling"): the inference engine packs multiple in-flight requests together. As one request finishes a token, the next request's tokens are slotted in. vLLM, TGI, TensorRT-LLM all implement this; without it, throughput is 5–10× lower.

KV cache

During generation, the model computes attention keys/values for every prior token in the context. Caching them avoids recomputation on each new token. The KV cache is the GPU's working memory; for big contexts (128k tokens), the KV cache itself can be tens of GB. Managing it across batched requests is what vLLM's "PagedAttention" optimises.

Prompt caching for common prefixes

System prompts are identical across users. Cache the KV state of the system prompt; new requests start from the cached state. Anthropic's Claude API and OpenAI both expose this — it cuts cost on the system-prompt portion by ~90%.

Model variants

Not every query needs the biggest model. Route trivial queries to a smaller, cheaper model; route complex ones to the big model. A small classifier on the input decides. Quality of cheap-model output for short replies is often indistinguishable; cost is 10× lower.

05Deep dive — token streaming.

SSE — Server-Sent Events

The model emits tokens one at a time. Stream them to the client via SSE (one HTTP request, server holds it open, sends data: token\n\n per token). The client renders progressively; perceived latency drops by ~100× vs. waiting for the full response. This is why ChatGPT feels fast even though responses take seconds. See Realtime Channels chapter for SSE specifics.

Time-to-first-token

The metric that matters. The model has to process the entire input prompt before emitting the first output token. Long prompts → slow TTFB. Prompt caching helps; smaller models for short queries help. Production target: ~500ms TTFB on small models, ~2s on large.

Mid-stream cancellation

The user closes the tab. The server should stop generating immediately. The HTTP request closing is the cancel signal; the inference engine releases the GPU slot. Without this, you generate 1000 tokens nobody reads, costing you GPU seconds.

06Deep dive — conversation memory.

Sliding window

Models have a fixed context window (~128k tokens for modern frontier models). A long conversation eventually exceeds it. Sliding window: keep the most recent N tokens; older messages drop off. Simple, lossy.

Summary memory

As the conversation grows, periodically compress old messages into a summary. Replace the old messages with the summary in the prompt. The model carries the gist of the conversation without the bytes. Implementation: run a "summarise the conversation so far" model call when the context budget gets tight.

Retrieval-augmented memory

Store every message in a vector database; on each new turn, retrieve the most relevant past messages and prepend them. Long-term recall: the user references something they said two weeks ago, the system retrieves it. See RAG chapter for the retrieval pattern.

07Scale & bottlenecks.

GPU supply is the cap

You cannot auto-scale GPUs in five minutes. Capacity has to be reserved. Spot-instance pools, multi-cloud agreements, and queueing during spikes are the operational reality. ChatGPT regularly has "we're at capacity" pages during peaks.

Cost optimisation is the product feature

Each query is dollars. The free tier exists because the per-query cost is low enough to subsidise. Smaller models, prompt caching, output truncation, batch limits — all are levers on the cost-per-query line. The financial model and the architecture are intertwined.

Safety filtering is async-friendly

Output safety check runs on the stream: as tokens arrive, a smaller classifier (or a list of patterns) inspects them. If a violation is detected, the stream is cut and the response is replaced with a refusal. Don't wait for the full response to filter; that doubles latency.

Tradeoffs to mention

Quality vs latency vs cost: bigger model = better answers = slower & more expensive. Picking is product policy. Stateful conversation vs stateless API: ChatGPT persists; the API is stateless and forces the caller to manage memory.

Open questions

Multi-modal input (images, audio): different model. Tool use / function calling: the model emits structured calls, the system executes them, results feed back. Fine-tuning per-user: cost-prohibitive at this scale, hence retrieval-based memory instead.

"This is the first product in our industry where the cost-of-goods-sold has been measured in dollars per request."
