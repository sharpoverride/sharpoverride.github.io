A helper next to every pod.

The sidecar pattern co-locates a helper process with your application — same pod, same lifecycle, separate process. Logging, tracing, mTLS, service discovery, secret rotation, retries — features your app gets for free, by living alongside something that does them properly.

What you outsource

Cross-cutting concerns: TLS termination, mTLS, retries, circuit breaking, observability, log shipping, config reload.

What you keep

Business logic. Domain code. The bits that are actually specific to your service.

01The shared-concerns problem.

Every service needs the same boring infrastructure features. mTLS to internal callers. Retry policies. Circuit breakers. Distributed traces. Metrics emission. Log shipping. Secret rotation. Implementing these in every service in every language is the polyglot tax — and even in a monoglot shop, it's a maintenance burden that scales with service count.

Three options for centralisation. Library: every service links the same SDK. Same code, but versioning across N services is painful; polyglot multiplies the work. Gateway: a central proxy at the edge does it. Works for ingress, doesn't work for service-to-service. Sidecar: every service runs a small helper process alongside it; the helper does the work. The sidecar wins because it covers service-to-service traffic without burdening the application.

"Outsource the boring problems. Keep the interesting ones."

02What a sidecar looks like.

Fig. 29The sidecar shares the pod's network namespace. The app talks to localhost; the sidecar handles everything that touches the wire.

Shared pod lifecycle

The sidecar starts before the app, terminates after it. Kubernetes models this as multiple containers in one pod, sharing the network namespace and (optionally) volumes. The app and the sidecar are deployed together, scale together, fail together. They are not "two services that happen to talk"; they are one operational unit.

03Service mesh = sidecar squared.

A service mesh — Istio, Linkerd, Consul Connect — is "sidecar deployed automatically to every pod, configured centrally." Each pod gets an Envoy (Istio) or linkerd2-proxy. The control plane pushes routing rules, mTLS certs, retry policies. The app code stops needing to think about mTLS at all; the sidecar mesh issues and rotates certs and terminates TLS at both ends of every internal call.

Sidecarless meshes — Cilium, Istio Ambient

Sidecars cost CPU and memory: ~50–100 MB per pod, sometimes more. At 10,000 pods, that's serious. Newer mesh designs move the proxy to a per-node daemon or kernel-level eBPF: Cilium (eBPF), Istio Ambient mode (per-node ztunnel + waypoint proxies). Same features, lower overhead. This is the active frontier in service-mesh design.

04In Kubernetes.

yaml   pod-with-otel-sidecar.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: orders-api
  labels: { app: orders }
spec:
  containers:
    - name: app
      image: ghcr.io/example/orders:v2
      ports: [{ containerPort: 5000 }]
      env:
        # App emits OTLP to localhost — the sidecar picks it up.
        - { name: OTEL_EXPORTER_OTLP_ENDPOINT, value: "http://localhost:4317" }

    - name: otel-collector
      image: otel/opentelemetry-collector-contrib:0.95.0
      ports:
        - { containerPort: 4317 }   # OTLP gRPC ingest
      volumeMounts:
        - { name: otel-config, mountPath: /etc/otelcol }
      args: ["--config=/etc/otelcol/config.yaml"]

  volumes:
    - name: otel-config
      configMap: { name: otel-collector-config }

The app exports traces, metrics, logs to localhost:4317 — no DNS lookup, no auth, no mTLS in app code. The sidecar forwards to the collector pool, adds enrichment, batches, exports to multiple backends. If the backend is down, the sidecar buffers; the app doesn't notice.

Dapr — sidecar as platform

Dapr is the most ambitious of the application-sidecar designs: one runtime sidecar provides service invocation, pub/sub, state management, secrets, and bindings via HTTP/gRPC APIs. Your app calls localhost; Dapr translates to whatever backend (Redis, Kafka, Azure Service Bus, etc.) is configured. Same code runs against different infrastructure by changing a YAML file. The cost: an extra abstraction layer, and you're tied to Dapr's API shape.

05The detail.

Resource overhead is real

An Envoy sidecar idles around 50–100 MB RAM and 5–10% of a core; under load, much more. For pod-dense clusters, the mesh tax is significant. Budget for it; benchmark before committing.

Sidecar startup order matters

The app starts and tries to call out — but the sidecar isn't ready yet. Connection refused. K8s 1.28+ has native sidecar lifecycle (the sidecar starts before app containers and runs in a separate init phase). Pre-1.28, the pattern was a startup script in the app that waits for sidecar readiness.

Don't try to debug through the sidecar

Production weirdness is harder when traffic flows through Envoy. Have a kill-switch that disables the mesh sidecar for one pod so you can compare bare-app behavior. Linkerd and Istio both support pod-level opt-out.

Sidecars vs daemonsets

A sidecar is per-pod. A daemonset is per-node. For genuinely shared services (a log forwarder, an eBPF probe), daemonset is cheaper: one instance per node, shared across pods. Use sidecars when per-pod isolation matters (mTLS identity, per-pod retry policy).

Ambient mode is the future

The sidecar-per-pod model is being replaced by per-node "ambient" proxies + waypoints for L7 logic. Istio Ambient, Cilium service mesh. For new mesh deployments in 2026, look at ambient before committing to sidecar-per-pod.

"The sidecar is the architecture pattern that admits cross-cutting concerns deserve their own runtime."

06Decision guide.

Observability for a polyglot fleet

OpenTelemetry Collector as a sidecar. App emits OTLP to localhost; collector batches and forwards. Language-agnostic.

Internal service-to-service mTLS

Linkerd is the simplest mesh. Istio if you need richer L7 policy. Both auto-issue and rotate certs.

Secret rotation

Vault Agent sidecar. Renews short-lived credentials; writes them to a tmpfs mount the app reads.

Want service-mesh features, cheaper

Istio Ambient or Cilium service mesh. Per-node ztunnel instead of per-pod sidecar. Active development; benchmark.

Polyglot app needs pub/sub, state, secrets

Dapr. One sidecar abstracts multiple backends; app calls localhost APIs.

Single-pod, no mesh needed

Don't add a sidecar. The libraries (OTel SDK, Polly) are fine when one process is doing the work.
