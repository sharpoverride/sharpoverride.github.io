The sort behind everything you've called sort() on.

Tim Peters invented Timsort for Python in 2002. It's now the default in Python's list.sort(), Java's Arrays.sort(), Android, Rust's slice.sort(), V8 (JavaScript). The trick: real-world data isn't random; it has runs of order; exploit them.

Why it won

O(n) on already-sorted-ish data. O(n log n) worst case. Stable. Conservative on extra memory. Pleasantly fast in practice.

What it's not

Quicksort. The lore in textbooks is quicksort; the production reality has been Timsort or a Timsort-derivative for over a decade.

01Why it exists.

Textbook sorts (quicksort, mergesort) treat the input as random. Real-world input isn't random — log files are partially sorted, sensor data trends, user data clusters. Timsort exploits the order that's already there. If the input is mostly sorted, it runs in O(n); if it's reverse-sorted, also O(n). Random input is the worst case at O(n log n).

It's also stable: equal elements maintain their original order. Mergesort is stable; quicksort isn't. For sorting tables by multiple columns or anything where stability matters, Timsort wins by default.

"The textbook sort is for the worst case. The production sort is for the common case."

02Runs & minrun.

Scan the array. Identify runs — already-sorted (or reverse-sorted) subsequences. Reverse the descending ones in place; you now have ascending runs. Most real-world arrays have long natural runs; Timsort builds on them rather than ignoring them.

Minrun

If a natural run is too short, Timsort extends it via binary insertion sort to a target size called minrun (typically 32–64). Why: very-short runs cause too many merges; binary insertion sort is O(n²) but fast for small n with low constant factors. The "merge runs" phase is the expensive part; minrun keeps the count manageable.

03Merge & galloping.

Push runs onto a stack. Maintain invariants: top run lengths obey size relationships. When invariants break, merge. The merging respects the stack invariant; Timsort's bookkeeping is what makes the total work O(n log n) rather than O(n²).

Galloping

While merging two runs A and B: if A keeps winning (many consecutive elements of A are smaller than B's head), switch to "galloping" mode — use binary search to find where B's head fits in A, copy that whole block of A at once. Turns a string of comparisons into one log-n search; when one run is much smaller than the other, this is a huge win.

The galloping threshold adapts

If galloping pays off, lower the threshold (be quicker to gallop next time). If it doesn't, raise it. Self-tuning to the data's actual shape. This is the dose of empiricism that makes Timsort feel "smart."

Why this matters for system design

Most language standard libraries default to Timsort or a variant. "Sort" is fast and stable by default; you don't usually need a custom sort. The rare exceptions: external sorting (data doesn't fit in RAM — merge sort with disk runs), radix sort for fixed-width keys, counting sort for small integer ranges, parallel sorts for many-core machines.

The 2015 Java bug

Timsort's invariant-keeping logic had an off-by-one that, on adversarial inputs, broke the invariants — and crashed Java's Arrays.sort with ArrayIndexOutOfBoundsException. Found by formal verification, fixed promptly. Even "trivial" sort code in 2015 still had bugs hiding.

"Quicksort is the algorithm in your textbook. Timsort is the algorithm in your runtime."
