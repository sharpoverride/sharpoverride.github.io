The slowest-moving part of engineering.

Most teams' velocity bottleneck isn't writing code; it's waiting for review. The cultural and tooling conventions around code review compound — small PRs reviewed in hours produce a different organisation than 2000-line PRs reviewed in days.

The biggest lever

PR size. A 100-line PR gets a careful review in 30 minutes. A 1000-line PR gets a glance in 30 minutes. Same author time; tenfold different review quality.

01PR size.

The studies are consistent: reviewer attention scales sub-linearly with diff size. Reviewers who see a 1500-line PR spend roughly the same time on it as on a 200-line one — and miss proportionally more bugs. Big PRs are reviewed badly; the second half gets skimmed.

Target: <400 lines of change per PR. Tooling note: that's lines of logic changed, not auto-generated files. Lockfile updates, codegen, formatting — exclude.

Stacked diffs

Splitting one big change into N small dependent PRs. Each PR reviewed independently; merged in order. See stacked diffs chapter. The technique that makes "small PRs only" feasible for big changes.

"Authors write the diff once. Reviewers see it cold. The size that respects that is small."

02Review depth.

What the reviewer is actually checking

LayerWhat
Tests passCI did this; reviewer can skip
Linting cleanCI did this; reviewer can skip
CorrectnessDoes this code do what its description says?
Edge casesEmpty input, null, concurrent calls, errors?
SecuritySQL injection, XSS, auth bypass?
Architecture fitDoes this belong here? Is there a cleaner shape?
Naming & readabilityWill the next engineer understand?
Style nitsDefer to the linter; manual nits = noise

The reviewer's value is in the middle four rows. CI handles the top; style is automated; the human is for architecture, correctness, edge cases. Don't waste review time on whitespace.

03Culture & tone.

"Why?" beats "no"

"This is wrong" stops the conversation. "What's the case for this approach over X?" opens it. The author often has context the reviewer doesn't.

Praise specific things

"Nice use of the visitor here" is worth a hundred "lgtm" comments. Reviewing is teaching; praise reinforces what to do more of.

Distinguish blocking from suggestion

"nit:" or "suggestion:" prefix for "I prefer this but you can ignore." "blocking:" or no prefix for "this must change." Conventions documented; reviewers stick to them.

Time-bound the review

Target: PRs reviewed within 4 hours of request during business hours. Above that, the author has switched context; the cost of resuming compounds. Set the team's SLA.

Async vs sync

Async (GitHub comments) is default. Sync review (over a call) for genuinely complex changes; saves a 30-comment thread.

04Tooling that helps.

Auto-formatters (Prettier, Black, gofmt): eliminate style review entirely. Linters with auto-fix: ditto for common bug patterns. Type checkers: catch a class of bugs before review. Pre-commit hooks: don't even open a PR with violations.

Coverage / mutation testing: tells you whether the new code is tested, beyond pass/fail. Static analysis (CodeQL, Semgrep): catches security smells. AI review (CodeRabbit, Greptile, Sourcery): a useful "first reviewer" that catches obvious issues so humans focus on architecture.

05Decision guide.

PRs sit unreviewed for >1 day

Set a team SLA. 4 hours for first response in business hours. Make it visible (dashboard, Slack reminders).

PRs are routinely huge

Adopt stacked diffs. Train the team to chop. 400-line ceiling for "non-mechanical" diffs.

Style-bikeshedding in reviews

Auto-formatter wins. Move the argument to the linter config; settle it once.

Senior engineers are review bottlenecks

Spread review load. Junior + senior pair on harder PRs; junior on routine. AI tools as first-pass screen.

Cross-team / cross-org PRs

CODEOWNERS file. Each file path has owners; PR can't merge without their approval. Github / GitLab built-in.

Repeated bug class in reviews

Promote to a lint rule. If you've caught the same mistake five times in PRs, automate the catch.

"Code review is the slowest part of engineering. Treat it like the production system it is."
