The version is the API.

"How do we evolve an API without breaking clients" is the question. Four strategies, each with passionate defenders. The honest answer: it depends on who your callers are and how much you can ask of them.

The four strategies

URL path (/v1/), URL host (v1.api.example.com), media-type header (Accept: application/vnd.example.v1+json), date-based version header (Stripe's choice).

01Why versioning is hard.

Once you publish an API, clients code against it. A breaking change forces every client to update. For internal APIs you control both ends; for public ones, you have hundreds or thousands of integrators. Some will never update; their app is in someone's basement; the company is gone. You still have to serve them or they 500.

Versioning lets you ship breaking changes without killing those clients — they keep hitting the old version while you ship a new one alongside.

"The API contract is the public face of the company. Breaking it without a path forward burns trust no marketing can rebuild."

02The four strategies.

URL path versioning

/v1/charges, /v2/charges. Easiest to understand; obvious in logs; easy to route. The downside: it confuses REST purists (URI identifies a resource; the resource is the same in v1 and v2). Most public APIs use this. GitHub, Twilio, Stripe (for major versions).

URL host versioning

v1.api.example.com. Lets you route entire versions to different infrastructure. Useful if v1 and v2 run on different stacks. Rare in practice; same shape as path versioning, more setup.

Media-type / Accept header

Accept: application/vnd.example.v1+json. The REST-pure way. Same URL; client requests the version. Hard to curl; works fine for SDKs. GitHub's "preview" headers use this for opt-in features.

Date-based versioning (Stripe-style)

Stripe-Version: 2024-11-20 header. Every breaking change gets a date. Clients pin a version once and stay on it until they're ready to upgrade. The API documents a changelog of "what changed in this version." New version is the default for new accounts. This is the most disciplined; also the most operational — the server has to maintain compat for every published date.

03What counts as breaking.

ChangeBreaking?
Add a new optional field to responseNo (clients should ignore unknown fields)
Add a new endpointNo
Remove a fieldYes
Rename a fieldYes
Change field type (string → int)Yes
Change a default valueYes (clients may rely on it)
Add a required parameterYes
Change pagination shapeYes
Change error code for an existing conditionYes
Tighten validationYes (formerly-valid requests now fail)

"Tolerant readers" don't save you for the tightening cases. Servers must enforce; clients break when enforcement tightens. Treat any visible-behaviour-change as breaking; if in doubt, version it.

04Sunset policy.

Announce deprecation publicly

Blog post, email to integrators, docs banner. Lead time: typically 6–12 months for public APIs.

Add Sunset and Deprecation headers

RFC 8594 (Sunset), draft (Deprecation). Tooling can detect and warn integrators.

Soft-disable before hard-disable

2 weeks before sunset, return success but inject a noisy log header. Forces noisy attention.

After sunset

Return 410 Gone with a body explaining what to upgrade to. Not 404 — 410 is the "permanently retired" signal.

05Decision guide.

Public API, broad integrator base

URL path versioning. Major versions only. Bump rarely. Date-based as an inner mechanism for finer control.

SDK-only API (no curl)

Date-based header versioning. The SDK pins on behalf of the user; users don't see the header.

Internal microservices

Don't version. Coordinate change across the org. If you can't, you have an organisational problem; versioning hides it.

Mobile-app-backing API

Mobile clients persist for years. Use sticky versioning; never break old versions until you can confirm zero traffic.

You shipped no versioning v1

Your v1 IS v1. Add /v2/ when you need the first breaking change; route everything else to v1. Don't try to retrofit versioning to v1 — you can't.

GraphQL

Don't version. Deprecate fields with @deprecated, add new ones in parallel. Schema is forever-extending.

"You don't pick a versioning strategy. You pick the support burden you're willing to live with for ten years."
