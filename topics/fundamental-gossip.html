Information spreads like a cold.

Gossip protocols spread information through a cluster by random pairwise exchange. No central coordinator. Information reaches all nodes in O(log N) rounds. Cassandra, Consul, Serf, Akka Cluster all use a variant. The math is elegant; the operational story is "it just works."

What gets gossiped

Membership (who's in the cluster), failure detection (who's down), state (versioned data per node).

Why it scales

Each node talks to O(1) peers per round. Information reaches N nodes in O(log N) rounds. Total messages: O(N log N).

01The membership problem.

A cluster of N nodes. A new node joins. An old node dies. How does everyone learn? Three options. (1) Central registry: every node tells one master, master tells everyone. Single point of failure; scaling bottleneck. (2) Broadcast: every node tells everyone. O(N²) messages per change; doesn't scale. (3) Gossip: every node tells a random peer; information spreads epidemically.

Gossip is the winner. No central coordinator; messages per round are O(1) per node; convergence is O(log N) rounds. The trade is eventual consistency on cluster state — different nodes briefly disagree about who's in the cluster. Almost always acceptable.

"Gossip is the protocol that says: nobody knows, but everybody asks somebody, and the truth wins."

02How it works.

The round

Every second (or so), each node picks a random peer and exchanges state. Push-pull: A sends its state to B; B merges; B sends its state back to A; A merges. Both ends are now consistent on each other's state.

State is per-key versioned. "My IP is 10.0.0.4, version 17". On merge: keep the higher version. Conflicts (two nodes claiming the same key) are resolved by timestamp or by a chosen tie-breaker.

Convergence math

If a single piece of info enters the cluster at one node, after each round the number of nodes that know it roughly doubles (the original tells one new; both tell one new each; etc.). After log₂(N) rounds, everyone knows. For N=1000, that's 10 rounds; at 1 second each, 10 seconds to converge.

03SWIM — the failure-detection trick.

SWIM (Scalable Weakly-consistent Infection-style Process Group Membership, 2002) is the most-cited gossip variant. The trick: don't ping every peer every round. Pick one peer; if it doesn't respond, ask a few other peers to ping it indirectly. One node's network blip doesn't cause it to be falsely marked dead — the indirect probe asks "can you reach it?" If three random peers all say "I can't reach it either," the consensus is that the node is truly down.

Phi accrual failure detector

Cassandra's variant. Instead of "alive or dead," it outputs a probability that the node is dead. Application sets a threshold; if confidence exceeds it, mark dead. Gives a smoother failure-detection curve under unreliable networks.

05Decision guide.

Cluster < 50 nodes

Gossip is overkill. A central registry (etcd, Consul KV) is simpler.

Cluster 50–10000 nodes, no central

Gossip is the right answer. Pick a library (memberlist for Go, Akka Cluster for JVM, custom small implementation if Rust/.NET).

Critical state (config, leader)

Don't gossip — use Raft / Paxos. Gossip is eventually-consistent; critical state needs linearisability.

Failure detection only

SWIM-style. Pings + indirect probes. Modern libraries (memberlist) are 1-line setup.

You want to build one

Read the SWIM paper. Build push-pull first; add indirect probing; add anti-entropy. Production-ready in ~2 weeks.

Auditing gossip in production

Log all state transitions. False-positive failure detection is the recurring bug; the logs are how you tune the thresholds.
