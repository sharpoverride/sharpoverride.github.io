The Django monolith that scaled.

Instagram went from launch (2010) to a billion users on a Python monolith. Three engineers built the original. The story has become folklore — and a case study in how far "boring tech" can go.

The original stack

Django, Postgres, Redis, Memcached. EC2. Nothing exotic.

What scaled where

Postgres sharded by user-id. Cassandra added for inbox/feed. Memcached for everything. Django ran as gunicorn workers across many instances.

The lesson

Most engineering teams reach for new tech long before they exhaust their current stack. Instagram exhausted Django and Postgres before adding Cassandra.

01From 0 to a billion.

Instagram launched on EC2 in 2010. Three engineers. Django. Within three months they had 1M users. Within two years, 30M. Within ten years, a billion. The "obvious" answers — rewrite in Java, switch to NoSQL — were rejected. The team optimised the Django+Postgres stack until it stopped paying.

The architectural moves were boring: add a cache layer, add a read replica, shard the database, add a queue. Each move bought 1–2 years of runway; the next was scheduled in advance.

"Boring tech scales further than tech blogs suggest. Most teams give up too early."

02Postgres sharding the Instagram way.

Instagram's published sharding scheme is the most-cited in the field. The trick: a logical shard ID embedded in every primary key, so any row's shard is computable from the ID alone.

The 64-bit ID

BitsWhat
41 bitsTimestamp (ms since custom epoch)
13 bitsLogical shard ID
10 bitsAuto-increment within shard

13 bits = 8192 logical shards. The number of physical Postgres machines is <8192; each machine hosts many logical shards. Re-sharding = moving logical shards to different physical machines; the IDs in the application don't change.

Sharded by user

The shard for a row is determined by the user-id (the row's owner). All of one user's data lives on one shard. Single-user queries are local; no cross-shard joins needed for "show me my profile."

03Feed and pull-on-read.

The home-feed problem (also see interview framework): show me posts from the people I follow. Fan-out-on-write (push to followers' inboxes at post time) doesn't scale for celebrity accounts. Instagram uses hybrid: regular users fan out at post; celebrities are pulled at read.

Memcached is the feed cache. Hit rate is the engineering metric; cache miss = expensive recomputation. Memcache cluster of thousands of nodes, sharded by key.

05The lessons.

Embed the shard ID in the primary key

Every row's shard is computable from its ID. Re-sharding doesn't break the application. This is the technique most people don't know about and shouldn't ship a sharded app without.

Memcached scales further than Redis for raw cache

Memcached's design is simpler. Instagram uses it for billions of get/set per second. Redis is fine; just don't assume it's mandatory because it's fancier.

Don't rewrite Python because it's "slow"

The bottleneck is almost always I/O — database, cache, network. Python being 5× slower than Java on CPU rarely shows up when each request is 95% waiting. Instagram on Django to 2B users is the proof.

Hybrid fan-out for read/write skew

Pure push doesn't work with celebrities. Pure pull doesn't work for active users. Hybrid: push for normal users, pull for the long-tail celebrities. Same shape Twitter uses.

"Instagram's scale story is not about new technology. It's about not throwing away old technology."
