The folder that syncs.

Dropbox shipped a magical product: a folder that "just stays in sync" between devices. Under the hood: chunked block storage, content-addressable hashing, delta sync, conflict resolution. And in 2016, a famously bold move — repatriating from AWS to their own storage system, Magic Pocket.

The sync protocol

File → split into 4 MB blocks → hash each block → upload only blocks the server doesn't have → metadata DB records "file = these blocks."

Magic Pocket

Dropbox's own object storage, replacing S3 for ~90% of their data. Erasure coding, cross-region replication, ~$75M/year savings.

01The sync protocol.

The desktop client watches the local Dropbox folder. On file change: split file into 4 MB blocks, hash each block, send block hashes to server. Server replies "I have these; I don't have these"; client uploads only the missing blocks. Metadata DB records "file at path X = blocks [a, b, c]."

On the receive side: client polls (or holds a long-poll connection) for changes. When a file change lands, server tells client "this file's block list changed to [a, b, d]." Client downloads any block it doesn't have locally; assembles the file.

"The first 100k users were powered by one Python script and rsync."

02Block-level dedup.

Content-addressable: a block's identity is its SHA-256. Two users upload the same file; the server stores one copy. Edit a 1 GB file's first byte; only the first 4 MB block changes; the rest dedup. The bandwidth savings are enormous, especially at corporate scale where shared documents proliferate.

Rolling hashes for partial changes

Inserting bytes in the middle of a file shifts everything; naive 4 MB chunking would invalidate every subsequent block. Content-defined chunking (rolling hash, Rabin fingerprint) finds chunk boundaries at content-dependent positions; an insert only invalidates the chunk it landed in.

03Magic Pocket.

Until 2015, Dropbox stored its blocks on S3. ~$1B annual AWS bill. Magic Pocket: their own object storage, custom-built on commodity hardware in Dropbox-leased datacenters. Migration took 2.5 years. ~90% of stored bytes now on Magic Pocket.

Architecture

SMR (shingled magnetic recording) disks for cold storage. Erasure-coded across many disks (Reed-Solomon style; 10+4 typical — survive 4 disk failures). Cross-region replication. Custom storage software, custom hardware ordering, custom-everything.

When does building your own storage make sense?

For Dropbox: ~exabyte of storage, predictable access patterns, billion-dollar AWS bill. The math worked. For everyone else: don't. The fixed cost is ~$1B in engineering + facilities; the per-GB savings only matter at exabyte scale. Magic Pocket is a case study in what's possible, not a recommendation.

05The lessons.

Block-level dedup is the architecture

The bandwidth savings, the storage savings, the resume-from-interruption — all flow from content-addressable blocks. Most modern file-sync systems (iCloud, Drive, etc.) work this way.

Content-defined chunking is the upgrade

Fixed boundaries are simple; content-defined boundaries are correct under inserts. The complexity is in the rolling hash; the wins are real.

Cloud → on-prem can be a win at extreme scale

Magic Pocket was the right move at $1B/year. The break-even is in the hundreds of millions; below that, the cloud is cheaper. Don't build your own storage to save money — build it because you've outgrown the elasticity of the cloud.

Sync is harder than it looks

Conflict resolution (two devices edit the same file offline). Permissions. Sharing. Trash. Versions. Every one of these is its own subsystem; the magic is the engineering work across them.

"Dropbox's product is a folder. Their architecture is everything that makes the folder feel like one place."
