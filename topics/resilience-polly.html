Chapter 11 Difficulty ★★★☆☆ Read ~13 min Stack Polly v8 · ASP.NET Core · HttpClient factory

Every downstream call will fail.

Networks partition. Disks fill. Pods restart. The interesting question is not whether the call will fail — it's how your service behaves while it is failing. Polly v8 gives you four strategies, composed into one pipeline, applied everywhere.

The problem in one line

A slow dependency takes your service down faster than a dead one. The threads pile up first; the page comes second.

The thesis

Four primitives — timeout, retry with jitter, circuit breaker, bulkhead — composed into a single pipeline per dependency. Configured by the boundary, applied by the framework. No reaching for try/catch in business code.

01The cascade.

Service A calls service B. B gets slow — not failed, just slow. A's threads pile up waiting. The thread pool grows. New requests to A start queuing. A's p99 latency climbs. A's clients (frontends, mobiles) retry. The retry storm hits A harder. A becomes unhealthy and its own callers start retrying it. Five minutes later, half the system is down because one downstream got slow.

This is the cascading-failure pattern. It killed Netflix before they wrote Hystrix; it killed Twitter regularly until they tightened deadlines; it killed every Friday-night cloud incident you remember by name. The cause is almost never the original fault — it's the absence of a wall between the fault and the rest of the system. A timeout is that wall. A circuit breaker is the wall that keeps remembering. A bulkhead is the wall that keeps one pool of threads from drowning the rest.

"You don't ship resilience to handle the failures you imagined. You ship it to survive the failures you didn't."

02Four primitives, one pipeline.

Polly v8 refactored an idea from Hystrix and Microsoft's own HttpClientFactory story: a resilience pipeline is just a chain of strategies wrapping a delegate. Each strategy decides whether to invoke the next one, and what to do with the result.

even start hitting the dependency. Circuit breaker next: short-circuits everything when the dependency is observably broken. Retry third: tries again on transient faults — counts as one trip from the circuit's view. Timeout innermost: bounds one attempt, so a single hung call doesn't keep the slot. Get the order wrong and the strategies fight each other.
Fig. 09The pipeline is read outer-to-inner. Each strategy makes a local decision about the inner strategy's outcome.

Order is the design decision

The single most common Polly bug is wrong-order strategies. Put the timeout outside the retry and the deadline applies to the entire retry chain — three attempts collapse to one, the retry never gets to fire. Put the circuit breaker inside the retry and the retry will trip the breaker on its own retries, opening it on every transient blip. Outer-to-inner: bulkhead → circuit breaker → retry → timeout → execute. Polly v8's builder API enforces this naturally; the v7 fluent Wrap() made it backwards by default. If you're reading old code that wraps in the other direction, expect bugs.

03Retries, jitter, half-open.

The naive retry strategy — "wait 1s, then 2s, then 4s" — is what causes the retry storm. A thousand pods all retry at the same moment, hammer the recovering downstream, push it back over the edge. Synchronised retries are worse than no retries.

Decorrelated jitter (AWS Architecture Blog, 2015) gives the recovering downstream the headroom it needs to actually recover.

Don't retry the unretryable

Retry only on transient faults: 5xx, network timeouts, 408, 429. Never retry 4xx other than 408 or 429 — the response will be the same the second time. Never retry POST without an Idempotency-Key (see Chapter 08); you'll charge twice. The classification logic is the most under-loved part of every retry policy; Polly v8 makes it explicit via the ShouldHandle predicate, and you should write that predicate by hand for every dependency.

The circuit breaker's three states

Closed: traffic flows; failures counted. Open: traffic rejected immediately with a synthetic exception; downstream is allowed to recover. Half-open: after a cooldown, one probing request is allowed through; if it succeeds, close the circuit; if it fails, back to open. The half-open probe is the part most homegrown circuit breakers get wrong — they either let the floodgates open all at once (and re-trip immediately) or they never reset because nothing tests the path. Polly's state machine does this for you.

Bulkhead is concurrency, not rate

A bulkhead caps concurrent in-flight calls to a dependency — typically at "saturation of the downstream divided by your pod count, plus a little headroom." If 50 is the maximum, the 51st caller gets BulkheadRejectedException immediately and the thread is freed. This is what stops a slow downstream from holding all your threads. A rate limiter, by contrast, caps per-second calls — different problem. Use rate limiting for protecting the downstream from your fleet's burst; use bulkheads for protecting your fleet from the downstream's slowness.

04In Polly v8.

A typed HttpClient with the full pipeline. Wired into the DI container, applied automatically to every call made through this client. No try/catch in business code.

C#   ResilienceSetup.cs 
// dotnet add package Microsoft.Extensions.Http.Resilience
using Microsoft.Extensions.DependencyInjection;
using Polly;
using Polly.CircuitBreaker;
using Polly.Retry;
using Polly.Timeout;

namespace Academy.Resilience;

public sealed class InventoryClient
{
    private readonly HttpClient _http;
    public InventoryClient(HttpClient http) => _http = http;

    public async Task<StockLevel> GetAsync(string sku, CancellationToken ct)
    {
        var resp = await _http.GetAsync($"/stock/{sku}", ct);
        resp.EnsureSuccessStatusCode();
        return (await resp.Content.ReadFromJsonAsync<StockLevel>(ct))!;
    }
}

public static class ResilienceRegistration
{
    public static IServiceCollection AddInventoryClient(
        this IServiceCollection services, string baseUrl)
    {
        services.AddHttpClient<InventoryClient>(c =>
        {
            c.BaseAddress = new Uri(baseUrl);
            // "Total deadline" for the whole call. Lives in HttpClient.Timeout — Polly per-attempt is shorter.
            c.Timeout = TimeSpan.FromSeconds(10);
        })
        .AddResilienceHandler("inventory", builder =>
        {
            // 1. Bulkhead — outermost. Cap concurrent in-flight calls per pod.
            builder.AddConcurrencyLimiter(permitLimit: 50, queueLimit: 0);

            // 2. Circuit breaker — trip after 50% failures over a 30s window of 100 calls.
            builder.AddCircuitBreaker(new CircuitBreakerStrategyOptions<HttpResponseMessage>
            {
                FailureRatio      = 0.5,
                MinimumThroughput = 20,
                SamplingDuration  = TimeSpan.FromSeconds(30),
                BreakDuration     = TimeSpan.FromSeconds(15),
                ShouldHandle = PredicateBuilder
                    .HttpStatusCodes(HttpStatusCode.InternalServerError,
                                      HttpStatusCode.BadGateway,
                                      HttpStatusCode.ServiceUnavailable,
                                      HttpStatusCode.GatewayTimeout)
                    .Handle<HttpRequestException>()
                    .Handle<TimeoutRejectedException>()
            });

            // 3. Retry — only transient. Decorrelated jitter backoff.
            builder.AddRetry(new RetryStrategyOptions<HttpResponseMessage>
            {
                MaxRetryAttempts = 3,
                BackoffType      = DelayBackoffType.Exponential,
                UseJitter        = true,           // decorrelated jitter built in
                Delay            = TimeSpan.FromMilliseconds(200),
                ShouldHandle = PredicateBuilder
                    .HttpStatusCodes(HttpStatusCode.RequestTimeout,
                                      HttpStatusCode.TooManyRequests,
                                      HttpStatusCode.InternalServerError,
                                      HttpStatusCode.BadGateway,
                                      HttpStatusCode.ServiceUnavailable,
                                      HttpStatusCode.GatewayTimeout)
                    .Handle<HttpRequestException>()
                    .Handle<TimeoutRejectedException>(),
                OnRetry = args =>
                {
                    Activity.Current?.AddEvent(new("polly.retry",
                        tags: new() { ["attempt"] = args.AttemptNumber }));
                    return ValueTask.CompletedTask;
                }
            });

            // 4. Per-attempt timeout — innermost. Each HTTP call gets 2 seconds.
            builder.AddTimeout(TimeSpan.FromSeconds(2));
        });

        return services;
    }
}

Register once in Program.cs and the handler is applied to every call through InventoryClient. No per-call wrapping needed — the pipeline lives in the message handler chain.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddInventoryClient(builder.Configuration["Inventory:BaseUrl"]!);

builder.Services.AddOpenTelemetry()
    .WithTracing(t => t.AddSource("Polly").AddAspNetCoreInstrumentation().AddOtlpExporter())
    .WithMetrics(m => m.AddMeter("Polly").AddOtlpExporter());

var app = builder.Build();

app.MapGet("/products/{sku}", async (string sku, InventoryClient inv, CancellationToken ct) =>
{
    try
    {
        var stock = await inv.GetAsync(sku, ct);
        return Results.Ok(stock);
    }
    catch (BrokenCircuitException)
    {
        // Graceful degradation: return cached "last known" or a friendly fallback.
        return Results.Ok(new { sku, available = (bool?)null, source = "unavailable" });
    }
});

app.Run();

05The detail.

The total deadline lives in HttpClient.Timeout

Polly's per-attempt timeout bounds one HTTP call. The total deadline for the operation — including all retries and the bulkhead queue — is HttpClient.Timeout. Set the outer larger than the inner × max attempts × max delay, or you'll see TaskCanceledException on the second retry of every call and wonder why your retry policy never gets to fire.

Circuit breakers belong per-dependency, not per-service

One breaker per downstream. Sharing a breaker across services means a slow inventory call trips the breaker for payment too. Polly's resilience handler is per HttpClient, which lines up naturally with per-dependency typed clients. Resist the urge to extract a single "global resilience policy" — the whole point is that each dependency fails differently.

Observability is half the value

Polly v8 emits Activity events and a Polly meter — wire them to OpenTelemetry and you'll see retry counts, circuit state transitions, and bulkhead rejections on the dashboard. Without telemetry, the breaker is a black box and the "intermittent failure" ticket will keep coming back. If you can't see when the circuit opened, you can't tune the thresholds.

Fallback is a separate concern

Polly v7 had a FallbackPolicy; Polly v8 deliberately removed it. Fallback is a business decision — return cached data, return a default, return 503 — and lives in your endpoint, not in the pipeline. The pipeline's job is to fail fast and clearly (BrokenCircuitException, BulkheadRejectedException). The endpoint translates that into a customer-friendly response.

Don't double-up with the service mesh

If you're on Istio or Linkerd with retry policies in the mesh, do not add retries in Polly — you'll get retries × retries = retry-of-retries and a 9× amplification on every blip. Pick one layer. The mesh wins for cross-language uniformity; Polly wins for per-dependency precision. Mixed, they're a footgun.

"A timeout you never set is a timeout the platform sets for you. The platform's choice is rarely better than yours."

07Decision guide.

.NET service calling a flaky HTTP dependency

Polly v8 typed-client with the four-strategy pipeline. Per-attempt timeout 2s, retry 3× with jitter, breaker 50% over 30s, bulkhead 50.

Pure idiomatic, willing to accept defaults

Microsoft.Extensions.Http.Resilience's AddStandardResilienceHandler() — one line, opinionated, fine for 80% of dependencies.

Polyglot fleet (C# + Go + Python)

Move retries into the mesh (Linkerd or Istio). Uniform policy across languages beats per-service drift. Remove Polly retries to avoid amplification.

Calls to your own database (EF Core)

EF Core has EnableRetryOnFailure() for transient SQL faults. Use it; don't wrap EF in Polly — the contexts don't survive disposal across retries.

Background worker → external API

Polly pipeline + longer per-attempt timeouts (10–30s). Retry budget should be lower than for online traffic; failure is not user-facing.

POST without idempotency key

Don't retry. Or, more precisely, generate an idempotency key and then retry. Polly will gladly charge a customer twice; the safe path is upstream of the retry policy.

Three things people get wrong

One — wrong strategy order. Timeout outside retry collapses the retry to one attempt. Retry outside breaker re-trips the breaker on its own backoff. Outer-to-inner: bulkhead, breaker, retry, timeout. Memorise it.

Two — retrying on 4xx. 400, 401, 403, 404, 422 will return the same thing the second time. Only 408 and 429 are retryable client errors. Spell out ShouldHandle explicitly for every dependency.

Three — double retries (Polly + mesh). Three Polly attempts × three mesh attempts = nine actual calls per logical request. A 1% transient failure rate looks like 9% to the downstream. Pick a layer.
