The USB-C of LLM context.

Model Context Protocol (MCP) is Anthropic's 2024 specification for how AI applications connect to data sources and tools. One protocol; many clients; many servers. A growing ecosystem of MCP servers gives any MCP-aware LLM client immediate access to a wide variety of capabilities — without each integration being bespoke.

The problem

Every AI app reinvents how it talks to GitHub, Slack, your filesystem, your database. The integration code is N × M (N apps × M data sources). MCP makes it N + M.

The shape

Client (the AI app, e.g. Claude Desktop, Cursor, ChatGPT clients) speaks MCP to one or more servers. Each server exposes tools, resources, and prompts. JSON-RPC over stdio or HTTP/SSE.

01The N × M problem.

Every AI client wants to talk to your tools. Claude Desktop wants GitHub access; so do Cursor, Continue, Cline, ChatGPT's desktop app, your in-house chatbot. Each app currently writes its own GitHub integration. Now imagine the matrix: N AI clients × M tools = N×M integrations.

MCP is the standard. Build one MCP server for GitHub; every MCP-aware client can use it. Build one Slack server; same. Build one for your internal CRM; ditto. The integration cost goes from N×M to N+M. This is the same play LSP made for IDEs (one language server, many editors); the same play USB-C made for cables.

"MCP is what tool calling looks like when you stop writing it inside each AI app."

02Client, server, primitives.

Fig. AOne AI client speaks MCP to many servers concurrently. Each server is a separate process with its own permissions and tool set.

Three primitives

Tools: callable functions. The LLM can invoke them. Same shape as the agent tool-calling pattern from the agents chapter — name, description, JSON-schema parameters. Resources: readable data with stable URIs. A file on disk, a row in a database, a Confluence page. The client can present these to the user or to the LLM as context. Prompts: reusable prompt templates the server publishes. The client surfaces them as commands ("@Repo summarise this PR").

Server scope and trust

Each MCP server is a separate process. Permissions are scoped per server: the GitHub server can do GitHub things; the filesystem server can read files; neither can do the other's job. Compromise of one server doesn't compromise the others. The client decides which servers to load, often per-project or per-conversation.

03The wire protocol.

JSON-RPC 2.0 over stdio or SSE

Messages are JSON-RPC requests and responses. Two transports: stdio (the server is a local subprocess; the client writes JSON-RPC to its stdin, reads responses from stdout) — the default for local servers. SSE / HTTP (the server is a remote service over HTTP, the client uses Server-Sent Events) — for cross-network servers.

The handshake

Client sends initialize; server responds with its capabilities (which tools/resources/prompts it provides). Client then sends tools/list, resources/list, prompts/list as needed. Tool calls are tools/call with the tool name and arguments; the server invokes the tool, returns the result.

json   a tool/call wire frame 
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": {
    "name": "github_create_issue",
    "arguments": {
      "repo": "acme/website",
      "title": "Banner not rendering on Safari",
      "body": "Steps to reproduce..."
    }
  }
}

04Building an MCP server.

The official Anthropic SDKs are Python and TypeScript; community SDKs exist in .NET, Go, Rust. A minimal MCP server in C# (using the community ModelContextProtocol.NET package):

C#   MyMcpServer.cs 
// dotnet add package ModelContextProtocol
using ModelContextProtocol.Server;
using System.ComponentModel;

[McpServerToolType]
public static class GitHubTools
{
    [McpServerTool, Description("Search for issues in a GitHub repo by query string.")]
    public static async Task<SearchResult> SearchIssues(
        [Description("owner/repo, e.g. acme/website")] string repo,
        [Description("Search query, GitHub search syntax")] string query,
        [Description("Max results, default 10")] int limit = 10)
    {
        var client = new GitHubClient(Environment.GetEnvironmentVariable("GITHUB_TOKEN")!);
        return await client.SearchIssuesAsync(repo, query, limit);
    }

    [McpServerTool, Description("Open a new GitHub issue.")]
    public static async Task<string> CreateIssue(
        [Description("owner/repo")] string repo,
        [Description("Issue title")] string title,
        [Description("Issue body in markdown")] string body)
    {
        var client = new GitHubClient(Environment.GetEnvironmentVariable("GITHUB_TOKEN")!);
        var url = await client.CreateIssueAsync(repo, title, body);
        return $"Created: {url}";
    }
}

// Program.cs — the server is a stdio-loop. The host harness handles JSON-RPC.
var builder = Host.CreateApplicationBuilder(args);
builder.Services
    .AddMcpServer()
    .WithStdioServerTransport()
    .WithToolsFromAssembly();
await builder.Build().RunAsync();

Configuring a client to use it

Claude Desktop, Cursor, etc. take a JSON config that lists MCP servers — the command to launch and environment variables. Once configured, the LLM in the client can see and call the server's tools.

json   claude_desktop_config.json 
{
  "mcpServers": {
    "github": {
      "command": "dotnet",
      "args": ["run", "--project", "C:/repos/mcp-github-server"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://localhost/mydb"]
    }
  }
}

05The detail.

stdio servers run with your privileges

A local MCP server inherits your user's file-system access, env vars, and credentials. Treat MCP server source like any code you run; an MCP server can be malicious. Only install from sources you trust.

Tool descriptions are prompts

The model picks tools based on their descriptions. Spend time on the descriptions. "Searches for files" is bad; "Lists files in a directory, supports glob patterns, returns up to 100 results sorted by mtime" is good. The model's tool-use quality depends on this directly.

Resources let you expose data without tools

If you want the LLM to read your knowledge base, a resource with stable URIs is better than a "search_kb" tool. The client lists resources, the user can pin one, the LLM gets the content as context. Resources are first-class; many clients let the user pin them as @-mentions.

SSE servers for cross-host

For servers that aren't on the user's machine — a cloud service, a remote DB — the SSE transport runs over HTTPS. Add auth at the HTTP layer; MCP itself doesn't define authentication beyond what the transport provides.

The ecosystem is exploding

By mid-2025, hundreds of MCP servers existed — GitHub, GitLab, Slack, Notion, Jira, Postgres, SQLite, Google Drive, Sentry, Datadog. Before you write one, search the registry. The official server catalog is at github.com/modelcontextprotocol/servers.

"MCP isn't a model. It's the pluggable connector for everything around the model."

06Decision guide.

Building an AI client / desktop app

Adopt MCP. You get the entire ecosystem of servers for free. Don't roll your own tool plumbing.

Exposing your SaaS to AI agents

Ship an MCP server. Any MCP-aware client (Claude Desktop, Cursor, ChatGPT clients) becomes a potential UI for your product.

Internal tools for your team

Write a local MCP server. Co-workers install it in Claude Desktop; the LLM can query your CRM, internal docs, build system.

Server-side agent platform

MCP works inside server agents too — Anthropic's SDK has client support. The agent can connect to MCP servers and use their tools.

Existing OpenAPI / gRPC service

Don't replace; wrap. A thin MCP server around your existing API exposes the right subset to LLMs with descriptions tuned for them.

Quick one-off tool

Skip MCP, use the LLM provider's native function-calling. MCP is for things you'll share across multiple AI apps.
