The Swiss army knife of infrastructure.

Redis is the most-used non-database in the field. Cache. Queue. Rate limiter. Leaderboard. Session store. Pub/sub. Lock. Stream. Counter. The data structures are the API; you compose them into whatever you need.

The data types

STRING, HASH, LIST, SET, SORTED SET, STREAM, BITMAP, HYPERLOGLOG, GEO. Each maps to a class of use case.

The hidden feature

Lua scripts. Atomic multi-step operations. Most production Redis use cases that aren't pure cache use Lua somewhere.

01Cache.

The original use case. SET key value EX ttl. Sub-millisecond reads, 100k QPS per node. Cache-aside is the most common pattern: app checks Redis first, falls back to DB on miss, writes back on the way out. See caching patterns.

02Rate limiter.

Token bucket or fixed/sliding window. Lua script enforces atomicity: read count, check limit, increment, return decision — all in one server-side operation. The whole evaluation is sub-ms; the answer is correct under concurrency.

See rate limiting chapter and Stripe's four limiters for the production design.

03Leaderboard.

Sorted set: ZADD leaderboard 1234 player_id. ZREVRANGE leaderboard 0 9 WITHSCORES returns top 10 in milliseconds, even with millions of entries. ZRANK leaderboard player_id gives a specific player's position. The "1B player ranking" interview question is one ZADD/ZRANK away from a working solution.

04Queue / stream.

List as queue

LPUSH / BRPOP: producers push to the head, consumers block-pop from the tail. Simple, FIFO, fast. Not durable beyond Redis's persistence settings; not great if you need replay or multiple consumer groups.

Streams

Redis Streams (5.0+) are a proper append-only log with consumer groups. Replaces simple List-based queues for any non-trivial messaging. Closer to Kafka in semantics; differences in throughput and durability. See message queues chapter.

05Sessions & locks.

Session store

HSET session:<id> user_id 42 logged_in 1 last_seen 1715000000. TTL'd. Stateless app tier; session in Redis — the same pattern every framework uses. See JWT vs sessions.

Distributed lock

SET lock:<name> <owner-id> NX EX 30. Acquire. Work. Release (carefully — verify owner before deleting, via Lua). Single-Redis lock is fine for most use cases. Redlock for multi-Redis (controversial; see leader election).

07Decision guide.

Just need a cache

Memcached or Redis. Redis wins on richer data types; Memcached wins on simplicity and raw throughput.

Need richer ops than cache

Redis. Sorted sets, streams, Lua scripts. The reason Redis won the field.

Highly available

Redis Sentinel for failover; Redis Cluster for horizontal scale + HA. Managed (Azure, AWS Elasticache) is the boring correct option.

Durable queue / event log

Streams, not Lists. Or just use Kafka for serious workloads. See Kafka chapter.

Geographic distribution

Redis active-active (Redis Enterprise) or per-region clusters. Cross-region replication has latency tradeoffs.

"Redis as primary database"

Rare but real (RedisJSON, durability tuning). For most teams, Postgres + Redis is a better split than Redis-only.

"Redis isn't a cache; it's a toolkit. Treat it as one."
