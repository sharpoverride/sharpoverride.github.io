Stateful in Redis, or signed in the cookie.

The choice between server-side sessions and JWTs is not which is more modern — it's which failure mode you can live with. Sessions die when Redis dies. JWTs are valid until they expire, even if the user is fired this morning. The honest tradeoff settles in the middle.

The problem in one line

HTTP is stateless. The server has to recognise the same user across requests without trusting whatever the client claims to be.

The thesis

Short-lived access JWT + opaque refresh token + revocation list is the default for SaaS/B2C. Server-side sessions in Redis remain the default for traditional web apps. Pure JWT-without-revocation is wrong for anything user-facing.

01Stateful HTTP from stateless HTTP.

HTTP delivers one request, one response, no memory. Every modern web app needs to know who's calling without re-authenticating on every click. The mechanism: after login, hand the client an opaque token; on every subsequent request, the client returns the token; the server resolves it to a user. Two shapes of token: a random string that points at server state (a session) or a signed payload that contains the state (a JWT).

That decision — point or contain — is the whole architectural question. Sessions push the cost to the server (a Redis lookup per request) and keep the source of truth there. JWTs push the cost to the client (the token is bigger) and put the source of truth on the wire. Every other tradeoff is downstream of which one you picked.

"A JWT is a session that doesn't know how to die."

02Session vs JWT.

Fig. 22The session asks the server every time. The JWT asks the cryptography. Each answers fast; only one knows about the user being fired this morning.

When sessions win

Traditional web apps, single backend, B2B internal tooling, anything where "log out everywhere now" is a feature you'll use. Cookie auth in ASP.NET Core is one line of AddAuthentication().AddCookie(); the framework handles ticket protection, expiration, sliding renewal. Redis or SQL holds the ticket data; the cookie is just an opaque ID. Revocation is a DELETE away. Most "we should JWT" decisions for monolithic apps are over-engineering.

When JWTs win

Multiple backends that need to authenticate independently — microservices, mobile + web + third-party, API consumers. The JWT is verified with public-key crypto; any service can verify without contacting a session store. The signing authority publishes a JWKS endpoint; each service fetches and caches the public keys. Federated identity (Azure AD, Google, Okta) ships JWTs because that's the only shape that works without every relying party calling back to the issuer.

The standard SaaS shape: access JWT + opaque refresh

The pragmatic middle ground. The access JWT lives 5–15 minutes; expiration is its only revocation. The refresh token is opaque (random string) and lives 30 days; it's stored in a server-side table and can be revoked instantly. On every refresh, the access token is reminted from the current claims. Stale-claim window is bounded to the access token's TTL. A revoked user keeps access for at most 15 minutes — usually acceptable, never indefinite.

03Revocation is the whole game.

Every JWT decision returns to this question. A signed token cannot be "unsigned" — until it expires, anyone holding it can use it. Five mitigations:

Short TTLs

Access token expires in 5–15 minutes. The blast radius of "user fired but token still valid" is the access TTL. The refresh-token call requires server contact, which means revocation can take effect on next refresh. This is the standard fix and the right one for most apps.

Revocation list (token deny list)

When a user is revoked, write the JWT's jti (or the user ID) to a Redis list with a TTL matching the access token's expiry. Every request checks the list. This re-introduces a per-request Redis lookup — exactly the cost a JWT was supposed to eliminate. But the list is tiny (only currently-revoked tokens), so the lookup is cheap. Most "we need instant revoke" production setups end up with this list anyway; combine it with short TTLs to keep the list small.

Versioned tokens

Add a token_version claim. The user row has current_token_version. On revoke, bump the row. Every JWT verify checks jwt.token_version == user.current_token_version from cache. Same cost as a revocation list, slightly more granular — you can invalidate one user's all-sessions without enumerating tokens. Often the cleanest implementation.

Sliding refresh with rotation

Each refresh issues a new refresh token and invalidates the old one. If an attacker uses a refresh token, the legitimate user's next refresh fails — both tokens get invalidated, and the user is forced to re-login. This is the de-facto standard for OAuth refresh-token flows and what Auth0, Okta, Cognito, Azure AD B2C all do by default.

Key rotation

Rotate the signing key periodically. Old keys remain in JWKS for the TTL of the longest outstanding token, then are removed. Any token signed with a retired key fails verification. This is the nuclear option for invalidating all tokens at once — useful for breach response, not for daily revocation. Plan key rotation from day one; retrofitting it after years of production tokens is painful.

04In ASP.NET Core.

Two complete patterns. Cookie auth for a traditional web app; JWT access + refresh + revocation list for an API. Same framework, different one-time setup.

C#   CookieAuthSetup.cs 
// Server-side session cookie auth — for traditional web apps.
using Microsoft.AspNetCore.Authentication.Cookies;
using Microsoft.AspNetCore.DataProtection;

var builder = WebApplication.CreateBuilder(args);

// Persist data-protection keys to a shared store (Redis, Azure Blob).
// Without this, every pod's cookies are invalidated on restart.
builder.Services.AddDataProtection()
    .SetApplicationName("shop")
    .PersistKeysToAzureBlobStorage(builder.Configuration["DP:BlobUri"]!);

builder.Services.AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddCookie(o =>
    {
        o.Cookie.Name             = "__Host-shop-auth";
        o.Cookie.SecurePolicy     = CookieSecurePolicy.Always;
        o.Cookie.HttpOnly         = true;
        o.Cookie.SameSite         = SameSiteMode.Lax;
        o.ExpireTimeSpan          = TimeSpan.FromHours(8);
        o.SlidingExpiration       = true;
        o.LoginPath               = "/login";

        // Validate the principal on every request — picks up role/permission changes.
        o.Events.OnValidatePrincipal = async ctx =>
        {
            var sub = ctx.Principal?.FindFirstValue("sub");
            var users = ctx.HttpContext.RequestServices
                .GetRequiredService<IUserStore>();
            if (sub is null || !await users.IsActiveAsync(sub))
                ctx.RejectPrincipal();   // instant revoke on next request
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();
app.Run();

For an API serving SPAs and mobile clients, JWT with a refresh-rotation flow. The access token is a JWT signed by your IdentityServer (or Azure AD B2C); the refresh token is opaque and stored server-side. A revocation check runs on every authenticated request — the Redis lookup is sub-ms.

C#   JwtAuthSetup.cs 
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;
using StackExchange.Redis;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IConnectionMultiplexer>(
    ConnectionMultiplexer.Connect(builder.Configuration.GetConnectionString("Redis")!));

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.Authority = builder.Configuration["Auth:Issuer"];      // fetches JWKS
        o.Audience  = builder.Configuration["Auth:Audience"];
        o.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateLifetime          = true,
            ClockSkew                 = TimeSpan.FromSeconds(30),
            RequireSignedTokens       = true,
            ValidateIssuerSigningKey  = true,
        };

        // Revocation check — fast Redis EXISTS on a deny-list keyed by jti.
        o.Events = new JwtBearerEvents
        {
            OnTokenValidated = async ctx =>
            {
                var jti = ctx.Principal?.FindFirstValue("jti");
                if (jti is null) { ctx.Fail("missing jti"); return; }

                var redis = ctx.HttpContext.RequestServices
                    .GetRequiredService<IConnectionMultiplexer>().GetDatabase();
                if (await redis.KeyExistsAsync($"revoked:{jti}"))
                    ctx.Fail("token revoked");
            }
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();

// Refresh endpoint: rotate the refresh token, mint a new access token.
app.MapPost("/auth/refresh", async (
    RefreshRequest req, ITokenService tokens, CancellationToken ct) =>
{
    var pair = await tokens.RotateAsync(req.RefreshToken, ct);
    return pair is null
        ? Results.Unauthorized()
        : Results.Ok(new { access = pair.Access, refresh = pair.Refresh });
});

// Logout endpoint: blacklist the current access token's jti until its exp.
app.MapPost("/auth/logout", async (
    HttpContext ctx, IConnectionMultiplexer redis, ITokenService tokens) =>
{
    var jti = ctx.User.FindFirstValue("jti");
    var exp = ctx.User.FindFirstValue("exp");
    if (jti is { } id && long.TryParse(exp, out var e))
    {
        var ttl = TimeSpan.FromSeconds(Math.Max(1, e - DateTimeOffset.UtcNow.ToUnixTimeSeconds()));
        await redis.GetDatabase().StringSetAsync($"revoked:{id}", "1", ttl);
    }
    await tokens.RevokeRefreshAsync(ctx.User.FindFirstValue("sub")!);
    return Results.NoContent();
});

app.Run();

05The detail.

Don't put JWTs in localStorage

Any XSS reads localStorage and exfiltrates the token. Use HttpOnly cookies (the JWT is in the cookie, browser sends it automatically) or memory-only storage with refresh on page-load. localStorage is for non-sensitive data. The "JWT in Authorization header" pattern is fine for native mobile clients; for browsers, cookie auth is safer.

Data-protection keys must be shared

ASP.NET Core encrypts cookies and antiforgery tokens with data-protection keys. Default storage is the local filesystem, which means each pod has different keys and cookies issued by pod-1 fail on pod-2. Persist keys to Redis, Blob storage, or a shared file system — and set SetApplicationName so multiple apps don't trash each other's keys.

JWT clock skew is real

Two servers in different regions can have clocks 100ms apart. A freshly-minted token can fail validation on a server with a slightly-slow clock. Set ClockSkew = TimeSpan.FromSeconds(30) in TokenValidationParameters. Default is 5 minutes (too loose); zero is too tight; 30s is the production default.

Refresh tokens deserve their own table

A row per refresh token with UserId, IssuedAt, ExpiresAt, RevokedAt, ReplacedBy. The ReplacedBy column makes rotation auditable. The RevokedAt column is what your "log me out everywhere" button updates. This table is the only stateful part of an otherwise stateless JWT setup, and that's fine.

Federated identity is almost always the answer

Don't write user authentication in 2026. Use Azure AD B2C, Auth0, Clerk, Cognito, Duende IdentityServer — they handle the JWKS, the password hashing, the MFA, the social logins, the compliance audits. Your app is a relying party that consumes JWTs. The home-grown user-table-plus-password-hash is the wrong default.

"The right authentication is the one you didn't write. The wrong authentication is the one you wrote yourself."

07Decision guide.

Traditional MVC / Razor web app

Cookie auth with server session. __Host- cookie, HttpOnly + SameSite=Lax + Secure. DataProtection keys persisted to Blob.

SPA + .NET API + mobile

JWT access (15min) + opaque refresh (30 days, rotated). Revocation list in Redis. Federated to Azure AD B2C / Auth0.

Microservices need to verify the same user

JWT signed by a central issuer. Each service validates via JWKS. Cache the public keys 1h. Don't re-implement per service.

B2B SaaS with SSO requirement

OIDC federation. Customer's IdP issues a JWT; your app trusts it. Don't store passwords. Don't onboard the customer with usernames.

"We need to log everyone out NOW"

Bump a global token-version key in Redis; every JWT validate checks it. Or rotate the signing key. Have one of these wired from day one.

Mobile app with offline support

JWT in secure storage (Keychain / Keystore), long-lived refresh, short access. Offline = valid until access expires; reconnect refreshes.

Three things people get wrong

One — JWTs in localStorage. One XSS and your tokens leave the building. Use HttpOnly cookies for browsers; localStorage is for non-sensitive data.

Two — pure JWT with no revocation. "We're stateless" until someone gets fired and the architecture review meets compliance. Add a revocation list or token-version check from day one.

Three — DataProtection keys on the local filesystem. Every pod issues its own crypto. Cookies break on the second pod. Persist to Blob or Redis with a shared application name.
