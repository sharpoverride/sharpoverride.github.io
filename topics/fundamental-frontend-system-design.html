The frontend has system design too.

"Frontend system design" used to be an oxymoron. In 2026, it's the discipline that decides Web Vitals scores, mobile abandonment rate, and how many engineers can ship in parallel on the same UI. The architectural decisions are different from the backend's; the rigour required is the same.

The questions

Render on server or client? How to split the bundle? How to manage state? How to ship a UI library across teams? How to keep p95 LCP under 2.5s on mobile?

The new vocabulary

SSR, CSR, RSC, hydration, islands, partial pre-rendering, route-based code splitting, micro-frontends.

01What changed.

Once: a page was server-rendered HTML, plus some jQuery. Then SPAs took over: the server returns a near-empty HTML, JS bootstraps everything. SPAs felt fast after the initial load but were brutal on first paint, SEO, and mobile. The current era is the synthesis: server-render the first paint, hydrate or progressively enhance, ship only the JS the route needs.

"The pendulum hasn't stopped swinging. It just got finer-grained."

02The rendering modes.

ModeWhat it doesStrengthWeakness
SSRServer renders HTML per requestSEO, fast FCP, accessible without JSServer cost, latency on cold caches
SSGStatic HTML at build timeFastest possible, CDN-servedStale until rebuild
ISRStatic, regenerated on demandStatic speed + freshnessComplex revalidation
CSRJS bundles render in browserApp-like interactivity, simple deploySlow FCP, JS-blind SEO, mobile cost
RSC (server components)Component-level "render on server, ship as data"Smaller JS, server-side data fetch in componentsNew mental model
IslandsMostly-static page with hydrated interactive islandsMinimal JS, static-fastLimits app-like interactions

Most production apps mix modes. Marketing pages SSG. Product pages SSR or ISR. App shell hydrated. Per-component islands for interactive bits. The framework (Next, Remix, Astro, Nuxt, SvelteKit) is the mode-orchestrator.

Bundle-size budgets

Initial JS bundle (uncached): <200 KB gzipped for a good FCP on mid-tier mobile. Aim for <100 KB if you can. Code-split by route; don't ship the dashboard's JS to the landing page.

Image performance

Modern formats (AVIF, WebP). Responsive srcset. loading="lazy" below the fold. Image weight dominates page weight; one un-optimised hero image can blow your LCP budget single-handedly.

Critical CSS inlined

The CSS for above-the-fold content goes in the HTML. The rest async-loaded. Tools (critical, criticalCSS) extract automatically. The single biggest LCP win on most sites is this.

04State management.

The three layers

Server state: data fetched from APIs. Use React Query / SWR / TanStack Query. Handles caching, refetch, optimistic updates. UI state: form inputs, toggles, modal open/close. useState in component, lift up as needed. Global UI state: theme, current user, cart contents. Zustand, Jotai, Redux. Keep global state minimal; most things are server state in disguise.

URL is state

Filter selections, sort order, pagination, modal-open — encode in the URL. The URL is the user's bookmark, their shareable link, their back-button history. Pattern: state hooks that wrap URLSearchParams.

05Cross-team frontend.

Design system

Shared component library (Button, Input, Card, Modal). Tokens (colors, spacing, typography). Single source of truth across product surfaces. Versioned like an API; breaking changes affect every consumer.

Micro-frontends — sometimes

Each team owns a UI slice; runtime composes them. Module Federation, Single SPA. Useful when teams ship at different cadences against one shell. Cost: shared dependency duplication, harder UX consistency, complex deploy story. Most teams should not start here.

Monorepo with packages

Less ambitious, more common. UI components in @org/ui; SDK in @org/api-client; apps depend on both. Nx, Turborepo, Bun workspaces. This is the structure most modern frontend orgs end up at.

06Decision guide.

Marketing site

SSG / Astro / Next.js with SSG mode. Static, CDN-served, fastest possible.

Content with personalisation

SSR or ISR. Next.js, Remix, Nuxt. Server-side per-user rendering with cache.

App-like (dashboard, design tool)

SPA or SSR-with-hydration. Vite + React, Next.js app router. Heavy client state, real-time updates.

SEO-critical

SSR or SSG. Never CSR for content that must be indexed. Google's JS rendering is real but unreliable.

Cross-team UI

Design system in a shared package. Tokens. Storybook for visibility. Micro-frontends only if you have a clear org-shaped reason.

Performance trouble

Measure with real-device telemetry (RUM). Don't optimise what you don't measure. The bug is almost always image weight or third-party scripts.

"Frontend system design isn't about rendering. It's about deciding which user pays which cost."
