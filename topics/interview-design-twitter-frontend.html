Design Twitter timeline — frontend.

The backend designs Twitter. The frontend designs the experience of Twitter: infinite scroll with no jank, optimistic likes that don't lie, real-time updates that don't surprise, image media at 60fps. Different problem, different metrics, different art.

The hard parts

Virtualised infinite list. Optimistic mutations. Real-time updates that respect scroll position. Media-heavy rendering at 60fps. Offline-tolerance.

Performance budgets

First Contentful Paint <1s. Largest Contentful Paint <2s. Interaction latency <100ms. Scroll at 60fps on a 4-year-old phone.

Tech you'll touch
React + virtual scrollerReact Query / SWRSSEService worker

01Clarify.

Scope: the home timeline view. Infinite scroll. Tap to like / retweet / reply. Real-time updates (new tweets appear). Media (images, video) inline. Works on mobile web. Not in scope: compose flow, profile pages, search, DMs.

Lighthouse / Web Vitals scores are the deliverable. The interviewer wants to hear which budgets you'd set before coding, not after.

03High-level architecture.

Three layers: view (React components, virtual list), state (React Query or Redux-like, normalised tweet store), transport (REST for pagination, SSE for real-time updates). Plus a service worker for offline-tolerant caching and a CDN for media.

Fig. AThe state layer is the single source of truth. Components subscribe to slices; mutations go through it; transport syncs in the background.

04Deep dive — virtualised list.

Don't render 10,000 DOM nodes

The timeline can be thousands of tweets long. Rendering all of them kills scroll perf and memory. Virtualise: render only the ~30 tweets currently visible + a small buffer above and below. Above and below the viewport, use spacer divs sized to the placeholder height. react-window, react-virtual, virtua are the libraries.

Variable-height items

Tweets have different heights — text vs image vs video vs quoted-tweet. Variable-height virtualisation: measure each item on first render; cache the height; use cached heights to compute scrollable space. Without this, scroll jumps because positions guess-and-shift.

Maintain scroll position on prepend

"5 new tweets ↑" prepends items. The user is mid-scroll on something they were reading; you do not want to suddenly jump them up. Maintain anchor: track the offset of a known tweet ID; after prepend, scroll so that tweet stays at the same screen Y. This is the difference between "polished" and "infuriating."

05Deep dive — optimistic UI.

Tap → mutate locally → send to server

User taps "Like." Don't wait for the server. Increment the like count in the local state immediately; the heart fills; the count goes up. Send the mutation to the server in the background. On success, no-op. On failure, revert.

Rollback gracefully

Server returns 503. The optimistic state was wrong. Revert with a brief toast: "Couldn't like. Tap to retry." Don't silently swallow the failure; the user pressed the button and expects feedback.

Idempotency in mutations

"Like" should be idempotent — clicking twice is one like, not toggling. Send an idempotency key (see idempotency chapter). The client maintains a per-action token; retries are safe.

06Deep dive — real-time updates.

SSE for "new tweets available"

Open an SSE connection (see realtime channels chapter). Server pushes "new tweet for your timeline" events. Don't auto-insert; show a "5 new tweets ↑" pill at the top of the list. User taps it; you prepend; you reset scroll anchor.

Live updates within visible tweets

A tweet you're looking at gets a new reply. Subscribe to per-tweet updates only when visible. Intersection observer fires when the tweet enters viewport; client subscribes. Leaves viewport; client unsubscribes. Bounds the WS / SSE bandwidth.

Reconnection

SSE drops; the browser auto-reconnects. Resume with Last-Event-ID: server replays events the client missed. Without resume, you have a "I was offline for 2 minutes and now I'm out of sync" bug.

Image lazy-loading

loading="lazy" on img tags. IntersectionObserver to start video downloads only when visible. Preload the next 2–3 tweets' images to avoid blanks on scroll.

Don't fetch what you already have

React Query / SWR keeps a normalised cache keyed by tweet ID. Re-rendering a tweet you've seen reuses the cached data; no extra fetch.
