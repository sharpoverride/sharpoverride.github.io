The reverse API call.

A webhook is an HTTP POST you didn't ask for — sent by a third party when something interesting happens to your account on their system. Stripe, GitHub, Twilio, Shopify all ship them. The protocol is HTTP. The interesting part is the contract: signing, retries, idempotency, replay protection.

What a webhook is

"Push instead of poll." The provider sends you an HTTP POST with a JSON event payload when something happens to your account. You acknowledge with 200 OK.

The four problems they solve

Signing (prove it came from the provider). Idempotency (handle retries). Ordering (or accept the lack of it). Backpressure (when you can't keep up).

01Why webhooks exist.

Your integration with Stripe needs to know when a payment intent succeeds. The polling answer: hit GET /payment_intents/{id} every few seconds. This wastes Stripe's API budget, your CPU, and is still seconds-of-latency stale. The push answer: Stripe POSTs your endpoint when the event fires. The latency is the network's; the cost is one request per event; the consumer (you) controls the URL.

The reason webhooks aren't just a different API: the security model is inverted. In a normal API call, the client proves who it is (with an API key). In a webhook, the server (the provider) has to prove who it is to the client (you). That's done by signing.

"A webhook is a request you accept from someone you don't trust, that you have to be sure came from someone you do."

02Signing & replay.

Fig. 30Standard webhook shape: HMAC over the body, timestamp to prevent replay, fast 200 OK, do the work asynchronously.

HMAC over the raw body

The provider shares a secret with you at signup. On every event, the provider computes HMAC-SHA256(secret, timestamp + body) and sends it in a header. You compute the same HMAC on receipt; if they don't match, reject with 401. Don't parse JSON before verifying — the canonical bytes are the signed thing. Use a constant-time comparison (CryptographicOperations.FixedTimeEquals); naive == leaks timing info.

Timestamp prevents replay

The signature includes the timestamp. An attacker who captures the POST in flight can't replay it later because you reject any webhook older than ~5 minutes. The provider's clock and your clock must agree to within tolerance. This is the second-cheapest improvement to your webhook handler; the first is signing.

Event ID for deduplication

Every event has a unique ID. The provider will retry; you must dedupe. Store processed event IDs in Redis with a TTL longer than the provider's retry window (typically 7 days). On receipt: SETNX event:<id>; if it was already set, ack but skip the work.

03Delivery semantics.

At-least-once

Every reasonable webhook provider promises at-least-once delivery. You will receive every event at least once, possibly more than once. Your handler must be idempotent. The dedupe-by-event-id pattern above is the standard implementation.

Ordering is not guaranteed

Stripe will sometimes deliver charge.refunded before charge.captured. Your handler must not assume order. Use the event's data, not the order it arrived, to reconstruct state. For ordering-sensitive workflows, fetch the current state from the provider's API after receipt rather than reconstruct from the event stream.

Return 200 fast

Most providers have a tight timeout (Stripe: ~30s, GitHub: ~10s, Twilio: ~15s). Acknowledge first, work second. Receive the webhook, validate the signature, persist to a queue or table, return 200. The actual processing happens in a background worker. If you do synchronous work on the webhook thread and timeout, the provider thinks you failed and retries — duplicates pile up.

Dead-letter for stuck deliveries

If your endpoint is unhealthy, the provider retries with exponential backoff for hours or days. After that, they give up. Stripe sends an email; GitHub keeps a "recent deliveries" page; some providers send to nothing. Build a dashboard that surfaces missed events; reconcile via the provider's API daily. Webhooks are best-effort; the API is the source of truth.

04In ASP.NET Core.

C#   StripeWebhook.cs 
using System.Security.Cryptography;
using System.Text;
using StackExchange.Redis;

app.MapPost("/webhooks/stripe", async (
    HttpRequest req,
    IConnectionMultiplexer redis,
    IConfiguration cfg,
    IPublishEndpoint bus,
    CancellationToken ct) =>
{
    // Read the raw body — signing is over the exact bytes.
    using var reader = new StreamReader(req.Body);
    var body = await reader.ReadToEndAsync(ct);
    var sigHeader = req.Headers["Stripe-Signature"].ToString();

    // Parse "t=…,v1=…" header format.
    var parts = sigHeader.Split(",")
        .Select(p => p.Split("=", 2))
        .ToDictionary(p => p[0], p => p[1]);

    if (!parts.TryGetValue("t", out var tsStr) ||
        !parts.TryGetValue("v1", out var sigHex) ||
        !long.TryParse(tsStr, out var ts))
        return Results.BadRequest();

    // Reject anything older than 5 minutes (clock-skew tolerant).
    if (Math.Abs(DateTimeOffset.UtcNow.ToUnixTimeSeconds() - ts) > 300)
        return Results.Unauthorized();

    // Recompute the HMAC over "t.body" with the shared secret.
    var secret = cfg["Stripe:WebhookSecret"]!;
    var signed = $"{ts}.{body}";
    using var hmac = new HMACSHA256(Encoding.UTF8.GetBytes(secret));
    var computed = hmac.ComputeHash(Encoding.UTF8.GetBytes(signed));
    var provided = Convert.FromHexString(sigHex);

    if (!CryptographicOperations.FixedTimeEquals(computed, provided))
        return Results.Unauthorized();

    // Parse the event JSON and dedupe by event id.
    var evt = JsonDocument.Parse(body);
    var eventId = evt.RootElement.GetProperty("id").GetString()!;

    var db = redis.GetDatabase();
    var first = await db.StringSetAsync(
        $"wh:stripe:{eventId}", "1",
        expiry: TimeSpan.FromDays(8),
        when: When.NotExists);

    if (first)
    {
        // First time seeing this event — queue the work, don't process inline.
        await bus.Publish(new StripeEventReceived(eventId, body), ct);
    }

    return Results.Ok();  // 200 fast — work happens in the consumer.
});

05The detail.

Verify before you parse

JSON parsers have bugs. Don't run yours against untrusted input before signature verification. The HMAC is over the raw bytes; verify it first, then parse.

FixedTimeEquals — always

Comparing signatures with byte[].SequenceEqual or == shortcuts on the first byte that differs. A patient attacker can time you and reconstruct the secret. Use CryptographicOperations.FixedTimeEquals in .NET (or hmac.compare_digest in Python, etc.).

Async after ack

The webhook endpoint should only verify, dedupe, queue. Do NOT do database writes or downstream API calls on the webhook thread. A failed downstream means you 5xx the provider; the provider retries; you process again; duplicate work everywhere.

Reconcile via the API daily

Webhooks miss events sometimes. Your provider's "list events since timestamp" API is the source of truth. A nightly reconciliation job fetches anything the webhook stream missed.

Standard Webhooks — a real spec

standardwebhooks.com is a 2023 spec proposed by Svix that standardises signing, retries, and headers across providers. Adopt it for your own webhook-publishing system; the spec captures everything this article describes and a few you'd otherwise miss.

"A webhook is an API call where you're the API. Treat it accordingly."

06Decision guide.

Receiving webhooks (Stripe, GitHub, etc.)

Verify HMAC against raw body with FixedTimeEquals. Reject old timestamps. Dedupe by event id in Redis. 200 fast; queue the work.

Sending webhooks to your customers

Adopt Standard Webhooks. HMAC + timestamp + event id. Retry with exponential backoff for ~24h, then DLQ. Expose recent-deliveries UI.

Customers complain webhooks are missing

They probably aren't (mostly). Add reconciliation: "list events since" API + diff. The missing webhooks are almost always 5xx-ed on the customer's side.

Self-hosted webhook dispatching

Svix open-source. Saves writing the retry/dedupe/UI infrastructure. Or roll your own with a Kafka topic + worker pool if you have specific needs.

Customer's endpoint is behind a firewall

Polling fallback. Or tunneled webhooks (Ngrok-style) — provider connects out from their side. Or move to a long-lived gRPC stream.

You want exactly-once semantics

Doesn't exist on the wire. At-least-once + idempotent handler = the practical equivalent. Dedupe by event id; reconcile via API.
