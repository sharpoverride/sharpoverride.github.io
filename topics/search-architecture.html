The query you wanted to write — not the one SQL allows.

Full-text search is the part of every product that's invisibly outsourced to a different database. Inverted indexes, BM25 relevance, the indexing pipeline that fights schema drift — the work is real, the cost is real, and the alternative is WHERE title LIKE '%foo%' at 30 RPS.

The problem in one line

A B-tree index supports prefix lookups and equality. It does not support "every document that mentions any of these tokens, ranked by relevance, with stemming, with faceted filters."

The thesis

Use Postgres FTS for "good enough" search inside a product you already own. Use Elasticsearch / OpenSearch / Azure AI Search when relevance tuning, scale, or ML-augmented retrieval is the requirement. Build neither yourself.

01Why SQL falls over.

The product manager asks for "search products by name." You ship WHERE name LIKE '%shoe%'. It works. Then they want a typo tolerance pass — "shooe" should match "shoe". Then synonyms — "trainer" should match "running shoe". Then faceted filters that update as the user types. Then "boost products in stock" and "deprioritise discontinued ones." Each request looks small; together they describe a search engine. A B-tree index gives you none of them.

The reason is structural. A B-tree maps a sorted key to row IDs. A search engine needs to map every token in every document to the documents that contain it, with scores, with positions, with field weights, with analyser pipelines. That data structure — the inverted index — is upside-down from the row-store layout. Trying to make Postgres do it with raw LIKE is the architectural equivalent of cutting bread with a hammer; trying to make it do it with tsvector works surprisingly well, up to a point.

"Search isn't a query problem. It's a different database in trench coat."

02The inverted index.

Forward index: document → words. Reverse it: word → documents. That single inversion is what every search engine is built on. Add positions, term frequencies, document frequencies, and you can score relevance with BM25. Add an analysis pipeline and you handle stemming and synonyms. Add sharding and you scale.

Fig. 19The same text passes the same analyser at index time and query time. Matching is on the tokens, not the raw words. Synonyms and stemming live in the analyser.

BM25 is the score you'll inherit

Elasticsearch, OpenSearch, Lucene, Postgres FTS, Azure AI Search — they all default to BM25 (a refinement of TF-IDF). The score for a document d against a query q rewards documents where query terms are frequent inside the document but rare across the corpus, and penalises long documents (so a 50-word product description with "shoe" twice ranks higher than a 5,000-word manual that also says "shoe" twice). You almost never need to tune BM25's k1 and b parameters; the defaults are good. What you tune is field weights — boost the title 3×, the brand 2×, the body 1×, the SEO tags 0.5×.

Hybrid retrieval is the 2026 default

Pure BM25 misses semantic matches ("car" vs "automobile"). Pure dense vectors (embeddings + ANN search) miss exact matches and rare terms ("XJ-470-A"). Production search in 2026 is hybrid: run both, combine with a reciprocal rank fusion (RRF) score. Elasticsearch 8, OpenSearch 2, and Azure AI Search all ship native hybrid retrieval. Build new systems around hybrid from the start; retrofitting it later is a project of its own.

03The indexing pipeline.

The index is a derived store. Keeping it fresh is the unsexy half of search architecture and the half that causes most production incidents.

Three options for keeping the index fresh

Synchronous indexing on write: save to the database, index immediately. Newest data is searchable instantly; failure to index now blocks the write. Don't do this in production; the search cluster becomes part of your write SLA.

Outbox + indexer: save to the database, write an outbox event (see Chapter 17), let a background indexer dispatch to the search cluster. Eventual consistency on the order of seconds, write path independent of search availability. The boring correct answer.

CDC → search: Debezium reads the WAL, transforms into search documents, ships to Elasticsearch via a sink connector. No app code. Works well for "the index is a mirror of the database table" use cases. Less great when you need to join across tables to produce a search document — joins live in the indexer, not in the WAL.

Denormalisation is forced

A search document is one self-contained record. A product's brand name, category path, available sizes, image URL — all flattened into one JSON document, even if they live in five SQL tables. The index doesn't join. When the brand name changes, every product document carrying it must be re-indexed. Plan the reindex paths from day one. Typical pattern: a "rebuild all products affected by brand X" job runs nightly + after any brand mutation.

Reindex is a routine event, not an outage

You'll change analysers, add fields, tweak mappings. Lucene-based engines can't change mappings on an existing index — you create a new index, reindex into it, atomically alias the old name to the new index, drop the old one. Elasticsearch's reindex API does this in place; the operation should be a deploy-time procedure, not a one-off heroic. Make "rebuild the search index from source of truth" a one-command operation and test it monthly.

Shards have memory; pick before you ingest

Elasticsearch defaults to one primary shard per index in modern versions, which is fine for <30 GB indexes. Bigger than that, you want sharding — but each shard adds JVM overhead. Rule of thumb: aim for shards of 10–50 GB each. Too few and you can't parallelise; too many and JVM heap dies a thousand cuts. Reshape requires reindex, so plan capacity before you commit.

04In .NET.

The modern .NET client for Elasticsearch is Elastic.Clients.Elasticsearch (or OpenSearch.Client for OpenSearch). The shape is the same: typed documents, an analyser-aware indexer, a query builder. Outbox handler subscribes to domain events and ships the affected document into the index.

C#   ProductSearchService.cs 
// dotnet add package Elastic.Clients.Elasticsearch
using Elastic.Clients.Elasticsearch;
using Elastic.Clients.Elasticsearch.QueryDsl;

namespace Academy.Search;

public sealed record ProductDoc
{
    public required Guid   Id      { get; init; }
    public required string Title   { get; init; }
    public required string Brand   { get; init; }
    public required string[] Categories { get; init; }
    public required decimal Price  { get; init; }
    public required bool    InStock { get; init; }
    public          DenseVector? Embedding { get; init; }   // optional, 768-d
}

public sealed class ProductSearchService
{
    private readonly ElasticsearchClient _es;
    private const string IndexName = "products";

    public ProductSearchService(ElasticsearchClient es) => _es = es;

    // Index — called from an outbox consumer, not from the HTTP write path.
    public async Task UpsertAsync(ProductDoc doc, CancellationToken ct)
    {
        var resp = await _es.IndexAsync(doc, i => i
            .Index(IndexName)
            .Id(doc.Id)
            .Refresh(Refresh.False),    // don't pay the refresh cost per call
            ct);

        if (!resp.IsValidResponse)
            throw new InvalidOperationException($"index failed: {resp.DebugInformation}");
    }

    // Hybrid BM25 + vector search with field boosts and a stock filter.
    public async Task<SearchResult> SearchAsync(
        string query, DenseVector? embedding, int page, CancellationToken ct)
    {
        const int pageSize = 20;

        var resp = await _es.SearchAsync<ProductDoc>(s => s
            .Index(IndexName)
            .From(page * pageSize)
            .Size(pageSize)
            .Query(q => q
                .Bool(b => b
                    .Must(m => m.MultiMatch(mm => mm
                        .Query(query)
                        .Fields(new[] { "title^3", "brand^2", "categories" })
                        .Fuzziness(new Fuzziness("AUTO"))))
                    .Filter(f => f.Term(t => t.Field(d => d.InStock).Value(true)))))
            .Knn(kn => embedding is { } emb ? kn
                .Field("embedding")
                .QueryVector(emb)
                .K(50)
                .NumCandidates(200) : kn)
            .Aggregations(a => a
                .Add("by_brand", agg => agg.Terms(t => t.Field(d => d.Brand).Size(10))))
            .Sort(so => so.Score(o => o.Order(SortOrder.Desc))),
            ct);

        return new SearchResult(
            Total: resp.Total,
            Hits:  resp.Documents.ToArray(),
            Facets: resp.Aggregations?.GetStringTerms("by_brand")?
                .Buckets.Select(b => (b.Key.Value, b.DocCount)).ToArray() ?? []);
    }
}

public sealed record SearchResult(long Total, ProductDoc[] Hits, (string, long)[] Facets);

The outbox consumer keeps the index fresh. On a domain event of "product updated", load the canonical record, project to ProductDoc, upsert. The consumer is idempotent — re-running on the same event id is safe.

C#   ProductIndexConsumer.cs 
public sealed class ProductIndexConsumer : IConsumer<ProductChanged>
{
    private readonly ProductSearchService _search;
    private readonly IProductRepository _repo;

    public ProductIndexConsumer(ProductSearchService s, IProductRepository r)
    { _search = s; _repo = r; }

    public async Task Consume(ConsumeContext<ProductChanged> ctx)
    {
        var product = await _repo.LoadAsync(ctx.Message.ProductId, ctx.CancellationToken);
        if (product is null) return;   // hard delete handled separately

        await _search.UpsertAsync(new ProductDoc
        {
            Id         = product.Id,
            Title      = product.Title,
            Brand      = product.Brand.Name,
            Categories = product.Categories.Select(c => c.Path).ToArray(),
            Price      = product.Price,
            InStock    = product.Stock > 0,
        }, ctx.CancellationToken);
    }
}

05The detail.

Refresh interval is your throughput knob

Elasticsearch's default refresh_interval=1s makes new docs searchable within a second. For bulk-load and reindex paths, set it to -1 and refresh manually at the end — 10× ingest throughput. Set back to 1s for steady-state. The single most useful operational setting in the engine.

Index name is not the alias name

Apps query products; the real index is products-2026-01; an alias maps the two. When you reindex into products-2026-02, atomically switch the alias. Zero downtime, zero app changes. Don't let app code ever reference the dated index directly — the alias is the contract.

Postgres FTS scales further than you think

Up to ~10M small documents on a beefy Postgres, with proper tsvector + GIN indexes, FTS matches Elasticsearch's relevance for most product-catalog workloads. No second cluster to operate, joins still possible, transactional with the row. Use Postgres FTS until you outgrow it; the migration to Elasticsearch is straightforward when the day comes.

Watch JVM heap and the field-data cache

Elasticsearch / OpenSearch nodes are JVMs. heap_used_percent > 75% sustained = you're a node restart away from a paging incident. Field-data cache (for aggregations on text fields) is the most common bloater. Limit aggregations to keyword fields; never to text fields without a hard size cap.

Vendor or self-host?

Self-hosting Elasticsearch is feasible at small scale but the operational footprint at production scale — JVM tuning, snapshot management, cluster rolling upgrades, security patches — eats a full engineer. Elastic Cloud / OpenSearch Service / Azure AI Search costs 2–3× the raw compute and replaces that engineer. The math almost always favours the vendor past ~3 nodes.

"Search is the only feature where the user notices ‘works most of the time’ instantly and stops trusting you forever."

07Decision guide.

SaaS app, <1M docs, search is a sideline

Postgres FTS with GIN indexes on tsvector. No second cluster, transactional with rows, joins still work. The boring correct answer.

E-commerce, >1M products, relevance matters

Managed Elasticsearch or Azure AI Search. Hybrid retrieval (BM25 + vectors). Outbox consumer for indexing. Reindex pattern from day one.

Logs / observability

OpenSearch on AWS, or Elastic's Observability tier. Different use case from product search; same engine.

RAG / AI search over docs

Azure AI Search's semantic + vector or Elastic's ELSER. Pure vector DBs (Qdrant, Pinecone) are fine but you'll want lexical too — hybrid wins.

Autocomplete in a marketing site

Algolia or Meilisearch. The DX and latency advantage is real for the "search box at the top of the page" pattern. Pay the tax.

Already on Cosmos DB

Azure AI Search with a Cosmos data source. Native indexer keeps the index in sync; no outbox to write.

Three things people get wrong

One — synchronous indexing on the write path. Search cluster blip = checkout outage. Always go through an outbox / CDC and let the index be eventually consistent.

Two — aggregating on text fields. Aggregations need keyword fields. Use text for analysis, keyword for grouping. Mixing them blows the field-data cache and OOMs the node.

Three — skipping the reindex muscle. The day you change an analyser, you'll need to rebuild from scratch. Build the alias-swap pattern early and exercise it monthly so the procedure is boring.
