The conventions that save a year.

Public APIs live for a decade. The shape you ship is the shape you maintain. Get the conventions right at the start — resource naming, errors, pagination, idempotency, versioning — and the next five teams that integrate with you will thank you. Get them wrong and you'll be writing migration guides forever.

The dozen conventions

Plural resources. Verbs in HTTP, not in URLs. Use status codes. Pagination is cursor-based. Errors are structured. Idempotency keys on writes. ISO timestamps. UUIDs. OpenAPI spec. Versioning by URL or header. Rate limits in headers. Webhooks signed.

01Resources & verbs.

Resources are plural nouns: /users, /orders, /charges. Verbs live in the HTTP method, not the URL. POST /charges creates a charge; GET /charges/{id} reads one; DELETE /charges/{id} deletes. "POST /createCharge" is the smell of an API designer who hasn't read the standards.

Sub-resources for relationships: GET /users/{id}/orders. Limit depth to two; deeper hierarchies are a pain to evolve. Actions that don't fit CRUD: use a verb sub-resource. POST /orders/{id}/cancel, POST /charges/{id}/refund. Better than PATCH with a magic body field.

Error body shape — structured, not a string:

json   standard error
{
  "error": {
    "code": "insufficient_funds",
    "message": "Card has insufficient funds for this charge.",
    "docs_url": "https://docs.example.com/errors/insufficient_funds",
    "request_id": "req_7c4a92"
  }
}

Code is the stable identifier — clients switch on it. Message is human-readable, can change. Request ID lets support trace.

03Pagination.

Cursor-based, not offset. ?limit=20&cursor=eyJpZCI6Li4ufQ. The cursor is an opaque opaque-to-client base64 of "where we left off." Offset pagination breaks on inserts (a new row at position 0 shifts everything; client sees duplicates on the next page). Cursors are stable.

Response includes next_cursor and has_more. Bonus: hypertext links (HATEOAS) — { "next": "/charges?cursor=..." }. Some clients use them; most don't; cheap to include.

04Idempotency.

Mutating POSTs should accept an Idempotency-Key header. The server stores the key + the response; a retry with the same key returns the cached response without re-executing the side effect. For payments, this isn't a feature; it's a requirement. See idempotency-keys chapter.

05Docs & spec.

OpenAPI 3.x. Generated from code or hand-written; the SDKs are generated from it. The spec is the contract; if it doesn't match the implementation, fix one or the other. CI lints for breaking changes against the previous version.

Examples, examples, examples. Each endpoint in the docs needs at least one full curl example with realistic payload. Engineers integrate by copying examples; the docs are the API in disguise.

06Decision guide.

Public API for customers

REST, JSON, OpenAPI, semantic versioning. Documented. Cursors. Idempotency on writes. The boring correct.

Internal service-to-service

gRPC if both sides are yours. REST if the boundary is the org. See gRPC vs REST vs GraphQL.

BFF for mobile / web

GraphQL fits. Per-screen-shaped responses; client picks fields.

Long-running operations

Return 202 Accepted with a status URL the client polls. Or use webhooks.

Bulk operations

Either accept arrays (most common) or expose a separate batch endpoint with its own rate limits.

You inherited a bad API

Don't rewrite. New version alongside; route old endpoints to a compat layer; deprecate over years. See API versioning.

"The conventions exist because every shop that didn't follow them rewrote their API in year three."
