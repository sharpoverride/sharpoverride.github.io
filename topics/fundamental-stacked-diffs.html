Many small PRs stacked in order.

A 1500-line PR is reviewed badly. Splitting it into five dependent 300-line PRs gives you five good reviews — and ships the first one as soon as it's approved, instead of waiting for the whole stack. Phabricator, Graphite, Sapling popularised this; modern teams that ship fast use it.

The model

Each PR is one commit. PRs depend on earlier ones. The bottom of the stack merges first; the next rebases onto main and merges; etc.

Why it's hard in vanilla git

Rebasing a stack on every merge is fiddly. Tools (Graphite CLI, ghstack, sapling) automate the rebase choreography.

01Why not a feature branch.

The traditional approach: branch off main, write 1500 lines, open one PR. Reviewer sees a wall of code; skims; approves; bugs slip through. Plus: the author is blocked on review for the entire change before they can ship any of it. Plus: the longer the branch lives, the more main has moved underneath; the rebase eventually becomes a project.

Alternative: split the work into independent commits, each reviewable on its own. Each commit becomes its own PR. The five PRs depend in order; reviewer can tackle them sequentially or even in parallel.

"The big PR is the version of the work that respects nobody's time."

02How it works.

You write commit A (refactor a helper). Commit B (use the helper in a new feature). Commit C (wire it to the UI). Open three PRs:

  • PR A targets main
  • PR B targets PR A's branch
  • PR C targets PR B's branch

Reviewer of PR A only sees commit A — small, focused. When PR A merges, the tool rebases PR B onto main, retargets it to main; B merges; rebases C; C merges. Author's "the whole thing is shipped" experience is identical to a single big PR; reviewers got three small reviews.

The rebase ballet

The painful part of vanilla git: when PR A's commit changes (a fixup during review), PR B is now built on a stale base. You have to rebase the rest of the stack on every change, by hand, every time. Graphite / Sapling automate this — one command rebases the whole stack and updates all the open PRs.

04Decision guide.

You routinely ship 500+ line PRs

Adopt stacked diffs. Graphite for a GitHub team; Sapling if you can switch source-control tooling.

Small team, simple changes

Don't. Single-commit PRs are already small; the overhead isn't worth it.

Big refactor coming

Strong case. Split into self-contained steps; ship the easy ones first; keep the risky ones for the end.

Your CI is slow

Stacks make this worse — each PR runs CI. Fix CI speed first if you can.

Reviewer pool is small

Stacks distribute review load — different reviewers per PR. Helps if you have the breadth.

You've never used it

Try Graphite on a single PR first. The mental model takes a week; the productivity gain compounds for years.

"Stacked diffs feel like the difference between batched and streaming. Same total throughput; entirely different perceived latency."
