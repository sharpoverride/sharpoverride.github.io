Never store the password.

Store an irreversible derivation. With a unique salt. Using a slow algorithm. Why? Because a database leak is a near-certain event over a long enough timescale, and the consequences of leaked passwords compound (users reuse). The cost of getting this right is one library and one config line.

The rules

(1) Never plain. (2) Never MD5/SHA1/SHA256-only. (3) Always per-user salt. (4) Always a slow algorithm: argon2id, scrypt, bcrypt. (5) Optionally pepper.

The default in 2026

argon2id with appropriate cost parameters. If you can't use argon2, bcrypt with cost 12+ is acceptable.

01What not to do.

Don't store plaintext. Anyone who reads the DB sees every user's password. Anyone — internal DBA, attacker, backup leak, log accident.

Don't store a plain SHA-256 / SHA-512 hash. Fast hashes are crackable: rainbow tables, GPU brute-force at billions/sec, custom ASICs. The attacker who steals your DB cracks the weak passwords in minutes and the medium ones in days.

Don't roll your own. Custom KDF schemes are crypto-faulty by default. Use a library that exposes argon2id, scrypt, or bcrypt. Pass a strong password, get back a single string that contains algorithm, parameters, salt, hash. Store that string. Verify by passing the plaintext input and the stored string to the same library.

"Storing a SHA-256 of the password is the security equivalent of leaving the door locked with cellotape."

02The correct shape.

argon2id (2026 default)

Winner of the 2015 Password Hashing Competition. Memory-hard (resists GPU/ASIC), iteration-bounded (resists time-based brute force). Use argon2id (the hybrid variant), not argon2i or argon2d alone. Libraries: libargon2 (C), Argon2.Net, node-argon2.

bcrypt

Still acceptable in 2026. Cost factor 12+. Maximum input length 72 bytes (truncates silently — pre-hash with SHA-512 if you must handle longer). Slightly less GPU-resistant than argon2id, but mature and ubiquitous.

scrypt

Memory-hard like argon2; older. Still secure with proper parameters. Less common in 2026; default to argon2id.

03Tuning the cost.

Slow algorithms are the point. Login should take ~250–500 ms on your auth server. Faster is too easy to brute-force; slower is bad UX.

AlgorithmParameters (2026)
argon2idm=64MB, t=3, p=4 (start; tune to ~300ms)
bcryptcost=12 (~250ms on modern CPU)
scryptN=2^17, r=8, p=1 (~250ms)

Re-tune every 2 years. CPUs get faster; what took 300 ms in 2024 takes 150 ms in 2026. On successful login, you can rehash with new parameters (your library exposes the stored parameters, so you can detect "this was hashed with old cost; rehash now").

04Pepper.

Optional. A site-wide secret added to the hash input. The salt is per-user and stored in the DB; the pepper is global and stored separately (env var, HSM, secret manager). If the attacker steals the DB but not the pepper, they can't crack passwords — they don't have the input.

The cost: rotating the pepper requires re-hashing everyone. Most operations stop at "salt + slow hash"; pepper is the additional defence for high-security applications.

05Breach response.

If you stored hashes correctly

Notify users, force password reset, monitor for credential-stuffing on neighbouring sites. The leaked hashes are slow to crack; attackers might give up rather than spend GPU-years.

If you stored hashes incorrectly

The leaked hashes are crackable in days. Every user's password is effectively in the wild. Force reset; assume any service those users share a password with is also compromised.

2FA changes the calculation

If users had MFA enabled, a leaked password isn't enough to log in. 2FA is the seatbelt for the day the password database leaks.

Passkeys are the future

WebAuthn / passkeys bypass passwords entirely. The DB stores a public key, not a password derivative. Even a complete DB leak doesn't give attackers a login capability. 2025+ is the year to start migrating.

"Every password breach in the news is the database breach you've already had, but with a slow hash so nobody noticed."
