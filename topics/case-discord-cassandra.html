Case study Years 2015 → 2023 Scale 1 trillion msgs Stack MongoDB → Cassandra → ScyllaDB

One trillion messages, two migrations.

Discord rewrote its message database twice. MongoDB cracked at 100M messages in 2015. Cassandra cracked at one trillion messages in 2022. ScyllaDB — the C++ rewrite of Cassandra — is what they're on now. The lessons are not about the databases.

What broke each time

MongoDB: working set exceeded RAM at 100M messages. Cassandra: GC pauses, hot partitions, "Cassandra was always the problem we were managing, not the problem we were using."

What it cost

Migration 1: 6 months, small team. Migration 2: a year, dedicated team. Avoided by waiting: years of operational pain that would have come either way.

Pull-quote

"We didn't pick the database. We picked the next database we could survive." — paraphrased from Discord's engineering blog.

01The trillion-message problem.

Discord stores every message ever sent: chat, DMs, server channels, voice text. The access pattern is unusual. Most messages are read once, within seconds, by users who happen to be online. A small fraction are read repeatedly — pinned messages, channel scrollback, search hits. A vanishing fraction are read after a year. The working set is the last ~hour of activity across the planet; the rest is cold storage that occasionally gets paged in.

The number that matters: by 2017, Discord was inserting 1 billion messages per month. By 2022, it was 4 billion per day — a trillion messages cumulative. Storage costs scale linearly with that. Read latency does not — a 1 ms p99 in 2015 has to remain a 1 ms p99 in 2025, on a corpus 10,000× bigger.

"The hard part of the database isn't the writes. It's that the writes don't stop and the reads never get easier."

02Architecture before.

Fig. AThe 2015 architecture was textbook: one replica set, document model. Discord ran it for two years before scale forced a redesign.

The 2015 design was a single MongoDB replica set. One document per message, indexed on {channel_id, _id} for the typical query "give me the last N messages in this channel before timestamp T". Reads scaled by load-balancing across secondaries; writes funneled to the primary. The schema was straightforward; the team was small; the architecture diagram fit on a napkin.

03The breaking points.

2015 — MongoDB's working set

At ~100M messages, the message index — the B-tree that maps (channel_id, _id) to document location — stopped fitting in RAM. Each query that wasn't perfectly hot triggered disk reads to walk the index. The p99 read latency climbed from 5ms to 300ms. It wasn't a query problem; it was a memory-shape problem. The fix in MongoDB land would have been sharding, but the access pattern (channel-scoped, time-ordered) didn't fit MongoDB's sharding model cleanly.

2017–2022 — Cassandra's hot partitions

Cassandra solved 2015's problem. Partition by (channel_id, time_bucket); each large channel's history lives on a specific node's disk; reads are sequential within a partition. This is the textbook good design for chat at scale.

The breaking point was hot partitions. A viral channel — a Twitch streamer's Discord, a major-game release, a leak — would suddenly receive 10,000× the normal write rate. One node hits CPU saturation; queries to it time out; replicas behind that node lag; the cluster's tail latency degrades for everyone. Java GC pauses compounded the problem: a 200ms GC pause during a hot-partition burst caused cascading queue buildups.

The compaction backlog

Cassandra periodically merges SSTables to reclaim space and improve read performance. Discord's write rate was high enough that compaction couldn't keep up on the hot nodes. Pending compactions backed up; disk usage ballooned; reads slowed because they had to consult more SSTables. Compaction tuning became a team responsibility. A team that didn't want to be Cassandra DBAs was becoming Cassandra DBAs.

The off-by-one cost

"It works, but it requires a team" is a real cost the architecture review doesn't price. Discord's published number: roughly 4 engineers full-time on Cassandra tuning at the peak. A managed equivalent (DataStax Astra) would have cost roughly the same in license fees; the self-host saved cash and spent headcount.

04Architecture after.

Fig. BTwo layers: a Rust request-coalescing service in front, a shard-per-core C++ database underneath. Discord rebuilt both.

Data Services: the coalescing layer

Discord noticed that hot partitions were largely an artefact of duplicate concurrent reads. When a viral message goes out, 10,000 clients fetch the same channel within milliseconds. The database serves 10,000 reads, identical, against a hot partition. Data Services is a Rust service that deduplicates these: incoming reads for the same key within a small window are coalesced into one backend read; everyone gets the same response. The hot partition stops being hot.

ScyllaDB: same API, different engine

ScyllaDB speaks the Cassandra wire protocol; the migration was query-compatible. The engine underneath is different: C++, shard-per-core, lock-free, no JVM. Discord runs ScyllaDB on ~177 nodes today, replacing ~900 Cassandra nodes at the migration boundary. The reduction is engine efficiency, not architectural change.

The 177 → 72 node reduction is the headline. The latency improvement is the user-facing result. The "1 FTE instead of 4" is the cost line nobody quotes but the CFO notices.

06The lessons.

Coalesce reads before you shard writes

Discord's biggest single win wasn't the database swap — it was the Data Services layer that deduplicates concurrent reads. A hot key with 10,000 concurrent readers can be served by one backend read; the fan-out happens in the proxy. Apply this everywhere: it's a few hundred lines of code that defers a database migration by years.

Migrating an API-compatible engine is a different beast

Mongo → Cassandra was a rewrite. Cassandra → Scylla was a configuration change. API-compatible replacements are a different class of migration; if the new engine speaks your old protocol, you're moving capacity, not redesigning.

JVM is a tax for stateful infrastructure

Cassandra, Elasticsearch, Kafka, ZooKeeper — all JVM, all share the GC-pause problem at the tail. Discord's move to a non-JVM engine reclaimed predictability they couldn't tune Cassandra into. If you're choosing stateful infra today and your tail latency matters, the C++ / Rust / Go alternatives deserve more weight than they did five years ago: Scylla (vs Cassandra), Redpanda (vs Kafka), QuestDB (vs InfluxDB).

Headcount is a cost; ignore it at your peril

The "free open-source database" running on your team's nights and weekends costs you a small team's salary. The managed equivalent (DataStax, Confluent Cloud, AWS MSK) costs you a similar number in vendor invoices. The choice is whether you pay it in headcount or in invoice line items; both are real money. Discord chose to engineer the headcount cost down by simplifying the engine.

"The database that scales is the one your team can operate. Everything else is choice paralysis."
