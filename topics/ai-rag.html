Give the model the notes first.

Retrieval-augmented generation: before answering, fetch relevant context from a knowledge base and stuff it into the prompt. The LLM stays general; the knowledge stays current. The interesting engineering is in the retrieval, not the generation.

What RAG solves

LLMs hallucinate. They're frozen at training time. RAG grounds them in fresh, owned, verified content — your docs, your codebase, your customer data.

The three stages

Indexing (offline) → Retrieval (online, ~50ms) → Generation (online, ~1–10s). Most engineering time goes into stage 2.

01Why retrieval beats finetuning.

You want an LLM that knows your company's documentation. Two options. Finetune: take a base model, train it on your docs. Costly (~$10k–$1M depending on size), slow to update (new docs = new training run), no source-of-truth (you can't see what the model "learned"). RAG: keep the base model as-is; at query time, retrieve the relevant docs and put them in the prompt. Cheap to update (re-index nightly), traceable (you can show which docs the answer came from), and works on any base model.

RAG won the architecture war for "ask my docs"–shaped products. Finetuning still wins for "change the model's style or domain," but for grounding in current data, retrieval is the default.

"Retrieval is how an LLM borrows your library card."

02The three stages.

Fig. ATwo pipelines: indexing batches ahead of time; query runs end-to-end on each user request.

Indexing — get the docs into a queryable shape

Docs come from various sources — Confluence, Notion, GitHub, S3, your DB. Chunk them into pieces small enough to fit in the LLM's context (typically 200–800 tokens). Embed each chunk into a vector (e.g. with OpenAI text-embedding-3-small or a local model like BGE). Index both the vector (in Qdrant, Pinecone, Weaviate, or pgvector) and the text (in Elasticsearch / Lucene for BM25).

Retrieval — find the relevant chunks

At query time: embed the user's question; do a vector search (top-50 by cosine similarity) and a BM25 search (top-50 by lexical match). Combine the two sets. Re-rank the combined set using a cross-encoder model that scores each candidate against the query. Take the top 3–10. This is hybrid retrieval with re-ranking — the modern default.

Generation — build the prompt

Build a prompt that includes the retrieved chunks as context, then the user's question. Pass to the LLM. Cite the sources — the model can reference the chunk IDs in its answer, and the UI can link to the original docs. Citation is the only protection against hallucination; without it, the user can't verify the answer.

03Hybrid retrieval & re-ranking.

Why hybrid beats pure vector

Pure vector search misses exact matches. The user types "Error code SQL-0931"; the embedding similarity says "this looks like a database error topic" and returns 100 generic DB chunks — none containing the literal "SQL-0931". BM25 catches the exact match. Hybrid takes the best of both: vector for semantic similarity, BM25 for exact terms. Combined with reciprocal rank fusion or score normalisation.

Re-rankers are the secret weapon

The top-50 from retrieval is noisy. A cross-encoder model (Cohere Rerank, BGE-Reranker, Voyage rerank-2) takes the (query, chunk) pair and produces a relevance score, jointly considering both. The top-5 after re-ranking is dramatically better than the top-5 from retrieval alone. Cost: ~50ms latency for re-ranking 50 candidates. Worth it.

Chunking is the make-or-break

Chunks too small (50 tokens) = no context, retrieval misses. Chunks too large (2000 tokens) = irrelevant context drowns the relevant signal. Sweet spot: ~300–500 tokens, with ~50 tokens overlap between chunks. Split on semantic boundaries (paragraphs, sections) when possible, not arbitrary character counts. Per-document strategies vary: code wants AST-aware chunking; markdown wants header-aware.

04In .NET.

C#   RagPipeline.cs (using Semantic Kernel + Qdrant) 
// dotnet add package Microsoft.SemanticKernel
// dotnet add package Qdrant.Client
using Microsoft.SemanticKernel;
using Qdrant.Client;

public sealed class RagPipeline
{
    private readonly QdrantClient   _vec;
    private readonly IElasticClient _es;     // for BM25
    private readonly IRerankerClient _rerank;
    private readonly Kernel         _kernel;  // SK with chat completion

    public async Task<RagAnswer> AskAsync(string question, CancellationToken ct)
    {
        // 1. Hybrid retrieval — vector + BM25, top 50 each.
        var queryEmbed = await _kernel.InvokeAsync<float[]>("embed", new() { ["input"] = question }, ct);

        var vectorTask = _vec.SearchAsync(
            "chunks", queryEmbed,
            limit: 50,
            cancellationToken: ct);

        var bm25Task = _es.SearchAsync<Chunk>(s => s
            .Index("chunks")
            .Size(50)
            .Query(q => q.Match(m => m.Field(c => c.Text).Query(question))));

        await Task.WhenAll(vectorTask, bm25Task);

        // 2. Reciprocal rank fusion → union top 100, score by 1/(rank+60).
        var fused = ReciprocalRankFusion(vectorTask.Result, bm25Task.Result);

        // 3. Re-rank fused set with a cross-encoder; keep top 5.
        var reranked = await _rerank.RerankAsync(question, fused, topK: 5, ct);

        // 4. Build the prompt with citations.
        var contextBlock = string.Join("\n\n", reranked.Select((c, i) =>
            $"[doc:{c.DocId}#{c.ChunkId}]\n{c.Text}"));

        var prompt = $"""
            Answer the question using ONLY the context below.
            Cite sources by [doc:ID#CHUNK] in the answer.
            If the context doesn't contain the answer, say so.

            === CONTEXT ===
            {contextBlock}

            === QUESTION ===
            {question}
            """;

        // 5. Generate the answer.
        var response = await _kernel.InvokeAsync<string>("chat", new() { ["prompt"] = prompt }, ct);

        return new RagAnswer(response, reranked.Select(c => c.DocId).ToArray());
    }
}

05The detail.

Citations are non-negotiable

Make the LLM cite sources by chunk ID. The UI links back to original docs. Without citations, users can't trust the answer; with them, they can verify and you can trace bad answers back to bad chunks.

Embedding model choice matters less than you'd think

OpenAI's text-embedding-3-small, BGE-large, Voyage-3, Cohere embed-v3 — all are within ~5% of each other on most benchmarks. Spend the optimization budget on chunking, re-ranking, and prompt engineering — not on chasing the latest embedding model.

Re-indexing is a regular operational task

Docs change. Embeddings need to be re-computed for changed chunks. Build a CDC-style pipeline (see CDC chapter): doc updates trigger chunk re-embedding; the vector DB is updated incrementally. Full re-indexing is expensive; do it only when you change the embedding model.

Hallucinations still happen

Even with grounding, LLMs invent details. Test with adversarial queries; have an eval suite of (question, expected sources, accepted answers) that you run on every prompt or model change. The eval is the QA gate.

Access control is your job

If only some users can see some docs, you have to filter at retrieval time. Pre-filter the vector search by user permissions (most vector DBs support metadata filters). Post-filtering — retrieving everything then filtering — leaks the existence of restricted content via timing.

"The model is the borrowed mouth. The retrieval is your voice."

06Decision guide.

"Ask my docs" chatbot

Hybrid retrieval (vector + BM25) → re-rank → grounded prompt with citations. Start with Azure AI Search or pgvector + Elasticsearch.

Vector DB choice

pgvector if you already have Postgres. Qdrant for self-hosted vector-first. Pinecone for managed. Azure AI Search for "vector + lexical in one box."

Small data, <100k chunks

Skip the vector DB. Embeddings in a list, compute cosine in-process. Stay simple until it hurts.

Multi-tenant SaaS RAG

Tenant-id as a metadata filter on every query. Cell-architecture model (see cell architecture) if tenant isolation must be hard.

Code search / repo Q&A

AST-aware chunking — chunk by function/method, not by line count. Use a code-specific embedding model (e.g. text-embedding-3-large with code-tuned prompts, or CodeBERT-style).

Real-time freshness needed

CDC pipeline doc → re-embed → vector DB. Get the lag down to seconds; don't try for milliseconds — the value isn't there.
