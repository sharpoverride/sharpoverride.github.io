A log, partitioned, replicated.

Kafka isn't a queue. It's an append-only log that pretends to be a queue when consumers want it to. Once you accept that — and understand partitions, offsets, ISR, and consumer groups — everything else falls out of the same primitives.

What Kafka is

Distributed, replicated, append-only commit log. Topics are logs. Partitions are shards. Consumer groups are sets of consumers that share a topic's partitions.

What it is not

A queue (messages aren't deleted on consume — they age out by retention). A database (no query language, no random access). RabbitMQ.

01Why an append-only log.

Traditional message brokers (RabbitMQ, ActiveMQ) treat messages as items in a queue: a producer puts one in, a consumer takes it out, it's gone. Kafka treats messages as entries in a log: a producer appends, every consumer reads forward through the log at their own pace, and messages remain on disk until they expire by retention policy (often 7 days, sometimes forever).

This sounds backwards until you list what it buys you. Multiple independent consumer groups can read the same stream — analytics, billing, search-indexing, each at their own offset. Replay is built-in: rewind a consumer to a previous offset to reprocess. Backpressure is asymmetric: producers don't wait for consumers; the log absorbs the gap. The disk is the queue.

"Kafka took the database's idea of a transaction log and made it the user-facing API."

02Partitions and offsets.

Fig. 26A topic is N partitions. Each partition is its own log with monotonically increasing offsets. Partition count is the unit of parallelism.

Offsets are the bookmark

Every record in a partition has an offset — a 64-bit integer that monotonically increases. Consumers track which offset they've processed. To resume after a crash, the consumer asks Kafka "where was I?" and continues from the stored offset. Offsets are stored in a special internal topic __consumer_offsets, committed by consumers as they make progress.

Partitioning is the throughput knob

One partition = one consumer maximum. If you need 10 parallel consumers, you need 10 partitions. Partition count is set at topic creation and is hard to change later (adding partitions changes the hash distribution and breaks per-key ordering). Pick generously: a topic with 24 partitions can support up to 24 parallel consumers; a topic with 1 partition can never go faster than one consumer.

03Replication and ISR.

Each partition is replicated across N brokers (typical: 3). One replica is the leader; the others are followers. Producers and consumers always talk to the leader. Followers fetch from the leader and stay in sync. An "in-sync replica" (ISR) is a follower that's caught up within a tolerable lag; if a follower falls too far behind, it drops out of the ISR.

Acks: the producer's safety vs. speed knob

Producer config acks controls when the leader acknowledges a write. acks=0: fire-and-forget, no durability. acks=1: leader has the record; if leader dies before replicating, data is lost. acks=all: every ISR has the record; durable across N-1 broker losses (for replication factor N). The standard production default is acks=all with min.insync.replicas=2 on a replication-factor-3 topic.

When the leader dies

The controller (in KRaft mode: one of the controller-quorum brokers) detects the leader failure via heartbeats. It promotes an ISR follower to leader. Failover happens in ~10 seconds in practice. The new leader has at least as much data as the old one (because ISR-only promotion). Producers retry against the new leader; consumers transparently switch.

KRaft — Kafka without ZooKeeper

Until 2022, Kafka required ZooKeeper for metadata. Operating ZooKeeper alongside Kafka was famously painful — two distributed systems to keep healthy. KRaft mode replaces ZK with a Raft consensus protocol inside Kafka itself; the controller quorum lives in the Kafka brokers. Modern Kafka (3.3+) defaults to KRaft. Plan to be on it; ZooKeeper-based clusters are being deprecated.

04Consumer groups.

A consumer group is a set of consumers that share a topic's partitions. Kafka assigns each partition to exactly one consumer in the group. If you have 10 partitions and 4 consumers, each consumer gets ~2.5 partitions. If you add a 5th consumer, Kafka rebalances; one consumer drops to 2 partitions. If you scale to 11 consumers, one sits idle (a partition can't be shared).

Multiple consumer groups can read the same topic independently — they each have their own offsets. This is how you do fan-out: orders topic feeds the billing-group, the analytics-group, and the search-index-group; each is at a different offset; each consumes the full stream.

Rebalancing — the operational gotcha

When a consumer joins or leaves the group, Kafka rebalances partition assignments. During the rebalance, consumption pauses for the whole group. A flaky consumer that connects-disconnects-reconnects can cause "rebalance storms" that bring throughput to a crawl. Modern Kafka has cooperative rebalancing (incremental, less disruptive); enable it.

05In .NET.

C#   OrderConsumer.cs 
// dotnet add package Confluent.Kafka
using Confluent.Kafka;

public sealed class OrderConsumer : BackgroundService
{
    private readonly IServiceProvider _sp;
    private readonly ConsumerConfig _config = new()
    {
        BootstrapServers   = "kafka-1:9092,kafka-2:9092,kafka-3:9092",
        GroupId            = "order-consumer",
        EnableAutoCommit   = false,                       // commit on success only
        AutoOffsetReset    = AutoOffsetReset.Earliest,
        PartitionAssignmentStrategy =
            PartitionAssignmentStrategy.CooperativeSticky,    // less rebalance pain
        IsolationLevel     = IsolationLevel.ReadCommitted,    // for txn writes
    };

    public OrderConsumer(IServiceProvider sp) => _sp = sp;

    protected override async Task ExecuteAsync(CancellationToken stop)
    {
        using var consumer = new ConsumerBuilder<string, string>(_config).Build();
        consumer.Subscribe("orders");

        while (!stop.IsCancellationRequested)
        {
            try
            {
                var cr = consumer.Consume(stop);
                if (cr.IsPartitionEOF) continue;

                using var scope = _sp.CreateScope();
                var handler = scope.ServiceProvider.GetRequiredService<IOrderHandler>();

                // Process. If this throws, we don't commit — Kafka redelivers.
                await handler.HandleAsync(cr.Message.Key, cr.Message.Value, stop);

                consumer.Commit(cr);                            // at-least-once delivery
            }
            catch (OperationCanceledException) { break; }
            catch (ConsumeException ex)
            {
                // Bad message format — push to DLQ; don't block the partition.
                await SendToDeadLetterAsync(ex.ConsumerRecord, ex.Error);
            }
        }
        consumer.Close();
    }
}

06The detail.

Commit offsets after successful processing

Default auto-commit (every 5s) breaks at-least-once delivery — you'll commit offsets for messages your handler hasn't actually processed. Disable auto-commit; commit explicitly after the handler succeeds. The handler must be idempotent because the same message can be redelivered on crash.

Pick partition count generously

Partitions are cheap; resizing isn't. A topic with 6 partitions can support 6 parallel consumers maximum. A topic with 24 partitions can support 24. Start with 12–24 unless you have a strong reason for fewer. The cost is metadata overhead, not disk or CPU.

Compact vs delete retention

Default retention is "delete after 7 days." Compacted topics retain the latest value per key forever — useful for state snapshots (current price per product, latest config per user). Pick compact for state-shaped topics; delete for event-shaped.

Exactly-once is real but constrained

Kafka transactions + idempotent producer + read-committed consumer = exactly-once within Kafka. It doesn't extend to your database unless you use the Kafka-Connect or Outbox pattern (see CDC chapter). "Exactly-once" in mixed-system pipelines requires more discipline than the flag.

Confluent / Redpanda are real alternatives

Confluent Cloud is managed Kafka with the Confluent ecosystem (Schema Registry, ksqlDB, Connect). Redpanda is a Kafka-API-compatible C++ rewrite with no JVM, smaller cluster, easier ops — same wire protocol, same clients. For new deployments, Redpanda is worth a serious look (it's the Scylla of the Kafka world).

"Kafka isn't fast because of clever code. It's fast because the disk is sequential and the consumer is patient."

07Decision guide.

Event log for microservices

Self-host or Confluent Cloud / MSK / Event Hubs. Replication 3, acks=all, min ISR 2. 24+ partitions.

You want Kafka API but easier ops

Redpanda. Same protocol, no JVM, fewer nodes. Or Confluent Cloud if you want hosted.

Stream of state changes (CDC)

Kafka + Debezium. Compacted topic per source table. See CDC chapter.

Simple work queue

Maybe not Kafka. Redis Streams, Service Bus, RabbitMQ might be a better fit for "task per message" semantics.

Low ops budget

Confluent Cloud or Azure Event Hubs (Kafka-compatible). The hosted offerings are worth the cost vs running a Kafka cluster yourself.

Sub-millisecond latency

Reconsider. Kafka's p99 is 5–50ms typically. For HFT or in-memory cache invalidation, a Redis pub/sub or NATS may be closer to what you want.
