Case study Year 2023 Scale 55M RPS Stack Postgres · pgbouncer · partitioning · NVMe

55 million requests per second. Fifteen Postgres clusters.

Cloudflare runs its core configuration store on a small number of large Postgres clusters. Not Spanner. Not Cassandra. Postgres — operated extremely well. The story is what "extremely well" actually means.

The surprise

15 Postgres clusters, each ~140 TB, handle the entire Cloudflare control plane. The data is critical; the volume is small relative to the request rate.

The architecture

One cluster per logical workload. pgbouncer for connection pooling. Heavy use of partitioning. Read replicas for everything that tolerates lag.

Not used

Cassandra, MongoDB, Spanner, DynamoDB. Cloudflare considered them and stayed with Postgres. The choice that scales is the one your team operates well.

01What 55M RPS actually means.

The 55M figure is request volume across Cloudflare's edge — the HTTP requests their CDN serves globally. Almost none of those requests touch Postgres. Postgres serves the control plane: configuration, DNS records, security policies, customer settings, billing state. That data is fetched once and cached aggressively at the edge.

But the control plane is critical. A config change has to land in seconds across the edge. A DNS record edited in the dashboard has to propagate. A WAF rule pushed by an enterprise customer can't wait. The Postgres clusters are the source of truth for everything the edge does; they don't serve the request volume, but they cannot be down.

"Cloudflare's database isn't fast because Postgres is fast. It's fast because the data fits — and the team knows how to keep it fitting."

02Postgres in a CDN's control plane.

Fig. APostgres is the source of truth. A sync layer pushes changes to the edge cache. The 55M RPS edge load never reaches a database.

The key architectural decision: Postgres is a control plane, not a data plane. Customer requests are answered at the edge from cached configuration. Postgres serves ~thousands of QPS, not millions. The 55M RPS number is what the edge serves; the database sees a small fraction of that.

03The discipline.

One cluster per logical workload

15 clusters, each dedicated to a specific concern: DNS records, page rules, security policies, billing, customer accounts. Failure of one cluster degrades one feature, not the platform. The cost is operational — 15 things to monitor, patch, upgrade. The benefit is blast-radius isolation.

Partitioning over sharding

Within a cluster, large tables are partitioned (Postgres native partitioning) by time or by tenant. Partitioning is cheaper than sharding: same database, same connection, same transactions. It scales reads and write distribution without the cross-shard query restrictions sharding imposes (see Figma). Cloudflare partitions aggressively; they shard only when partitioning ceases to suffice.

NVMe and right-sized hardware

Cloudflare's Postgres clusters run on dedicated bare metal with NVMe SSDs — not RDS, not cloud VMs. The hardware is sized so the hot working set fits in RAM. The fundamental performance trick is the same as 2015: keep the index hot. Bare metal with NVMe is the modern way to do it.

04Connection pooling.

Postgres at scale dies on connections, not on queries. Each connection consumes ~10 MB of RAM; thousands of app instances open thousands of connections; the server's memory is consumed by idle connection state. pgbouncer is non-optional: a connection-pool layer that fans out a small number of real Postgres connections among many client sessions.

Cloudflare's pgbouncer runs in transaction-pooling mode: each transaction gets a connection from the pool, returns it on commit. This breaks some Postgres features (session state, prepared statements stored on the connection) — application code has to be written aware of it. The performance benefit is worth the discipline.

06The lessons.

Control plane vs data plane

The headline number (55M RPS) is the data plane. The Postgres clusters are the control plane. They are different problems with different solutions. Don't size your database for traffic the database isn't serving.

Partition before you shard

Postgres's native partitioning gets you ~10× scale-up with the same DB. Sharding is an order of magnitude more work for an additional 10×. If you haven't partitioned the largest table, you haven't earned the right to discuss sharding.

pgbouncer or you die on connections

Direct app-to-Postgres connections do not scale past a few hundred. pgbouncer in transaction mode is the standard answer. Adopt it on day one; retrofitting after the connection table fills is a panicky afternoon.

Specialise the cluster, not the database

15 Postgres clusters each for one workload beats "one big Postgres for everything" for blast radius, deploy independence, and tuning. The cost is operational headcount; the benefit is that one bad query in billing doesn't touch DNS.

Bare metal still wins for stateful

RDS is convenient. NVMe on bare metal is 3–10× the IOPS for a third of the cost — and for stateful workloads, IOPS is the bottleneck. The cloud's win is the stateless tier; for stateful, ownership often still pays. Cloudflare's choice; not everyone's. But worth knowing the option exists.

"The hardest thing about scaling Postgres is admitting you don't need to scale Postgres."
