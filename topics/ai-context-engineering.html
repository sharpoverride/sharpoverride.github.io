Prompt engineering grew up.

"Prompt engineering" was about clever wordings of one prompt. Context engineering is about everything the model sees on a call: the system prompt, the retrieval, the recent conversation, the tools available, the structured input. Production LLM systems live or die on this.

The shift

2023: "rewrite this prompt." 2025: "redesign the context pipeline." The center of gravity moved from words to architecture.

The five levers

(1) System prompt. (2) Retrieved context. (3) Conversation memory. (4) Tool descriptions. (5) Structured I/O.

01From prompt to context.

The early-2023 advice was "tell the model what role it is, what format to use, give it examples." All correct. But shipping a real LLM feature is mostly not about that. It's about: which user-specific data do you retrieve? How do you summarise yesterday's session? Which tools are exposed? How do you format the response so downstream code can use it?

Context engineering names this larger discipline. The model is the substrate; the context pipeline is the product.

"Prompt engineering writes a sentence. Context engineering builds an information pipeline."

02The five levers.

System prompt

Role, rules, output format, refusal policy. Short and concrete beats long and aspirational. A 200-word system prompt outperforms a 2000-word one almost always; the model attends to it better. Cache it via prompt-caching APIs — it's identical across calls.

Retrieved context

For "answer about my data" use cases: RAG. Pick the chunks that maximise relevance, minimise noise. Re-rank harder than you index; the 5 best chunks beat the 50 mediocre ones. See RAG chapter.

Conversation memory

Long conversations exceed context windows. Three strategies: sliding window (lossy), summary memory (compress old turns), retrieval memory (vector-index past turns; fetch on demand). Mix and match per use case.

Tool descriptions

For agents: the description of each tool is the prompt. "Searches for files" is bad; "Lists files matching a glob pattern; returns up to 100 results sorted by modified time" is good. The model picks tools by their descriptions; the time you spend on these compounds.

Structured I/O

JSON schema for output. Don't parse free-form text; constrain the generation. OpenAI's response_format, Anthropic tools, Gemini's structured output — all enforce shape. Downstream code becomes deterministic.

Fig. AEach lever is independently tunable. Most quality wins come from one of the middle three, not the system prompt.

03Common anti-patterns.

Stuffing the system prompt

"Just one more rule" creeps to a 4000-word system prompt that the model attends to poorly. Cap it. Move rules into structured tool definitions or examples.

Over-retrieving

20 chunks of 500 tokens = 10k tokens of context. The signal is in 2–3; the rest is noise that dilutes. Re-rank, prune, summarise.

All of memory, always

Including the entire conversation history on every call. Fine for the first 5 turns; disastrous by turn 50. Summarise older turns; retrieve on-demand.

Tool descriptions written for humans

"Calls the search API." Tells the model nothing. Describe inputs, outputs, when to use it, when not to. The description is prompt; write it like one.

Parsing JSON from chat output

The model returns "Sure! Here's the JSON: {...}". You parse, mostly. Sometimes the wrapping breaks; you ship a hotfix. Use structured output APIs; let the runtime enforce the schema.

"Every quality bug is a context bug somewhere up the pipeline."

04Decision guide.

Quality is poor

Profile your context. Print the actual prompt going to the model. The bug is almost always in what's being included, not in the model.

Long-running agents

Summarise older trajectory. Vector-index past steps. Don't carry the whole tape into every call.

Multiple tools confuse the model

Reduce. 5–10 tools is the sweet spot. Tighten descriptions. Group rarely-used tools behind a "look-up" indirection.

JSON output unreliable

Switch to structured output API. Don't parse chat-format JSON.

System prompt too long

Move examples into few-shot turns. Move rules into tool definitions. Aim for ~200 words.

Costs are high

Cache the system prompt. Shrink retrieval. Use a smaller model for triage; route to big model only for complex queries.
