Case study Year 2014–2018 Scale 1B txn/day · 8 VMs Stack Akka · Scala · actor model

One billion transactions a day. Eight VMs.

PayPal's payment-flow service handled a billion transactions a day on eight VMs. The trick wasn't bigger hardware — it was the actor model. Same hardware, different concurrency primitive, two orders of magnitude more throughput.

The setup

Java/Tomcat thread-per-request stack hit a wall: thousands of threads idle on I/O, GC pauses, context switching dominating CPU.

The move

Rewrite on Akka actors (JVM-based actor library). One actor per request, no thread per request. Backed by Scala for the message-passing model.

The number

Pre-Akka: ~50 VMs for the workload. Post-Akka: 8 VMs for the same workload. 5–6× density improvement, same hardware.

01The thread-per-request wall.

A payment in PayPal touches 5–10 downstream services synchronously: fraud, ledger, risk scoring, partner banks, notifications. Each is an HTTP call; each takes ~10–100ms. A single payment request holds a Java thread for the duration, mostly idle on I/O. At 10k concurrent payments, you've consumed 10k threads. The JVM defaults aren't ready for it; thread switching dominates CPU; GC pauses lengthen under thread-creation pressure.

The standard fixes — bigger thread pools, async I/O with futures — improve throughput by a factor of 2–3. They don't change the basic model. PayPal needed a 10× improvement, not a 3× one. The architectural rewrite was the only path.

"A thread is 1 MB of stack. A million concurrent requests don't fit in any machine you can buy."

02What the actor model gives you.

Fig. AActors are lightweight (~300 bytes vs a thread's 1 MB stack), scheduled by the JVM onto a small thread pool, communicate by message-passing instead of blocking I/O.

The key shift: messages instead of method calls

An actor receives messages, processes them, sends messages, returns. It never blocks waiting for a response. When a payment actor needs the fraud service's verdict, it sends a message and registers a callback; it goes back to processing other messages until the response arrives. The JVM's small thread pool stays busy with CPU work; nothing idles on I/O.

This is structurally similar to Node.js's event loop or Go's goroutines — same idea, different shape. The actor model adds isolated mutable state per actor; each actor's state is private, modified only by that actor in response to messages. No locks, no race conditions, no memory model debates. The trade is verbose: every interaction is a message type with a handler.

03PayPal's deployment.

PayPal didn't rewrite the whole platform on Akka. They identified the highest-volume, most-I/O-bound path — payment authorisation — and rewrote that. The rewrite was Scala on Akka; everything else stayed on Java/Tomcat. The two systems coexisted; the actor-based service handled the hot path; the Java tier handled the rest.

This is the right pragmatic move: actor-model code requires a different mental model and toolchain. Imposing it on the entire engineering organisation would have been a multi-year project. Limiting it to a specific component contained the scope.

04The cost of switching.

Hiring

Scala + Akka engineers cost ~20–30% more than Java engineers and are harder to find. PayPal could absorb that cost; smaller teams cannot. The talent market is a real input to architectural choices; a brilliant technical decision that depends on hiring twenty Akka specialists isn't viable for most companies.

Debugging

Async, message-passing code has stack traces that don't read like procedural code. The "what called this" question is harder. Distributed tracing (OpenTelemetry, Zipkin) becomes non-optional. Hours that would have gone into feature work go into improving observability of the actor mesh.

Akka licensing

In 2022, Lightbend (Akka's commercial sponsor) relicensed Akka under BSL — no longer freely usable for commercial production. Companies on Akka had to either pay for a commercial license or migrate. Open alternatives (Pekko, an Akka fork) exist but the ecosystem fragmented. The PayPal story predates this shift; if you're starting today, weigh it. .NET's Akka.NET, Erlang/OTP, and Microsoft Orleans are the closest equivalents that remain freely usable.

06The lessons.

The concurrency primitive is the architecture

Thread-per-request and actor-model produce different ceilings on the same hardware. If your workload is I/O bound and your throughput is capped, the concurrency model is usually the cause. Async/await (modern C#, Node.js), goroutines (Go), actors (Akka, Orleans), virtual threads (JVM 21+) — all variations of the same insight.

Rewrite the hot path, not everything

PayPal rewrote one service, not the platform. The rest of the system kept running. Architectural-pattern changes are almost always wrong as platform-wide mandates; they're right as targeted interventions where the workload demands them.

Modern .NET makes most of this free

The PayPal story is from a pre-async-everywhere world. async/await in .NET (and Kotlin coroutines, modern Java's virtual threads) give you the I/O-non-blocking benefit without the actor-model rewrite. If you're starting today on a modern language, you already have most of what PayPal moved to. Pure actor models (Orleans, Akka.NET) are now justified only by additional benefits — location transparency, supervision trees, distributed state.

Watch the license

Akka's commercial relicense in 2022 caught many teams flat-footed. Stateful infrastructure choices have a vendor-dependency profile; check the licence before you build on it. Open alternatives age in and out of viability; build with replaceability in mind where you can.

Density is a cost line

50 VMs → 8 VMs is ~$25k–$50k/month of compute saved at production scale, plus the cascading savings (smaller cluster = simpler ops). The architecture review should price compute density as a feature, not just throughput.

"More hardware is the answer until the architecture is the problem. The trick is knowing which you've got."
