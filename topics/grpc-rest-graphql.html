Three protocols. Three conversations.

REST won the public-internet era. gRPC won the cluster-internal era. GraphQL won the BFF era — and lost the simple-CRUD era. LinkedIn cut 60% latency moving from JSON to Protobuf and paid a tooling bill nobody talks about. The honest choice depends on who's on the other side of the wire.

The problem in one line

"What format do we put on the wire?" looks like a tooling question. It's actually a question about who owns the contract.

The thesis

REST for public APIs and anything humans debug. gRPC for service-to-service inside a cluster. GraphQL for client-driven aggregation across multiple services. Mix all three — most production systems do.

01The contract is the protocol.

The wire format determines what the contract can express, how clients evolve, and who debugs at 3am. REST's resources align with HTTP verbs — natural for documents, awkward for actions. gRPC's service Methods() aligns with RPC — natural for invocations, awkward for caching. GraphQL's single endpoint with a typed query aligns with client-driven aggregation — natural for mobile BFFs, awkward for write-heavy command APIs.

None of these are wrong; each is right for a specific shape of conversation. The recurring bug is picking by hype ("we should be on gRPC") or by lowest common denominator ("REST is what everyone knows"). Pick by who's calling and what they need to express.

"REST is the protocol of the internet. gRPC is the protocol of the cluster. GraphQL is the protocol of the screen."

02Wire shapes.

Fig. 23Same business operation, three idioms. The wire size, the cacheability, and the contract surface are all consequences of the protocol choice.

REST: the universal protocol

HTTP verbs map to operations. Resources have URLs. Responses are JSON. Every client on Earth speaks it, every proxy caches it, every monitoring tool understands it. REST is the lowest-common-denominator protocol — and the lowest common denominator is exactly what you want for a public API. Stripe, Twilio, GitHub, every major SaaS publishes a REST API even when they have gRPC internally. The wire is bigger; the trust surface is universal.

gRPC: the cluster-internal protocol

Protobuf-encoded frames over HTTP/2. Generated client and server stubs in 11 languages. Streaming in both directions. ~10× smaller payloads than equivalent JSON; ~10× faster serialization. LinkedIn published a 2018 number — 60% latency reduction moving internal service-to-service from REST to Protobuf. The cost is the tooling: .proto files, codegen pipelines, no browser support without grpc-web, harder ad-hoc debugging.

GraphQL: the client-driven protocol

The client describes the shape of the response in the query; the server returns exactly that shape. No over-fetch, no under-fetch, no v2/v3 endpoint sprawl. Born at Facebook for mobile clients on slow networks; now powers most BFF (backend-for-frontend) layers at companies with multi-screen surfaces. The tradeoff is operational: every query is unique, HTTP caching doesn't apply, N+1 query problems are real, query complexity limits are a deliberate engineering activity.

03Schema, streaming, tooling.

Schema discipline by force vs by convention

gRPC and GraphQL require a schema (.proto file or SDL). Generated stubs and types make breaking changes loud at compile time. REST's schema (OpenAPI) is optional — most teams write it after the fact, and most clients don't enforce it. Schema-required protocols catch breaking changes; schema-optional protocols hope developers do it. The reality is mixed: with discipline, REST + OpenAPI is fine. Without discipline, gRPC and GraphQL make discipline non-negotiable.

Streaming

gRPC has first-class server-streaming, client-streaming, and bidirectional streaming over HTTP/2. GraphQL has subscriptions (over WebSocket or SSE). REST has SSE for server-streaming and not much else; for bi-directional, you reach for WebSockets (see Chapter 10). For high-fan-out push, gRPC is the most ergonomic, GraphQL subscriptions the most prevalent on the web.

Browser story

Browsers cannot speak gRPC directly — HTTP/2 frame access isn't exposed to JS. grpc-web is the workaround: a proxy translates gRPC-web's restricted protocol to grpc-native. Cloudflare, Envoy, and Linkerd all support it. The DX is worse than native; for browser clients, REST or GraphQL is the smoother path. Mobile and server clients have no such restriction.

N+1 and query complexity

GraphQL's killer footgun: a query that traverses many relations can fan out to many database calls per request. Use DataLoader (or HotChocolate's batching primitives) to coalesce reads within a single resolver execution. Set a maximum query depth and complexity score; reject queries above it. Without these, a malicious client can ship a single query that DOSes your database.

REST is not "no schema"

OpenAPI v3 with codegen (NSwag, Refit) gets you most of gRPC's tooling benefits over plain HTTP. ASP.NET Core's minimal APIs generate OpenAPI automatically. If you've been hand-rolling REST clients in 2026, you've been doing it wrong. Treat the OpenAPI spec as the contract; generate clients; review breaking changes in CI.

04In .NET.

Three of the same service. Minimal API for public REST, gRPC for the cluster, HotChocolate for the BFF. They all share the same domain layer — only the wire shape differs.

C#   RestEndpoints.cs (minimal APIs) 
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddOpenApi();

var app = builder.Build();
app.MapOpenApi();

app.MapGet("/v1/orders/{id:guid}", async (
    Guid id, IOrderService svc, CancellationToken ct) =>
{
    var o = await svc.GetAsync(id, ct);
    return o is null ? Results.NotFound() : Results.Ok(o);
})
.WithName("GetOrder")
.Produces<OrderDto>(200)
.Produces(404);

app.MapPost("/v1/orders", async (
    PlaceOrderRequest req, IOrderService svc, CancellationToken ct) =>
{
    var id = await svc.PlaceAsync(req, ct);
    return Results.Created($"/v1/orders/{id}", new { id });
})
.WithName("PlaceOrder")
.Produces(201);

app.Run();
proto   orders.proto + OrdersService.cs (gRPC) 
// orders.proto — the contract. Lives in a shared package, codegen-ed for every consumer.
syntax = "proto3";
package academy.orders.v1;

service Orders {
  rpc GetOrder(GetOrderRequest) returns (Order);
  rpc PlaceOrder(PlaceOrderRequest) returns (Order);
  rpc WatchOrder(GetOrderRequest) returns (stream OrderEvent);
}

message GetOrderRequest    { string id = 1; }
message PlaceOrderRequest  { string customer_id = 1; repeated LineItem items = 2; }
message LineItem           { string sku = 1; int32 qty = 2; }
message Order              { string id = 1; string status = 2; repeated LineItem items = 3; }
message OrderEvent         { string id = 1; string type = 2; google.protobuf.Timestamp at = 3; }

// OrdersService.cs — server-side. Generated base class is <Service>.<Service>Base.
public sealed class OrdersGrpcService : Orders.OrdersBase
{
    private readonly IOrderService _svc;
    public OrdersGrpcService(IOrderService svc) => _svc = svc;

    public override async Task<Order> GetOrder(
        GetOrderRequest req, ServerCallContext ctx)
    {
        var dto = await _svc.GetAsync(Guid.Parse(req.Id), ctx.CancellationToken)
            ?? throw new RpcException(new(StatusCode.NotFound, "order not found"));
        return new Order { Id = dto.Id.ToString(), Status = dto.Status };
    }

    // Server-streaming: push events to the caller as the order progresses.
    public override async Task WatchOrder(
        GetOrderRequest req, IServerStreamWriter<OrderEvent> stream, ServerCallContext ctx)
    {
        await foreach (var ev in _svc.WatchAsync(Guid.Parse(req.Id), ctx.CancellationToken))
            await stream.WriteAsync(new OrderEvent
            {
                Id = ev.OrderId.ToString(),
                Type = ev.Type,
                At = Timestamp.FromDateTimeOffset(ev.At)
            });
    }
}

// Program.cs adds:
builder.Services.AddGrpc();
app.MapGrpcService<OrdersGrpcService>();
C#   GraphQL via HotChocolate 
// dotnet add package HotChocolate.AspNetCore
using HotChocolate;
using HotChocolate.Types;

namespace Academy.Bff;

public sealed class Query
{
    public async Task<OrderDto?> GetOrder(
        Guid id, [Service] IOrderService svc, CancellationToken ct)
        => await svc.GetAsync(id, ct);
}

// Resolver for the nested Product field — uses DataLoader to batch lookups.
public sealed class LineItemType : ObjectType<LineItemDto>
{
    protected override void Configure(IObjectTypeDescriptor<LineItemDto> d)
    {
        d.Field("product")
         .Resolve(async (ctx, ct) =>
         {
             var item = ctx.Parent<LineItemDto>();
             var loader = ctx.DataLoader<ProductByIdBatchLoader>();
             return await loader.LoadAsync(item.Sku, ct);
         });
    }
}

public sealed class ProductByIdBatchLoader : BatchDataLoader<string, ProductDto>
{
    private readonly IProductService _products;
    public ProductByIdBatchLoader(IBatchScheduler s, IProductService p,
        DataLoaderOptions o) : base(s, o) { _products = p; }

    protected override async Task<IReadOnlyDictionary<string, ProductDto>>
        LoadBatchAsync(IReadOnlyList<string> skus, CancellationToken ct)
        => (await _products.GetManyAsync(skus, ct))
            .ToDictionary(p => p.Sku);
}

// Program.cs adds:
builder.Services
    .AddGraphQLServer()
    .AddQueryType<Query>()
    .AddType<LineItemType>()
    .AddDataLoader<ProductByIdBatchLoader>()
    .SetPagingOptions(new() { MaxPageSize = 50, DefaultPageSize = 20 })
    .AddMaxExecutionDepthRule(8);

app.MapGraphQL("/graphql");

05The detail.

gRPC over HTTP/2 has a connection-balancing footgun

HTTP/2 multiplexes everything down one connection. Behind a standard K8s ClusterIP service, one client → one connection → one pod. The pod gets all the load. Use headless services + client-side load balancing, or an L7 proxy / service mesh (see Chapter 15). The "90% of traffic on one pod" symptom is universal until fixed.

GraphQL needs cost analysis from day one

A naive GraphQL endpoint can be DOSed by a single deep query. HotChocolate, Apollo Server, and most production servers ship cost/complexity analysis: each field has a cost, queries above a threshold are rejected. Configure it before exposing the endpoint, not after the first incident.

REST + OpenAPI is more rigorous than people give it credit for

Minimal APIs generate OpenAPI. NSwag generates typed C# clients from OpenAPI. Spectral lints the OpenAPI for breaking changes in CI. You can have schema-required REST without writing a line of contract code. The "REST has no schema" framing is a 2014 framing.

grpc-web is a real, useful escape hatch

Browsers can't speak native gRPC, but grpc-web (proxied through Envoy or Connect) gives you 95% of the type-safety with a slightly heavier wire format. For an internal admin tool, grpc-web saves you a separate REST surface; for a public website, REST or GraphQL is still the smoother default.

Don't use GraphQL for write-heavy APIs

GraphQL mutations work but lose the protocol's main advantage. Each mutation is its own RPC; you don't benefit from client-driven query shape on writes. Mix freely: GraphQL for reads, REST/gRPC for writes, on the same service. The "GraphQL all the way" purity has cost most teams that tried it.

"REST is a contract for humans. gRPC is a contract for machines. GraphQL is a contract for product managers."

07Decision guide.

Public API for partners and integrations

REST + OpenAPI. JSON over HTTP. Cacheable, debuggable, universal. Don't try to ship gRPC to your customers' procurement teams.

Service-to-service inside a cluster

gRPC. The 10× wire-size win pays for itself; the schema-required discipline keeps the contract honest. Use headless services for LB.

Mobile + web BFF aggregating multiple services

GraphQL with HotChocolate. One endpoint, client-driven shape, DataLoader for N+1, complexity caps. The BFF's natural shape.

High-throughput stream of events to a client

gRPC server-streaming if the client is server/mobile; SSE if browser (see Chapter 10). Both win over polling REST.

Write-heavy command API

REST with idempotency keys (Chapter 08). GraphQL mutations are fine but you lose the protocol's edge.

You have one of each already

That's the production reality. REST for external, gRPC for internal, GraphQL for the BFF. Same domain layer behind all three.

Three things people get wrong

One — picking gRPC for the public API. Your customers don't want a Protobuf SDK. They want curl. REST is the right boundary at the edge of your organisation, every time.

Two — picking GraphQL for everything. Mutations don't benefit from client-driven shape; the operational cost of N+1 protection and cost analysis is real. Use GraphQL where its shape pays — read-heavy multi-source aggregation.

Three — gRPC behind a standard K8s service. HTTP/2 multiplexing concentrates traffic on one pod. Use headless services with client-side LB, or an L7 mesh.
