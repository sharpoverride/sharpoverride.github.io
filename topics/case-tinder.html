Academy /Case Studies/ Tinder

Case study Years 2012 → today Scale ~75M users · 1.6B swipes/day Stack Elasticsearch · MongoDB · Redis · Go

One billion swipes a day.

Tinder's swipe gesture is the world's most-used UI mechanism. Underneath: a geo-indexed candidate pool, a match-detection engine that fires on mutual right-swipes, and a write-heavy load most apps never see.

The pattern

Show next candidate → user swipes → record swipe → if other party already right-swiped you, create a match. Repeat ~30 times per session.

The hot path

Almost all swipes are no-match. The expensive lookup ("did the other side right-swipe me?") happens once per match — <1% of swipes.

01Discovery.

"Who shows up next on the deck?" — a recommendation problem. Inputs: location, age range, gender preferences, recent activity, ML-derived attractiveness signals. Elasticsearch is the candidate index; geo + age + gender filters, score by recommendation model, return top 50. User pre-fetches the next batch while swiping the current one.

"The hard problem isn't the swipe. It's making sure the next face is worth swiping on."

02The swipe write path.

1.6B swipes/day = ~20k writes/sec average; ~80k at peak. Each swipe is a small write: (user_id, target_id, direction, timestamp). Per-user partitioned: one user's swipes live on one shard. Read pattern is rare per swipe (mostly write-only); the heavy reads are during match detection.

The data is somewhat ephemeral — months-old swipes don't usually affect future discovery (the swiped person may be inactive, may have changed photos, may have left the app). TTL on cold swipes keeps the dataset bounded.

03Match detection.

When Alice right-swipes Bob, check: has Bob already right-swiped Alice? One lookup against Bob's "people I liked" index. If yes, it's a match; create a match row; send push notifications to both; open a chat.

The asymmetric data

The "did Bob right-swipe Alice" lookup is against Bob's data, not Alice's. Cross-shard read. Mitigation: a separate "right-swiped-by" index per user, denormalised. Two writes per right-swipe (Alice's "I liked Bob"; Bob's "Alice liked me"); one read per match check. Trades write for read.

Push notification

Match → notification within seconds. Push to both devices via APNs/FCM. The user engagement hook is timing; a match they hear about an hour later doesn't generate a conversation. Notification is the conversion event.

05The lessons.

Denormalise for hot reads

"Right-swiped-by-me" and "right-swiped-me" are different indexes; one's cheap per write, the other's cheap per read. Tinder maintains both. Storage costs nothing; latency costs everything.

Pre-fetch the next batch

While the user swipes one, fetch the next 5. Mobile UI feels instant; the network is hidden.

TTL the cold data

Swipes from a year ago rarely matter. Trim them; keep the dataset bounded. Same pattern as TTL on Redis keys.

The recommendation is the product

Showing better candidates beats showing more candidates. ML model + ranking is the engineering investment that compounds.

"Tinder's tech is unremarkable except for the candidate ranking. The candidate ranking is the entire product."
