Case study Years 2006 → today Scale 3B docs · 1B users Stack Operational Transform (OT) · WebSocket · Spanner

Twenty people typing in the same paragraph.

Google Docs' real-time multiplayer document editing was a 2007 product miracle. The underlying technique — Operational Transformation — predated Docs and is still the production answer for many collaborative editors. CRDTs are the modern alternative; both solve the same problem; both have tradeoffs.

The problem

Two editors type at the same position simultaneously. Whose change wins? Without coordination — both. Convergence is the goal.

OT vs CRDT

OT (operational transformation): the server transforms ops to apply them consistently. CRDT: ops are designed to be commutative.

01The convergence problem.

Alice and Bob are editing the same paragraph. Alice inserts "X" at position 5. Bob inserts "Y" at position 5. Naively, both ops sent to each other; each applies the other's; final state diverges.

The convergence requirement: after every editor's ops have been delivered to every other editor, in any order, all editors converge to the same document. Without this, your "real-time collaboration" is "real-time divergence."

"Real-time collab isn't about speed. It's about converging on the same answer."

02Operational transformation.

Each edit is an op: insert("X", pos: 5), delete(pos: 7, len: 1). When ops arrive out of order, the server transforms them. If Alice's insert at 5 came before Bob's insert at 5, Bob's op gets transformed to insert at 6 (because Alice's "X" shifted everything by one). The transformation function depends on the data type; for text, it's well-studied.

Server is the truth

Google Docs has a central server. Clients send ops; server linearises them (gives each a sequence number); server transforms incoming ops against any ops that landed since the client's last known sequence; broadcasts the transformed ops to all clients. The server is the convergence point.

Why OT is hard

The transformation functions for text editing are subtle. "insert and delete interact in 9 different ways depending on relative positions and lengths." Implementing them correctly is a multi-year project; Google has spent serious engineering effort on Docs' OT engine.

03CRDTs — the modern alternative.

Conflict-free Replicated Data Types. Operations are designed to be commutative by construction: the order in which they're applied doesn't change the outcome. No transformation needed. Each replica can apply ops as they arrive; convergence is mathematically guaranteed.

For text editing: tree-based CRDTs (RGA, Yjs, Y-CRDT, Automerge). Each character has a stable identifier in a partial order; ops insert/delete by identifier, not by integer position. No transformation; the IDs encode the ordering.

CRDTs vs OT — the practical comparison

AspectOTCRDT
Server requiredYes (central transformer)No (peer-to-peer works)
Complexity to implementVery highModerate
Memory per docBoundedGrows with edits (compression mitigates)
Offline editsHardNative

Modern collaborative editors — Figma, Linear, Notion's newer surfaces, Tldraw — use CRDTs. Google Docs still uses OT because the rewrite cost outweighs the benefits at their scale. The 2007 design has been load-bearing for nearly two decades.

05The lessons.

Pick the convergence mechanism early

OT and CRDT have different implications for offline support, server architecture, and op shapes. Choosing late = rewrite.

CRDTs are the modern default

For new collaborative editors in 2026, CRDTs (Yjs, Automerge, Loro) are the right starting point. Offline-first; peer-to-peer-capable; mature libraries.

OT can outlive its replacement

Google Docs still uses OT. The right answer for a new product isn't always "rewrite legacy products"; Docs ships.

Presence is a separate problem

Cursors, selection ranges, who-is-editing-what — different concerns from the convergence of edits. Usually solved via a separate ephemeral channel (Awareness in Yjs land).

"Real-time collaboration looks magical because the underlying algorithm has to be."
