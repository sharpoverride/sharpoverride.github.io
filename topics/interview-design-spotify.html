Design Spotify.

A music streaming service plus recommendations. The streaming half looks like a simpler YouTube; the recommendation half is the actual product. Where you spend your time depends on what the interviewer signals.

Anchor numbers

~600M users · 100M+ tracks · ~250 KB/min audio at 128 kbps · ~50B plays/month · catalog roughly static (curated, not user-uploaded).

The two halves

Streaming (small catalog, predictable access pattern, CDN-friendly). Recommendations (the secret sauce — Discover Weekly, Daily Mix, Now Playing radio).

Out of scope (probably)

Podcasts, video, social features. Pin scope; play to your strengths.

The catalog is tiny by streaming-service standards: ~1 PB total, compared to YouTube's PB/day. The whole catalog fits on CDN. That changes the architecture significantly — there is no "cold storage tier" of long-tail content.

03High-level architecture.

Catalog ingestion: labels submit tracks; ingestion service validates, transcodes to multiple bitrates, writes to object storage, pushes to CDN, indexes metadata in search. Playback: client requests track → metadata API returns CDN URL → client streams via HLS or progressive download. Recommendations: an offline ML pipeline computes per-user playlists nightly; an online ranker re-ranks the top of feeds in real time.

04Deep dive — streaming.

CDN with full catalog at most POPs

The catalog is small enough (~1 PB) that Spotify can keep most of it on CDN. Long-tail tracks pre-cached as well: when a user requests a track not at the edge, fetch from origin and cache. The miss rate is low because the catalog is closed (no user uploads), and popularity is power-law — top 10k tracks dominate but the next 1M still account for serious volume.

Predict the next track

Spotify knows what you're playing next — it's in the queue. Pre-fetch the next track's first chunk while the current track is playing. Gapless playback; sub-100ms transition. Same trick on shuffle: pre-fetch the 1–2 most likely candidates.

Offline downloads

Premium users can download tracks for offline play. Downloads are DRM-protected; the encryption key is bound to the user's subscription. Periodic re-validation: the app calls home every ~30 days; if the subscription lapsed, the keys expire and the downloads stop working. Caching tier for downloads is the same CDN.

05Deep dive — recommendations.

Two-stage system

Offline candidate generation (batch, ~daily): for each user, generate a few thousand candidate tracks using collaborative filtering, content-based features, and embedding nearest-neighbour. Store in user_candidates table. Online ranking (real-time): when the user opens the app, take the candidates and re-rank using fresher signals (current session, time of day, device). The offline step does the heavy ML; the online step does the personalisation.

Embeddings everywhere

Each track is a vector in some learned embedding space (~128 dims). Each user is a vector — typically the average of recent-play track embeddings. Similarity is cosine distance. Tracks "near" the user's vector are candidates. The embeddings are computed by a model that ingests listening history, playlist co-occurrence, audio features, label metadata.

Discover Weekly

The flagship — 30 tracks personalised weekly. Generated by a batch pipeline: for each user, take their recent listening, find tracks "near" in embedding space that the user has not heard, filter out content they're predicted to dislike, sample diverse-but-coherent set. The trick is balancing familiarity with discovery; pure nearest-neighbour gives you songs that sound like what you already love (boring); too much exploration gives you songs you'll skip.

Feedback loop

Every play, skip, like, save is a signal. Recent signals weighted heavier; a skipped track within 5 seconds is strong negative; a track completed is strong positive. These flow into the next day's batch computation and the same day's online ranking. The system that doesn't have this feedback loop produces recommendations that go stale within a week.

06Scale & bottlenecks.

Recommendation compute is the cost line

Generating candidates for 600M users daily is a large batch job. Approximately 600M × 1k candidates = 600B candidate scores per day. Done on a Hadoop/Spark/Beam cluster. Scaling is by adding nodes; the bottleneck is the embedding index lookups (use Faiss or ScaNN for ANN search).

Cold-start is the hard product problem

A user with no listening history has no embedding. Cold-start strategy: ask them genres/artists at signup; fall back to popular-by-region. After ~10 plays the model has enough signal to personalise.

Tradeoffs to mention

Catalog freshness vs cache hit rate: new releases need to be pushed to CDN promptly. Storage strategy: 3 bitrates × 100M tracks = ~3 PB total; could go higher with lossless tier (which Spotify added in 2024). Per-region catalog: licensing means some tracks are unavailable in some countries; metadata API filters before returning URLs.

Open questions

Live audio (concert streams, radio): different bitrate, lower latency. Social features (friends listening, shared playlists): adds a fanout problem. Podcasts: different content shape, longer durations, ad insertion — Spotify built a separate ingestion pipeline.

"Streaming is the table stakes. The product is what plays next."
