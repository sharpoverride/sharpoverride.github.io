Chapter 16 Difficulty ★★★★☆ Read ~13 min Stack DistributedLock · Redis · ZooKeeper · etcd

Exactly one pod does the work.

A scheduled job. A queue drain. A cache warm-up. The work has to happen, but only on one of the N pods running. The trick is picking the one — and keeping the answer correct when the picker crashes mid-pick.

The problem in one line

"Only one of these should run" is easy on one pod. It is the hardest distributed-systems problem to get correct on three.

The thesis

Use a hardened lock store. SQL Server's sp_getapplock or a Redis-backed lock via Medallion.Threading covers 95% of needs. Reach for Raft (etcd, ZooKeeper) only when you have correctness invariants that distributed locks can't carry.

01Why one of N is hard.

Three pods of the same service. A scheduled job runs every minute. Without coordination, all three pods run it, three times the work, three times the bill, and if the job sends an email, three emails. The naive fix — "let one pod run it" — collapses the moment you ask which one, and what happens when that one dies.

The problem decomposes into four sub-problems. Election: pick a leader. Liveness: detect when the leader dies so a new one can take over. Safety: never have two leaders at the same time. Recovery: when the new leader takes over, make sure the old one cannot still be doing work. Safety and recovery are the ones that produce the famous bugs.

"Two leaders is worse than no leader. No leader does nothing; two leaders disagree."

02Locks, leases, Raft.

Three tools, increasing in cost and correctness guarantees.

Fig. 16Every leader-election system has these three states. What differs is the lock store underneath — and whether it tolerates partition.

A distributed lock with a TTL is enough most of the time

"Get a key, hold it, do work, release it. If the holder dies, the TTL releases it for the next claimant." That's the model behind sp_getapplock in SQL Server, Redis SET NX EX, Azure Blob leases, Consul sessions. For workflows where the worst case of "two pods briefly run the job" is acceptable — duplicate emails, double-decremented counters that can be reconciled — this is the right tool. It's cheap, idiomatic, and the failure mode is bounded.

A Raft lease when "two leaders" is unsafe

etcd and ZooKeeper give you linearisable leases. The cluster's Raft consensus guarantees that the lease can be held by at most one client at any moment in the cluster's logical time. This is what you need when "two leaders" corrupts data — a single-writer database replica, a Kafka partition leader, the coordinator of a distributed transaction. The cost is operating a Raft cluster of 3 or 5 nodes; the payoff is correctness under partition.

Most application-level workloads don't need Raft. The instinct to reach for ZooKeeper is the systems-engineer's equivalent of the over-architect's instinct to reach for microservices: it's technically correct, but the cost is paid forever for a problem you don't have.

03Fencing and split-brain.

Martin Kleppmann's 2016 article "How to do distributed locking" is required reading: the lock is not the safety mechanism. Consider this sequence. Pod A acquires the lock with a 30s TTL. Pod A pauses for 45 seconds (GC pause, kernel scheduling, VM live-migration). Pod A's lock expired at 30s. Pod B acquires the same lock at 35s. Pod B starts working. Pod A's pause ends at 45s. Pod A is still convinced it holds the lock. Both pods do the work. The lock did nothing.

Locks bound the optimistic case, not the worst case. Anything that pauses the holder past the TTL breaks the safety guarantee.

Fencing tokens

The fix is a fencing token: every acquire returns a monotonically increasing number. Pod A gets token 42, pod B gets token 43. The downstream resource — the database, the storage layer — refuses any write whose token is older than the highest it has seen. Pod A's stale write with token 42 is rejected by the database, because the database has already accepted a write with token 43. The token is the safety mechanism; the lock is the optimisation.

Fencing requires the downstream to participate. etcd and ZooKeeper give you the token natively (the lease's mod-revision). SQL applocks give you the connection's transaction ordering. Redis SETNX gives you nothing — the famous "Redlock is broken" debate hinges on whether Redis alone can produce a fencing token, and the answer is no. If your workload needs fencing, don't pick Redis as the lock store.

When two leaders is acceptable

Be honest about the worst case. A scheduled job that sends a marketing email and the worst case is "user gets two emails" → a Redis-based lock with no fencing is fine. A scheduled job that issues refunds → no, you need fencing or idempotency keys (Chapter 08). A single-writer to a distributed database → Raft lease and fencing tokens, no exceptions. Pick the lock primitive by the worst case of split-brain, not by the steady-state.

04In .NET.

The most pragmatic .NET library is Medallion.Threading.DistributedLock: a single abstraction over SQL Server applocks, Postgres advisory locks, Redis Redlock, and ZooKeeper. The leader hosted service runs the work; BackgroundService coordinates the lease loop.

C#   LeaderHostedService.cs 
// dotnet add package DistributedLock.SqlServer
// (or DistributedLock.Redis, DistributedLock.Postgres, DistributedLock.ZooKeeper)
using Medallion.Threading;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;

namespace Academy.Election;

public abstract class LeaderHostedService : BackgroundService
{
    private readonly IDistributedLockProvider _locks;
    private readonly ILogger _log;
    private readonly string _lockName;

    protected LeaderHostedService(IDistributedLockProvider locks,
        ILogger log, string lockName)
    { _locks = locks; _log = log; _lockName = lockName; }

    protected override async Task ExecuteAsync(CancellationToken stop)
    {
        var myId = $"{Environment.MachineName}-{Guid.NewGuid():N}";
        while (!stop.IsCancellationRequested)
        {
            IDistributedSynchronizationHandle? handle = null;
            try
            {
                // Try to acquire. If another pod holds it, wait up to 5s, then loop.
                handle = await _locks.AcquireLockAsync(
                    _lockName,
                    timeout: TimeSpan.FromSeconds(5),
                    cancellationToken: stop);

                if (handle is null)
                {
                    _log.LogDebug("{Me}: another pod is leader, will retry", myId);
                    continue;
                }

                _log.LogInformation("{Me}: ELECTED as leader for {Lock}", myId, _lockName);

                // Run the leader workload until we're cancelled or the lock is lost.
                using var linked = CancellationTokenSource.CreateLinkedTokenSource(
                    stop, handle.HandleLostToken);
                try
                {
                    await RunAsLeaderAsync(linked.Token);
                }
                catch (OperationCanceledException)
                {
                    _log.LogWarning("{Me}: lost leadership or shutting down", myId);
                }
            }
            catch (Exception ex) when (!stop.IsCancellationRequested)
            {
                _log.LogError(ex, "{Me}: error in leader loop", myId);
                await Task.Delay(TimeSpan.FromSeconds(2), stop);
            }
            finally
            {
                if (handle is not null) await handle.DisposeAsync();
            }
        }
    }

    protected abstract Task RunAsLeaderAsync(CancellationToken ct);
}

// A concrete leader workload — drain a queue, run a cron, ingest from a single stream.
public sealed class CronLeader : LeaderHostedService
{
    private readonly IServiceProvider _sp;

    public CronLeader(IDistributedLockProvider l,
        ILogger<CronLeader> log, IServiceProvider sp)
        : base(l, log, "cron:leader") { _sp = sp; }

    protected override async Task RunAsLeaderAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            using var scope = _sp.CreateScope();
            var work = scope.ServiceProvider.GetRequiredService<IDailyReportJob>();
            await work.RunAsync(ct);
            await Task.Delay(TimeSpan.FromMinutes(1), ct);
        }
    }
}

Wire-up: pick the lock provider that matches your existing infra. SQL Server is the lowest-cost choice if you have it; Redis is the lowest-latency; ZooKeeper is the strongest-correctness.

C#   Program.cs 
var builder = Host.CreateApplicationBuilder(args);

// Choose ONE provider. Backed by the same IDistributedLockProvider interface.
builder.Services.AddSingleton<IDistributedLockProvider>(_ =>
    new SqlDistributedSynchronizationProvider(
        builder.Configuration.GetConnectionString("Locks")!));

builder.Services.AddScoped<IDailyReportJob, DailyReportJob>();
builder.Services.AddHostedService<CronLeader>();

builder.Build().Run();

05The detail.

Observe the HandleLostToken

Every IDistributedSynchronizationHandle exposes a HandleLostToken that fires if the library decides leadership has been lost (TTL not renewed, connection lost, etc.). Link it to your work's cancellation token so the workload aborts immediately when leadership ends — don't keep writing after you've lost.

Workload must be idempotent or fenced

Even with a perfect lock, network pauses and pod restarts mean the leader can change mid-batch. Every operation the leader performs must be idempotent or must include a fencing token that the downstream rejects on staleness. If neither, you're betting on never having a pause longer than the TTL — and you'll lose that bet.

Pick the lock store you already operate

Adding a Redis cluster just for leader election is over-investment. SQL Server's sp_getapplock works fine if you have SQL. Postgres advisory locks if you have Postgres. Azure Blob lease if you're on Azure. The right answer is almost always the data store that already has a 24/7 oncall rotation.

Heartbeats too short = elections too often

A 1-second lease with a 500ms renewal interval means leadership churns on any 1.5s network blip. A 30-second lease with a 10s renewal interval gives you stable leadership at the cost of slower failover. Match the lease duration to your failover SLO: SLO=60s of duplicate work? 30s lease. SLO=10s? 5s lease and a tight renewal loop.

Don't elect for things that don't need it

"Only one pod should drain this queue" — actually the broker handles that for you (Kafka partition assignment, Service Bus session locks). "Only one pod should be the cron" — Kubernetes CronJob already runs once. Reach for leader election when the platform doesn't have a primitive for the thing you're trying to do.

"A leader election is a state machine you don't get to test in production until production tests it."

07Decision guide.

Cron job, two-leaders briefly is OK

DistributedLock on whichever data store you already operate. SQL applock is the cheapest correct answer.

Queue drain (Kafka, Service Bus)

Don't elect. The broker's partition / session assignment is the election. Subscribe and let the broker pick.

Single-writer to a shared resource

Raft lease (etcd or ZooKeeper) plus fencing tokens at the downstream. The Redlock debate exists; this is the side of it without the debate.

On K8s with no Redis

etcd lease via the K8s leader-election library, or DistributedLock with Postgres if you have it. Don't add Redis just for this.

On Azure with nothing else

Blob lease. Simple, native, cheap, in the SLA you already pay for. 15-second lease, renew every 5.

"It just needs to be one of the pods"

K8s StatefulSet with one replica, plus a recovery policy. Single-pod is the right shape if scale isn't needed.

Three things people get wrong

One — assuming the lock is the safety mechanism. GC pauses and VM live-migrations break locks the same way as a partition. Fencing tokens at the downstream are what actually keep two writers from colliding.

Two — operating Raft for non-Raft problems. ZooKeeper is wonderful and expensive. Use it when the data corruption from split-brain is unacceptable, not because "leader election sounds like a Raft problem."

Three — renewing too eagerly or too lazily. Renewing every 100ms thrashes the lock store; renewing every TTL/2 with no jitter creates synchronised renewals. Renew at TTL/3 with jitter; you'll live a long life.
