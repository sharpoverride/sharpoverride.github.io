Ten bits per item. No false negatives.

A bloom filter answers one question — "have you maybe seen this before?" — at one percent the memory of a hash set. Cassandra uses one per SSTable. Chrome's safe-browsing list ships one to your laptop. Every URL shortener relies on one. The math is older than you are; the use cases keep multiplying.

The problem in one line

"Is this key on disk?" is expensive. "Is this key DEFINITELY NOT on disk?" can be answered from a bit array smaller than your terminal window.

The thesis

Trade exactness for size. A bloom filter says "no" with certainty and "maybe" with a tuneable false-positive rate. Use it as the first hop in any read path where the negative case dominates.

01The negative case.

A URL shortener generates a candidate slug. Is it taken? Almost certainly not. The keyspace is huge, the existing set is finite. A direct database lookup answers "no" 99% of the time and costs you 5ms of round-trip plus an index seek. Multiply by a billion candidate generations and you've spent six figures on the word "no". The same shape repeats everywhere: spam URL check, cached password leak check, "have I emailed this lead before", "is this row in any SSTable on this disk" — they all dominate on negatives.

A hash set would solve it, but a hash set of a billion 64-byte URLs is 64 GB. You can't ship 64 GB to each pod. A bloom filter holds the same billion entries in 1.2 GB at a 1% false-positive rate. The negatives are confirmed for free; the 1% of "maybe" are passed through to the real database lookup. Net effect: 99× fewer database hits, at no correctness cost.

"The cheapest read is the one that says ‘definitely not’ before you touch the disk."

02How the bits encode the set.

A bloom filter is a bit array of length m and a set of k independent hash functions. To add an item, hash it k ways, set the k resulting bits to 1. To check, hash and read the bits; if any is 0, the item is definitely not in the set. If all are 1, the item is probably in the set — could be a real hit, could be a collision.

Fig. 14k hash functions decide k bit positions. Insert flips them; lookup reads them. Any zero bit means definitely not in the set.

Why no false negatives

If an item was inserted, every one of its k bits was set to 1. Subsequent inserts can only turn other bits from 0 to 1; they cannot turn 1 to 0. So a lookup for the inserted item reads k ones — every time, forever. The false-negative rate is zero. This is the property that earns the bloom filter its place in the read path. A negative answer is a real negative; the database doesn't need to confirm it.

Why false positives exist

Two items can hash to the same k bits, or a checker can land on k bits that were each set by different other items. The bloom filter cannot tell the difference between "you inserted me" and "the bits that would prove me were collectively set by other inserts." A false positive means: pass the query through to the actual database lookup. Correctness is preserved; you just did the work you were trying to skip.

03Sizing it correctly.

Three numbers: n (items you'll insert), p (acceptable false-positive rate), m (bits) and k (hash functions). The classic formulas:

m = −(n · ln p) / (ln 2)²

k = (m / n) · ln 2

For 1 million items at 1% FP rate: m ≈ 9.6 Mbits ≈ 1.2 MB; k ≈ 7. For 1 billion items at the same rate: m ≈ 9.6 Gbits ≈ 1.2 GB; k ≈ 7. Memory grows linearly with n; k stays roughly constant in the useful range.

Pick p from the cost ratio

The false-positive rate is a knob, not a fixed constant. Pick it from (cost of the unnecessary downstream lookup) ÷ (memory cost of additional bits). If a database hit is 5ms and your downstream is provisioned for 1k RPS, a 1% FP rate is fine — that's 10 spurious lookups per second per pod. If the downstream is provisioned for 100k RPS and a hit is 50ms, you want closer to 0.01% — pay the extra megabytes to save the seconds.

You cannot remove items

A standard bloom filter doesn't support deletion. Removing an item would mean clearing its k bits — but those bits may belong to other items, and clearing them would give those other items false negatives, which breaks the guarantee. Three workarounds: counting bloom filter (each "bit" is a small counter; decrement on delete — costs 4× memory), scalable bloom filter (start a new filter when full; query both — useful for append-only workloads), cuckoo filter (a different probabilistic structure that supports deletion natively).

The cardinality you committed to matters

The filter is sized for n. Insert 10× n and the FP rate climbs catastrophically — the bit array saturates with 1s and almost every lookup returns "maybe." Plan for growth; oversize on creation; or use a scalable bloom filter that adds layers as you exceed the original cardinality. Capacity planning is the part everybody forgets until the filter degrades and nobody knows why the cache hit rate is suddenly 0%.

04In C#.

A from-scratch bloom filter using xxHash for cheap, well-distributed hashes. One BitArray. Single-pod, in-memory. ~80 lines. Ship it as a typed dependency.

C#   BloomFilter.cs 
// dotnet add package System.IO.Hashing  -- for XxHash3
using System.Collections;
using System.IO.Hashing;
using System.Text;

namespace Academy.Bloom;

public sealed class BloomFilter
{
    private readonly BitArray _bits;
    private readonly int _k;
    private readonly int _m;

    public BloomFilter(int expectedItems, double falsePositiveRate = 0.01)
    {
        if (expectedItems <= 0)        throw new ArgumentOutOfRangeException(nameof(expectedItems));
        if (falsePositiveRate <= 0 || falsePositiveRate >= 1)
            throw new ArgumentOutOfRangeException(nameof(falsePositiveRate));

        // m = -n * ln(p) / (ln 2)^2 — round up to the nearest 8 for byte alignment
        _m = (int)Math.Ceiling(-expectedItems * Math.Log(falsePositiveRate) / (Math.Log(2) * Math.Log(2)));
        _m = (_m + 7) & ~7;
        // k = (m / n) * ln 2 — clamp to a sensible band
        _k = Math.Max(1, (int)Math.Round((_m / (double)expectedItems) * Math.Log(2)));
        _bits = new BitArray(_m);
    }

    public void Add(ReadOnlySpan<byte> key)
    {
        foreach (var idx in Indices(key)) _bits[idx] = true;
    }
    public void Add(string s) => Add(Encoding.UTF8.GetBytes(s));

    public bool MightContain(ReadOnlySpan<byte> key)
    {
        foreach (var idx in Indices(key))
            if (!_bits[idx]) return false;  // any zero = definitely not in set
        return true;                       // all ones = probably in set
    }
    public bool MightContain(string s) => MightContain(Encoding.UTF8.GetBytes(s));

    // Kirsch–Mitzenmacher trick: simulate k hash functions from two real ones.
    // XxHash3 with two different seeds gives statistically independent h1/h2.
    private IEnumerable<int> Indices(ReadOnlySpan<byte> key)
    {
        var h1 = XxHash3.HashToUInt64(key, seed: 0xC0FFEEUL);
        var h2 = XxHash3.HashToUInt64(key, seed: 0xDEADBEEFUL);
        for (var i = 0; i < _k; i++)
            yield return (int)(unchecked(h1 + (ulong)i * h2) % (ulong)_m);
    }

    // Estimate of current false-positive rate based on bits set.
    public double EstimatedFalsePositiveRate()
    {
        var setBits = 0;
        foreach (bool b in _bits) if (b) setBits++;
        var fraction = setBits / (double)_m;
        return Math.Pow(fraction, _k);
    }

    public (int M, int K) Shape => (_m, _k);
}

For a multi-pod world, an in-process bloom filter won't do — each pod sees a different state. The fix is RedisBloom, the official Redis module that provides server-side bloom filters with the same constant-time guarantees and a few extras (count-min sketch, cuckoo, top-k). One filter, every pod hits the same state.

C#   RedisBloomClient.cs 
// dotnet add package StackExchange.Redis
using StackExchange.Redis;

namespace Academy.Bloom;

public sealed class RedisBloomClient
{
    private readonly IDatabase _db;
    private readonly RedisKey _key;

    public RedisBloomClient(IConnectionMultiplexer mux, string filterName)
    { _db = mux.GetDatabase(); _key = filterName; }

    // One-time setup. Idempotent at the Redis side (re-running is a no-op).
    public async Task EnsureCreatedAsync(long capacity, double errorRate = 0.01)
    {
        // BF.RESERVE key error_rate capacity
        try
        {
            await _db.ExecuteAsync("BF.RESERVE", _key, errorRate, capacity);
        }
        catch (RedisServerException ex) when (ex.Message.Contains("already exists"))
        { /* idempotent */ }
    }

    public async Task<bool> AddAsync(string item)
        => (bool)(await _db.ExecuteAsync("BF.ADD", _key, item));

    public async Task<bool> MightContainAsync(string item)
        => (bool)(await _db.ExecuteAsync("BF.EXISTS", _key, item));

    // Batch versions exist (BF.MADD / BF.MEXISTS). Use them — 10× throughput.
}

05The detail.

Two hashes, k indices: the Kirsch–Mitzenmacher trick

Computing seven independent hash functions is expensive. The 2006 Kirsch–Mitzenmacher paper showed that g_i(x) = h1(x) + i · h2(x) is statistically indistinguishable from k independent hashes, as long as h1 and h2 are themselves independent. Every production bloom filter uses this trick. Two hashes, k indices, constant time. The sample code above does it explicitly.

Don't use HashSet<T> for "large" sets

"How large?" you ask. The crossover is around 100k items. Below that, a HashSet<string> is faster and uses less memory than a bloom filter, because the per-string overhead dominates. Above 1M items, the bloom filter wins on memory by an order of magnitude. Between 100k and 1M, profile both — the right answer depends on item size and FP requirement.

Cuckoo filter when you need delete

If you need to delete items, a counting bloom filter costs 4× memory; a cuckoo filter costs ~1.5× and supports delete natively. Cuckoo is also slightly faster on lookup (one cache line vs k cache lines). The downside: cuckoo has an insertion failure mode if it overfills, which a bloom filter doesn't. Pick cuckoo when delete is a requirement, bloom when append-only is the access pattern.

Don't ship the filter; ship the bytes

A bloom filter serializes to a fixed-size byte array. Persist it to disk, ship it to clients, mmap it. Cassandra's bloom filters live on disk next to the SSTable, paged in on demand. Chrome's safe-browsing filter is a downloaded byte array that the browser checks against locally before consulting the cloud. The bit array is the entire artefact.

Monitor saturation

The estimated false-positive rate climbs as the filter fills. Emit setBits / m as a gauge — when it exceeds the value you sized for, the filter is overfull and the FP rate has degraded. This is the single most useful metric you can ship for a bloom filter and the one nobody adds until the cache hit rate cratered last Friday.

"The bit array is the entire object. The math is the entire algorithm. Everything else is plumbing."

07Decision guide.

URL-shortener slug availability check

RedisBloom shared across pods. Size for 10× current cardinality at 0.1% FP. Pre-warm from the existing slug table on deploy.

Cache-skip pattern ("is this key on disk?")

In-process bloom per pod, rebuilt on cold start from the database's distinct keys. Cassandra and Postgres ship this pattern internally.

"Have we seen this email/phone before?"

Shared RedisBloom. The negative case dominates; the 1% false positive falls through to a real database lookup. Net 99× fewer DB hits.

Set with deletes (revocation, blocklists)

Cuckoo filter (CF.* commands in RedisBloom). Same lookup profile, supports remove without 4× memory of a counting bloom.

Set fits in RAM (<100k items)

Use HashSet<T>. The bloom filter's constant overhead loses to a hash set's directness at small N.

Counting distinct, not membership

HyperLogLog (PFADD / PFCOUNT). Different problem, different probabilistic structure. 12 KB to count a billion distinct items at 0.81% error.

Three things people get wrong

One — undersizing. Filter sized for a million, real cardinality grows to ten million, FP rate climbs from 1% to 60% — cache hit rate craters. Plan capacity; instrument setBits / m.

Two — deleting. Clearing the k bits of an item destroys the no-false-negative guarantee for every other item that shared a bit. If you need delete, use cuckoo or counting bloom; never a plain one.

Three — using cryptographic hashes. SHA-256 is overkill and slow. xxHash3, MurmurHash3, or even FNV-1a are statistically fine and 10–50× faster. The bloom filter doesn't care about pre-image resistance.
