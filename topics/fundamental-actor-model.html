One mailbox, one state, one thread of execution.

Carl Hewitt formulated the actor model in 1973. Erlang built a phone system on it in 1986. PayPal ran a billion transactions/day on it in 2014 (see PayPal case). The primitive is unusual; the production track record is excellent.

What an actor is

(1) Private mutable state. (2) A mailbox of messages. (3) A handler that runs one message at a time. (4) The ability to send messages to other actors (by reference).

What you don't have

Shared memory. Locks. Direct method calls between actors. The constraints are the point.

01The four primitives.

An actor owns its state. No other actor can read or modify it. Communication is message-passing only: I send you a message; my code returns; later your handler runs, reads the message, may mutate your state, may send messages of its own.

The handler runs one message at a time. No locks are needed; concurrency-inside-an-actor doesn't exist. Concurrency-between-actors is the whole game; the runtime schedules thousands of actors across a small thread pool.

Mailbox

Messages queued in the order received. FIFO per sender typically; ordering across senders is undefined. The mailbox is unbounded by default; production systems usually bound it to detect runaway producers.

Sending is fire-and-forget

Send returns immediately. The receiver may process the message in microseconds or seconds. The sender does not wait for the response. If a reply is needed, the actor includes a reply address; the receiver sends back a separate message.

"The actor model says: state is private, computation is local, communication is the only public surface."

02Supervision and "let it crash".

Erlang's defining idea: don't try to handle every error in-process; let the actor crash and have its supervisor restart it. Hierarchical supervision trees. Each supervisor watches a set of children; on crash, applies a strategy: restart one, restart all, escalate.

This sounds reckless and is the opposite. Bugs you didn't anticipate are bugs you can't write defensive code against. Crashing reliably is better than corrupting silently. The system recovers; the audit log records; you fix root cause later.

03Location transparency.

An actor address looks the same whether the actor is in this process, on another machine, or in another datacenter. Send the message; the runtime routes it. Cross-network is just slower than in-process; the API is identical.

This is what makes distributed actor systems (Akka Cluster, Orleans, Erlang's distributed mode) feel coherent. You design as if everything were local; the runtime handles the wire.

04Decision guide.

You have one entity per real-world thing (user, device, order)

Actors fit naturally. Microsoft Orleans calls these "grains"; one actor per logical entity scales to millions of entities across a cluster.

High concurrency, I/O-bound

Actors thrive. Each actor is cheap (~hundreds of bytes). Millions per machine. See PayPal.

Stateful long-running workflows

Actors are durable state machines. Saga-style flows fit, especially with Orleans virtual actors (lazy-activated).

CPU-bound batch processing

Actors are the wrong shape. Use thread pools, MapReduce, dataflow systems.

Already on .NET

Microsoft Orleans is mature, well-documented, used in production at Microsoft. Akka.NET if you want the Akka API.

Already on JVM

Akka is the standard — note 2022 license change (BSL); Pekko is the open fork.

"Actors are a model where the concurrency is obvious and the bugs are testable."
