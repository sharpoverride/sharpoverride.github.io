Team chat that won by staying fast.

Slack's architectural story is unusual: PHP. Sharded by workspace. The architecture was built for a particular shape — small-to-mid-size teams in their own private universe — and held up as that shape spread to enterprise scale.

The shape

Workspace-shard model. Each workspace lives on a single shard end-to-end. No cross-workspace queries on the hot path.

The transport

WebSockets for real-time; SQS/Kinesis for inter-service; flannel (Slack's homegrown LB) for the WebSocket fleet.

The search

Per-workspace Solr indexes. Each workspace is a self-contained search universe.

01Why workspace-sharded.

Slack's data is naturally partitioned by workspace. A user in Acme Corp's Slack never queries data from Globex Corp's Slack. No cross-workspace search. No cross-workspace anything. This is the same cell-architecture pattern as Shopify (see Shopify case) — partition by tenant, isolate end-to-end.

The win: a Slack workspace with 100k users can grow without affecting workspaces with 50 users on the same physical infrastructure. Each shard is sized for the workspace; the small ones share machines, the big ones get dedicated capacity. Blast radius is one workspace.

"Per-tenant is the unit of fail. Anything bigger than one tenant is over-scoped."

02Real-time messaging.

WebSocket per session

Each open Slack tab holds a WebSocket to a Slack-managed fleet. flannel: Slack's homegrown WebSocket proxy. It terminates the WS, fans incoming messages out to relevant subscribers, caches "what does this client need." A workspace's edges scale independently of its data tier.

Channel as topic

Every channel is a pub/sub topic internally. When a message lands, look up subscribers, fan out via the WebSocket fleet. The data tier doesn't push to clients; the WebSocket tier does, after being told.

Presence

"Is so-and-so online?" — the always-asked question. Per-user TTL'd records in a presence cache. Slack famously redesigned this when it didn't scale; the original "broadcast presence to every user in your workspace" pattern was O(N²) and crushed huge teams. The fix: lazy presence — only compute for users you're actively looking at.

03Search.

Per-workspace Solr index. The whole workspace's message history is in its own Solr collection. Search queries scope to the workspace; no global search. This is why Slack's search feels "fast for me" — your search universe is your workspace, not the whole product.

Indexing pipeline: message lands → published to a queue → indexed into Solr asynchronously. Lag of a few seconds to "searchable" is acceptable; instant message delivery is not. The two paths are decoupled.

05The lessons.

Choose a partition key that matches the access pattern

Slack's workspace boundary is also the access boundary. Almost all queries are intra-workspace. The shard key is the access pattern.

Decouple delivery from durability

WebSocket fanout for "you see it now." Async queue + Solr for "you can search for it tomorrow." Same message, two paths, different SLAs.

Presence is O(N²) if you let it be

The naive "everyone subscribes to everyone's presence" pattern collapses at enterprise scale. Lazy presence — only compute for active observers — is the fix.

PHP is fine

The Slack backend remains substantially PHP/Hack. The bottleneck isn't the language; it's the architecture. Don't rewrite the language unless you've exhausted the architecture.

"Slack succeeded at scale by not pretending to be a single global system."
