Case study Years 2020 → 2024 Scale 4M+ users · 10× DB load / yr Stack Postgres · pgbouncer · application sharding

Vertical, then horizontal.

Figma's database was one Postgres instance. It scaled vertically — bigger machine — for years, doubling capacity faster than they consumed it. When vertical scaling stopped paying, they sharded horizontally. The story is what they did between those two states.

The constraint

Postgres on RDS. Single primary. CPU climbing 10× per year. Vertical scale ran out in 2022; the largest RDS instance wasn't going to last another year.

The move

Vertical partitioning first — split large tables off into per-table databases. Then horizontal sharding — split the largest table across shards by hash of a logical "shard ID".

What didn't happen

They did not switch to NoSQL. They did not move to Aurora or Spanner. The migration was Postgres → Postgres + a routing layer.

0110× per year.

Figma is a real-time multiplayer design tool. Every keystroke, every cursor move, every shape edit is a row written to Postgres. The collaborative model means every active session multiplies database load: a Figma file with 10 designers active has 10× the row writes of a single-user editing session. By 2020, Figma's database CPU was doubling every six months — a 10× growth per year.

The textbook answers — "go NoSQL", "event-source everything", "move to Aurora" — all looked correct on the slide and wrong in the implementation. Figma's data was relational: files have nodes, nodes have properties, properties have history, teams own files. The application code assumed transactions and foreign keys. Moving off relational was a multi-year rewrite.

"Scaling Postgres is engineering. Replacing Postgres is also engineering — but the kind that takes three years."

02Architecture before.

Fig. ASingle primary, fronted by pgbouncer for connection pooling, with read replicas. The standard Postgres-at-scale shape — and the shape that runs into a wall.

The 2020 architecture was a single Postgres primary on the largest RDS instance available, behind pgbouncer for connection pooling, with several read replicas behind a load balancer. Writes funneled to the primary; reads scaled horizontally by adding replicas. Vertical scaling was the easy lever: bigger machine, more RAM, faster CPU. Each upgrade bought roughly a year of headroom.

03The vertical ceiling.

CPU at 80%, no bigger machine

By 2022, Figma was on AWS's largest RDS instance — 128 vCPUs, 1 TB RAM, ~$25k/mo. CPU was running at 65% steady-state, peaking at 95% during traffic spikes. There was no bigger machine to upgrade to. The runway was measured in months. The team had two choices: rearchitect, or hit a wall.

The vacuum and the table size

Postgres needs to vacuum tables — reclaim space from dead tuples — to maintain performance. Vacuums on multi-terabyte tables took days. During vacuum, IOPS competed with user traffic. "Tables that take 3 days to vacuum cannot be the foundation of a real-time product." Splitting tables would also split the vacuum cost.

The replica lag

Read replicas lagged the primary by ~50ms in steady state and several seconds during write bursts. For "show me my latest edits", Figma had to route reads to the primary, which removed the replica capacity advantage. The replicas were absorbing 70% of read load — but only the 70% that could tolerate stale data. The other 30% was funneling back to the already-strained primary.

04Architecture after.

Step one — vertical partitioning

Before sharding the hot table, Figma split cold tables off into their own Postgres clusters. The audit log, the comments table, the activity feed — none of these needed to live alongside the multiplayer editing data. Each got its own RDS instance. Six months of work bought ~30% CPU back on the primary, deferring the harder horizontal-sharding migration.

Step two — horizontal sharding by logical ID

The remaining hot table — the one holding live editing state — got sharded. The shard key was a synthetic "shard ID" assigned per file or per team, not the primary key directly. This decoupled the sharding scheme from the application's identifier shapes; future re-sharding becomes a metadata change, not a schema migration.

Fig. BApplication routing layer translates logical shard ID to physical Postgres instance. Cold tables remain in their own non-sharded databases.

Cross-shard queries: don't

The hardest design decision was the constraint: application code is not allowed to write a query that crosses shards. Anything that needs data from multiple shards is computed by fan-out at the application tier, not by JOIN in the database. Many features had to be rewritten to fit this constraint; some were dropped. The discipline is the architecture. Without it, you sharded the storage but kept the bottleneck.

06The lessons.

Vertical first, horizontal second

Going to AWS's biggest instance is the cheapest way to buy time. Vertical scale is straightforward; horizontal sharding is a project. Use the easy lever until it runs out; only then start the hard one. Figma got 2+ years of vertical runway before sharding.

Vertical partitioning before horizontal sharding

Splitting cold tables into their own databases is half a year of work and buys 30%. Sharding the hot table is two years of work and is irreversible. Do the cheap one first.

Logical shard ID, not natural key

Don't shard by user_id directly. Add a shard_id column that the routing layer translates. Future re-sharding becomes a metadata change; the application's identifiers don't carry the sharding scheme.

Cross-shard queries are bugs in disguise

The discipline of "no cross-shard JOIN" is what makes sharding worth doing. Without it, you have a federated database that is slower than the original monolith. Enforce the constraint in code review; cross-shard logic lives in the app, not in SQL.

You don't have to leave your database

"Switch to NoSQL" or "move to Spanner" were the alternatives. Both were rewrites. Figma's migration was Postgres → Postgres. If your data is relational, scaling-by-replacing your database is the most expensive option on the menu. Make sure you've exhausted the cheaper ones.

"You don't replace Postgres at scale. You teach the application to live with fewer assumptions about it."
