Mobile is different.

Backend system design assumes infinite RAM, infinite connectivity, infinite background time. Mobile system design assumes none of that. Battery, intermittent network, app lifecycle, store review — every constraint flips. The framework adapts.

The big differences

Offline-first by default. Background-execution is rationed. Push notifications are the wake-up mechanism. App-store updates take days.

The framework

(1) Scope. (2) Estimate (storage, battery, network). (3) Local DB schema. (4) Sync protocol. (5) UI + state mgmt. (6) Offline / failure modes.

01What makes mobile different.

Network is unreliable. Trains, tunnels, elevators, planes. The app must keep working. Battery is finite. Spinning the radio costs real watts. Background is rationed. iOS will suspend you; Android will kill you under memory pressure. Storage is limited. Not infinite cloud; a few GB you share with photos.

The framework changes accordingly. Local DB isn't a cache — it's the source of truth for offline. Sync isn't an afterthought — it's the architecture. Lifecycle isn't a polite suggestion — it's the operating system's policy you live under.

"Backend system design assumes the network. Mobile system design assumes the network might not be there."

02The 6-step adaptation.

Step 1 — Scope

Same as backend: pin features, users, platforms. iOS only? Android only? Both? The answer changes the design (cross-platform vs native). Tablet support, watch support, foldables are scope-cuts.

Step 2 — Mobile-specific estimation

QuantityWhy it matters
Local DB size2 GB is a lot of phone storage
Sync payload sizeBattery and bandwidth, especially cellular
Push notification ratePer-user, per-day; affects wake costs
Cold-start timeCritical for engagement; aim <2s
App binary sizeStore download limits, install rate

Step 3 — Local DB schema

The phone has a SQLite DB (Room on Android, Core Data / SwiftData / GRDB on iOS). Design it like a real database: indexes for the queries your UI does, denormalisation where the joins are too slow on mobile CPUs. The schema lives locally; it doesn't have to match the server's exactly.

Step 4 — Sync protocol

How do client and server reconcile state? The hard subsystem. Bidirectional, idempotent, conflict-aware. See section 3.

Step 5 — UI + state management

Single source of truth (the local DB). Reactive layer (Compose / SwiftUI / Redux-like). The UI binds to local data; sync happens underneath. Optimistic updates: mutate locally first, send to server, reconcile on response.

Step 6 — Offline / failure modes

What does the user see when offline? Read works (from local DB). Writes queue (in an outbox table on-device). "Stale data" indicator if data is older than X. Reconnection drains the outbox; conflicts surface to the user.

03Sync is the hard part.

Delta sync, not full sync

"Fetch everything" works once; doesn't work daily. Track a sync token (timestamp or sequence number); server returns only changes since the token. Saves bandwidth, battery, time.

Bidirectional with conflict resolution

Both client and server can have made changes. Last-write-wins is the simplest; sometimes wrong (see geographic distribution chapter). Per-field LWW. CRDTs for collaborative apps. User-prompt resolution for the rare collisions.

Background sync windows

iOS: BGAppRefreshTask, ~30s every ~6 hours. Android: WorkManager with constraints (charging, Wi-Fi). You don't control when sync happens; you describe the constraints, the OS schedules.

Push as the wake signal

Server has new data → silent push notification → OS wakes app briefly → sync runs → UI updated when app opens. Silent push is rationed too (iOS will deliver only some); use it as a hint, not a contract.

Outbox for writes

Pending writes go into a local outbox table. On connectivity, drain in order with idempotency keys. Same outbox pattern as backend (see CDC chapter), applied client-side.

04Anti-patterns.

No offline mode

The interviewer expects you to handle it. A backend-shaped answer ("call the API on tap, show spinner, retry") is a fail.

Polling instead of push

Polling drains battery. Use push. If push isn't possible, poll on app foreground only, not in background.

No optimistic UI

User taps, sees spinner, waits for server. Mobile users abandon. Mutate locally; reconcile on response.

Treating the local DB as a cache

If it's a cache, you'll re-fetch on every cold start; battery dies. Local DB is the truth. Sync is the freshener.

Forgetting app store update lag

You can't roll back a buggy version like a server. Feature flag dangerous changes; server-side kill switch is the actual emergency control.

05Decision guide.

Read-heavy app (news, social)

Local DB + delta sync. Cache aggressively. Pre-fetch on Wi-Fi.

Collaborative app (notes, docs)

CRDT-based sync. Server is convergence point, not source of truth. Real-time channel for low-latency sync.

Messaging

WebSocket / long-poll when foregrounded; push when backgrounded. Local outbox for unsent messages.

Native vs cross-platform

React Native / Flutter / KMP if the team is small and the UI is straightforward. Native if the platform-specific UX or performance matters.

Need real offline-first

Local-first stack: Realm, Couchbase Mobile, Cloud Firestore offline, custom SQLite + sync. The trick is consistent sync semantics.

Watch / wearable

Even more constrained. Sync paired phone, not server directly. Cache aggressively; expect bursts of connectivity.
