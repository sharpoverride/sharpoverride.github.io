Design a web crawler.

A polite, large-scale crawler is a frontier queue, a duplicate detector, a politeness scheduler, and an army of fetchers — all of which scale on different bottlenecks. The unsexy parts (deduplication, politeness, robots.txt) win the interview.

Anchor numbers

~10B pages on the surface web · ~1 MB avg HTML · ~10 PB total · crawl-budget targets vary (Google: re-crawl every few days for important pages; once a month for cold ones).

The four subsystems

(1) URL frontier (queue). (2) Fetchers. (3) Parser + URL extractor. (4) Duplicate detector. The politeness scheduler glues them.

The hidden constraint

Politeness: don't hammer any single site. Robots.txt: respect crawl rules. Get either wrong and you become an attacker.

03High-level architecture.

Fig. ALoop: take URL → fetch → parse → extract links → add new links back to frontier. Dedup, politeness, and storage hang off each step.

04Deep dive — URL frontier.

Two-tier queue: priority & politeness

The frontier needs to balance two things. Priority: important pages (high PageRank, frequent change) should be visited more often. Politeness: don't hit any one host too fast. The classic shape: a front queue per priority level (1=high, 2=medium, 3=low) and a back queue per host that buffers URLs ready to fetch at the next polite time.

Storage

100M URLs in flight; ~10B total ever seen. The frontier doesn't fit in RAM; back it with a distributed store (Cassandra, RocksDB, or a partitioned key-value store). Each URL's record: URL, priority, last-fetched timestamp, retry count, host. Hot queue partitions stay in RAM via the dedup bloom filter (see below).

05Deep dive — deduplication.

URL dedup

"Have I queued this URL?" — checked on every link extraction. For 10B URLs, a hash set in RAM is impossible (~640 GB). Use a bloom filter (see bloom filters chapter): ~12 GB at 1% false-positive rate, distributed across the cluster. False positive = skip a real URL (acceptable; tolerated for the savings). Persistence-backed for restart.

Content dedup

Different URLs can serve the same content (mirrors, tracking-parameter variants). Hash the page content and dedup by content hash. SimHash for near-duplicate detection. This is where you avoid storing the same page a hundred times.

Canonical URL normalisation

Before hashing, normalise the URL: lowercase host, strip default ports, sort query parameters, strip utm_*. Different normalised forms of the same logical URL collapse to one frontier entry. Without this, the dedup bloom filter degrades — you store 10× the URLs you need.

06Deep dive — politeness.

Robots.txt

Before crawling a host, fetch /robots.txt; cache it for 24 hours. The file tells you which URLs to skip and (optionally) the crawl delay. Honor it strictly; failing to do so gets you blocked.

Per-host rate limiting

One in-flight request per host, with a delay between requests (default ~5s; respect Crawl-delay from robots). A back-queue per host enforces this: URLs are pulled from the back-queue at the rate the host allows. A small site has 5 URLs and takes 30 seconds to crawl; a large site has 10M URLs and takes a year. The crawler's fairness is per-host, not per-URL.

Identify yourself

Set a clear User-Agent with contact info. Most webmasters distinguish "legitimate crawler" from "scraper" by that header. Without it, you'll be IP-blocked by everyone with a WAF.

07Scale & bottlenecks.

Politeness caps absolute crawl rate

You can't crawl Wikipedia faster than Wikipedia allows. Total crawl rate is bounded by the count of distinct hosts × per-host rate. Want more pages/sec → crawl more hosts in parallel; you can't crawl any single one faster.

JavaScript rendering changes the math

Modern web pages are JS-heavy; raw HTML doesn't reveal the content. Rendering means running a headless browser per page; that's 10–100× the CPU. If JS rendering is in scope, the architecture changes: a separate render fleet, render-only when a page is identified as JS-heavy.

Re-crawl strategy matters

"Crawl every page once" is not the goal. "Re-crawl popular pages frequently" is. Use change-frequency hints (last-modified, ETag), HTTP 304 responses, and a freshness scoring model. A news homepage gets re-crawled hourly; a 2007 forum post gets re-crawled annually.

Storage is colossal

5 PB of HTML. Compress aggressively; gzip is 70% reduction on HTML. Cold tier for old crawls; hot tier for recent. WARC format (the web-archive standard) is the canonical container — multiple records in one file, indexable offset.

Tradeoffs to mention

Depth-first vs breadth-first: BFS gives broader coverage early; DFS digs into one site fast. Most crawlers are BFS with priority. Storage vs re-crawl: pages stored cheaply once; if frequently revisited, the cost shifts to compute. Polite crawler vs hostile crawler: polite is right; hostile crawls get blocked.

Open questions

Auth-walled content (login required) — out of scope. Anti-bot defences (Cloudflare bot management) — even legitimate crawlers fight this; need fingerprint diversification or partnerships. International domains — DNS resolution latency, IDN handling, character encoding gotchas.

"A crawler isn't a tight loop — it's a polite, slow conversation with a million unhappy webmasters."
