Limit the blast radius.

Cell-based architecture partitions a multi-tenant platform into independent, identical units called cells. A cell handles a slice of customers end-to-end. When one cell fails, the rest keep running. AWS, Shopify, and Slack all converged on this — it's the most underrated SaaS architecture pattern of the last decade.

The premise

A single big shared system fails for everyone when it fails. N small identical systems with disjoint customer sets fail for 1/N of customers at a time.

The key constraint

Cells share nothing in the hot path. No shared database. No shared cache. No shared message broker. The router is the only shared layer.

01The shared-fate problem.

A traditional multi-tenant SaaS is one big system serving thousands of customers. Big database; big app fleet; big cache. When the database has a hot lock, everyone is slow. When the cache cluster fails, everyone is down. When a noisy tenant's batch job consumes the worker pool, everyone waits. The blast radius of any single failure is the entire customer base.

Cells replace the big shared system with many small identical ones. Each cell serves a partition of tenants, end-to-end: its own database, its own app fleet, its own cache, its own message broker. A failure in one cell is invisible to tenants in other cells. Per-customer reliability stops being a function of platform health; it's a function of the cell's health, which is a function of one cell-worth of customers.

"In a shared system, every customer is one tenant's bad day away from a Sev-2."

02What a cell contains.

Fig. 31Each cell is a complete stack. Tenants are pinned to a cell. The only thing shared is the router that knows which cell owns which tenant.

A cell is a full vertical slice

API tier, database, cache, message broker, background workers — all running per-cell. The application code is identical across cells; the cell is a deployment unit, not a different product. A new feature is deployed to every cell (in waves: canary → 10% of cells → all cells). A schema migration runs in every cell's database.

Cells are dimensioned for one cell-worth of customers

Sizing decision: how big should a cell be? Too small and you have many cells to operate. Too big and the blast radius approaches "platform-wide". Typical: cells sized to hold ~5–10k tenants, supporting a few thousand RPS. AWS's published guidance: a cell should be small enough that you can manually reason about it, large enough that you don't have 1000 of them.

03The router is the only seam.

The cell router is the only shared component. It looks up tenant → cell and forwards the request. Keep it boring: a small, highly-available service with a tiny mapping table cached aggressively. If the router has a hot path that calls a database, the router itself becomes the bottleneck the cells were supposed to avoid.

Where the mapping lives

Three storage options, in order of operational simplicity. DNS: each tenant gets {tenant}.api.example.com resolving to a cell-specific IP. Zero hot-path lookup; DNS TTL is the propagation knob. Path-based: /{tenant}/... and the router parses the URL. JWT: the user's auth token carries the cell ID; the router validates and routes. Pick by what you can secure cleanly; DNS-based routing is the most popular.

Cell migration

Tenants grow. Cells get unbalanced. Build the "move tenant from cell A to cell B" tooling early. The procedure: snapshot the tenant's data in cell A; replay onto cell B; switch the router; verify; clean up cell A. Automate it; run it routinely; don't let it become a one-off engineering project. Shopify moves merchants between pods all the time.

04In practice.

C#   CellRouterMiddleware.cs 
public sealed class CellRouterMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ICellMap _map;

    public CellRouterMiddleware(RequestDelegate next, ICellMap map)
    { _next = next; _map = map; }

    public async Task InvokeAsync(HttpContext ctx, HttpClient http)
    {
        // Resolve tenant from JWT, subdomain, or path.
        var tenant = ctx.User.FindFirstValue("tenant") ?? "anonymous";
        var cell = await _map.GetCellForTenantAsync(tenant);

        // Forward to the cell-specific origin. In production this is a YARP route
        // or front-door rule; the middleware is illustrative.
        var upstream = $"https://{cell}.internal{ctx.Request.Path}{ctx.Request.QueryString}";

        using var req = new HttpRequestMessage(new(ctx.Request.Method), upstream);
        foreach (var h in ctx.Request.Headers)
            req.Headers.TryAddWithoutValidation(h.Key, h.Value.ToArray());

        if (ctx.Request.ContentLength > 0)
            req.Content = new StreamContent(ctx.Request.Body);

        using var resp = await http.SendAsync(req, ctx.RequestAborted);
        ctx.Response.StatusCode = (int)resp.StatusCode;
        foreach (var h in resp.Headers)
            ctx.Response.Headers[h.Key] = h.Value.ToArray();
        await resp.Content.CopyToAsync(ctx.Response.Body, ctx.RequestAborted);
    }
}

In production you'd use YARP (.NET) or Envoy / NGINX with a small Lua/Rust extension that looks up the tenant→cell map from a cache. The router itself must be horizontally scalable and stateless; the mapping is in Redis or DynamoDB, the router pods just look it up.

05The detail.

Cells don't talk to each other

Cell 1 should not query Cell 2's database, even indirectly. Cross-cell queries reintroduce shared fate. If you need cross-tenant analytics, run it in a separate analytics pipeline that reads from each cell asynchronously. Cell-to-cell is a smell; alarm on it.

Cell capacity is dimensioned for failure

Each cell runs at ~50% of its theoretical max. When a cell fails, its tenants need to be rebalanced into the others; the others must absorb the load without saturating. Running cells at 80% is fragile; running them at 50% costs more compute but makes the failure recovery routine.

The control plane is its own concern

Provisioning a cell, deploying code to all cells, running schema migrations across cells — this is the "control plane" of the cell architecture. Build it deliberately; don't let it become 100 shell scripts. AWS calls this the "cell-router" and "cell-orchestrator" layers; treat them with the rigor of any production system.

"Sticky" vs "balanced" cells

Tenant → cell mapping is sticky by default — once you're in cell 7, you stay there. Rebalancing is the operational work that makes the architecture sustainable. Plan for it; tooling beats heroism.

Cells fit naturally with AWS Wisdom paper

AWS's published "Wisdom from cell-based architecture" paper formalises the patterns. Required reading for anyone designing this from scratch. The summary: "all blast-radius reduction is partitioning. All partitioning has a cost. Cell-based architecture is the partitioning that's worth paying for in multi-tenant SaaS."

"The interesting question isn't ‘does the cell scale to one million customers?’ — it's ‘does the platform survive when this cell doesn't?’"

06Decision guide.

Multi-tenant B2B SaaS, >1000 tenants

Cell architecture is the right default. ~5–10k tenants per cell. Router via DNS or JWT. Cell migration tooling from day one.

Single tenant per customer, <100 customers

Probably overkill. One database with strong tenant isolation columns is fine. Revisit if you grow past a few hundred.

B2C with billions of users

Different shape. Sharding by user-id is the typical pattern (see Figma). Cell architecture mainly suits B2B with named tenants.

You have one big shared database now

You can retrofit. Start by extracting one tenant cohort into a new cell; build the migration tooling there. Don't try to cell-ify everything at once.

Need cross-tenant analytics

Build a separate analytics pipeline that reads from every cell. Don't cross-query cells in the hot path. CDC (see CDC chapter) feeds a warehouse.

Compliance / data-residency

Cells map cleanly: EU tenants in EU cells, US in US, etc. Data never crosses regions because the cell never crosses regions.
