Rate limiting.

Five algorithms, one distributed reality, and the Redis Lua script that lets a single API handle a billion daily requests without melting downstream.

The problem

Your API has a noisy neighbour. A free-tier user runs a tight retry loop. A scraper hits the search endpoint at 50 RPS. Without a limiter, one client takes down the whole platform.

The thesis

Cap requests per key, per window. Push state to a place every replica can see. Reject early — at the edge — and emit headers that teach good clients to back off.

01Why limit at all.

A rate limiter answers one question, asked once per request: has this client used more than its share? The answer must be cheap, correct under concurrency, and survive a process restart. That is harder than it sounds.

Three forces push for it. Cost — every unauthenticated request you serve costs CPU, bandwidth, and downstream calls you pay for. Fairness — one greedy tenant should not starve the others. Stability — when your DB starts to wobble, the limiter is the bulkhead that keeps the wobble from becoming a fall.

"A rate limiter is not a security feature. It is a fairness feature with security side-effects."

The keys you limit on matter at least as much as the algorithm. Limit on IP and you punish coffee shops; limit on api_key and you punish the customer; limit on (api_key, endpoint, action) and you can be surgical. Stripe famously runs four independent limiters per request: total request rate, concurrent request count, expensive-endpoint rate, and burst protection.

02Five algorithms.

Every limiter is a bookkeeping problem: how do you remember how many requests have passed? The five canonical answers below trade memory, smoothness, and burst behaviour against each other.

Fig. 01Five algorithms · limit = 5 req/s · burst & memory profile

The right pick is rarely the most accurate one. For a public API, sliding-window counter and token bucket win — they're O(1) per key, allow modest bursts that match how real clients actually call you, and play well with horizontal scaling. AWS API Gateway, Stripe, GitHub, and Cloudflare all ship variants of these two.

  • Fixed window — one counter per (key, window). Cheap, but at the boundary a client can fire 2× the limit by hitting the last 500 ms of window N and the first 500 ms of window N+1.
  • Sliding log — store every request timestamp in a sorted set. Exact, but memory scales with traffic. Use for billing-grade accuracy.
  • Sliding counter — store two integers (current window + previous), weight by overlap. The sweet spot.
  • Token bucket — refill at rate r, capacity b. Allows bursts up to b, then enforces r. The textbook public-API limiter.
  • Leaky bucket — FIFO queue draining at rate r. Smooths traffic. Use upstream of brittle services that can't tolerate spikes.

03Going distributed.

A single in-process limiter is trivial: one ConcurrentDictionary<string, Counter> and you're done. The moment you have two replicas, you have a problem.

If each replica enforces 100 req/s, ten replicas allow 1,000 req/s — ten times the contract. So state must move out of the process. You have three options:

Option A

Sticky routing

  • Hash the API key to a specific pod
  • That pod owns the counter, in-memory
  • Zero shared state, lowest latency
  • But: deploys = counter loss, hot keys = hot pods, K8s rolls hurt
Option B · ★

Central store (Redis)

  • One Redis cluster sees every increment
  • Lua script makes check-and-increment atomic
  • ~0.5 ms p99 for limit decision
  • Trade: +1 hop per request, Redis is now critical-path
Option C

Local quota + gossip

  • Each pod gets a slice of the global budget
  • Pods exchange usage via gossip every N ms
  • No critical-path Redis
  • But: drift between pods, hard to reason about, niche

Option B wins for 95% of teams. Redis is mature, the round-trip is fast, and your ops team probably already runs it for caching. The remaining 5% are operating at a scale where Redis itself becomes the bottleneck — at which point you typically also shard the limiter by key prefix and run a second tier of approximate limiting at the edge (Cloudflare, Envoy, NGINX).

Production note

Fail open, not closed. If Redis returns an error or times out, allow the request and emit a metric. A broken limiter that refuses every request looks identical to a global outage. Stripe's playbook is explicit on this — the limiter is a guardrail, not a gate.

04In ASP.NET Core.

Two paths. Pick the one that matches your scale and your team's appetite for owning Redis.

ASP.NET Core 7 shipped Microsoft.AspNetCore.RateLimiting. It's solid for single-instance services, internal admin tools, and dev environments. It does not coordinate across replicas.

C#   Program.cs 
// Built-in: Microsoft.AspNetCore.RateLimiting (single-node only)
using System.Threading.RateLimiting;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddRateLimiter(o =>
{
    o.RejectionStatusCode = StatusCodes.Status429TooManyRequests;

    // Free tier — token bucket, 60 req/min, burst 20
    o.AddTokenBucketLimiter("free", opt =>
    {
        opt.TokenLimit          = 20;        // bucket capacity
        opt.TokensPerPeriod     = 60;        // refill
        opt.ReplenishmentPeriod = TimeSpan.FromMinutes(1);
        opt.QueueLimit          = 0;         // reject instead of queue
        opt.AutoReplenishment   = true;
    });

    // Pro tier — sliding window, 600 req/min
    o.AddSlidingWindowLimiter("pro", opt =>
    {
        opt.PermitLimit        = 600;
        opt.Window             = TimeSpan.FromMinutes(1);
        opt.SegmentsPerWindow  = 6;          // 10-sec sub-windows
    });

    // Partition each tier by API key from header
    o.GlobalLimiter = PartitionedRateLimiter.Create<HttpContext, string>(ctx =>
    {
        var apiKey = ctx.Request.Headers["X-Api-Key"].ToString();
        var tier   = ctx.User.FindFirst("tier")?.Value ?? "free";
        return RateLimitPartition.GetSlidingWindowLimiter(
            partitionKey: $"{tier}:{apiKey}",
            factory: _ => new SlidingWindowRateLimiterOptions
            {
                PermitLimit       = tier == "pro" ? 600 : 60,
                Window            = TimeSpan.FromMinutes(1),
                SegmentsPerWindow = 6,
            });
    });

    o.OnRejected = async (ctx, ct) =>
    {
        ctx.HttpContext.Response.Headers["Retry-After"] = "60";
        await ctx.HttpContext.Response.WriteAsJsonAsync(
            new { error = "rate_limited", retry_after_s = 60 }, ct);
    };
});

var app = builder.Build();
app.UseRateLimiter();
app.MapGet("/v1/charges", () => Results.Ok(new { ok = true }))
   .RequireRateLimiting("pro");
app.Run();

When this is enough: single-pod service, dev environment, internal tool. When it isn't: any service running >1 replica behind a load balancer. Three replicas = 3× the contracted limit.

For multi-replica production, push state to Redis. The script below is a complete sliding-window counter with token-bucket refill semantics — drop-in, ASP.NET Core middleware, emits the four headers good clients expect, fails open.

C#   RedisRateLimiter.cs 
// dotnet add package StackExchange.Redis
using StackExchange.Redis;
using Microsoft.Extensions.Logging;

namespace Academy.RateLimiting;

public sealed record RateLimitDecision(
    bool Allowed,
    long Limit,
    long Remaining,
    TimeSpan RetryAfter);

public sealed class RedisRateLimiter(
    IConnectionMultiplexer redis,
    ILogger<RedisRateLimiter> log)
{
    // Sliding window counter with token-bucket-like refill.
    // KEYS[1] : per-tenant bucket key
    // ARGV    : limit | window_ms | now_ms | cost
    // Returns : { allowed, remaining, retry_after_ms }
    private const string Script = @"
        local key      = KEYS[1]
        local limit    = tonumber(ARGV[1])
        local window   = tonumber(ARGV[2])
        local now      = tonumber(ARGV[3])
        local cost     = tonumber(ARGV[4])
        local cutoff   = now - window

        redis.call('ZREMRANGEBYSCORE', key, '-inf', cutoff)
        local count = redis.call('ZCARD', key)
        if count + cost > limit then
            local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
            local retry  = window - (now - tonumber(oldest[2] or now))
            return { 0, limit - count, retry }
        end
        for i = 1, cost do
            redis.call('ZADD', key, now, now .. ':' .. i)
        end
        redis.call('PEXPIRE', key, window)
        return { 1, limit - count - cost, 0 }
    ";

    public async Task<RateLimitDecision> CheckAsync(
        string key, long limit, TimeSpan window, int cost = 1)
    {
        try
        {
            var db = redis.GetDatabase();
            var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
            var result = (RedisValue[]?)await db.ScriptEvaluateAsync(
                Script,
                new RedisKey[] { $"rl:{key}" },
                new RedisValue[] { limit, (long)window.TotalMilliseconds, now, cost });

            if (result is null || result.Length < 3)
                return new RateLimitDecision(true, limit, limit, TimeSpan.Zero);

            var allowed   = (long)result[0] == 1;
            var remaining = Math.Max(0, (long)result[1]);
            var retryMs   = (long)result[2];

            return new RateLimitDecision(
                allowed, limit, remaining, TimeSpan.FromMilliseconds(retryMs));
        }
        catch (RedisException ex)
        {
            // FAIL OPEN — never let a broken limiter break the API.
            log.LogWarning(ex, "rate limiter unavailable; allowing {key}", key);
            return new RateLimitDecision(true, limit, limit, TimeSpan.Zero);
        }
    }
}
C#   RateLimitMiddleware.cs 
public sealed class RateLimitMiddleware(
    RequestDelegate next,
    RedisRateLimiter limiter,
    IOptionsMonitor<RateLimitOptions> opts)
{
    public async Task InvokeAsync(HttpContext ctx)
    {
        // Build a key per (tenant, endpoint, tier). Don't limit by IP alone.
        var apiKey = ctx.Request.Headers["X-Api-Key"].ToString();
        if (string.IsNullOrEmpty(apiKey)) { await next(ctx); return; }

        var policy = opts.CurrentValue.PolicyFor(ctx.Request.Path, apiKey);
        var key    = $"{policy.Name}:{apiKey}";

        var r = await limiter.CheckAsync(key, policy.Limit, policy.Window);

        ctx.Response.Headers["X-RateLimit-Limit"]     = r.Limit.ToString();
        ctx.Response.Headers["X-RateLimit-Remaining"] = r.Remaining.ToString();
        ctx.Response.Headers["X-RateLimit-Reset"]     =
            DateTimeOffset.UtcNow.Add(r.RetryAfter).ToUnixTimeSeconds().ToString();

        if (!r.Allowed)
        {
            ctx.Response.StatusCode = StatusCodes.Status429TooManyRequests;
            ctx.Response.Headers["Retry-After"] =
                ((int)Math.Ceiling(r.RetryAfter.TotalSeconds)).ToString();
            await ctx.Response.WriteAsJsonAsync(new {
                error = "rate_limited",
                limit = r.Limit,
                retry_after_seconds = (int)Math.Ceiling(r.RetryAfter.TotalSeconds)
            });
            return;
        }
        await next(ctx);
    }
}

05Redis Lua, line by line.

The Lua script above is doing three things atomically — and atomicity is the entire point. Without it, two replicas would read the same count, both pass the check, both increment, and both let the request through. With it, Redis executes the script as a single command; nothing else runs on that key until the script returns.

  1. ZREMRANGE
    BYSCORE    Sweep the window. Remove every entry older than now - window_ms. The sorted set now contains only requests inside the active window.
  2. ZCARDCount. O(1) read of how many requests survived the sweep. This is your current usage.
  3. if > limitDecide. If adding cost would breach the limit, peek at the oldest timestamp to compute when the client should retry, return {0, remaining, retry_ms} and don't increment.
  4. ZADD ×costCommit. Otherwise add cost timestamps (so an expensive endpoint can consume multiple) and bump the TTL.
  5. PEXPIREReap. Set the key TTL to the window. Idle keys vanish — memory stays bounded.
Why not INCR + TTL?

The classic INCR + EXPIRE pair gives you fixed-window semantics — and the 2× burst at the boundary. The sorted-set approach costs roughly 3× the bytes per key but gives you proper sliding-window precision for free. At p99 it adds ~0.3 ms. Worth it.

06Tradeoff matrix.

Where to enforce the limit matters as much as how. The same algorithm has very different cost and latency depending on whether it runs at the CDN, the gateway, or in your service. Here's how six common choices stack up.

The honest production stack is layered, not one of the rows above:

  • Cloudflare for crude per-IP and known-bot rules — drop the obvious garbage before it reaches your origin.
  • Redis + Lua inside your services for the per-tenant, per-endpoint contract enforcement that actually maps to your pricing tiers.
  • Polly in your outbound clients so you're a good citizen to your dependencies.

NGINX is a defence against unauthenticated abuse, not a replacement for application-level limits. APIM makes sense only when you already need its other features (subscription keys, developer portal, OAuth flows) — paying its price tag just for rate limiting is a poor trade.

07Decision guide.

Single replica, internal tool

Use the built-in AddSlidingWindowLimiter. Don't add a Redis just for this. Revisit when you scale out.

Public B2B API, >1 replica

Redis + Lua sliding window. Partition by (tenant, endpoint, action). Emit standard X-RateLimit-* headers.

Pricing has tiers

Limit policy comes from claims, not config. Free / Pro / Enterprise each get a different (limit, window) at runtime from the JWT.

Expensive endpoint (search, reports)

Charge multiple tokens per call via the cost param. A heavy search might cost 10; a status check, 1.

Burst tolerance matters

Token bucket. Sets capacity (allowed burst) independently from refill rate (sustained throughput).

Smoothness matters (queue feeder)

Leaky bucket on the producer. Downstream sees a steady drip even if upstream is bursty.

"Fail open. A broken limiter that refuses every request is indistinguishable from a global outage — and a lot harder to roll back."
Three things people get wrong

① Limiting by IP. NAT exists. Mobile carriers exist. Your customers' offices exist. Limit by api_key or user_id; use IP only as a coarse anti-abuse signal.

② No headers. Without X-RateLimit-* headers, well-behaved clients can't back off gracefully. They'll just retry and you'll just keep 429'ing them.

③ Same limit for every endpoint. A status check costs 1 ms; a CSV export costs 5 s. Charging them the same token is throwing away most of your headroom.
