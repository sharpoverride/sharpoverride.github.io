Message queues.

Logs versus brokers, the outbox pattern that fixes the only hard problem, and how to pick between Kafka, Service Bus, RabbitMQ and SQS without lying to yourself about cost.

The two models

Log (Kafka) — append-only, consumers track offsets, replay is free. Broker (RabbitMQ, SB, SQS) — messages have lives; acks delete them; replay needs work.

The thesis

Pick the broker model unless you actually need replay or fan-out. Run an outbox. Make consumers idempotent. Everything else is decoration.

01What queues solve.

A message queue is not a clever piece of infrastructure. It's a piece of infrastructure that lets you say no later. The producer hands a message to the broker, the broker takes responsibility, and the producer goes back to serving its next request. Five problems collapse into one shape:

  • Decoupling. The order service doesn't know how many things care about a new order — email, inventory, fraud, analytics. It publishes; consumers self-select.
  • Smoothing. A 10× traffic spike at the producer does not become a 10× spike at every downstream. The queue absorbs the difference.
  • Retry. If the email service is down for 8 minutes, the message waits 8 minutes. No retry loop in your hot path.
  • Ordering. Per-partition ordering — every event for order:42 goes to the same consumer instance, in order.
  • Audit. Every state change is, by construction, a durable record.
"A queue is what you reach for when 'try again' is no longer a good enough answer."

02Log vs broker.

The single most useful mental model for messaging: does the broker delete messages after consumers ack them?

Fig. 03The log model fans out and replays; the broker model competes and forgets
Log model wins

Replay, fan-out, audit

  • Multiple independent consumer groups (analytics, search, billing) all read the same stream
  • Add a new consumer 6 months later — replay from offset 0
  • Event sourcing / CDC / streaming joins
  • Per-partition ordering by key
Broker model wins

Work queues, RPC, DLQ

  • "Send this email" is consumed once and gone
  • Priority queues, scheduled delivery, TTL per message
  • Native DLQ semantics — poison messages have a home
  • Lower operational burden than running Kafka
Don't conflate

The wrong choice hurts

  • Using Kafka as a work queue — no priority, awkward retries, no native scheduled delivery
  • Using SQS for event sourcing — replay requires rebuilding all the infrastructure Kafka gives you free
  • "We'll just use Postgres as a queue" — works to ~1k msg/s, then breaks in confusing ways

03The reliability triangle.

Three guarantees, pick two — but with messaging, the unpicked one is always "exactly once," and that's because exactly-once delivery does not exist. It's a marketing word. Every production system that says "exactly once" means at-least-once delivery + idempotent processing. Internalise that and the rest falls into place.

Fig. 04 · The only two semantics you can build withat-least-once + idempotency = "exactly once"

At-least-once delivery is on by default in every broker worth using. Idempotency is your job. There are three places to put it:

  • At the message. Stamp every message with a stable messageId at publish time. Consumer stores processed IDs (a fast Redis set, or a unique-constrained table).
  • At the operation. Make the side-effect itself idempotent: UPSERT instead of INSERT; SET counter = 5 instead of INCR.
  • At the boundary. Pair the message with the outbound side-effect's natural idempotency key — Stripe's Idempotency-Key header, an order number, a tax-document UUID.

04The transactional outbox.

The most common bug in event-driven .NET code: "we saved the order, but the OrderCreated event never went out". Or worse: "we sent the event but the DB transaction rolled back". These are two faces of one problem — you're trying to atomically update two systems (the database and the broker) without a distributed transaction.

The fix has a name. The outbox pattern writes the message to an outbox table in the same EF Core transaction as the business write. A separate process drains the outbox to the broker. The database becomes the single source of truth for both state and intent.

C#   OutboxMessage.cs · DbContext 
namespace Academy.Messaging;

// The outbox row. One per message. Single table for all message types.
public sealed class OutboxMessage
{
    public Guid      Id            { get; set; } = Guid.NewGuid();
    public string    Type          { get; set; } = default!;   // e.g. "order.created.v1"
    public string    Payload       { get; set; } = default!;   // JSON
    public string    PartitionKey  { get; set; } = default!;   // for ordering
    public DateTimeOffset CreatedAt  { get; set; } = DateTimeOffset.UtcNow;
    public DateTimeOffset? PublishedAt { get; set; }
    public int       Attempts      { get; set; }
}

public sealed class ShopDbContext(DbContextOptions<ShopDbContext> opts) : DbContext(opts)
{
    public DbSet<Order>          Orders => Set<Order>();
    public DbSet<OutboxMessage> Outbox => Set<OutboxMessage>();

    protected override void OnModelCreating(ModelBuilder b)
    {
        b.Entity<OutboxMessage>(e =>
        {
            e.HasIndex(x => new { x.PublishedAt, x.CreatedAt })
             .HasFilter("\"PublishedAt\" IS NULL");    // partial index — fast drain
            e.Property(x => x.Payload).HasColumnType("jsonb");
        });
    }
}
C#   CreateOrderHandler.cs 
// Notice: no broker call here. Just the DB. SaveChangesAsync commits both rows.
public sealed class CreateOrderHandler(ShopDbContext db)
{
    public async Task<Order> HandleAsync(CreateOrderCommand cmd, CancellationToken ct)
    {
        var order = new Order {
            Id = Guid.NewGuid(),
            CustomerId = cmd.CustomerId,
            Total      = cmd.Total,
            Status     = OrderStatus.Pending,
        };
        db.Orders.Add(order);

        db.Outbox.Add(new OutboxMessage
        {
            Type         = "order.created.v1",
            Payload      = JsonSerializer.Serialize(new {
                                  order.Id, order.CustomerId, order.Total
                              }),
            PartitionKey = order.CustomerId.ToString(),   // keep customer events ordered
        });

        await db.SaveChangesAsync(ct);   // ★ both rows in one tx
        return order;
    }
}
C#   OutboxPublisher.cs · BackgroundService 
// One per pod. Drains the outbox in small batches; idempotent on retry.
public sealed class OutboxPublisher(
    IServiceScopeFactory scopes,
    IMessageBroker broker,
    ILogger<OutboxPublisher> log) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            try
            {
                using var scope = scopes.CreateScope();
                var db = scope.ServiceProvider.GetRequiredService<ShopDbContext>();

                // SKIP LOCKED — many pods can drain concurrently, each grabs disjoint rows.
                var batch = await db.Outbox
                    .FromSqlInterpolated($@"
                        SELECT * FROM ""Outbox""
                        WHERE ""PublishedAt"" IS NULL
                        ORDER BY ""CreatedAt""
                        FOR UPDATE SKIP LOCKED
                        LIMIT 100")
                    .ToListAsync(ct);

                if (batch.Count == 0)
                {
                    await Task.Delay(TimeSpan.FromMilliseconds(200), ct);
                    continue;
                }

                foreach (var m in batch)
                {
                    try
                    {
                        // Pass m.Id as broker's message-id so consumer can dedupe.
                        await broker.PublishAsync(
                            topic: m.Type,
                            messageId: m.Id,
                            partitionKey: m.PartitionKey,
                            payload: m.Payload,
                            ct);
                        m.PublishedAt = DateTimeOffset.UtcNow;
                    }
                    catch (Exception ex)
                    {
                        m.Attempts++;
                        log.LogWarning(ex, "outbox publish failed {Id} attempt={Attempts}",
                            m.Id, m.Attempts);
                        if (m.Attempts >= 10) {
                            // Move to a separate dead-letter table; alert.
                        }
                    }
                }
                await db.SaveChangesAsync(ct);
            }
            catch (Exception ex)
            {
                log.LogError(ex, "outbox loop crashed; backing off");
                await Task.Delay(TimeSpan.FromSeconds(2), ct);
            }
        }
    }
}

Three details that separate a textbook outbox from a production one:

  • SKIP LOCKED. Without it, every pod grabs the same first 100 rows, three pods see 33 each through MVCC, and you serialise. With it, each pod gets a disjoint slice. Postgres ≥ 9.5 supports it; SQL Server uses READPAST.
  • Partial index on PublishedAt IS NULL. A million published rows shouldn't slow down the drain query. The index only stores the few thousand un-published.
  • The outbox row's Id becomes the message-id. Consumers can dedupe with it. Re-publishing the same row is harmless — at-least-once delivery, idempotent consumer.
When CAN you skip the outbox?

If the consumer can tolerate some events being lost and you never need to correlate a DB state to an event — for example, fire-and-forget metrics, request logs, low-stakes notifications. In every other case, the outbox is the smallest amount of complexity that buys you sanity.

05Idempotent consumers.

The other half of the contract. If the producer can deliver twice, the consumer must handle twice. Two patterns; pick by data volume.

C#   IdempotentConsumer.cs 
// Pattern A — dedupe table. Strong, audit-friendly, costs a row per message.
public sealed class OrderEventConsumer(ShopDbContext db, IPaymentClient pay)
{
    public async Task HandleAsync(Guid messageId, OrderCreated evt, CancellationToken ct)
    {
        // Atomic: insert dedupe row + do business work in the same tx.
        await using var tx = await db.Database.BeginTransactionAsync(ct);
        try
        {
            db.ProcessedMessages.Add(new ProcessedMessage(messageId, DateTimeOffset.UtcNow));
            await db.SaveChangesAsync(ct);     // unique key on Id → 23505 if duplicate

            await pay.ChargeAsync(evt.OrderId, evt.Total, ct);
            await tx.CommitAsync(ct);
        }
        catch (DbUpdateException ex) when (IsUniqueViolation(ex))
        {
            // Already processed; ack and move on.
            await tx.RollbackAsync(ct);
        }
    }

    static bool IsUniqueViolation(DbUpdateException ex) =>
        ex.InnerException is PostgresException { SqlState: "23505" };
}

// Pattern B — natural idempotency. No dedupe row needed at all.
// Just write the operation to be the same regardless of how many times it runs.
public sealed class ProductPriceConsumer(ShopDbContext db)
{
    public async Task HandleAsync(ProductPriceUpdated evt, CancellationToken ct)
    {
        // SET, not INCR. Conditional on version, not on prior state.
        await db.Products
            .Where(p => p.Id == evt.ProductId && p.Version < evt.Version)
            .ExecuteUpdateAsync(s => s
                .SetProperty(p => p.Price,   evt.NewPrice)
                .SetProperty(p => p.Version, evt.Version), ct);
    }
}

Pattern A is universal. Pattern B is better when you can pull it off — no extra table, no extra round-trip. The event payload simply contains everything needed for the operation to be a no-op on replay (a monotonic Version, an absolute NewPrice rather than a delta).

How to read the table. The honest cost of each row isn't the broker bill — it's the people-time. Confluent Cloud at $500/mo is cheaper than one quarter-time platform engineer babysitting your own Kafka. Service Bus Standard at $10 is cheaper than the RabbitMQ HA cluster nobody on your team wants to upgrade.

The fork in the road is clean: log model if multiple independent consumer groups will read the same stream and you need replay; broker model for everything else, which is most things. People over-pick Kafka because it sounds like the serious answer. Service Bus or SQS is the serious answer 80% of the time.

07Decision guide.

"Run this work asynchronously"

SQS or Service Bus standard queue. Outbox to it. Single competing-consumer pattern. Don't reach for Kafka.

"Multiple services react to this event"

Event Hubs / Kafka / Service Bus topic (with subscriptions). Each consumer group tracks its own offset.

"We might rebuild a downstream from history"

Log model (Kafka / Event Hubs) with long retention. Set retention high enough to outlive your worst outage.

"Strict per-customer ordering"

Partition by customer id (Kafka) or use Service Bus sessions. Ordering is per-partition / per-session, never global.

"Scheduled / delayed delivery"

Service Bus scheduled enqueue or SQS message-timer. Not Kafka — its scheduling story is bolted on.

"< 5k msg/s, want to ship fast"

Postgres + SKIP LOCKED as a queue. Plan the exit ramp on day one. When you outgrow it, the outbox carries over.

Three production lessons, learned the hard way

① Always have a DLQ. A queue without a dead-letter queue is a queue that will eventually back up to infinity over one bad payload. SB and SQS give you DLQs free; Kafka requires you to build it.

② Don't put schemas in messages. Use Schema Registry (Confluent / Apicurio) or version your event types in the topic name (order.created.v1). Forwards/backwards compat is a contract, not a hope.

③ Consumer lag is the only SLI that matters. Alert on lag, not on producer rate. Lag tells you the system is in trouble before users notice.
