What an LLM is, for the rest of us.

Tokens, context windows, attention, sampling, temperature, prompt caching. The dozen concepts you actually need to know to build with LLMs effectively — without claiming you understand transformer internals.

The honest scope

What you need to ship LLM features and not be confused by the API docs. Not training, not architecture papers.

The mental model

An LLM is a function: text in, probability distribution over the next token out. Wrap that in a loop and you have generation. Almost every concept derives from that primitive.

01Tokens are the unit.

An LLM doesn't see characters or words; it sees tokens. A token is roughly a sub-word — common words ("the", "system") are single tokens; rare words ("antidisestablishmentarianism") split into multiple. For English text, 1 token ≈ 4 characters ≈ 0.75 words; 1000 tokens ≈ 750 words. For other languages, the ratio is worse (often 2–3× more tokens per character).

Pricing is per-token. Context-window limits are per-token. Throughput is per-token. Every operational metric in the LLM ecosystem is in tokens. When you read "GPT-4 has a 128k context", that's 128,000 tokens — roughly 96,000 English words.

Why this matters

A 10-page document is ~6,000 tokens. A long codebase file is 10–30k tokens. A long agent conversation accumulates tokens fast. The first cost-optimization is "are we sending tokens we don't need to send?" Trim system prompts, summarize history, prune retrieval results.

02Context window.

The context window is the maximum number of tokens the model can attend to in one call — both your input and its output. Modern frontier models: 128k to 2M tokens. Exceeding it errors out. Approaching it degrades quality (most models attend less reliably to the middle of very long contexts — the "lost in the middle" effect).

Practical context budgeting

Reserve space for output. If the model's window is 128k and you expect 4k tokens of response, leave 4k headroom — don't fill input to 128k. Plan for ~75% input, ~25% output as a starting heuristic, adjust by use case.

"More context is not always better"

Adding more retrieved chunks to the prompt past a point hurts quality. The signal-to-noise ratio matters more than the absolute amount of information. The art of RAG is putting the few most-relevant chunks in front; the art of prompt design is keeping the system prompt tight.

03Sampling, temperature, top-p.

At each step, the model outputs a probability distribution over the next token. Sampling picks one. Three knobs:

Temperature

Reshapes the distribution. temperature=0 picks the highest-probability token always (deterministic-ish, repetitive). temperature=1 uses the raw distribution. temperature=2 flattens — picks more variety, more creative, more hallucinations. For structured tasks (extraction, classification, code): 0–0.3. For creative tasks: 0.7–1.0.

Top-p (nucleus sampling)

Cuts off the tail of the distribution. top_p=0.9 means "sample from the smallest set of tokens whose cumulative probability exceeds 0.9." Filters out implausible tokens. Often used together with temperature; top_p=0.95 + temperature=0.7 is a common default for chat.

Top-k

Even simpler: only consider the top K most-likely tokens. top_k=40 is a default; rarely tuned in modern APIs.

Stop sequences

Strings that, if generated, cause the model to halt. Use them for structured output: "stop on \n---\n", "stop on </code>". Without stop sequences, the model wanders past your intended response.

04Prompting moves that matter.

System prompt — set the persona, the rules, the output format

The system message defines who the model is. Short and concrete beats long and vague. "You are a customer support agent for an e-commerce store. Respond in 2–3 sentences. If you're unsure, escalate to a human." Six lines is often plenty.

Few-shot examples

Show, don't tell. Give 2–5 examples of input → desired output. The model learns the pattern from the examples better than from instructions. For classification, extraction, and formatting tasks, 3 examples often beat any amount of prose.

Structured output

"Respond in JSON" works inconsistently. Use the API's structured output feature: OpenAI's response_format, Anthropic's tool use, Gemini's structured generation. The model is constrained to emit valid JSON matching your schema. Don't parse free-form JSON from chat responses; let the runtime enforce the schema.

Chain-of-thought

"Think step by step before answering." Gives the model space to reason; improves accuracy on math, logic, multi-step questions. Modern models do CoT automatically (especially "reasoning" models like o1, Claude 4 thinking, Gemini Pro). You don't usually need to prompt for it; sometimes you need to prompt against it ("answer directly, no preamble").

05Cost & latency.

QuantityTypical value (2025)
Input tokens / sec (small models)~50k
Output tokens / sec (frontier)50–200
TTFB (time to first token)~300ms – 2s
Cost / 1M input tokens (frontier)~$3 – $15
Cost / 1M output tokens (frontier)~$15 – $75
Cost / 1M tokens (small)~$0.10 – $0.50

Output tokens cost ~5× input tokens

The asymmetry is a constraint. Keep responses short when you can; reply length is the dominant cost driver in chat. "Be concise" in the system prompt actually saves you money.

Prompt caching

Modern APIs (Anthropic, OpenAI) cache the KV state of repeated prefixes. If your system prompt is identical across calls, cache it once and pay ~10% for re-reads. Big savings on chat-style workloads with a fixed system prompt. Use cache markers in the API as documented.

Batch API for non-realtime

OpenAI and Anthropic both have batch APIs that process requests within 24h at 50% cost. For evals, content generation, document processing — anything where you can wait — use batch. The cost saving compounds at scale.

Model tiering

Don't use the biggest model for everything. Route by complexity: a small classifier triages the request; trivial ones go to a small fast model; complex ones to the big model. Saves 10× cost with minimal quality drop on the trivial cases.

06Tradeoff matrix.

Choice What you gain What you give up
Frontier modelOpus / GPT-4 / Gemini Pro Best reasoning, most reliable 10× cost, 5× latency
Small modelHaiku / GPT-4o-mini Fast, cheap, fine for simple tasks Worse on reasoning, more hallucinations
Reasoning modelo1 / Claude thinking Better at hard reasoning Slower (extended thinking time), higher cost

High temperature Creative variety Less reliable, hard to test
Low temperature Reproducible, structured Repetitive, less interesting
Five practical rules

One — model tiering saves money. Two — structured output APIs beat parsing JSON yourself. Three — temperature 0 for extraction, 0.7 for chat. Four — cache the system prompt. Five — evals or it's not engineering.

"The model is the substrate. The product is the prompt, the retrieval, the tools, and the eval."
