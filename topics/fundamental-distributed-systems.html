The eight fallacies, and what to do about them.

"The network is reliable." "Latency is zero." "Bandwidth is infinite." Peter Deutsch's 1994 list of fallacies still names the bugs production teams ship today. The discipline of distributed systems is mostly the discipline of not believing them.

The fallacies

Network reliable. Latency zero. Bandwidth infinite. Network secure. Topology stable. One admin. Transport cost zero. Network homogeneous.

The acronyms to know

CAP, PACELC, FLP, BASE, ACID. Each names a tradeoff that becomes architectural.

02CAP and PACELC.

CAP

Brewer's theorem. During a network partition, you must choose: Consistency (refuse writes) or Availability (accept writes, reconcile later). You cannot have both. The "C" in CAP is linearisability; weaker consistencies don't trigger the trade.

Dynamo chose A. Spanner chose C. Most modern databases let you tune it. See consistency patterns for the spectrum.

PACELC

CAP only describes partition behaviour. PACELC extends: during a partition (P) pick A or C; else (E) pick latency (L) or consistency (C). In normal operation, faster-but-eventual or slower-but-strong is also a choice. Cassandra is PA/EL; HBase is PC/EC; MongoDB is PA/EC.

03Time is the hard one.

In a single process, time is monotonic. In a distributed system, every node's clock disagrees by milliseconds (NTP-synced) or hours (drifting). You cannot order events from different nodes by wall-clock without ambiguity.

Lamport clocks

Logical counter per node, incremented on each event, attached to every message. On receive, set local counter to max(local, received) + 1. Establishes happens-before ordering for events that are causally related; says nothing about concurrent events.

Vector clocks

Per-node counter for every node in the cluster. Detects concurrent (causally-unrelated) writes; gives clients the data to merge. Dynamo used these. Application has to know how to reconcile concurrents.

TrueTime

Google's hardware-clock-bounded interval (see Spanner paper). Returns "now is between X and Y." Lets the database wait out clock uncertainty to provide global ordering. The cheating answer: pay for atomic clocks.

HLC — hybrid logical clocks

Wall-clock combined with a logical counter. Stays close to physical time but corrects for skew. CockroachDB, YugabyteDB use HLCs — the software-only approximation of TrueTime.

04Consensus.

The problem: N nodes must agree on a value (the next entry in a log, the current leader, the cluster config). FLP impossibility (1985): in an asynchronous network with even one possible crash, no deterministic protocol can guarantee consensus in bounded time. Practical protocols (Paxos, Raft) work because they assume eventual progress, not bounded time.

Paxos

The 1989 protocol. Two phases: propose, accept. Notoriously hard to understand. Multi-Paxos extends to a sequence of values (a log). Underneath Google Spanner, Chubby, Zookeeper's Zab.

Raft

2014 reimagining of Paxos for understandability. Roles: leader, follower, candidate. The leader replicates a log to followers; commit when a majority acks. Election by timeout + vote. The protocol that most modern systems use — etcd, Consul, TiKV, CockroachDB.

Why quorum?

For N nodes, a majority (N/2+1) is the minimum that guarantees consistency. Two majorities cannot exist simultaneously in a partition — at most one side has the majority; the other side correctly refuses to act. This is the deepest property of consensus systems.

06Decision guide.

Designing fresh

Start with single-machine. Add distribution only when the workload requires it. Most products never need actual distribution.

Need consistency

Use a system that provides it (Spanner, CockroachDB, classic SQL with replication). Don't reinvent consensus.

Need availability under partition

Pick an AP system (Cassandra, Dynamo, Riak). Plan for conflict resolution. Use CRDTs where the data fits.

Suspect ordering bugs

You're probably trusting wall-clock somewhere. Look for cross-node timestamp comparisons; replace with logical clocks or fencing tokens.

Cluster membership

Use what your platform provides (K8s, Consul). Don't roll your own gossip without a strong reason.

Want to learn

Read Kleppmann's "Designing Data-Intensive Applications." It's the modern textbook. Then the original Paxos and Raft papers.
