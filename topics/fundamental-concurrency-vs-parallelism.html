"Doing many things" vs "doing many things at once".

Rob Pike's distinction is one of the most useful sentences in the field. Concurrency is structure: many independent things in flight. Parallelism is execution: many things running simultaneously on different cores. Concurrency is a design choice; parallelism is an implementation detail.

The Pike line

"Concurrency is about dealing with many things at once. Parallelism is about doing many things at once."

Why it matters

Concurrent design opens the door to parallel execution. But you can be concurrent without parallel (1 core, event loop) and parallel without concurrent (vector instructions, GPU). The terms aren't synonyms.

01The two words.

Concurrency is a structural property of a program: independent computations that can interleave. A single-core machine running an event loop is concurrent — many in-flight requests, only one executing at any instant. A web server with 1000 open connections is concurrent regardless of how many cores it has.

Parallelism is a hardware property of execution: computations physically running at the same wall-clock moment. Requires multiple cores (or vector lanes, or GPU threads). A program with no concurrent structure can still be parallel — e.g., matrix multiplication on a GPU.

The matrix

StateWhat it looks likeExample
NeitherOne thing at a timeBash script
Concurrent, not parallelMany in flight, one coreNode.js, Python asyncio on 1 CPU
Parallel, not concurrentOne task across many coresNumPy / BLAS, GPU shader
BothMany concurrent tasks across many coresGo web server, Erlang, modern .NET
"Concurrency enables parallelism. It doesn't require it. The distinction is the design tool."

02Concurrency models.

How do you express "many things in flight"? Different languages picked different answers.

Threads

OS-level threads. Heavyweight (~1 MB stack each), scheduled by the kernel. The default in older Java, C, C++. Hard to use safely at high counts. Limited to thousands per process.

Event loop

Single-threaded, callback-driven. Concurrency without parallelism. Node.js, classic Python single-threaded servers. Simple mental model; CPU-bound work blocks the loop.

Async / await

Cooperative concurrency expressed with await points. Modern .NET, Python asyncio, JavaScript async, Rust async. Avoids callback hell; runtime decides scheduling. Can be parallel if the runtime uses multiple threads.

Goroutines / virtual threads

Lightweight runtime-managed "user-space threads". Go's goroutines (~few KB), JVM 21+ virtual threads. Scale to millions per process. Programmer writes blocking-looking code; runtime multiplexes onto OS threads.

Actors

See actor model chapter. Concurrency expressed as isolated actors + message passing.

03Parallelism models.

SIMD / vector

One instruction on many data points. CPU vector units (AVX, NEON). Compilers auto-vectorise simple loops. Big speedup for matrix math, image processing, string scanning.

Multi-core (SMP)

Threads or processes pinned to different cores. The standard parallelism story. Bottleneck shifts from compute to memory bandwidth and lock contention.

GPU

Thousands of simple cores. Massive parallelism for embarrassingly-parallel problems (rendering, neural networks). Different programming model (CUDA, Metal); the algorithm must fit the architecture.

Distributed

Parallel across machines. MapReduce, Spark, Dask. Network is the new bottleneck. Parallelism that crosses the network is a different design problem entirely; see distributed systems chapter.

04Decision guide.

I/O-bound web service

Concurrency without much parallelism. Async/await or goroutines. The work is waiting on the network, not on CPU.

CPU-bound number crunching

Parallelism, not concurrency. SIMD where it fits; multi-core thread pools; GPU for the hot inner loop if you're in that league.

Mixed

Most production systems. Async/await with a pool of CPU-bound workers. Modern .NET and JVM excel at this shape.

Many independent stateful entities

Actor model. Each entity is concurrent and independently routable; the runtime parallelises across cores.

Heavy math, big data

GPU for inner loops; cluster compute (Spark, Ray) for the volume. Numpy / Polars in Python for in-memory analytics.

"I have a parallelism problem"

Profile first. Usually it's a concurrency problem misdiagnosed. The fix is async/await, not bigger thread pool.

"Pick concurrency for structure. Pick parallelism for speed. They're different choices."
