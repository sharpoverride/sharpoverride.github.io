"The site is down." Top to bottom.

A user can't reach your website. Where's the problem? Use the request path as the troubleshooting tree: DNS, TCP, TLS, HTTP, app, database. Test each step; the first failure is the symptom; the symptom often is the bug.

The toolkit

dig, curl -v, traceroute, nc, openssl s_client. Plus the browser DevTools network panel.

The discipline

Bisect. Don't guess. Each test eliminates half the possible causes.

01The decision tree.

Walk the request from top to bottom. Each layer either works or doesn't; the first layer that doesn't is where to dig.

  1. DNS — does the name resolve?
  2. TCP — does the port accept connections?
  3. TLS — does HTTPS complete a handshake?
  4. HTTP — does the server respond with anything?
  5. App — is the response correct, or 5xx?
  6. DB / downstream — is the app failing because something it depends on is?

Test each. The first "no" is the failure layer.

"Diagnose by bisection. Half-eliminate at every step."

02DNS.

sh   dig
$ dig example.com
$ dig @1.1.1.1 example.com         # bypass local resolver
$ dig +trace example.com           # walk the hierarchy
$ dig AAAA example.com             # IPv6

If dig against a known-good public resolver (1.1.1.1, 8.8.8.8) returns the right IP but your local stub doesn't, the bug is in your network's DNS chain. If dig against the authoritative NS fails, the bug is at the DNS host. See DNS chapter.

03TCP.

sh   connect
$ nc -zv example.com 443           # can we reach the port?
$ traceroute example.com           # where in the path
$ mtr example.com                  # continuous traceroute, packet loss

If nc connects, TCP is fine. If it hangs, packets aren't getting through; traceroute finds where. Watch out for asymmetric routing — packets get there; replies take a different path and time out.

04TLS.

sh   openssl s_client
$ openssl s_client -connect example.com:443 -servername example.com
  ... verify return code: 0 (ok)             # what you want
  ... verify return code: 18 (self-signed)   # cert chain broken
  ... verify return code: 10 (expired)       # cert expired

Cert expired is the famous one. Cert mismatch (no SAN for the domain) is the second. Chain broken — an intermediate is missing — is the silent killer (works in browsers because they fetch the chain on their own; fails in libraries that don't).

05HTTP.

sh   curl -v
$ curl -v https://example.com/
  > GET / HTTP/2
  < HTTP/2 502 Bad Gateway              # upstream is down
  < HTTP/2 504 Gateway Timeout          # upstream too slow
  < HTTP/2 500 Internal Server Error    # app bug
  < HTTP/2 503 Service Unavailable      # rate-limited or maintenance

Each HTTP error has a different cause class. 502 ≠ 503 ≠ 504; reading the code is the diagnosis. curl -v shows headers — X-Request-ID is useful for tracing in logs.

06App / DB.

If HTTP returns and the body is wrong (or 5xx), you've narrowed to "the app misbehaves." Now logs, metrics, traces.

The standard log triage

Filter by request-id from the failing request. Follow the trace through the app, into downstream calls, into the database. The first error in the trace is usually root cause. 5xx without a stack trace is a config / infra problem; with a stack trace, an app bug.

Database

Slow queries, locks, connection-pool exhausted, replica lag. Check active connections — if the count is at the pool max and growing, the bottleneck is downstream. SELECT * FROM pg_stat_activity (Postgres) or equivalent.

"It works on my machine"

The failure is environment-specific. DNS, TLS, network policies, secrets. The bisection still applies; just compare your machine's path against the broken one.

The intermittent bug

Half the requests fail. Likely one of N replicas is broken — load balancer routes randomly. Check per-replica error rates; isolate the bad pod; drain.

"Troubleshooting is a skill. The skill is the discipline of testing each layer instead of guessing."
