Caching patterns.

Five patterns, two hard problems, and the L1/L2 architecture that takes Postgres from "the bottleneck" to "the durable backup."

The two hard things

Phil Karlton was right. Naming things and cache invalidation. We can't help with the first one. The second one is mostly what this chapter is about.

The thesis

Pick the pattern by who owns the writes. Cache-aside when reads dominate. Write-through when freshness is law. Write-behind when throughput is law and you can afford to lose a write.

01Why we cache.

A cache is a wager. You bet that the cost of maybe being slightly wrong is less than the cost of always being slow. For 99% of read traffic on 99% of products, that wager pays off — Postgres takes ~3 ms for an indexed lookup, Redis takes ~0.2 ms, and IMemoryCache takes ~50 ns. Three orders of magnitude buys you a lot of forgiveness.

But you don't add a cache to solve a problem. You add a cache because you've already measured three things:

  • Read/write ratio. If you write as often as you read, the cache hit-rate plummets and you're paying for two systems to do the job of one.
  • Working set size. If the hot working set fits in RAM, you'll get 95%+ hit rates. If it's the whole database, you'll get 5%.
  • Staleness budget. How wrong are you allowed to be, for how long? "Product price" is 5 minutes. "Account balance" is 0 ms.
"A cache without a measured hit-rate is just a way to make production outages more interesting."

02Five patterns.

Caching patterns differ on one axis: who is responsible for keeping the cache and the database in agreement? The application, the cache, or nobody (let it expire).

Fig. 02Five patterns · data-flow · where the latency lives

Cache-aside is the default. The app reads the cache, falls back to the database on miss, fills the cache on the way back. It's the only pattern where the cache being down is merely "slow" rather than "broken." This is the pattern you'll write in 90% of services.

Read-through hides the database from the app — the cache library fetches on miss. Tidier code, but you're coupled to a cache that knows how to query your DB. (Most useful when the cache is a managed product like Cosmos integrated cache or DAX.)

Write-through guarantees the cache is never stale. Every write goes to both stores synchronously. Pay for the freshness with doubled write latency.

Write-behind (aka write-back) acknowledges the write to the client after the cache write, batches to the DB asynchronously. Wonderful throughput; if the cache process crashes before the batch flushes, you've just lost writes. Use only when the source of truth is, in fact, the cache — counters, leaderboards, session data.

Refresh-ahead proactively refreshes a key before it expires. The hot key never thunders past your DB. Great for predictable, expensive-to-compute objects: homepage feeds, ranking results.

03The L1/L2 stack.

One cache is good. Two caches — one in-process (L1), one networked (L2) — is better, and is how every serious service operates above ~10k RPS. The in-process cache absorbs the absurdly hot keys (the top 0.1%) at memory-bus speed; the Redis tier holds the warm working set; Postgres remains the durable truth.

Fig. 03 · L1 sits inside the pod; L2 is shared across all pods~50 ns    0.5 ms    3 ms

The interesting design question isn't the architecture, it's the invalidation propagation. If pod A writes to Postgres and invalidates Redis, how do pods B and C know to evict their L1 entry? Three workable answers:

  • Short L1 TTL (30–60 s). Crude but free. Accept the stale window; works for product catalogues, feature flags, user profiles.
  • Pub/sub invalidation. Pod A publishes cache:invalidate product:42; every pod subscribes and evicts. Use Redis pub/sub or Azure Service Bus topic.
  • Client-side caching (RESP3). Redis 6+ pushes invalidations to clients automatically — the L1 becomes a passive mirror. StackExchange.Redis supports this but the ergonomics are still rough.

04Cache-aside, in ASP.NET Core.

The code below is the cache-aside pattern at production strength: L1 + L2, single-flight (only one DB query when N concurrent requests miss the same key), pub/sub invalidation across pods, and stale-while-revalidate so a request never waits for a refresh it could read a slightly-old value through.

C#   TwoTierCache.cs 
// dotnet add package Microsoft.Extensions.Caching.Memory StackExchange.Redis MessagePack
using System.Collections.Concurrent;
using MessagePack;
using Microsoft.Extensions.Caching.Memory;
using StackExchange.Redis;

namespace Academy.Caching;

public sealed class TwoTierCache(
    IConnectionMultiplexer redis,
    IMemoryCache l1,
    ILogger<TwoTierCache> log) : IAsyncDisposable
{
    private readonly IDatabase _db    = redis.GetDatabase();
    private readonly ISubscriber _bus = redis.GetSubscriber();
    private readonly ConcurrentDictionary<string, Task> _inflight = new();
    private const string InvalidateChannel = "cache:invalidate";
    private static readonly MessagePackSerializerOptions MpOpts =
        MessagePackSerializerOptions.Standard.WithCompression(MessagePackCompression.Lz4BlockArray);

    // One subscriber per pod, wired up in DI startup.
    public Task StartAsync() => _bus.SubscribeAsync(InvalidateChannel, (_, key) =>
    {
        l1.Remove((string)key!);
    });

    // The headline API. Single call site for the whole cache-aside pattern.
    public async Task<T> GetOrSetAsync<T>(
        string key,
        Func<CancellationToken, Task<T>> factory,
        TimeSpan ttl,
        CancellationToken ct = default) where T : class
    {
        // L1 — in-process; 50 ns, no network.
        if (l1.TryGetValue(key, out T? hit) && hit is not null)
            return hit;

        // L2 — Redis; ~0.5 ms.
        var raw = await _db.StringGetAsync(key);
        if (raw.HasValue)
        {
            var value = MessagePackSerializer.Deserialize<T>(raw!, MpOpts);
            l1.Set(key, value, TimeSpan.FromSeconds(Math.Min(ttl.TotalSeconds, 30)));
            return value;
        }

        // MISS — single-flight: only one factory() per key in-process.
        var task = _inflight.GetOrAdd(key, k => Task.Run(async () =>
        {
            try
            {
                var fresh = await factory(ct);
                if (fresh is null) return (object?)null;

                var bytes = MessagePackSerializer.Serialize(fresh, MpOpts);
                await _db.StringSetAsync(k, bytes, ttl);
                l1.Set(k, fresh, TimeSpan.FromSeconds(Math.Min(ttl.TotalSeconds, 30)));
                return (object?)fresh;
            }
            finally { _inflight.TryRemove(k, out _); }
        }, ct));

        var result = await task;
        return (T)result!;
    }

    // Call after EVERY successful write to the source of truth.
    public async Task InvalidateAsync(string key)
    {
        await _db.KeyDeleteAsync(key);
        await _bus.PublishAsync(InvalidateChannel, key);  // fan out to all pods
    }

    public async ValueTask DisposeAsync() => await _bus.UnsubscribeAllAsync();
}
C#   ProductService.cs 
// Pattern in use: a repository that goes through the cache.
public sealed class ProductService(TwoTierCache cache, ShopDbContext db)
{
    private static string Key(Guid id) => $"product:v3:{id}";
    private static readonly TimeSpan Ttl = TimeSpan.FromMinutes(10);

    public Task<Product?> GetAsync(Guid id, CancellationToken ct) =>
        cache.GetOrSetAsync(
            Key(id),
            async token => await db.Products
                .AsNoTracking()                            // no change-tracker overhead
                .FirstOrDefaultAsync(p => p.Id == id, token),
            Ttl, ct);

    public async Task UpdatePriceAsync(Guid id, decimal price, CancellationToken ct)
    {
        var rows = await db.Products
            .Where(p => p.Id == id)
            .ExecuteUpdateAsync(s => s.SetProperty(p => p.Price, price), ct);
        if (rows > 0)
            await cache.InvalidateAsync(Key(id));
    }
}

Three production-grade things this snippet quietly gets right:

  • Single-flight in L1. If 500 requests hit a cold key simultaneously, exactly one runs factory() and the other 499 await its task. Without this, you've built a "thundering herd amplifier," not a cache.
  • MessagePack with LZ4. ~5× smaller than JSON for typical entities; ~10× faster to (de)serialise. Memory and CPU both matter at Redis scale.
  • Versioned key prefix. product:v3: — bump the version when the schema changes and the entire old cache becomes inert. No mass FLUSHALL, no thundering migration.

05Invalidation strategies.

If you only remember one thing from this chapter: every cached entry is either explicitly invalidated or implicitly trusted to be slightly wrong. There is no third option. Pick the right mix.

TTL only

Cheapest. Set an expiry; let it die naturally. Use for things where "a few minutes stale" is fine — feature flags, product listings, search facets, public counts.

Write-invalidate

The default for transactional data. After every write, delete the key. Pair with TTL as a safety net for bugs ("the time you forgot to invalidate").

Versioned keys

Encode a schema version (product:v3:42) or an entity version (product:42:v8) into the key. Writes bump the version; old keys age out by TTL.

Event-driven (CDC)

Postgres → Debezium / Azure SQL CDC → Kafka → cache invalidator service. Writes don't need to know about the cache. Operationally heavy; only justified at scale.

Tag-based

Each entry has tags (tags=[user:42, org:8]). Invalidate by tag, fan out to all matching keys. Redis modules (RediSearch) or your own tag-set in a hash.

Stale-while-revalidate

Serve the stale value and kick off a background refresh. A reader never blocks on the DB; the refresh thread eats the latency. Pairs beautifully with refresh-ahead.

Three honest opinions.

  • Redis won. Memcached was the right answer in 2010 — Redis is the right answer in 2026. The "Redis costs more" argument vanishes the first time you reach for sorted sets, pub/sub, or Lua.
  • The CDN is a cache. Treat your CDN as the first tier of your cache hierarchy, not as a deploy artefact. Cache-Control headers earn their keep; stale-while-revalidate and stale-if-error are free wins.
  • Don't cache what's already fast. A 2 ms Postgres lookup behind a 50 ms LLM call does not need a cache. Cache the LLM call.

07Decision guide.

Read-heavy, small mutable

Cache-aside · L1 + L2 Redis · write-invalidate. The textbook pick. Product catalogues, user profiles, feature flags.

Read-heavy, immutable

CDN with long max-age + versioned URLs. JS bundles, static config, public API responses keyed by query string.

Counter / leaderboard / session

Cache is the source of truth (write-behind or skip the DB entirely). Persist with Redis AOF; snapshot to durable storage every N minutes.

Strong freshness required

Write-through with single-writer per key. Or don't cache. "We need it fresh" and "we need a cache" are often in opposition.

Expensive computation (feed, ranking)

Refresh-ahead at 70% TTL. Hot key is always warm; cold computation runs out-of-band.

Multi-tenant SaaS

Prefix every key with tenant:. Set per-tenant memory limits in Redis. Hot-key isolation is a regulatory issue, not a perf issue.

The two failure modes you must design for

Thundering herd. A hot key expires; 10k requests miss simultaneously; your DB falls over. Mitigations: single-flight (above), probabilistic early expiration, refresh-ahead.

Cache stampede on restart. A pod boots with a cold L1 and immediately receives 1k RPS. Mitigations: warm L1 from L2 on startup; rolling deploys (not all-at-once); pre-warm critical keys.
