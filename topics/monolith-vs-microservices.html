Most teams should not split.

Shopify processes 30 TB per minute on a Rails monolith. Stack Overflow ran a single .NET app for a decade. The instinct to split is almost always older than the evidence that the split would help. The right time to extract a service is when the modular monolith stops being modular — and not a day before.

The problem in one line

Distribution is a tax you pay on every line of code that crosses the boundary, every deploy, every page, every onboarding ticket. The cheque clears whether or not you needed to.

The thesis

Three shapes exist: single-process monolith, modular monolith, microservices. Start at the modular monolith. Extract a service only when the cost of the boundary is observably less than the cost of not having one.

01The default-to-split bias.

The team is small. The codebase is six months old. The product hasn't shipped. Somebody draws "auth service · orders service · catalog service · notification service" on the whiteboard. Two months later the team has four repos, three Kubernetes namespaces, a saga (Chapter 09), and a permanent backlog of cross-service migrations. The product still hasn't shipped. The split was the answer to a question the team didn't have yet.

Microservices solve a deployment-coupling problem. Two teams want to ship at different cadences against the same codebase, the build is twenty minutes, the test suite touches every module, a regression in checkout blocks a fix in search. That is the moment a split pays for itself. None of those problems exist at 8 engineers. Splitting before they exist creates them.

"You can move a class between modules in an afternoon. You cannot move a row between databases without a six-week dual-write migration."

Shopify's playbook is the most-quoted because it's the most honest: they ran one Rails monolith for fifteen years, took it to 30 terabytes of traffic per minute, and only then started extracting services. Not because the monolith couldn't take it — it could. Because the team had grown past the point where one codebase per shipping team was sustainable. Scale was the symptom; organisational coupling was the disease.

02Three shapes.

The choice isn't binary. There's a useful middle: the modular monolith — one deployable, many bounded modules with explicit contracts between them. It's where most teams should live.

Fig. 11The modular monolith keeps the contracts of microservices and the latency of a method call. Extract only when you can name the dependency on independent deploy.

What "modular" actually means in code

Three things, in order of importance: (1) one project per module, each with a small public contract assembly and a larger internal implementation assembly. Other modules reference the contracts; nothing else. (2) One DbContext per module; foreign keys never cross. The orders module owns its Orders table; the catalog module owns its Products table; an order's product reference is a value ProductId, not a navigation property. (3) One message contract per cross-module call, even in-process. Use MediatR or a similar mediator so the call site looks the same as the eventual extraction.

That last point is the kicker. When the day comes to extract orders into its own service, the inside of OrdersModule doesn't change. The mediator dispatch becomes an HTTP call. Everything else stays. The extraction is a deployment-topology change, not a rewrite.

03The cost ledger.

The split adds capability. It also adds cost. The ledger is rarely written down because each item is small; the total is the production reality.

Two teams can't ship is a real entry on the buy side. It'll scale better someday is not.

The four real reasons to split

One — independent deploy cadence. Two teams want to ship daily but can't because their tests are entangled. The deploy is the constraint; the split frees them.

Two — different scale axis. Auth handles 100× the RPS of admin; checkout has 10× the latency budget of recommendation. One scale knob can't satisfy both; physical separation lets each scale independently.

Three — different runtime requirements. The ML scoring path needs Python and a GPU node pool; the transactional path is .NET on commodity. There's no reasonable polyglot monolith here.

Four — security or compliance isolation. The PCI-scoped path needs its own network segment, its own audit, its own deploy approval chain. Mixing it with non-scoped code means everything inherits PCI scope. That's expensive.

If you can name one of these clearly, extract. If you can't, you're paying the tax for nothing.

How to extract without breaking everything

The strangler-fig pattern is the only one that consistently works: in a modular monolith, change the mediator dispatch for one module to call an HTTP endpoint that wraps the same module code in a new deployable. Run both — in-process and over-the-wire — behind a feature flag. Cut traffic over gradually. Delete the in-process module when 100% of calls are over the wire. Do not write a new service alongside the old code and run a six-month dual-write migration. That is how every "modernisation" lands two years late and 50% done.

04In ASP.NET Core.

A module's public surface is a single Module.cs that registers the module's services and endpoints. Cross-module calls go through MediatR. The host project knows nothing about any module's internals; it just calls app.UseModule<OrdersModule>().

C#   OrdersModule/OrdersModule.cs 
using MediatR;
using Microsoft.AspNetCore.Builder;
using Microsoft.AspNetCore.Routing;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;

namespace Academy.Modules.Orders;

// Internal — nothing outside this assembly may reference these.
internal sealed class OrdersDbContext : DbContext
{
    public OrdersDbContext(DbContextOptions<OrdersDbContext> o) : base(o) {}
    public DbSet<Order> Orders => Set<Order>();
}

internal sealed class Order
{
    public Guid Id { get; set; }
    public Guid ProductId { get; set; }  // id only — no FK to catalog
    public decimal Amount { get; set; }
    public string Status { get; set; } = "Pending";
}

// Public — the only types other modules see.
public sealed record PlaceOrder(Guid ProductId, decimal Amount) : IRequest<Guid>;

internal sealed class PlaceOrderHandler : IRequestHandler<PlaceOrder, Guid>
{
    private readonly OrdersDbContext _db;
    private readonly IMediator _mediator;

    public PlaceOrderHandler(OrdersDbContext db, IMediator mediator)
    { _db = db; _mediator = mediator; }

    public async Task<Guid> Handle(PlaceOrder r, CancellationToken ct)
    {
        // Cross-module call via mediator — same call site after extraction.
        var price = await _mediator.Send(new Academy.Modules.Catalog.GetPrice(r.ProductId), ct);
        if (price != r.Amount) throw new InvalidOperationException("price changed");

        var order = new Order { Id = Guid.NewGuid(), ProductId = r.ProductId, Amount = r.Amount };
        _db.Orders.Add(order);
        await _db.SaveChangesAsync(ct);
        return order.Id;
    }
}

public static class OrdersModule
{
    public static IServiceCollection AddOrdersModule(
        this IServiceCollection services, string conn)
    {
        services.AddDbContext<OrdersDbContext>(o => o.UseSqlServer(conn));
        services.AddMediatR(c => c.RegisterServicesFromAssembly(typeof(OrdersModule).Assembly));
        return services;
    }

    public static IEndpointRouteBuilder MapOrdersEndpoints(this IEndpointRouteBuilder app)
    {
        app.MapPost("/orders", async (PlaceOrder req, IMediator m, CancellationToken ct) =>
        {
            var id = await m.Send(req, ct);
            return Results.Created($"/orders/{id}", new { id });
        });
        return app;
    }
}

The host's Program.cs wires every module the same way. The day the orders module needs to be extracted, you change the orders IRequestHandler registration to a thin HTTP client that posts to order-svc:5000 — call sites are identical.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddOrdersModule(builder.Configuration.GetConnectionString("Orders")!);
builder.Services.AddCatalogModule(builder.Configuration.GetConnectionString("Catalog")!);
builder.Services.AddAuthModule(builder.Configuration.GetConnectionString("Auth")!);

var app = builder.Build();

app.UseAuthentication();
app.MapOrdersEndpoints();
app.MapCatalogEndpoints();
app.MapAuthEndpoints();

app.Run();

05The detail.

No foreign keys across modules

The instant you add builder.HasOne(o => o.Product).WithMany().HasForeignKey(o => o.ProductId), you've welded orders to catalog. Extracting becomes a schema migration on every release. Keep cross-module references as plain ids. Validate via a domain service that calls the other module, not via a database constraint.

One DbContext per module

Modules share the SQL Server instance, not the schema. Each module's DbContext has its own migrations, its own schema name, its own connection string. Tables across modules don't talk to each other in SQL. The cost: you can't write a single JOIN across modules. The benefit: when orders gets extracted, its database is already a separate database — the connection string is the only thing that changes.

Use internal ruthlessly

Default to internal on everything in a module. Only the contract record types and the registration extensions are public. C# enforces the boundary at compile time — the other module literally cannot type the name of your Order entity. This is the cheapest, most under-used architecture tool in the language.

The mediator is the dispatch boundary

Every cross-module call goes through IMediator.Send(). In-process, that's a method dispatch by type. After extraction, swap the handler registration for a typed HTTP client that serialises the request, calls the new service, and deserialises the response. The call site never changes. That is the entire architectural payoff of a modular monolith.

Don't extract for "scale" without measurement

"We're splitting because the monolith won't scale" is almost always wrong. Profile first. The hot path is usually one slow query, one missing index, or one synchronous call that should be async. A modular monolith on a beefy VM scales further than any team thinks — Stack Overflow ran on nine servers at 6,000 RPS for years. Extract for org reasons, not throughput reasons.

"Microservices are an answer to an organisational problem dressed up as a technical decision. Diagnose the problem you actually have."

07Decision guide.

1–4 engineers, pre-PMF

Single-process monolith. Skip even the modular structure until you have two distinct domains that need to evolve at different speeds.

5–30 engineers, single product

Modular monolith. One deploy, one DB instance, multiple DbContexts and module assemblies. The shape that scales to 100k RPS without anyone noticing.

Two teams blocking each other's deploys

Extract the blocking module first via strangler. Run side-by-side for a quarter behind a flag, then cut over. Don't extract everything.

PCI / HIPAA scoped path

Extract the scoped path into its own service and namespace. Compliance isolation is one of the few clean-cut reasons to split early.

ML / batch workload alongside web tier

Separate process and node pool. ML doesn't share runtime requirements with transactional code; co-locating them costs you both.

"We need microservices for scale"

Show me the profile. Nine times out of ten, the bottleneck is one query. Optimise first; if the modular monolith still can't fit, then extract.

Three things people get wrong

One — extracting too early. Two months in, the team splits into four services and inherits four pipelines. The product hasn't shipped. Distribution should follow demonstrated need, not architecture-astronaut aesthetics.

Two — foreign keys across the boundary. An Orders.ProductId FK to Catalog.Products welds the modules together at the database level. When you eventually extract, you discover you can't — every migration is cross-service. Keep cross-module references as ids only.

Three — extracting only for "team velocity". If the team is one team, splitting the codebase doesn't help anyone deploy faster — it just adds coordination overhead. The split pays off when two distinct teams own two distinct services. Below that line, the modular monolith wins on every dimension.
