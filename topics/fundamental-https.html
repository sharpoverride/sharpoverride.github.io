What the lock actually means.

HTTPS is HTTP inside a TLS tunnel. The browser, the server, and a chain of certificate authorities cooperate so that nobody between them can read or modify the traffic. The handshake takes one round-trip in 2026 — down from three in 2010.

What HTTPS gives you

Three properties: confidentiality (eavesdroppers see ciphertext), integrity (tampering is detected), authenticity (the server is who it claims).

What it doesn't

It doesn't authenticate the client unless mTLS is configured. It doesn't hide who you're talking to (SNI is visible, though encrypted-SNI is shipping). It doesn't protect against a compromised server.

01Three problems, one protocol.

Plain HTTP travels in the clear. A coffee-shop Wi-Fi router, your ISP, every transit network between you and the server can read every byte. They can also modify it — inject ads, replace responses, capture passwords. None of these adversaries are theoretical; all of them have been documented in the wild.

HTTPS solves three problems that have to be solved together. Confidentiality: ciphertext on the wire. Integrity: a MAC detects tampering. Authenticity: the server proves it owns the domain via a certificate signed by a trusted CA. Drop any one of these and the other two are nearly worthless — encrypted traffic to an attacker is still talking to an attacker; an authenticated server you can't trust isn't who you wanted.

"The padlock isn't a feature. It's the absence of a long list of attacks."

02The TLS 1.3 handshake.

Fig. 24TLS 1.3 finishes the handshake in one round-trip. TLS 1.2 needed two. The win came from removing legacy key-exchange options and inlining the cert.

What's in the ClientHello

The client announces: which TLS versions it supports, which cipher suites, the hostname it wants (SNI), and — new in TLS 1.3 — a key share for the ECDHE key exchange. That key share is the trick that compresses the handshake. Instead of waiting for the server to pick a cipher, the client speculatively sends a key offer for the most common curves. The server picks one and replies with its own key share in the same round-trip.

ECDHE: ephemeral key exchange

Elliptic-curve Diffie–Hellman ephemeral. Each side generates a temporary key pair for this connection. They exchange the public halves; each side computes the same shared secret without ever transmitting it. The session key is ephemeral — discarded when the connection ends. This is what gives TLS 1.3 forward secrecy: even if the server's private key is later compromised, traffic recorded today cannot be decrypted retroactively.

0-RTT resumption

For a returning client (same browser, same server, recent connection), TLS 1.3 supports 0-RTT: send the HTTP request together with the ClientHello, using a pre-shared key from the previous session. The HTTP response comes back in 1 RTT total. This is fast but has a replay caveat: an attacker who captures the 0-RTT data can replay it; only use 0-RTT for safely-replayable requests (GETs of public resources). POSTs go through the standard 1-RTT path.

03The trust chain.

Certificate authorities

Your browser trusts a list of root CAs — Microsoft, DigiCert, Let's Encrypt's parent ISRG, etc. — baked in. A server proves its identity by presenting a certificate signed by one of these (or by an intermediate CA, in turn signed by a root). The browser walks the chain: server cert → intermediate → root. If the root is on the trusted list and every signature validates, the browser accepts.

How Let's Encrypt changed the game

Pre-2016: a TLS cert from a commercial CA cost $50–$500 per year, required manual application, was renewed by emailing PDFs. Let's Encrypt made it free and automated. The ACME protocol lets a server prove control of a domain (via DNS or HTTP challenge) and receive a 90-day certificate in seconds. 90 days is a feature: the short lifetime forces automated renewal, which forces correct configuration, which makes the ecosystem safer.

Certificate transparency

Every certificate issued by a public CA is logged to append-only certificate-transparency logs. Browsers refuse certs not present in a log. This makes mis-issued certificates discoverable. If a CA was tricked into issuing a cert for your domain to an attacker, you can detect it because the log is public. Without CT, mis-issuance was undetectable; with CT, it's a public event.

04In ASP.NET Core.

Kestrel's defaults are mostly correct. The configuration you actually need to touch: HSTS, certificate provisioning, and TLS version pinning.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddHsts(o =>
{
    o.Preload           = true;
    o.IncludeSubDomains = true;
    o.MaxAge            = TimeSpan.FromDays(365);
});

builder.Services.AddHttpsRedirection(o =>
{
    o.HttpsPort         = 443;
    o.RedirectStatusCode = StatusCodes.Status308PermanentRedirect;
});

builder.WebHost.ConfigureKestrel(o =>
{
    o.ConfigureHttpsDefaults(https =>
    {
        // TLS 1.2 minimum; let OS negotiate 1.3 when supported.
        https.SslProtocols = SslProtocols.Tls12 | SslProtocols.Tls13;
    });
});

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseHsts();             // browsers will refuse plain HTTP for 1 year
    app.UseHttpsRedirection(); // 308 redirect plain HTTP to HTTPS
}

app.MapGet("/", () => "hello over TLS");
app.Run();

Certificate provisioning

Three production options: terminate TLS at the load balancer (Azure Front Door, AWS ALB, NGINX) and let it handle certs — this is the default; ACME-Caddy / Win-ACME as a sidecar that automates Let's Encrypt renewal; or certbot with a cronjob. Do not generate certs by hand for production. The 90-day expiry will catch you.

05The detail.

HSTS preload is one-way

Adding your domain to the HSTS preload list means browsers refuse plain HTTP to it forever (until the list is rebuilt). Be confident before you submit; rolling back means waiting for browser updates. Set max-age=300 first, verify, then bump to a year and request preload.

Cipher suites are mostly auto-managed

TLS 1.3 ships with five cipher suites; all are good. TLS 1.2 has dozens, most of which you don't want. Trust the OS's defaults on modern .NET. The "let me hand-pick my cipher suites" instinct is from 2010; it's a footgun now.

SNI is in the clear (for now)

The hostname (Server Name Indication) is sent unencrypted in the ClientHello — so observers can see which site you're visiting, even if they can't read the traffic. Encrypted Client Hello (ECH) is shipping; it's the right way to hide SNI but isn't universally deployed yet. Don't rely on SNI privacy.

Mutual TLS for service-to-service

For internal service-to-service auth, mTLS is excellent: both ends present certs, both verify, no shared secrets in code. Service meshes (Linkerd, Istio) automate the cert rotation. For internet-facing, mTLS is overkill; for cluster-internal, it's the modern default.

Pin the chain, not the leaf

Certificate pinning used to mean "burn the leaf cert's fingerprint into the app." This made cert rotation a nightmare. Modern guidance: pin the issuer (intermediate or root). The leaf can rotate freely; the chain anchor stays stable. Or use HPKP — which is officially deprecated and nobody should use it now — and just rely on CT logs.

"TLS gets the protocol details right. The application engineer's job is to not undo any of it by mistake."

06Decision guide.

Public website / API

Terminate TLS at Front Door / CloudFront / ALB. Let it handle certs. HSTS with preload. TLS 1.2+.

Self-hosted, single VM

Caddy as the front proxy. Free Let's Encrypt certs, auto-renewed. Zero config; it just works.

Kubernetes

cert-manager + Let's Encrypt issuer. Renewal is automatic. Mount certs as secrets, rotate without restart.

Internal service-to-service

mTLS via service mesh (Linkerd or Istio). The mesh handles cert minting and rotation; your app just enables it.

You inherited cert pinning

Move to chain pinning. Plan the migration carefully — leaf pinning blocks cert rotation; you'll need a brick-by-brick mobile-app rollout.

Custom CA (corporate, dev)

Run a small private PKI (smallstep, HashiCorp Vault PKI). Distribute the root via MDM. Don't try to bypass cert validation in code.
