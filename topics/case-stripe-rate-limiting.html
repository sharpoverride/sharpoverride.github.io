Case study Year 2017 → 2024 Scale 1B+ daily requests Stack Redis + Lua · 4 distinct limiters

Four rate limiters, one request.

Stripe runs four different rate-limiters on every incoming API call. Each protects against a different failure mode. The public blog post from 2017 is the most-cited rate-limiting design in the industry — and the only one where the limit values are calibrated for cardholders.

What runs on every request

(1) Per-user request rate, (2) per-user concurrency, (3) fleet-wide overall, (4) per-endpoint backstop. Each can independently refuse the request.

Why four

Each protects a different resource: customer fairness, downstream concurrency, total fleet capacity, hot-endpoint protection. One limiter is too coarse to be useful at this scale.

Where it runs

Redis with Lua scripts. Atomic, low-latency, surviveable for a global API gateway. Sub-millisecond p99 on the limit check.

01The payment-API problem.

Stripe's API is the company. Slow it down by 50ms across the board and the merchants notice. Take it down for two minutes and the news writes about it. The job of the rate limiter isn't "be fair"; it's "survive everything in the wild while still being fair." A misbehaving integration that loops POSTing charges. A merchant during their Black Friday sale that suddenly sends 100× their normal load. A scraper trying to enumerate customers. A new partner that didn't read the docs.

One rate limit ("100 requests per second per user") is too coarse. It protects the global fleet from one bad user; it does nothing to protect Stripe from thousands of users each doing 99 RPS. And it doesn't differentiate "100 RPS of GET /charges" from "100 RPS of POST /charges." The cost to Stripe of those two patterns differs by a factor of 100: reads hit a cache, writes hit a payment processor.

"Rate limiting is the only feature you ship that customers hate, and you ship anyway, because the alternative is everyone hating you instead."

02The naive limiter.

The starting point everyone reaches first: a token bucket per user, in Redis, refilled at a fixed rate. The API gateway calls INCR user:requests:<id>, checks the count against a limit, decides allow/deny. This is the example you'll find in every "build a rate limiter" tutorial. It works at small scale. It misses three problems that show up at Stripe scale.

03Three failure modes one limiter misses.

Failure mode 1 — long-running requests

A user is "within their RPS limit" but every request they make takes 30 seconds — they're calling a slow endpoint, or their requests get queued behind a slow downstream. The user is consuming a server thread for 30 seconds at a time; ten of them in flight monopolises ten threads regardless of RPS. The limiter that counts requests-per-second is blind to concurrent in-flight requests.

Failure mode 2 — many users, one global cap

10,000 users at 99 RPS each = ~1M RPS globally. Even if each user is "well behaved", the fleet can be overwhelmed. The limiter that's per-user has no notion of total capacity; nothing throttles a sudden synchronised burst across many users (e.g. when Stripe publishes a new feature and every integration starts polling for it).

Failure mode 3 — uneven endpoint costs

GET /customers/cus_X costs Stripe ~1ms of CPU. POST /charges costs ~200ms, touches a payment network, and consumes external partner credit. "100 RPS" of one is fine; "100 RPS" of the other is an incident. The limiter that's blind to endpoint can't price the difference.

04The four-layer design.

Fig. AThe four limiters run in series in the gateway. Each is cheap; the combination is what makes the API both fair and survivable.

Limiter 1: per-user request rate

Token bucket per user, refilled at the user's RPS limit, burst capacity of 5–10×. Implemented as a 5-line Lua script on Redis for atomicity. This is the limiter every tutorial covers and the one users actually see most. The published default for Stripe is ~100 RPS per account, raised on request for high-volume merchants.

Limiter 2: per-user concurrency

A counter incremented on request start, decremented on response. If concurrent in-flight requests for one user exceed (typically) 20, new requests are refused. This is the limiter that catches the "30-second requests at 1 RPS" pattern: rate is fine, concurrency is melting. Redis stores the counter; a TTL ensures it self-heals if the decrement is lost.

Limiter 3: global fleet cap

A counter for total fleet RPS, refreshed periodically against a budget. If the fleet is at 95% of capacity, the limiter starts load shedding — typically priority-aware: failing low-priority traffic (background polling, dashboard refreshes) before mission-critical traffic (charges). This is the protection that prevents a coordinated traffic surge from melting Stripe.

Limiter 4: per-endpoint backstop

Each endpoint has its own absolute ceiling, regardless of who's calling. POST /charges might be capped at fleet-wide 10k RPS — high but finite. The backstop prevents a single endpoint from absorbing the global budget, which would otherwise cause one feature's spike to starve everything else.

06The lessons.

One limiter is not enough

Rate, concurrency, fleet, endpoint — each catches what the others miss. Adding any one of the missing three to a single-limiter system measurably improves resilience for a day's work of code. Adopt them in order: rate first, then concurrency, then fleet, then per-endpoint.

Lua scripts on Redis are the right primitive

Token bucket logic must be atomic — check, decrement, decide — without a network round-trip per step. Redis Lua scripts run on the Redis server atomically; this is the only standard primitive that meets the latency and consistency requirements. You'll see this implementation everywhere; Stripe didn't invent it but they made it famous.

Tell the caller which limiter they hit

The X-RateLimit-Bucket response header naming the limiter is the difference between "your client is broken" and "you need to retry differently." A client hitting limit 1 (rate) can back off; a client hitting limit 2 (concurrency) needs to reduce parallelism, not slow down. The diagnostic header is a customer-experience feature, not a debug aid.

Priority-aware shedding

When you're over global capacity, drop traffic by class — let merchant charge requests succeed; drop the dashboard's polling. This requires the request to carry a priority tag from the client. Without priorities, "shed load" means "make the most important things fail too."

"You can't talk a customer out of a 429. You can give them a header that tells them which 429 they got."
