White paper Year 2012 Authors Corbett et al., Google Stack TrueTime · Paxos · 2PC · external consistency

Globally distributed, linearisable.

"Spanner: Google's Globally-Distributed Database" (OSDI 2012). The paper that did what Dynamo said was impossible — global ACID transactions with external consistency. The trick: a global synchronised clock built from GPS and atomic hardware.

The thesis Dynamo rejected

Strong consistency across continents. Linearisable transactions. SQL. Eventually-consistent NoSQL was the 2007 answer; Spanner argued there was a better one — if you could pay the engineering bill.

The key idea

TrueTime: a clock API that returns an interval, not an instant. By bounding clock uncertainty to a few milliseconds, Spanner can assign globally-meaningful timestamps to transactions.

What it influenced

CockroachDB, YugabyteDB, TiDB, Aurora DSQL, Cosmos DB strong consistency. The "NewSQL" category exists because of this paper.

01The "CAP says no" challenge.

The 2000s consensus, codified by Dynamo: choose two of consistency, availability, partition-tolerance. In a wide-area network, partitions happen, so it's A or C. Most NoSQL systems chose A. Spanner refused the dilemma. Spanner is consistent and available — except during partitions, where it sacrifices availability to preserve consistency. That's CAP-allowed; it's just that nobody had built it for a global system.

Why bother? Google had workloads where eventual consistency was unacceptable: AdWords, where a customer's ad spend has to be globally accurate. F1, the relational store that replaced MySQL for AdWords, was the first internal customer. The cost of getting consistency wrong was direct revenue loss; the team got the budget to build the impossible.

"CAP doesn't say you can't have strong consistency at global scale. It says when partitions happen, somebody pays — and Spanner decided that somebody is availability."

02TrueTime.

The single most-cited innovation in the paper. Spanner needs to order transactions globally. NTP-synchronised clocks across continents have ~100ms uncertainty — not good enough for a database. Google deployed GPS receivers and atomic clocks in every datacenter, built a daemon that aggregates them, and exposed an API:

api   TrueTime.now() 
// Returns an interval, not a single timestamp.
TT.Interval { earliest, latest }

// Where (latest - earliest) is the current clock uncertainty,
// typically <7 ms in practice. The interval is GUARANTEED to contain
// "now" — both clocks above earliest and below latest are physically wrong.

Commit-wait: the trick that makes TrueTime useful

A transaction picks a commit timestamp s. Before releasing locks, Spanner waits until TrueTime says s is definitely in the past (i.e., until TT.now().earliest > s). The wait is ~7ms typically. This guarantees that any later transaction's timestamp is greater than s, even on a different node. External consistency emerges from this single discipline.

Why intervals, not instants

Lying about the clock has a cost — a node thinking it's 100ms ahead would assign timestamps from the future, breaking ordering. By exposing uncertainty as an interval, Spanner forces all the algorithms to handle it explicitly. The clock isn't precise; the protocol acknowledges that.

03Transactions across the world.

Paxos for each tablet

The data is partitioned into "tablets". Each tablet's replicas form a Paxos group — typically 5 replicas across regions. Within a tablet, writes are linearisable: Paxos ensures a consistent order. This is the local consistency story.

2PC across tablets

A transaction that touches multiple tablets uses two-phase commit, coordinated by one of the participating Paxos groups. 2PC is rehabilitated: the famous "blocking on coordinator failure" problem is mitigated because the coordinator is a Paxos group (it doesn't crash; the leader rolls over). The cost is latency — wide-area 2PC pays cross-region round-trips.

External consistency via timestamps

Every transaction gets a timestamp from TrueTime. The commit-wait discipline guarantees that timestamps respect real-time ordering: "if transaction T1 commits before T2 starts, then T1's timestamp is less than T2's". This is stronger than linearisability — it's external consistency, also called strict serialisability. The strongest consistency property in the distributed systems literature.

04What it costs.

Hardware investment is real

GPS receivers, atomic clocks, the TrueTime daemon — Google built and operates physical hardware in every datacenter. This is not a software-only solution. Cloud variants (Cloud Spanner) handle it for you; on-prem implementations of similar designs (CockroachDB, YugabyteDB) use software-based clock-sync (hybrid logical clocks) at the cost of slightly looser guarantees.

Write latency includes commit-wait

Every write pays the ~7ms wait. Bursty workloads don't notice; high-throughput workloads do. Spanner trades absolute write throughput for global ordering correctness.

Cross-region writes hit physics

A transaction spanning North America and Europe pays the cross-region round-trip on its 2PC. ~150ms minimum. Spanner doesn't change the speed of light; it just makes "we have to pay it" the obvious choice when the consistency requirement demands it.

Most apps don't need it

Global linearisable transactions are valuable when you have them, but most applications can tolerate eventual consistency for most data. The Spanner team knew this; the paper was about proving it was possible, not arguing it was always necessary.

06What still applies in 2026.

External consistency is a real product feature

For workloads where audit-trail correctness matters — finance, healthcare, regulatory — the "happens-before" guarantee Spanner gives is the only thing that makes the system defensible. Auditors want serialisable transactions; pay the milliseconds.

Hybrid logical clocks where TrueTime is impractical

If you can't deploy atomic clocks (i.e., you're not Google), HLCs give you most of the benefit. CockroachDB and YugabyteDB both use HLCs. The guarantees are slightly weaker (causal consistency, not strict external) but acceptable for almost everything.

2PC isn't dead

The "2PC is broken" critique of the 2010s assumed an unreliable coordinator. 2PC on top of Paxos is fine because the coordinator is itself fault-tolerant. Modern distributed transactions still use this pattern.

Don't reach for Spanner-class for a one-region workload

If your workload is single-region, a regular SQL database is simpler, faster, and cheaper. Spanner's design overhead is justified by the global-strong-consistency requirement; in-region, you don't need it.

"Spanner is the answer when you need every property the textbook said you couldn't have at once. It's the wrong answer when you don't."
