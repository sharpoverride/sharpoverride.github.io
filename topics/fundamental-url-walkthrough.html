The fifteen steps.

The canonical interview question. The user types example.com and presses enter. What happens? — fifteen separate things, each its own subsystem. The depth at which you can describe each is the depth at which you understand the web.

The categories

(1) Resolve. (2) Connect. (3) Transfer. (4) Render. Each has 3–5 substeps.

The interesting parts

DNS caching, TLS 1.3 1-RTT, HTTP/2 multiplexing, the critical rendering path. Every step has been optimised for milliseconds.

01Resolve (0 – ~30 ms).

1. Browser URL parser

The browser parses the input: protocol, host, port, path, query. If no protocol, it tries https:// first. If the input looks like a search query, it goes to the default search engine instead. This is also where HSTS kicks in — if the host is on the HSTS preload list or the browser has remembered an HSTS directive, plain HTTP is rewritten to HTTPS before any network call.

2. Browser cache check

Already have this page cached and fresh? Serve from disk. Done. The fastest "request" is no request.

3. DNS lookup

OS resolver checks /etc/hosts → local stub cache → recursive resolver (configured DNS server, often the ISP's or 1.1.1.1/8.8.8.8). Recursive walks the hierarchy: root → TLD → authoritative. Returns an A or AAAA record. Cached at every layer; uncached lookups are rare in practice. See DNS chapter.

02Connect (~30 – ~100 ms).

4. TCP handshake

SYN → SYN-ACK → ACK. One round-trip. The client now has an open TCP socket to the server's IP:port. For HTTP/3, this is replaced by QUIC, which combines transport + crypto handshake.

5. TLS handshake

TLS 1.3: ClientHello (with key share, SNI) → ServerHello + cert + Finished, all in one round-trip. Client verifies the cert chain against trusted CAs. Application data flows next. For a recently-visited site, 0-RTT resumption can attach the first HTTP request to ClientHello — saves the round-trip. See HTTPS chapter.

6. HTTP/2 connection setup

Inside the TLS tunnel, HTTP/2 negotiates settings, opens streams. Multiplexing: many requests share one connection. HTTP/3 (QUIC) replaces both TCP and TLS with a single UDP-based protocol; for ~30% of modern web traffic, the above two steps are merged into one.

03Transfer (~100 – ~500 ms).

7. Send the request

GET / HTTP/2 Host: example.com User-Agent: … Accept: … Cookie: …. Compressed via HPACK (HTTP/2's header compression). The cookies attached are determined by the cookie-jar rules: same site, path, scheme, expiration.

8. Through the network

Through the ISP's network, peering points, possibly a CDN edge POP, possibly transit ASNs, into the destination's cloud region's load balancer. The bytes travel at near light-speed; the latency floor is physics, not bandwidth.

9. Load balancer → application

L7 LB (Azure Front Door, ALB, NGINX) inspects the request, picks a backend instance, forwards. Cache check at the CDN happens first; cache hits return locally without touching origin (see CDN chapter).

10. Application + database

App tier picks up the request, authenticates, queries the database (possibly via a cache), renders an HTML response or returns JSON. This is where the system-design content of every other deep dive in this academy applies.

11. Response back

Bytes flow back through the LB, the CDN may cache, through the network, back to the browser. Compressed with brotli or gzip.

04Render (~500 – ~2000 ms).

12. HTML parsing

Browser parses HTML into a DOM tree as bytes arrive (streaming parser). When it encounters <script> tags, it blocks (or doesn't, with async/defer). When it encounters <link rel="stylesheet">, it kicks off the CSS download in parallel.

13. Critical resources

CSS, fonts, hero images. The "critical rendering path" is the set of resources that block first paint; minimising it is half of frontend performance work. <link rel="preload"> hints to the browser to fetch a resource early.

14. Layout, paint, composite

Browser computes box positions (layout), paints pixels into layers, composites layers into the final frame. JavaScript that mutates the DOM invalidates layout; doing this on every scroll = jank.

15. Interactive

The JS runs; event handlers wire up; the page becomes interactive (TTI — Time To Interactive). For SPAs, this is often the moment the user can actually do things.

Each step is somebody's full-time job
