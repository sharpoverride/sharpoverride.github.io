Design WhatsApp.

A real-time messaging system at planetary scale. The interesting parts: connection management for hundreds of millions of concurrent devices, end-to-end encryption, group fan-out, and offline message delivery.

Anchor numbers

2B users · 100B msgs/day · 1.2M msgs/sec avg · ~10M concurrent connections peak per cluster. ~80% mobile, varying network quality.

The hard parts

(1) Keeping hundreds of millions of TCP connections alive. (2) Delivering messages to offline recipients. (3) Group fan-out at low cost. (4) End-to-end encryption.

Out of scope (probably)

Voice/video calling, status stories, business API, search. Pin scope early.

03High-level architecture.

Fig. AMessage hops: sender pod → dispatch → recipient pod → recipient. If recipient is offline, message goes to the offline queue.

Components, briefly

Connection tier: holds persistent TCP/WebSocket connections to clients. Stateless beyond the connection itself; ~10k pods, ~50k connections each. Router/dispatch: receives an incoming message, looks up the recipient's current connection pod, forwards. Session directory: small Redis-shaped store mapping user_id → connection_pod. Updated on connect/disconnect. Offline queue: per-user durable queue of pending messages, drained when the user reconnects. Message store: long-term storage for chat history (if WhatsApp stored content — it doesn't, see below).

04Deep dive — connections.

The connection tier is where the design lives or dies. 500M concurrent TCP connections is the unusual constraint. A pod can hold ~50k connections at most with tuning — Linux socket limits, file descriptors, ephemeral ports, conntrack table size. Each connection idles on a few KB of kernel buffer; aggregate memory is ~10 GB per 50k connections.

Keepalives must be cheap

Most connections sit idle most of the time. Keepalives — ping/pong — are needed to detect dead clients. Tune them long: every 30–60 seconds. Shorter wastes battery on mobile and bandwidth at scale; longer means dead connections linger. WhatsApp uses a custom protocol (modified XMPP); the keepalive cadence is part of the protocol.

Sticky routing

The session directory says "user 12345 is connected to conn-pod-7". When a message arrives for user 12345, the router forwards to conn-pod-7 directly. Reconnect → updates the directory. A user with two devices has two entries; messages are sent to both (or to one and synced). The directory is a Redis hash, sharded, replicated.

05Deep dive — message delivery.

Online recipient

Sender → sender's conn pod → router → recipient's conn pod → recipient. p99 < 500ms is achievable end-to-end in-region; cross-region adds ~150ms. The router can run in the sender's region; the recipient lookup hits the global session directory, which is regionally replicated.

Offline recipient

If the session directory shows no active connection, the message goes to the offline queue (one queue per user, stored durably — typically in a sharded key-value store). When the user reconnects, the conn pod drains their queue in order, delivers the messages, gets ACKs, deletes from the queue.

Delivery and read receipts

Three states: sent (server has it), delivered (recipient device has it), read (user opened the chat). Each transition is its own small message back to the sender. Receipts are just inverted messages: sender becomes recipient, payload is a status update. Goes through the same path.

Group messages — server-side fanout

Group of 256 members. Sender posts one message. Server fans out to each member's conn pod (or offline queue). WhatsApp's published number: groups of up to 1,024 members supported (some tiers higher). Fanout is O(group size) per message; for huge groups, this is a cost knob.

06Deep dive — end-to-end encryption.

WhatsApp uses the Signal protocol. Each pair of users derives a session key via X3DH (Extended Triple Diffie-Hellman) using their long-term public keys (registered at signup, on WhatsApp's key server) and per-session ephemeral keys. The server never has the plaintext. It routes ciphertext.

What's encrypted

The message body. Metadata (sender, recipient, timestamp) is unencrypted — the server needs it to route. This is the fundamental privacy compromise of E2EE messaging: content is private; the social graph is not.

Group keys

Groups use a Sender Keys protocol: each member encrypts their messages with a per-group symmetric key shared with the other members. Adding a member requires re-keying; this is why some E2EE group operations are slow.

Server is a routing fabric, not a content store

Critical consequence: the server does not store message history. Once delivered, the message is gone from the server. The history lives on the user's devices, optionally backed up to user-controlled cloud storage (iCloud, Google Drive) encrypted with a user-held key. This is also why WhatsApp doesn't have search across the whole archive on the server — the server can't see it.

07Scale & bottlenecks.

Connection tier scales horizontally — almost linearly

Add more pods; the session directory updates. The real ceiling is the OS: file descriptors, ephemeral ports. Custom kernels and careful tuning let WhatsApp historically hit ~1M connections per machine in tests; production typically runs lower for headroom.

Session directory is the hot path

Every message does a lookup. Cache the directory aggressively; the router holds a local cache of recent recipients. Misses go to Redis. A directory shard handling 1M lookups/sec is the realistic capacity per shard.

Group fanout is the cost line

A 1000-member group means 1000 fanout writes per message. For a small percentage of users in many groups, this is the dominant cost. Tune by capping group size; if business needs huge groups, build a separate broadcast-list product (one-way, not many-way), which is what WhatsApp did.

Tradeoffs to mention

E2EE limits server features: no server-side search, no spam classification on content, no moderation. Eventual consistency on read receipts: a slow recipient may have read receipts arriving out of order. Offline queue is durable but bounded: if a user is offline for months, the queue is pruned.

Open questions you'd raise

How do we handle multi-device sync? (WhatsApp's recent multi-device support uses an extra "primary device" pairing.) How do we handle voice/video? (Different protocol path; uses WebRTC peer-to-peer with TURN relay fallback.) How do we handle abuse / spam reports? (Hash-based; users can report messages, which forwards the hash and metadata to the moderation team.)

"The hard part isn't the messaging. The hard part is keeping 500 million sockets open and knowing where every one of them is."
