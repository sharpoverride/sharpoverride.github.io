Chapter 10 Difficulty ★★★★☆ Read ~14 min Stack SignalR · Redis · Azure SignalR Service · WebSocket

The server talks first.

HTTP is request/response. The browser asks; the server answers. Real-time inverts the conversation — the server pushes when something happens. WebSockets, Server-Sent Events, and long polling are three different ways to make TCP carry that shape.

The problem in one line

You can't ask the server "what just changed?" every 200ms and call yourself a real-time app. It works. It also costs 200× your bandwidth bill.

The thesis

WebSockets for two-way (chat, collab), SSE for one-way (feeds, dashboards), long polling only when neither survives your network. Scaling any of them requires a backplane the moment you run more than one pod.

01Why polling loses.

The naive answer is "poll every two seconds." For a thousand users on a dashboard, that's 500 requests per second. Most of them return 304 Not Modified or an empty payload. You've turned the absence of news into network traffic. Every request pays for the TCP handshake (in browsers without HTTP/2 reuse), the TLS overhead, the auth check, the database read, the response serialization. The DBA paging you about a hot SELECT loop is the one polling cost you can measure; the bandwidth bill is the one you can't avoid.

The push problem isn't that updates need to arrive fast. It's that the absence of updates should cost nothing. Polling makes silence expensive. Real-time transports invert that: idle is free, traffic is proportional to events. A chat room with no new messages costs the same as one with one user as one with a million users in the room (until somebody types).

"Polling is the architecture you ship when you can't ship the architecture you wanted. Sometimes that's the right call."

02Three transports.

WebSocket, Server-Sent Events, long polling. Same goal, three positions on the duplex-and-complexity spectrum.

Fig. 07The three transports differ in how much TCP is held open and which direction the bytes flow. WebSocket and SSE win on idle cost; long polling wins on compatibility.

Pick by direction, not by hype

If the client never has to send anything mid-session — token streams from an LLM, market prices, notification fan-out — SSE is the right answer. It's plain HTTP, every CDN handles it, every load balancer understands it, every browser supports EventSource natively, no framing protocol to learn. Don't reach for WebSocket because it sounds modern. SSE is half the operational surface for the same UX.

If the client needs to send messages on the same channel — chat, collaborative cursors, in-game inputs — WebSocket. The full-duplex frame protocol is what you actually need; you'd otherwise reinvent it on top of two HTTP connections and get it half-right.

If you cannot upgrade to either because a corporate proxy strips Upgrade: websocket or buffers SSE responses — and only then — long polling. SignalR's transport negotiator does this automatically; it's a fallback, not a target.

03Scaling out the channel.

Single pod, single connection: trivial. The connection state lives in the process. Now scale to three pods behind a load balancer. A client connected to pod A wants a message that pod B just received from another client. Pod A has no idea pod B exists. The connection state is partitioned across pods that don't share memory. You need a backplane — a pub/sub layer that every pod subscribes to — so that when pod B receives "broadcast to room-42", every pod hears it and fans out to its local sockets.

Sticky session for the connection, broadcast plane for the messages. Most "scale-out is broken" incidents are one of these two mis-wired.

Sticky sessions, but only for the socket

WebSocket and SSE are long-lived TCP connections. They must terminate on a specific pod and stay there. The load balancer needs sticky sessions — usually via a cookie set during the initial HTTP handshake. Without stickiness, a reconnect lands on a different pod, the new pod doesn't know about the connection's room subscriptions, and the user mysteriously stops receiving messages. Stickiness is a connection-layer concern. The messages still need to flow across pods via the backplane.

The backplane is where Redis earns its keep

SignalR ships first-party support for a Redis backplane. Every pod subscribes to a small set of pub/sub channels; every "broadcast to group X" call publishes to Redis; every pod's SignalR engine consumes and forwards to its local connections. At ~10k connections per pod and ~1k messages/second per pod, a single Redis primary handles the broadcast traffic with headroom. Beyond that — say, Slack-scale fan-out — the path is sharded backplanes or a managed service like Azure SignalR Service that does the sharding for you.

Connection cost is real money

An idle WebSocket isn't free. Each connection holds a file descriptor, a TLS context, a tiny per-connection memory footprint, and a place in the worker thread's event loop. Practical limits on a modern .NET pod: ~50k–100k concurrent connections with tuning, ~10k–20k without thinking about it. Azure SignalR Service charges roughly $50 per "unit" supporting 1,000 concurrent connections — so 100k users = $5k/mo just for the channel. Worth knowing before you ship "show online status of everyone you've ever DM'd."

04SignalR + Redis backplane.

A multi-replica chat hub. JWT auth, room-scoped groups, presence broadcast, Redis backplane so any pod can broadcast to any user. Drop in front of an ingress with sticky cookies enabled.

C#   ChatHub.cs 
// dotnet add package Microsoft.AspNetCore.SignalR.StackExchangeRedis
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.SignalR;
using Microsoft.Extensions.Logging;

namespace Academy.Realtime;

public sealed record ChatMessage(Guid Id, string User, string Text, DateTimeOffset At);

[Authorize]
public sealed class ChatHub : Hub
{
    private readonly IPresenceTracker _presence;
    private readonly ILogger<ChatHub> _log;

    public ChatHub(IPresenceTracker presence, ILogger<ChatHub> log)
    { _presence = presence; _log = log; }

    public override async Task OnConnectedAsync()
    {
        var userId = Context.UserIdentifier!;
        await _presence.JoinAsync(userId, Context.ConnectionId);
        await Clients.All.SendAsync("PresenceChanged", userId, "online");
        await base.OnConnectedAsync();
    }

    public override async Task OnDisconnectedAsync(Exception? ex)
    {
        var userId = Context.UserIdentifier!;
        var stillOnline = await _presence.LeaveAsync(userId, Context.ConnectionId);
        if (!stillOnline)
            await Clients.All.SendAsync("PresenceChanged", userId, "offline");
        await base.OnDisconnectedAsync(ex);
    }

    public async Task JoinRoom(string roomId)
        => await Groups.AddToGroupAsync(Context.ConnectionId, $"room:{roomId}");

    public async Task LeaveRoom(string roomId)
        => await Groups.RemoveFromGroupAsync(Context.ConnectionId, $"room:{roomId}");

    public async Task Send(string roomId, string text)
    {
        if (string.IsNullOrWhiteSpace(text) || text.Length > 2000)
            throw new HubException("message too long or empty");

        var msg = new ChatMessage(
            Guid.NewGuid(),
            Context.UserIdentifier!,
            text,
            DateTimeOffset.UtcNow);

        // Group send. Redis backplane broadcasts to every pod hosting members.
        await Clients.Group($"room:{roomId}").SendAsync("Message", msg);
    }
}

Wire-up in Program.cs. The Redis backplane is one line; the rest is JWT bearer for the WebSocket handshake — which is the one auth quirk people miss, because browsers can't set Authorization headers on a WebSocket. The token goes in the query string and the bearer handler reads it.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSignalR(o =>
    {
        o.EnableDetailedErrors = builder.Environment.IsDevelopment();
        o.KeepAliveInterval     = TimeSpan.FromSeconds(15);
        o.ClientTimeoutInterval = TimeSpan.FromSeconds(60);
    })
    .AddStackExchangeRedis(
        builder.Configuration.GetConnectionString("Redis")!,
        opts => opts.Configuration.ChannelPrefix = RedisChannel.Literal("chat:"));

builder.Services.AddSingleton<IPresenceTracker, RedisPresenceTracker>();

builder.Services.AddAuthentication("Bearer").AddJwtBearer(o =>
{
    o.Authority = builder.Configuration["Auth:Authority"];
    o.Audience  = builder.Configuration["Auth:Audience"];
    o.Events = new JwtBearerEvents
    {
        // WebSocket handshake can't set Authorization header → query-string fallback
        OnMessageReceived = ctx =>
        {
            var token = ctx.Request.Query["access_token"];
            if (!string.IsNullOrEmpty(token) &&
                ctx.HttpContext.Request.Path.StartsWithSegments("/hubs"))
                ctx.Token = token;
            return Task.CompletedTask;
        }
    };
});

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();
app.MapHub<ChatHub>("/hubs/chat");
app.Run();

05The detail.

Sticky cookies on the ingress

For SignalR without the Redis backplane, sticky sessions are mandatory; with the backplane, they're still recommended because the initial negotiate handshake is a separate HTTP request from the WebSocket upgrade. On Azure Application Gateway: enable cookie-based affinity. On NGINX: ip_hash or a per-route cookie module. Get this wrong and SignalR will silently fall back to long polling and your CPU bill will tell you.

KeepAliveInterval < load-balancer idle timeout

Azure Load Balancer kills idle TCP at 4 minutes. AWS NLB at 350 seconds. If the SignalR keepalive ping is sparser than the LB's idle timeout, your connections die at exactly the timeout boundary. Default keepalive is 15 seconds — leave it. The bug shows up as "everyone disconnects at the four-minute mark on Mondays" and takes a week to diagnose.

Backpressure is yours to handle

If a client's downlink is slow, the server's send buffer fills up. SignalR will block the send, then start dropping in the worst case. For high-fan-out flows (game state, financial ticks), use IAsyncEnumerable<T> streaming with bounded channels and explicit drop-old policies. Don't broadcast 10kB of state at 60Hz to a 3G client — buffer, coalesce, drop intermediate frames.

Presence is harder than it looks

"Show online users" sounds trivial. A user with two tabs has two connections. A user on a flaky train reconnects every 30 seconds. "Online" must be reference-counted per user-id, not per connection-id, with a TTL'd Redis hash and a debounce of ~10 seconds on "offline" broadcasts. Otherwise the green dot will flicker on every metro tunnel.

When to use Azure SignalR Service instead

Above ~10k concurrent connections per region, the math tips. Self-hosting means tuning kernel parameters, sizing the Redis backplane, and answering pages about idle WebSocket TIME_WAIT counts. Azure SignalR Service moves the channel out of your pod entirely: your app sends messages to the service, the service holds the WebSockets. Your pods become stateless again. Cost: ~$50/mo per 1k concurrent connection units. Worth it the moment a single pod can't hold the population.

"The cheapest real-time architecture is the one where idle costs nothing. Everything else is an optimisation around that."

07Decision guide.

Chat / collab app, <10k concurrent

SignalR self-hosted with Redis backplane and sticky sessions. JWT in the query string for the upgrade handshake. The default .NET answer.

Chat / collab app, >10k concurrent

Azure SignalR Service. Pods stay stateless, connection scaling becomes a knob in the portal, the bill is ~$50/1k units.

LLM token streams to the browser

SSE. Stream data: {...}\n\n chunks; the browser reconnects on its own. No WebSocket plumbing needed for a one-way flow.

Live dashboard / status board

SSE again. Push deltas as events; if a tab goes idle, it auto-reconnects and resumes with Last-Event-ID. One HTTP route.

Hostile network (corp firewalls / proxies)

Let SignalR negotiate down to long polling. Don't ship long-polling as the design — ship the negotiation so it survives the next strict proxy.

Game / 60Hz state sync

Raw WebSocket with a custom binary framing protocol and explicit backpressure. SignalR is the wrong abstraction at that data rate.

Three things people get wrong

One — no sticky session. Three pods, one user reconnects to a new pod, the new pod has no knowledge of the user's groups, the messages stop arriving. Sticky session cookie at the LB. Always.

Two — JWT in the Authorization header. Browsers can't set headers on a WebSocket upgrade. Token must go in the query string and be read by an OnMessageReceived event handler. Otherwise the handshake gets a 401 and the SDK silently retries.

Three — broadcasting raw state, not deltas. Pushing the whole document on every change works for 10 users and melts at 10,000. Push deltas with a sequence number; let the client reconcile from a baseline snapshot on reconnect.
