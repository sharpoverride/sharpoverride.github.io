Pick the algorithm by the backend.

Round-robin works for stateless homogeneous backends. Least-connections beats it when request durations vary. Power-of-two-choices beats both when you can afford the lookup. Consistent hashing wins when state lives on the backend. Choose by what your backends are, not what's familiar.

The four to know

Round-robin. Least-connections. Power-of-two-choices (P2C). Consistent hash. Each shines on a different backend shape.

The classic mistake

Defaulting to round-robin everywhere. Round-robin is the textbook example, not the production answer for most real workloads.

01What "balance" actually means.

"Spread requests evenly across N backends" is what people say. The interesting word is "evenly". Evenly by request count? Evenly by CPU consumed? Evenly by current concurrent connections? Each definition produces a different algorithm and a different bug when the assumption is wrong.

Round-robin is even by request count. Least-connections is even by in-flight count. Least-latency is even by recent response time. If your requests cost wildly different amounts, request-count fairness is unfairness. One backend serving 100 quick reads is doing less work than another serving 100 slow uploads.

"Round-robin only looks fair because we agreed not to measure what we cared about."

02Four algorithms.

Round-robin

Cycle through backends in order: 1, 2, 3, 1, 2, 3. Stateless on the load balancer; trivial to implement; predictable. Wrong when request costs vary. Works fine for homogeneous stateless backends serving uniform requests — e.g. a fleet of static-file servers.

Least-connections

Pick the backend with the fewest currently-active connections. Naturally adapts to variable request duration: a slow backend accumulates connections, gets less new traffic, recovers. The cost: the LB has to track per-backend connection counts. Most production LBs do this efficiently. Default in modern HAProxy and recommended in NGINX least_conn.

Power-of-two-choices (P2C)

For each new request, pick two backends at random; send to whichever has fewer connections. Sounds dumb, performs near-optimally. Avoids the herd effect of strict least-connections (where every LB picks the same backend simultaneously), and avoids the worst-case overload that round-robin produces. Cheap to compute, distributed-friendly, asymptotically optimal for most real workloads. Used in Linkerd's data plane by default.

Consistent hashing

Pick the backend based on a hash of some request key — user ID, session ID, partition key. Same key always goes to the same backend (modulo backend additions/removals). Essential when backends hold per-user state in memory (session affinity, in-process caches). See the consistent-hashing deep dive for the math.

03L4 vs L7.

Load balancers operate at one of two layers. L4 (TCP): balances connections without inspecting payloads. Fast (line-rate), simple, blind to content. AWS NLB, Azure Load Balancer, Cloudflare Spectrum. Great for non-HTTP protocols (databases, gRPC, raw TCP). L7 (HTTP): terminates the connection, parses HTTP, can route by URL/header/cookie. AWS ALB, NGINX, YARP, Envoy. Required for HTTP-aware features: per-route timeouts, header rewriting, cookie-based affinity.

The gotcha with HTTP/2 / gRPC: L4 LBs see one TCP connection and stick to one backend, so request multiplexing concentrates on one pod. For gRPC, L7 is mandatory — Envoy or Linkerd, not NLB.

YARP's policies

YARP ships RoundRobin, LeastRequests (=least-connections), PowerOfTwoChoices, Random, FirstAlphabetical. Use PowerOfTwoChoices unless you have a specific reason; it dominates round-robin on real workloads.

05The detail.

Slow start vs cold backend

A newly-added backend with zero connections looks like "least connections" — every LB sends to it, it gets crushed. Enable "slow start": the new backend ramps from 0% to full weight over a few minutes. NGINX slow_start, Envoy slow_start_config. Without this, every deploy briefly overloads the new pods.

EWMA — adaptive least-time

Exponentially weighted moving average of response time per backend. The LB sends new traffic to the backend with the lowest recent latency. Used in Linkerd's L7 LB. The "I noticed this backend got slow" signal is faster than waiting for connection counts to diverge.

Sticky sessions are an anti-pattern (mostly)

Sticky sessions are how you preserve in-memory session state across requests for the same user. The cost: failover loses the state. Modern shape: stateless backends + sessions in Redis (see JWT vs Sessions). Use sticky only when needed: WebSocket connections (the TCP must terminate on one pod) and a few legacy stateful protocols.

Health checks: active and passive

Active: LB polls /healthz every N seconds. Catches "completely dead" fast. Passive: LB tracks recent failures from real traffic; ejects a backend after N consecutive errors. Use both: active is reliable; passive is responsive.

gRPC needs client-side LB or L7

HTTP/2 multiplexes a single TCP connection; an L4 LB sticks one client to one pod for life. Either use an L7 LB (Envoy, Linkerd) that load-balances per-stream, or use gRPC's client-side LB with a headless service (DNS returns all pods, the gRPC client picks).

"The load balancer is the thing that turns a fleet into a service. Get it wrong and you have a list of computers."

06Decision guide.

Stateless HTTP API, uniform requests

Round-robin or P2C. AWS ALB / Azure Front Door / NGINX. Slow-start enabled. Active + passive health checks.

HTTP API with variable request cost

P2C or least-connections. Linkerd / Envoy / YARP with PowerOfTwoChoices. Lets slow backends shed load naturally.

gRPC service

L7 LB (Envoy / Linkerd) or client-side LB. L4 doesn't balance multiplexed HTTP/2 properly.

WebSocket / SSE

Sticky session cookie at the LB. Long-lived connections must stay on their pod. Pair with Redis backplane for state sharing (see Realtime Channels).

Per-user state lives in backend memory

Consistent hashing on user-id. Same user always lands on same backend. See consistent hashing deep dive.

TCP / DB / non-HTTP

L4 LB. AWS NLB, Azure Load Balancer, HAProxy in TCP mode. L4 is faster and content-blind, which is what TCP wants.
