Chapter 15 Difficulty ★★★☆☆ Read ~12 min Stack Kubernetes · Consul · Service Fabric · DNS

How does service A find service B?

In 2010 you wrote the IP into a config file. In 2016 you ran Eureka or Consul. In 2026 the sidecar already did it for you. The interesting question is no longer which registry — it's which layer in the stack owns the lookup and what it costs you when that layer hiccups.

The problem in one line

Pods come and go on random IPs. A stable name is a contract between caller and callee. The registry is the body that keeps the contract.

The thesis

If you're on Kubernetes, the answer is "use the cluster DNS." If you're not, the answer is "use the registry that's already on your platform." A custom service discovery layer in 2026 is almost always wrong.

01The instability of an IP.

Each pod gets its own IP. The pod restarts and the IP changes. The deployment scales from three replicas to ten — seven new IPs that didn't exist a second ago. Hardcoding 10.0.4.17 into a config file is correctness theatre; it works until the first deploy. The address has to be derived from a name. The system that derives it is the service-discovery layer.

Two shapes of solution, settled by 2018, refined since. DNS-based: the platform runs a DNS server that maps service names to current pod IPs; clients use ordinary getaddrinfo. Registry-based: services register themselves with a discovery service (Consul, Eureka, etcd); clients query the registry directly via its API. The cloud era picked DNS, hard, because every TCP client already speaks it.

"Hardcoding an IP is faster to write and faster to break. The discovery layer trades one keystroke for one outage."

02DNS vs registry.

Kubernetes ships with both. The cluster's CoreDNS maps orders.default.svc.cluster.local to a virtual ClusterIP, which iptables / IPVS load-balances to a pod. No client-side knowledge of the pod set, no registry API, just DNS.

Fig. 15DNS puts the load-balancer in the platform; the registry puts it in the client library. Both lookup; only one pushes per-call decisions to the caller.

DNS in Kubernetes is almost always enough

Every pod's resolv.conf points at CoreDNS, which holds the canonical mapping of service.namespace.svc.cluster.local → ClusterIP. The ClusterIP is a virtual address; the kube-proxy's iptables / IPVS rules round-robin or random-pick a ready endpoint. Health is owned by the readiness probe — an unready pod doesn't appear in the endpoint list. You write http://payments in your config and the platform figures out the rest. No client library, no agent, no Raft cluster of your own to operate.

The DNS approach has one famous weakness: connection-level load balancing. The DNS lookup returns one VIP; once a TCP connection is open, it sticks. HTTP/1 sees this and it's fine (each request opens a new connection-ish). HTTP/2 multiplexes everything down one connection; gRPC inherits the problem; the pod on the other end gets all the load, the rest sit idle. Fix: client-side load balancing via a library, or use headless services (one DNS record per pod) and let the gRPC client do the picking.

Registries shine when DNS isn't enough

Consul, Eureka, etcd, Service Fabric naming — all variations on "a key-value store that knows about service health and pushes updates to subscribers." They give you: multi-datacenter awareness (DNS in K8s is per-cluster), rich metadata (tags, weights, versions), watch APIs (be told when the set changes), and client-side load balancing with health awareness. The cost is a real cluster to operate, a client library to integrate, and the failure mode where the registry partition becomes your outage.

03Health checks and TTL.

Discovery only matters if the addresses returned actually work. Three ways a service goes "unhealthy" — process dead, port not listening, listening but broken — and the registry has to detect each. The detection layer is where most real-world bugs live.

Active vs passive checks

Active (registry → service): "hit my /healthz every 5s." Detects all three failure modes; costs registry CPU; can flap. Passive (clients report success/failure): "this caller saw three 503s in a row, mark unhealthy." Detects the failures that actually hurt traffic; risks evicting on transient blips. Production systems use both. Kubernetes uses active (readiness probe) and ignores passive — fine because the kube-proxy is connection-stupid. Consul does active by default and supports passive eviction. Envoy/Istio uses both.

TTLs are the tuning knob

How fast can a dead pod be removed from the rotation? In K8s, it's readinessProbe.periodSeconds (default 10s) × failureThreshold (default 3) = 30 seconds of bad traffic. Plus kube-proxy's iptables sync time. Tune the readiness probe to your latency requirements; default values are conservative. Consul's check intervals are similar — defaults send maybe 30 seconds of traffic to a dead pod. If your SLO is 99.95%, that's the budget for the year you spend in a single rolling restart that hits 1% of traffic for 30 seconds.

DNS caching is the silent killer

The registry can update in milliseconds. The client's DNS resolver caches the answer for the TTL, typically 30 seconds. The JVM caches it for 30 seconds by default. .NET's HttpClient caches it indefinitely until you set SocketsHttpHandler.PooledConnectionLifetime — a default that has bitten more on-call engineers than any other in the stack. Set PooledConnectionLifetime to 2 minutes minimum on every HttpClient that talks to a discovered name. Without it, your client will happily call a dead pod for an hour after it died.

04In .NET.

K8s DNS just works — write the service name as the host. The important code is the HttpClient configuration that prevents stale DNS from causing outages. Add Consul integration alongside when you need cross-cluster or rich metadata; the registration runs at startup.

C#   ServiceDiscoverySetup.cs 
// dotnet add package Microsoft.Extensions.ServiceDiscovery
// dotnet add package Consul                  -- only if using Consul
using Consul;
using Microsoft.Extensions.Hosting;

namespace Academy.Discovery;

public static class DiscoveryRegistration
{
    // The most important .NET service-discovery code is this method, full stop.
    // Without PooledConnectionLifetime, DNS changes are invisible to HttpClient.
    public static IServiceCollection AddDiscoveryAwareClient<TClient>(
        this IServiceCollection services, string baseUrl) where TClient : class
    {
        services.AddHttpClient<TClient>(c => c.BaseAddress = new Uri(baseUrl))
            .ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
            {
                // DNS re-resolution every 2 minutes — the difference between
                // "noticed the pod died" and "kept dialling a corpse for an hour."
                PooledConnectionLifetime = TimeSpan.FromMinutes(2),

                // Idle connection cleanup — kept conservative for keep-alive friendliness.
                PooledConnectionIdleTimeout = TimeSpan.FromMinutes(5),
            });
        return services;
    }
}

// Self-register with Consul on startup, deregister on shutdown.
public sealed class ConsulSelfRegistration : IHostedService
{
    private readonly IConsulClient _consul;
    private readonly IHostApplicationLifetime _life;
    private readonly ILogger<ConsulSelfRegistration> _log;
    private readonly string _serviceId = $"orders-{Environment.MachineName}-{Guid.NewGuid():N}";

    public ConsulSelfRegistration(IConsulClient consul,
        IHostApplicationLifetime life, ILogger<ConsulSelfRegistration> log)
    { _consul = consul; _life = life; _log = log; }

    public async Task StartAsync(CancellationToken ct)
    {
        var reg = new AgentServiceRegistration
        {
            ID      = _serviceId,
            Name    = "orders",
            Address = Dns.GetHostEntry(Dns.GetHostName()).AddressList[0].ToString(),
            Port    = 5000,
            Tags    = new[] { $"version={ThisAssembly.InformationalVersion}" },
            Check   = new AgentServiceCheck
            {
                HTTP     = $"http://{Environment.MachineName}:5000/healthz",
                Interval = TimeSpan.FromSeconds(5),
                Timeout  = TimeSpan.FromSeconds(2),
                DeregisterCriticalServiceAfter = TimeSpan.FromMinutes(1),
            }
        };
        await _consul.Agent.ServiceRegister(reg, ct);
        _log.LogInformation("Registered {Id} with Consul", _serviceId);
    }

    public async Task StopAsync(CancellationToken ct)
    {
        await _consul.Agent.ServiceDeregister(_serviceId, ct);
        _log.LogInformation("Deregistered {Id}", _serviceId);
    }
}

The Microsoft.Extensions.ServiceDiscovery package (introduced with .NET 8 / Aspire) abstracts the discovery source — DNS, Kubernetes, configuration, Consul — behind one resolver. Production-ready, single line in Program.cs, and the same call sites work across environments.

C#   Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddServiceDiscovery();   // resolves http://payments → real endpoints

builder.Services.AddDiscoveryAwareClient<PaymentsClient>("http://payments");
builder.Services.AddDiscoveryAwareClient<InventoryClient>("http://inventory");

builder.Services.ConfigureHttpClientDefaults(http => http.AddServiceDiscovery());

// If you're on K8s, you're done. If you also want Consul:
if (builder.Configuration.GetValue<bool>("Consul:Enabled"))
{
    builder.Services.AddSingleton<IConsulClient>(_ =>
        new ConsulClient(c => c.Address = new(builder.Configuration["Consul:Url"]!)));
    builder.Services.AddHostedService<ConsulSelfRegistration>();
}

var app = builder.Build();

app.MapGet("/healthz", () => Results.Ok(new { status = "healthy" }));
app.Run();

05The detail.

PooledConnectionLifetime is the most important line

Default HttpClient keeps a connection forever once opened. DNS changes are invisible. Set PooledConnectionLifetime = TimeSpan.FromMinutes(2) and the socket is closed after 2 minutes — the next request re-resolves the DNS. Without this line, every other line in this chapter is decoration.

Use the platform's discovery, not a custom one

In 2026, "I'll write a service registry" is a code smell. K8s ships one. Service Fabric ships one. Nomad+Consul ships one. AWS App Mesh and Azure Service Bus do equivalents. Pick the platform's primitive and only escape it when you have a named cross-platform requirement.

gRPC needs client-side load balancing

Standard K8s DNS with HTTP/2 sticks one client to one pod for the connection's life. Solutions: (1) use a headless service so DNS returns all pod IPs, and configure gRPC's client-side LB; (2) put an L7 proxy (Envoy, Linkerd) in front; (3) use a service mesh. The "I'm getting 90% of traffic on one pod" bug is the canonical symptom.

Deregister on graceful shutdown

If a pod terminates without deregistering, clients see "instance up" for up to a TTL × failure-threshold. Hook deregistration into IHostApplicationLifetime.ApplicationStopping; in K8s use preStop + terminationGracePeriodSeconds so the readiness probe fails before the process exits.

Cross-cluster discovery is a different problem

K8s DNS is intra-cluster. If you have multi-region or multi-cluster, you need either a global DNS layer (Route 53 health checks, Azure Front Door) or a federated registry (Consul with WAN gossip). Don't lift-and-shift in-cluster discovery to cross-cluster; the latencies and failure modes are different.

"In 2010 we wrote service discovery. In 2026 we configure it."

07Decision guide.

On Kubernetes, single cluster

CoreDNS + ClusterIP services. Microsoft.Extensions.ServiceDiscovery for indirection. Don't add Consul.

On Kubernetes, gRPC heavy

Headless service + gRPC client-side load balancing, or a service mesh (Linkerd, Istio). Standard ClusterIP hands too much load to one pod.

Multi-cluster / multi-region

Consul with WAN gossip, or a global DNS layer (Front Door, Route 53) with health checks. Intra-cluster DNS won't reach across.

VM-based fleet, no orchestrator

Consul is the cleanest answer. Agent on every VM, server cluster of 3–5, integrated health checks.

Service Fabric stack

Built-in naming service. Don't bolt anything on; it has more capability than most homegrown solutions.

Dev / staging / prod parity

Use the Microsoft.Extensions.ServiceDiscovery abstraction. Dev: config-based. Staging/prod: K8s DNS. Same call sites.

Three things people get wrong

One — forgetting PooledConnectionLifetime. Discovery works, DNS updates, but the open socket keeps dialling the dead pod. Set the lifetime to 2 minutes on every HttpClient.

Two — building a custom registry. Almost always wrong. The platform's registry has been hardened by 10× the engineering effort you have. Use it; escape only with a named reason.

Three — assuming DNS is enough for gRPC. HTTP/2 multiplexing sticks all traffic on one connection, so DNS round-robin doesn't actually round-robin. Headless services or an L7 proxy.
