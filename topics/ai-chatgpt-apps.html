OpenAI's third attempt.

Plugins (2023) failed because nobody could find them. GPTs (2024) failed because they were closed-ecosystem. Apps (2025) are the third try — built on MCP, OAuth-integrated, rendered inline. Whether this one sticks depends on the same thing it always did: discoverability.

The shape

An app is a server (your code, your data) that exposes a set of operations to ChatGPT. The user invokes it conversationally; the operation runs server-side; the response renders inline.

The protocol

Built on MCP (see MCP chapter). Plus OpenAI-specific extensions for OAuth, custom UI, and ChatGPT's app store.

01Three iterations.

Plugins (2023): developers built OpenAPI-described services; ChatGPT learned to call them. Users had to enable plugins manually; nobody did; usage stayed tiny. Discontinued in 2024.

GPTs (2024): "custom GPTs" — a system prompt plus optional tool calls. The store was OpenAI-only; the model couldn't introspect actions cleanly. Persisted, but mostly as a curiosity.

Apps (2025): built on MCP (open protocol), with first-class UI rendering and OAuth user-account integration. The bet: by standing on an open protocol and improving discoverability, the third iteration can stick.

"The protocol is the easy part. The directory is the product."

02What an app contains.

An MCP server

The core is an MCP server (see MCP chapter) that exposes tools, resources, and prompts. If you've built an MCP server, you've built 80% of an app.

An OAuth flow

Users authenticate to your service from ChatGPT. Standard OAuth 2.1 with PKCE; the token belongs to the user's ChatGPT account scoped to your app. Refresh tokens, revocation, the usual. ChatGPT brokers the popup; your server validates.

A response renderer

Beyond text, apps can return structured UI — cards, tables, forms, embedded interactive components. ChatGPT renders them inline. This is the new bit vs MCP alone: rich responses, not just text.

A manifest

An app.json (or equivalent) declares: name, icon, scopes, OAuth endpoints, the MCP server URL. Submitted to OpenAI's app directory; reviewed; published.

Fig. AThe app sits between ChatGPT and your backend. Rich responses render in the conversation, not as plain text.

03OAuth & the user.

User connects once

First invocation pops OAuth. User signs into your service. Token cached in ChatGPT. Subsequent calls reuse it transparently.

Scopes are tight

App declares minimum scopes in the manifest. Don't ask for more than you need; review will reject overreach. Users care about the consent screen.

Revocation is on you

Users can revoke from your side or from ChatGPT. Honor both. Webhook on revocation; clear caches; refuse subsequent API calls.

Don't store ChatGPT tokens long-term

The token represents a session, not a service identity. Don't use it to take actions outside the conversation. Background jobs run against your own user accounts, not the OpenAI-issued token.

"OAuth is the part that turns 'AI demo' into 'product with users.'"

04Decision guide.

Existing SaaS, wants AI surface

Build a ChatGPT app. Wrap your existing API in an MCP server; add OAuth; submit.

Cross-AI-client reach

Build an MCP server first; ChatGPT app is one consumer. Claude Desktop, Cursor, etc. become additional surfaces for free.

Internal use only

Skip the app store. Run an MCP server locally; configure Claude Desktop / ChatGPT Enterprise to use it.

Rich UI matters

The ChatGPT app's inline-render advantage applies. For pure-text use cases, a plain MCP server is fine.

You're new to LLM integration

Start with an MCP server. It teaches the shape. The app wrapper is mostly business stuff (manifest, OAuth, review).

Worried about lock-in

Standing on MCP is the safest move. The transport, the directory, the user account are OpenAI-specific; the protocol is open.
