The app is a constrained client.

Mobile architecture is what happens when the client owns state, the network is unreliable, the OS rations CPU, and the user can install the update next month or never. The interview framework (see Mobile SD framework) covers what to say in 45 minutes; this chapter is what to actually build.

The non-negotiables

Local DB. Sync protocol. Lifecycle handling. App-store-aware feature flags. Crash reporting.

The patterns

MVVM or unidirectional data flow. Clean / hexagonal layering. Repository pattern over the DB. Coroutines / async/await over callbacks.

01The layers.

Modern mobile apps converge on three layers regardless of platform. UI layer (SwiftUI / Compose / React Native screens). Domain layer (use cases, business logic, framework-free). Data layer (repositories, local DB, network clients).

UI subscribes to streams from the domain layer; mutations flow down. The domain layer is testable without a simulator. The data layer hides the network/DB split — callers ask the repository, not the API.

"Mobile is the only client tier where the architecture is also a survival strategy."

02State & persistence.

Single source of truth: the local DB

SQLite (Room on Android, GRDB / SwiftData / Core Data on iOS). Don't treat it as a cache; treat it as the truth. The UI reads from the DB; sync writes to the DB; the network is the freshener. This is the offline-first pattern, applied as a default rather than a special case.

Reactive flow

The DB is observed. Compose has Flow; SwiftUI has Combine / Observation; React Native has React Query subscriptions. When the DB row changes, the UI updates. The data-modification path doesn't have to explicitly tell the UI; the reactive plumbing handles it.

Optimistic mutations

User taps "like." Update local DB immediately; UI reflects. Send the API call in the background. On success, mark the row as confirmed. On failure, revert (with a toast). Mobile users perceive latency below 100ms as instant; anything above feels broken.

03Network.

Repository abstraction

Callers ask the repository for data. Repository decides: serve from local DB? Refresh from network? Pull both and merge? The UI doesn't know. Standard pattern; underused by teams that ship "loading spinner everywhere" UX.

Retries built in

The network drops mid-request. The library (URLSession with backgroundConfiguration, Retrofit + OkHttp interceptor, Ktor with retry plugin) retries automatically with exponential backoff. Idempotency keys on mutations; the retry isn't a double-charge.

Background sync windows

iOS: BGAppRefreshTask (~30s every several hours). Android: WorkManager (constraint-driven). You don't decide when sync runs; the OS does. Express constraints (charging, unmetered network); the OS schedules.

04Lifecycle.

The states

Active → backgrounded → suspended → terminated. Plus "killed by OS due to memory pressure." Plus "phone rebooted." Don't assume your process lives between two foregrounds. State must survive across process death.

State restoration

User returns to your app after a week. The UI should look like they left it. Save view state (scroll position, form input, selected tabs) to disk on backgrounding; restore on cold launch. SwiftUI @SceneStorage; Compose rememberSaveable; framework-specific patterns.

Push notifications as the wake signal

Backend has news; can't poll. Send a push. Silent push (data-only) lets the app fetch data without showing a notification; iOS rations these; treat as best-effort hints, not delivery guarantees.

05Decision guide.

Native iOS + Android

SwiftUI + Compose. Two codebases, two best-in-class experiences. The right pick for performance-critical or platform-feature-heavy apps.

One team, two platforms

Kotlin Multiplatform (KMP) shares business logic; native UI per platform. Or Flutter / React Native for shared UI too — accept some platform polish loss.

Heavy real-time / multiplayer

WebSocket or QUIC for the live channel. Local DB for offline. Same shape as the Slack/WhatsApp story but client-side.

Offline-first product

CRDT-based sync (Yjs, Automerge), or a custom delta-sync protocol. Server is the convergence point.

Crash reporting / observability

Sentry, Firebase Crashlytics, Bugsnag. Don't ship without one; mobile crashes are silent unless you're listening.

Force update / kill switch

Server-side feature flag for every consequential client behaviour. App store update lag is real; the kill switch is your emergency control.

"Mobile architecture is server architecture's rules — written by someone who has never been to your data center."
