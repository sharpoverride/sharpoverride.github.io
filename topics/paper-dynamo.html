White paper Year 2007 Authors DeCandia et al., Amazon Stack Consistent hashing · vector clocks · sloppy quorum

The paper that birthed NoSQL.

"Dynamo: Amazon's Highly Available Key-value Store" (SOSP 2007). The shopping-cart store that ran Amazon's checkout. Influenced Cassandra, Riak, Voldemort, and the entire eventually-consistent NoSQL category. The contributions are still relevant; the operational compromises are the warning label.

The problem it solved

Amazon's shopping cart must be writable even during partitions. "Add to cart" can never fail. Choose availability + partition tolerance; give up linearisability.

The five techniques

(1) Consistent hashing. (2) Sloppy quorum + hinted handoff. (3) Vector clocks for versioning. (4) Merkle trees for anti-entropy. (5) Gossip for membership.

What it influenced

Cassandra (most directly). Riak (almost a literal implementation). DynamoDB (Amazon's own managed version — different code, same philosophy).

01The shopping-cart problem.

By 2006, Amazon was running on relational databases with vertical scaling. The cracks showed in the checkout path: a database partition (network split) or replica failover would cause "Add to cart" to fail. For Amazon, every failed cart write is lost revenue — measurably, immediately. The team needed a store that would accept writes even during partitions, and reconcile inconsistencies later.

The Dynamo paper formalises that tradeoff. "In the trade-off between consistency and availability, our customers demand the highest level of availability." The cart store can accept conflicting writes from different replicas, return success to the user, and reconcile during reads (or by background repair). The customer sees "added"; the system promises eventual convergence.

"The decision to be available, even at the cost of consistency, was made before the architecture was."

02The five core ideas.

Consistent hashing for partitioning

The cluster is a ring; data is placed by hash of key onto the ring; each node owns a range. Adding or removing a node moves only ~1/N of the data, not the whole dataset. Virtual nodes (each physical node owns many ring positions) smooth load imbalances. This is the technique every modern partitioning system uses — see the consistent-hashing deep dive for the math.

Sloppy quorum + hinted handoff

A strict quorum write requires W replicas to acknowledge. If one of the original N replicas is down, a strict system blocks. Dynamo's sloppy quorum: if the intended replica is unreachable, write to the next-available node and store a "hint" that it belongs to the intended one. When the intended replica recovers, the holder hands off the data. This is what makes Dynamo writable during partitions: a write can always find some healthy node.

Vector clocks for versioning

Two replicas accept concurrent writes for the same key. On read, the client sees both versions. Which is the latest? Vector clocks: each replica tags its writes with a per-replica counter; concurrent writes have non-comparable clocks. The client must reconcile — typically by merging (cart: union both sets of items). This is the mechanism behind "eventually consistent with client-side reconciliation."

Merkle trees for anti-entropy

Two replicas drift over time. To detect and repair: each replica builds a Merkle tree over its key range; replicas exchange tree roots; mismatches trigger drill-down to find the diverged keys; repair shipping only the missing data. The Merkle tree compresses "what do you have that I don't" into log(N) network exchanges.

Gossip protocol for membership

How does a node learn that a peer joined or left the cluster? Gossip: each node periodically swaps membership state with a random peer. Information spreads epidemic-style; in O(log N) rounds, the cluster converges on a consistent view. No central coordinator; every node is equal.

03The CAP choice.

Dynamo's tuning is exposed: N (replicas), R (read quorum), W (write quorum). The customer picks. For "always-writable, eventually-consistent": N=3, R=2, W=1. For "strong consistency on reads": R+W > N. The same database, different consistency, by config.

Fig. AWith N=3, R=2, W=2: the read and write quorums must overlap on at least one replica. That overlap is what gives read-your-writes.

The Dynamo design says: let the application decide. The cart picks low-latency, high-availability (W=1, R=2); a "view the latest order" path picks stronger consistency (W=2, R=2). The database is a knob, not a fixed posture.

04What it influenced.

SystemWhat it took from DynamoWhat it changed
Cassandra Consistent hashing, gossip, tunable consistency Wider data model (column family), per-query consistency level
Riak Nearly the whole paper, as Erlang Pluggable backends, CRDT data types
Voldemort Same key-value model LinkedIn's own; later abandoned
DynamoDB Brand and philosophy Different internals (B-tree storage, single-master per partition, stronger consistency by default)
ScyllaDB Cassandra protocol → Dynamo lineage C++ shard-per-core (see Discord case study)

The interesting note: Amazon doesn't run Dynamo today. They run DynamoDB, which has different internals — single-master per partition, strong consistency by default, no vector clocks. The paper's ideas won; Dynamo-the-system was retired internally before most people heard of it.

05What still applies in 2026.

Consistent hashing is universal

Every modern distributed system that partitions by key uses a variant. The technique generalised; the specific Dynamo implementation is one of many. See the consistent-hashing chapter.

Tunable consistency is mostly a misfeature

The "let the application decide" promise turned out to be too much rope. Most applications can't reason about R/W/N tradeoffs; they pick badly. Modern databases (DynamoDB, Cosmos DB) ship with strong default consistency and let you weaken it explicitly — opposite default. The Dynamo paper's flexibility became its operational hazard.

Vector clocks lost to CRDTs

Vector clocks require the client to reconcile. Engineers got this wrong. CRDTs encode the reconciliation in the data type; the merge is automatic. For counters, sets, maps, CRDTs are the better answer (Riak adopted them; modern systems start with them).

Sloppy quorum + hinted handoff still in production

Cassandra and Riak still use this. "Always writable" is a real property; payments-shaped workloads (where lost writes are intolerable) still benefit. The mechanism is the same Dynamo described.

Gossip protocols are everywhere

Consul, Cassandra, Akka cluster, HashiCorp Serf. The "no central coordinator" property is still desirable; the mechanics are the Dynamo recipe.

"Dynamo isn't a database you run. It's a vocabulary the field uses."
